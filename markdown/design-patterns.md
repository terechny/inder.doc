
## Порождающие паттерны (Creational Patterns)

Порождающие паттерны (Creational Patterns) — это категория паттернов проектирования, которые управляют процессом создания (инстанцирования) объектов. Они делают систему независимой от того, как именно создаются, компонуются и представляются её объекты, изолируя клиентский код от прямого использования оператора new.

------------------------------
##### 1. Обзор основных порождающих паттернов

##### Singleton (Одиночка)

* Суть: Гарантирует, что у класса существует только один экземпляр на всё время работы приложения, и предоставляет к нему глобальную точку доступа.
* Как работает: Конструктор класса делается приватным (private), исключая создание объекта извне. Доступ к объекту осуществляется через статический метод (например, getInstance()), который создает объект при первом вызове, а при последующих — возвращает уже созданный.
* Пример в Laravel: Логирование, кеш или Service Container работают как синглтоны. Когда вы вызываете app('db'), Laravel возвращает один и тот же объект подключения к БД на протяжении всего жизненного цикла запроса.

##### Factory Method (Фабричный метод)

* Суть: Определяет интерфейс создания объекта, но делегирует выбор конкретного класса для инстанцирования своим подклассам (наследникам).
* Как работает: Базовый класс объявляет абстрактный метод создания, а дочерние классы реализуют его, возвращая нужный тип продукта. Подробнее тема описана в разделе Factory Pattern.

##### Abstract Factory (Абстрактная фабрика)

* Суть: Предоставляет интерфейс для создания целых семейств взаимосвязанных объектов, не привязываясь к конкретным классам продуктов.
* Как работает: Объединяет несколько фабричных методов в один интерфейс. Используется, когда коду нужно работать с разными линейками продуктов (например, UI-элементы для Windows и Mac, или драйверы логистики: курьер + машина для DHL и курьер + машина для FedEx).

##### Builder (Строитель)

* Суть: Позволяет создавать сложные объекты пошагово. Паттерн разделяет логику конструирования сложного объекта от его непосредственного представления, благодаря чему один и тот же процесс сборки может приводить к разным результатам.
* Как работает: Предоставляет цепочку методов (Fluent Interface) для конфигурации объекта перед его финальной сборкой (метод build()). Предотвращает появление "абсурдных" конструкторов с 15 аргументами.
* Пример в Laravel: Query Builder. Вы конструируете тяжелый SQL-запрос пошагово: `$query->where(...)->join(...)->orderBy(...)`, и только метод get() осуществляет финальную сборку и выполнение.

##### Prototype (Прототип)

* Суть: Позволяет копировать (клонировать) существующие объекты, не вдаваясь в подробности их реализации и не привязываясь к их конкретным классам.
* Как работает: Объект сам реализует интерфейс клонирования. В PHP для этого используется встроенный метод clone.
* Пример в Laravel: Дублирование моделей Eloquent через метод `$newUser = $user->replicate();`. Laravel создает точную копию объекта в оперативной памяти (без ID), готовую к изменению и сохранению как новая строка в БД.

------------------------------
##### 2. Сравнительная таблица порождающих паттернов

| Паттерн | Какую проблему решает? | Когда использовать? |
|---|---|---|
| Singleton | Контроль доступа к разделяемому ресурсу (БД, Логгер, Конфиги). | Нужен строго один экземпляр объекта на всю систему. |
| Factory Method | Изоляция создания одного продукта; выбор реализации на основе условий. | Заранее неизвестны типы объектов, с которыми должен работать код. |
| Abstract Factory | Создание целых наборов (семейств) совместимых друг с備 другом продуктов. | Система должна быть независима от способов создания её продуктов. |
| Builder | Избавление от "телескопического" конструктора; пошаговая сборка. | Объект собирается из множества опциональных элементов и конфигураций. |
| Prototype | Создание объекта "с нуля" слишком дорогое/сложное. | Проще скопировать существующий настроенный экземпляр, чем собирать новый. |

##### 3. Практический пример: Паттерн Builder (Строитель)

Реализуем сборку сложного объекта Invoice (Счет-фактура), который может содержать скидки, налоги, разную валюту и список позиций.

##### Шаг 1: Сложный целевой объект

```php

class Invoice {
    public string $currency = 'USD';
    public float $taxRate = 0;
    public float $discount = 0;
    public array $items = [];
    public ?string $recipient = null;

    // Конструктор пустой, объект настраивается извне
    public function calculateTotal(): float { /* Логика расчета */ return 100.0; }
}

```

##### Шаг 2: Класс Строителя (InvoiceBuilder.php)

```php

class InvoiceBuilder {
    protected Invoice $invoice;

    public function __construct() {
        $this->invoice = new Invoice(); // Начинаем чистую сборку
    }

    // Методы возвращают $this для поддержки цепочки вызовов (Fluent Interface)
    public function to(string $recipient): self {
        $this->invoice->recipient = $recipient;
        return $this;
    }

    public function inCurrency(string $currency): self {
        $this->invoice->currency = $currency;
        return $this;
    }

    public function withTax(float $rate): self {
        $this->invoice->taxRate = $rate;
        return $this;
    }

    public function withDiscount(float $amount): self {
        $this->invoice->discount = $amount;
        return $this;
    }

    public function addItem(string $name, float $price): self {
        $this->invoice->items[] = ['name' => $name, 'price' => $price];
        return $this;
    }

    // Финальный метод сборки
    public function build(): Invoice {
        if (!$this->invoice->recipient) {
            throw new Exception("Невозможно собрать инвойс без указания получателя.");
        }
        return $this->invoice;
    }
}

```

##### Шаг 3: Использование в клиентском коде (Контроллере)

```php

class InvoiceController extends Controller {
    public function generate() {
        // Красивое, читаемое, пошаговое создание сложного объекта
        $invoice = (new InvoiceBuilder())
            ->to('ООО Вектор')
            ->inCurrency('EUR')
            ->withTax(20)
            ->addItem('Разработка ПО', 5000)
            ->addItem('Поддержка сервера', 500)
            ->build();

        return response()->json(['total' => $invoice->calculateTotal()]);
    }
}

```

## Структурные паттерны проектирования (Structural Patterns)

Структурные паттерны (Structural Patterns) — это категория паттернов проектирования, которые отвечают за построение удобных и эффективных взаимосвязей между классами и объектами. Они помогают объединить разные интерфейсы и объекты в крупные, гибкие структуры, не увеличивая при этом связность системы, и позволяют координировать совместную работу старого (легаси) и нового кода.

------------------------------
##### 1. Обзор основных структурных паттернов

##### Adapter (Адаптер)

* Суть: Позволяет объектам с несовместимыми интерфейсами работать вместе. Он выступает в роли «переводчика» или переходника, преобразуя интерфейс одного класса в интерфейс, который ожидает клиентский код.
* Как работает: Создается класс-адаптер, который имплементирует интерфейс, понятный клиенту, но внутри себя содержит ссылку на несовместимый объект и перенаправляет ему вызовы, преобразуя форматы данных.
* Пример в Laravel: Драйверы кеша или очередей. У вас есть общий контракт CacheServiceProvider, но под капотом через адаптеры могут работать совершенно разные библиотеки: Redis, Memcached или обычные файлы.

##### Decorator (Декоратор)

* Суть: Позволяет динамически добавлять объектам новые обязанности и поведение на лету, не изменяя при этом код самого класса и не используя наследование.
* Как работает: Объект-декоратор оборачивает целевой объект (так как они реализуют один и тот же интерфейс). Декоратор перехватывает вызовы методов, выполняет свою добавочную логику (до или после основного вызова) и делегирует выполнение обернутому объекту.
* Пример в Laravel: HTTP Middleware (Посредники). Они работают по принципу матрешки-декоратора. Запрос проходит сквозь слои посредников (AuthMiddleware -> LogMiddleware), каждый из которых добавляет поведение (проверяет сессию, логирует время), прежде чем запрос дойдет до контроллера.

##### Facade (Фасад)

* Суть: Предоставляет простой и лаконичный интерфейс (одну точку входа) к сложной системе классов, библиотеке или фреймворку.
* Как работает: Скрывает за одним методом вызовы десятков методов других внутренних подсистем, избавляя клиентский код от необходимости знать, как устроена сложная подсистема изнутри.
* Пример в Laravel: Встроенные Laravel Facades (Cache::get(), Log::info(), Route::post()). Они предоставляют статический интерфейс к сложным сервисам, зарегистрированным в Service Container, упрощая написание кода.

##### Proxy (Заместитель)

* Суть: Позволяет подставлять вместо реального объекта специальный объект-заменитель. Этот заместитель контролирует доступ к оригинальному объекту, позволяя выполнять логику до или после запроса.
* Как работает: Заместитель реализует тот же интерфейс, что и оригинальный объект. Клиент не замечает подмены. Используется для ленивой загрузки (Lazy Loading), логирования, контроля прав доступа или кеширования.
* Пример в Laravel: Eloquent Relations (Ленивая загрузка отношений). Когда вы обращаетесь к $user->posts, Laravel не делает запрос в БД мгновенно при создании пользователя. Поле posts защищено прокси-логикой (Lazy Loading Proxy), которая выполнит SQL-запрос только в тот момент, когда вы реально попытаетесь прочитать эти данные.

##### Composite (Компоновщик)

* Суть: Позволяет сгруппировать объекты в древовидную структуру для представления иерархии «часть-целое». Он позволяет клиентам работать с единичными объектами и группами объектов одинаково.
* Как работает: И одиночные элементы, и контейнеры (группы) реализуют один и тот же интерфейс. Когда метод вызывается у контейнера, он автоматически транслирует этот вызов всем своим дочерним элементам.
* Пример в практике: Структура категорий каталога (категория может содержать товары и подкатегории) или дерево меню, где узел меню и отдельный пункт реализуют общий интерфейс рендеринга.

------------------------------
##### 2. Сравнительная таблица структурных паттернов

| Паттерн | Основная цель | Ключевое отличие (ментальный якорь) |
|---|---|---|
| Adapter | Изменение / подгонка интерфейса. | Связывает два разных интерфейса, которые изначально не подходили друг другу. |
| Decorator | Расширение обязанностей без наследования. | Не меняет интерфейс, а наращивает функционал (оборачивает объект). |
| Facade | Упрощение работы со сложной системой. | Создает новый упрощенный интерфейс поверх множества старых. |
| Proxy | Контроль доступа и ленивое управление. | Полностью копирует интерфейс объекта, перехватывая управление им. |
| Composite | Работа с деревьями и иерархиями. | Позволяет применять полиморфизм к одному объекту и к группе объектов одинаково. |

##### 3. Практический пример: Паттерн Decorator (Декоратор)

Реализуем задачу: Отправка сообщения. Базовый класс отправляет чистый текст, но нам нужно иметь возможность динамически кодировать его в Base64 или оборачивать в HTML-теги перед отправкой.

##### Шаг 1: Интерфейс и базовый компонент

```php

interface MessageSenderInterface {
    public function format(string $text): string;
}

// Базовый класс: возвращает текст как есть
class PlainTextSender implements MessageSenderInterface {
    public function format(string $text): string {
        return $text;
    }
}

```

##### Шаг 2: Базовый класс Декоратора и его расширения

```php

abstract class SenderDecorator implements MessageSenderInterface {
    // Декоратор обязательно хранит ссылку на оборачиваемый объект
    public function __construct(protected MessageSenderInterface $wrapper) {}
}

// Декоратор 1: Обертывает текст в HTML-тег strong
class BoldTitleDecorator extends SenderDecorator {
    public function format(string $text): string {
        // Вызываем логику предыдущего слоя и добавляем свою
        return "<strong>" . $this->wrapper->format($text) . "</strong>";
    }
}

// Декоратор 2: Шифрует итоговую строку в Base64
class Base64JsonDecorator extends SenderDecorator {
    public function format(string $text): string {
        return base64_encode($this->wrapper->format($text));
    }
}

```

##### Шаг 3: Динамическая сборка (композиция) в рантайме

Мы можем комбинировать слои поведения в любом порядке, не создавая подклассы вроде PlainBoldBase64Sender.

```php

class NotificationController extends Controller {
    public function send() {
        $rawText = "Hello World";

        // 1. Просто текст
        $sender = new PlainTextSender(); 
        
        // 2. Оборачиваем в HTML теги
        $boldSender = new BoldTitleDecorator($sender); 
        
        // 3. Оборачиваем HTML-сендер в Base64 шифровальщик (слоеная структура)
        $secureSender = new Base64JsonDecorator($boldSender);

        // Результат пройдет через всю цепочку декораторов
        $result = $secureSender->format($rawText); 

        return response()->json(['output' => $result]);
    }
}

```

## Behavioral Patterns

Поведенческие паттерны (Behavioral Patterns) — это категория паттернов проектирования, которые определяют алгоритмы, распределение обязанностей и способы взаимодействия между объектами. Они фокусируются не на том, как объекты создаются (порождающие) или как они соединяются (структурные), а на том, как они общаются между собой, обеспечивая максимальную гибкость и слабую связанность (Low Coupling).


##### 1. Обзор основных поведенческих паттернов

##### Strategy (Стратегия)

* Суть: Определяет семейство схожих алгоритмов, инкапсулирует каждый из них в отдельный класс и делает их взаимозаменяемыми в рантайме.
* Как работает: Класс контекста хранит ссылку на интерфейс стратегии. Клиент передает в контекст нужный объект-реализацию. Это избавляет код от условных операторов if/else или switch, реализуя принцип OCP из SOLID.
* Пример в Laravel: Способы авторизации в Passport/Sanctum или драйверы экспорта данных (в PDF, Excel, CSV).

##### Observer (Наблюдатель)

* Суть: Позволяет объектам подписываться на события, происходящие в другом объекте (издателе), и автоматически реагировать на них.
* Как работает: Подробно разобран в разделе Observer Pattern (Laravel Events, Listeners, Observers).

##### Command (Команда)

* Суть: Превращает запрос, действие или операцию в отдельный самостоятельный объект (DTO + метод выполнения).
* Как работает: Инкапсулирует параметры вызова и само действие. Это позволяет передавать действия в качестве аргументов, ставить их в очередь, логировать или поддерживать отмену операций.
* Пример в Laravel: Artisan-команды (app/Console/Commands) или классы Jobs (Джобы) для очередей. Каждая джоба — это изолированная команда со своим методом handle().

##### Chain of Responsibility (Цепочка обязанностей)

* Суть: Позволяет передавать запросы последовательно по цепочке обработчиков. Каждый обработчик решает, может ли он сам обработать запрос, и стоит ли передавать его дальше по цепочке.
* Как работает: Каждый элемент цепочки имеет ссылку на следующий элемент. Запрос движется по конвейеру, пока не встретит отказ или финальную обработку.
* Пример в Laravel: HTTP Middleware (Посредники). Запрос проходит сквозь цепочку CheckAge -> VerifyCsrf -> Terminate. Метод return $next($request); — это и есть передача запроса следующему обработчику цепочки.

##### Template Method (Шаблонный метод)

* Суть: Определяет костяк (скелет) алгоритма в базовом классе, делегируя реализацию отдельных шагов этого алгоритма подклассам (наследникам).
* Как работает: Базовый класс содержит финальный метод, который вызывает цепочку других методов (некоторые из них абстрактные, некоторые имеют дефолтную реализацию). Наследники переопределяют шаги алгоритма, но не могут изменить его общую структуру.
* Пример в Laravel: Базовый класс тестов Tests\TestCase или консольная команда Laravel, где базовый класс настраивает окружение, а вы переопределяете только метод handle().

##### Iterator (Итератор)

* Суть: Позволяет последовательно обходить элементы составного объекта (коллекции), не раскрывая его внутреннее представление (структуру данных: массив, дерево, граф).
* Как работает: Предоставляет стандартные интерфейсы (в PHP это встроенный Iterator или IteratorAggregate) с методами current(), next(), key(), valid(), rewind().
* Пример в Laravel: LazyCollection и генераторы (выборка cursor() из БД), позволяющие обходить миллионы строк СУБД в цикле foreach без переполнения памяти.

##### State (Состояние)

* Суть: Позволяет объекту менять свое поведение в зависимости от внутреннего состояния. Изнутри кажется, что объект изменил свой класс.
* Как работает: Вместо хранения десятков флагов и if ($status === 'draft') внутри одного класса, каждое состояние выносится в отдельный класс-наследник интерфейса State. Объект контекста просто делегирует выполнение текущему объекту состояния.
* Пример в практике: Управление жизненным циклом заказа (DraftState -> PaidState -> ShippedState -> DeliveredState).

##### Mediator (Посредник)

* Суть: Снижает связанность множества классов между собой, заставляя их общаться не напрямую, а только через один специальный объект-посредник.
* Как работает: Объекты не знают о существовании друг друга. Они лишь отправляют сигналы Посреднику, а тот координирует действия (кто должен ответить и что сделать).
* Пример в Laravel: Event Dispatcher (Диспетчер событий). Когда вы вызываете event(new UserRegistered($user)), модель пользователя не знает, какие классы на нее подписаны. Диспетчер событий выступает медиатором — он сам находит нужные листнеры и передает им данные.

##### 3. Практический пример: Паттерн Strategy (Стратегия)

Реализуем расчет стоимости доставки посылки в зависимости от выбранной транспортной компании (ТК).

##### Шаг 1: Интерфейс стратегии и конкретные реализации

```php

interface DeliveryStrategyInterface {
    public function calculateCost(float $weight, float $distance): float;
}

class DhlDeliveryStrategy implements DeliveryStrategyInterface {
    public function calculateCost(float $weight, float $distance): float {
        return ($weight * 1.5) + ($distance * 0.5) + 100; // Тарифы DHL
    }
}

class RussianPostStrategy implements DeliveryStrategyInterface {
    public function calculateCost(float $weight, float $distance): float {
        return ($weight * 0.8) + ($distance * 0.2) + 20;  // Тарифы Почты
    }
}

```

##### Шаг 2: Класс Контекста (DeliveryCalculator.php)

```php

class DeliveryCalculator {
    // Хранит текущую выбранную стратегию
    private DeliveryStrategyInterface $strategy;

    public function setStrategy(DeliveryStrategyInterface $strategy): void {
        $this->strategy = $strategy;
    }

    public function calculate(float $weight, float $distance): float {
        // Делегируем расчет текущей стратегии
        return $this->strategy->calculateCost($weight, $distance);
    }
}
```

## Шаг 3: Использование в контроллере Laravel

```php

class OrderDeliveryController extends Controller {
    public function getShippingCost(Request $request) {
        $calculator = new DeliveryCalculator();

        // Динамически выбираем стратегию на основе ввода пользователя
        $strategy = match ($request->input('carrier')) {
            'dhl'  => new DhlDeliveryStrategy(),
            'post' => new RussianPostStrategy(),
            default => throw new InvalidArgumentException("Неизвестный перевозчик"),
        };

        $calculator->setStrategy($strategy);
        
        $cost = $calculator->calculate(
            $request->input('weight'),
            $request->input('distance')
        );

        return response()->json(['shipping_cost' => $cost]);
    }
}

```
