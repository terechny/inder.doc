## Database Optimization


##### База знаний: Оптимизация баз данных (Database Optimization)

Оптимизация базы данных — это комплекс мер по уменьшению времени выполнения запросов, снижению нагрузки на CPU/Disk/RAM и обеспечению масштабируемости системы при росте объемов данных. Основная цель — перейти от линейного или квадратичного роста времени выполнения ($O(N)$ или $O(N^2)$) к логарифмическому ($O(\log N)$) или константному ($O(1)$).

------------------------------
##### 1. Проблема N+1 (N+1 Problem)## Суть проблемы

Ситуация, когда для извлечения связанных данных приложение делает один стартовый запрос, а затем выполняет по одному дополнительному запросу для каждой полученной строки. Вместо одного эффективного запроса база получает $N+1$ мелких, что приводит к сетевым задержкам (network round-trips) и перегрузке пула соединений.

##### Как возникает

Обычно это результат неявной ленивой загрузки (Lazy Loading) в ORM (Hibernate, Entity Framework, Django ORM, Sequelize).

##### Пример (Антипаттерн)

Приложение хочет вывести список из 100 постов и авторов каждого поста.

   1. SELECT * FROM posts LIMIT 100; (1 запрос — вернул 100 записей).
   2. ORM в цикле для каждого поста запрашивает автора: SELECT * FROM users WHERE id = :author_id; (Выполняется 100 раз).
   Итого: $1 + 100 = 101$ запрос.

##### Способы решения

* Eager Loading (Жадная загрузка): Принудительное соединение таблиц на уровне БД через JOIN (LEFT JOIN FETCH в JPA, .select_related() в Django).
* Запрос: SELECT * FROM posts p LEFT JOIN users u ON p.author_id = u.id LIMIT 100; (1 запрос).
* Batch Fetching (Пакетная выборка): Сбор идентификаторов и вызов одного запроса через IN (.prefetch_related() в Django).
* Запрос 1: SELECT * FROM posts LIMIT 100;
   * Запрос 2: SELECT * FROM users WHERE id IN (1, 5, 12, ...); (Всего 2 запроса).

------------------------------
##### 2. Индексы (Indices)## Что это такое

Специальные структуры данных, которые создаются поверх таблиц для ускорения поиска строк. Без индекса БД выполняет полное сканирование таблицы (Full Table Scan / Seq Scan), что критично для больших таблиц.

##### Основные типы индексов

* B-Tree (Сбалансированное дерево): Используется по умолчанию. Идеален для поиска по точному совпадению (=), диапазонам (>, <), сортировки (ORDER BY) и оператора LIKE 'abc%'. Скорость поиска: $O(\log N)$.
* Hash-индекс: Использует хэш-таблицы. Эффективен только для точного совпадения (=). Не поддерживает диапазоны и сортировку. Скорость: $O(1)$.
* GiST / GIN: Используются для полнотекстового поиска, массивов, JSON-документов и геоданных (в PostgreSQL).

##### Стратегии индексирования

* Составные индексы (Composite Index): Индекс по нескольким колонкам (A, B, C).
* Правило левого префикса: Индекс сработает для условий WHERE A = 1 или WHERE A = 1 AND B = 2, но не сработает для WHERE B = 2.
* Покрывающие индексы (Covering Index): Индекс, который содержит в себе все поля, запрашиваемые в SELECT. БД берет данные напрямую из индекса (Index Only Scan), не обращаясь к самой таблице (Heap).

##### Избыточное индексирование

Каждый индекс замедляет операции записи (INSERT, UPDATE, DELETE), так как БД обязана обновлять дерево индекса при каждом изменении данных. Также индексы занимают место на диске и в оперативной памяти.

------------------------------
##### 3. Оптимизация запросов (Query Optimization)

Оптимизация начинается с анализа плана выполнения запроса с помощью команд EXPLAIN или EXPLAIN ANALYZE (в PostgreSQL/MySQL).

##### Основные маркеры в плане запроса

* Seq Scan / Table Scan (Плохо для больших таблиц): Полное сканирование. Требует создания индекса.
* Index Scan (Хорошо): Поиск по индексу с последующим переходом к физической странице диска за остальными данными.
* Index Only Scan (Отлично): Данные прочитаны целиком из индекса.

##### Практические приемы оптимизации

* Избегайте SELECT *: Перечисляйте только нужные колонки. Это экономит память, пропускную способность сети и позволяет использовать покрывающие индексы.
* Фильтрация до сортировки: Операция ORDER BY над неиндексированным полем заставляет БД выполнять тяжелую сортировку в памяти или на диске (Subselect/FileSort). Сначала максимально сужайте выборку через WHERE.
* Избегайте функций в WHERE: Запрос WHERE YEAR(birthday) = 2026 заставит БД проигнорировать стандартный индекс по полю birthday.
* Решение: Переписать под диапазон: WHERE birthday >= '2026-01-01' AND birthday <= '2026-12-31'.

------------------------------
##### 4. Оптимизация JOIN

При соединении таблиц планировщик БД выбирает один из трех алгоритмов. Понимание этих алгоритмов помогает правильно проектировать индексы.

##### Алгоритмы JOIN

   1. Nested Loop (Вложенный цикл): Для каждой строки внешней таблицы сканируется внутренняя таблица. Отлично работает, если внутренняя таблица имеет индекс по ключу соединения ($O(M \cdot \log N)$).
   2. Hash Join: БД строит хэш-таблицу по меньшей таблице в оперативной памяти, а затем сканирует большую таблицу. Эффективно для больших объемов данных без индексов при соединении по равенству (=).
   3. Sort Merge Join: Обе таблицы сортируются по ключу соединения, а затем сливаются. Применяется для очень больших таблиц, или если данные уже отсортированы (например, по индексу).

##### Правила оптимизации JOIN

* Индексируйте внешние ключи (Foreign Keys): Поля, по которым происходит ON t1.id = t2.user_id, обязательно должны быть проиндексированы.
* Размер таблиц: Фильтруйте таблицы с помощью WHERE до того, как объединять их, если это возможно, либо доверяйте оптимизатору, не усложняя подзапросы без необходимости.

------------------------------
##### 5. Агрегаты (Aggregates)

Расчет агрегатных функций (COUNT, SUM, AVG, GROUP BY) на лету по миллионам строк блокирует ресурсы БД.

##### Способы оптимизации агрегатов

* Индексы для группировки: Индекс по полю GROUP BY позволяет БД не сортировать данные заново, а считывать их последовательно (GroupAggregate).
* Материализованные представления (Materialized Views): Физическое сохранение результатов тяжелого агрегатного запроса на диск. Представление обновляется по расписанию (cron) или триггерам. Подходит для аналитики (Dashboard), где допустима небольшая задержка актуальности данных.
* Денормализация (Денормализованные счетчики): Добавление колонки comments_count прямо в таблицу posts. Значение инкрементируется при добавлении нового комментария. Избавляет от необходимости вызывать COUNT(*) при каждом просмотре поста.

------------------------------
##### 6. Кеширование запросов (Query Caching)

Кеширование предотвращает повторное выполнение тяжелых запросов к СУБД.

##### Уровни кеширования

* Внутреннее кеширование СУБД: Буферный пул (PostgreSQL Shared Buffers, MySQL InnoDB Buffer Pool). БД хранит часто запрашиваемые страницы данных в RAM. В современных версиях MySQL встроенный Query Cache (кеш текстовых запросов) удален, так как он блокировал таблицы при любой записи.
* Внешнее кеширование (Прикладной уровень): Использование Redis или Memcached перед базой данных.

##### Паттерн Cache-Aside (Классический подход)

   1. Приложение получает запрос на данные.
   2. Проверяет наличие ключа в Redis.
   3. Если данные есть (Cache Hit) — возвращает их клиенту.
   4. Если данных нет (Cache Miss) — делает запрос в БД, записывает результат в Redis с заданным временем жизни (TTL) и возвращает клиенту.

------------------------------
##### 7. Пул соединений (Database Connection Pool)## Проблема «На каждый запрос — новое соединение»

Установление TCP-соединения с СУБД и процесс аутентификации (handshake) — крайне ресурсоемкие операции. Если создавать и закрывать соединение на каждый HTTP-запрос, приложение будет тратить больше времени на подключение, чем на выполнение SQL-команд. В PostgreSQL каждое соединение — это отдельный процесс ОС, который потребляет около 10–20 МБ RAM.

##### Что такое Connection Pool

Брокер соединений (компонент внутри приложения или отдельный прокси-сервер, например PgBouncer, HikariCP).

##### Механизм работы

   1. При старте приложение создает фиксированное количество долгоживущих соединений с БД (например, 20 штук) и держит их в "пуле".
   2. Когда поступает запрос, приложение арендует свободное соединение из пула.
   3. Выполняет SQL-запрос.
   4. Освобождает соединение и возвращает его обратно в пул (физического закрытия сетевого сокета не происходит).

```
[Клиенты] ---> [Приложение] ---> [ Пул соединений (HikariCP/PgBouncer) ] ---> [ База данных ]
                                  (Держит 20 открытых сокетов)
```

------------------------------
##### ✅ Заключение

Оптимизация базы данных — это иерархический процесс. Сначала устраняются архитектурные ошибки кода (проблема N+1) и настраиваются пулы соединений. Затем оптимизируются точечные запросы через добавление индексов и рефакторинг SQL. На финальном этапе, когда ресурсы железа исчерпаны, внедряется кеширование и материализованные агрегаты.
Если вам необходимо, мы можем углубиться в устройство B-Tree индексов, разобрать синтаксис EXPLAIN ANALYZE на конкретном примере или составить конфигурацию для пула соединений PgBouncer. Что из этого разобрать подробнее?


## Query Optimization

##### База знаний: Оптимизация баз данных. Часть 2

------------------------------
##### 1. Query Optimization (Оптимизация запросов). Что это?

Query Optimization — это процесс выбора СУБД наиболее эффективного способа выполнения SQL-запроса.
Когда вы отправляете запрос, встроенный компонент БД (Оптимизатор / Query Optimizer) анализирует его структуру, проверяет доступные индексы, собирает статистику распределения данных в таблицах и строит несколько планов выполнения. Он оценивает их условную стоимость (Cost) в операциях ввода-вывода (I/O) и процессорного времени, после чего запускает самый «дешевый» план.
Задача разработчика — написать запрос и спроектировать структуру таблиц так, чтобы оптимизатор мог легко выбрать идеальный путь, избегая «тяжелых» операций.

------------------------------
##### 2. EXPLAIN и EXPLAIN ANALYZE

EXPLAIN — это главная команда для аудита производительности. Она показывает, какой именно план выполнения выбрал оптимизатор.

##### Разница между командами

* EXPLAIN <query>: Показывает теоретический план на основе накопленной статистической информации. Сама команда запрос не выполняет.
* EXPLAIN ANALYZE <query>: Физически выполняет запрос, замеряет реальное время, количество обработанных строк и сравнивает их с ожиданиями оптимизатора. Внимание: если делать EXPLAIN ANALYZE для DELETE или UPDATE, данные изменятся/удалятся.

##### Ключевые термины в выводе EXPLAIN (на примере PostgreSQL)

* cost: Условная стоимость (первое число — старт запроса, второе — полное выполнение).
* rows: Ожидаемое/реальное количество отфильтрованных строк.
* Filter: Дополнительная фильтрация, которая происходит после чтения данных (часто указывает на отсутствие нужного индекса).
* Actual time: Реальное время выполнения в миллисекундах.

------------------------------
##### 3. Database Profiling и Laravel Debugbar

Database Profiling (Профилирование) — это непрерывный сбор и анализ метрик работы БД: времени выполнения запросов, объема потребляемой памяти, количества дублирующихся или медленных запросов.

##### Инструменты профилирования

* Системные (СУБД): slow_query_log в MySQL, pg_stat_statements в PostgreSQL. Они собирают все запросы, которые выполнялись дольше заданного лимита (например, >200 мс).
* Прикладные (Application Level): Специальные панели разработчика.

##### Laravel Debugbar (Barryvdh/laravel-debugbar)

Это пакет для экосистемы Laravel, который встраивает панель внизу страницы браузера для моментального профилирования в режиме разработки.
Что он позволяет отслеживать в контексте БД:

* Queries (Вкладка запросов): Показывает точный SQL каждого запроса, выполненного при рендере текущей страницы.
* Дубликаты: Подсвечивает абсолютно одинаковые запросы (первый признак отсутствия кеширования или неверной логики).
* N+1 проблемы: Визуально выделяет лавинообразные запросы (например, 50 одинаковых SELECT * FROM roles WHERE id = ...).
* Timeline: Показывает, сколько времени ушло на выполнение PHP-кода, а сколько — на ожидания ответов от базы данных.

------------------------------
##### 4. Оптимизация WHERE

Эффективность блока WHERE определяет, сколько лишних данных СУБД отсечет на самом раннем этапе.

* SARGable условия: Запросы должны быть построены так, чтобы индекс мог использоваться (Search Argumentable).
* Плохо: WHERE MyFunction(column) = 'value' или WHERE column + 10 > 100. Индекс отключается.
   * Хорошо: WHERE column = 'value' или WHERE column > 90.
* Оператор LIKE:
* Поиск по префиксу LIKE 'Иван%' использует стандартный B-Tree индекс.
   * Поиск по подстроке LIKE '%Иван%' или суффиксу LIKE '%Иван' игнорирует B-Tree индекс (требуется полное сканирование или GIN-индекс).
* Индексирование NULL: В PostgreSQL WHERE column IS NULL отлично использует индекс. В старых версиях MySQL/Oracle с этим могут быть нюансы, но современные СУБД индексируют NULL-значения по умолчанию.

------------------------------
##### 5. Оптимизация JOIN

* Порядок соединения: Оптимизатор сам пытается переставить таблицы так, чтобы сначала уменьшить выборку малой таблицей, а затем джойнить большую. Но при использовании LEFT JOIN порядок жестко задан разработчиком — левая таблица обязана быть прочитана первой.
* Индексы на связующие поля: Поля, указанные в секции ON t1.user_id = t2.id, должны иметь индексы с обеих сторон. Обычно t2.id — это Primary Key (индекс есть), а вот про t1.user_id (Foreign Key) разработчики часто забывают.
* Типы данных: Поля соединения должны иметь строго одинаковый тип данных (например, INT и INT, или UUID и UUID). Если джойнить VARCHAR с INT, СУБД будет принудительно приводить типы на лету для каждой строки, отключая индексы.

------------------------------
##### 6. Оптимизация сортировки (ORDER BY)

Сортировка — одна из самых ресурсоемких операций, так как при нехватке оперативной памяти (work_mem в Postgres) СУБД сбрасывает временные данные на диск (файловая сортировка / FileSort).

* Сортировка по индексу: Если для поля в ORDER BY создан B-Tree индекс, данные в нем уже отсортированы. БД просто читает их по порядку ($O(1)$ на сортировку).
* Сортировка составным индексом: Если у вас есть индекс (status, created_at), то запрос WHERE status = 'active' ORDER BY created_at DESC отработает мгновенно.
* Направление сортировки: Запрос ORDER BY score DESC, date ASC не сможет эффективно использовать стандартный индекс (score, date). Направление колонок в индексе должно совпадать с запросом, либо индекс должен быть объявлен как (score DESC, date ASC).

------------------------------
##### 7. Оптимизация COUNT

* COUNT(*) против COUNT(column):
* COUNT(*) или COUNT(1) считает общее количество строк. Оптимизатор выбирает для этого самый "узкий" и легкий индекс.
   * COUNT(column) считает только строки, где значение column NOT NULL. БД тратит ресурсы на проверку каждой ячейки.
* Проблема больших таблиц: В СУБД с поддержкой MVCC (PostgreSQL) COUNT(*) без условий всегда делает полное сканирование (Index Only Scan всей таблицы), так как СУБД должна проверить видимость каждой строки для текущей транзакции.
* Оптимизация:
* Если нужна точная цифра: использовать денормализованные счетчики (триггеры, инкремент в приложении).
   * Если нужна примерная цифра (для пагинации на миллионах строк): брать системную статистику СУБД. В PostgreSQL: SELECT reltuples FROM pg_class WHERE relname = 'users';.

------------------------------
##### 8. Оптимизация UPDATE и DELETE

Модификация данных работает медленнее чтения, так как включает механизмы блокировок строк/таблиц, запись в журнал транзакций (WAL) и перестройку индексов.

* Модификация по индексу: Любой `UPDATE`/`DELETE` содержит в себе скрытый `SELECT`. Запрос `UPDATE users SET status = 0 WHERE email = 'test@test.com'` должен моментально найти строку по индексу email. Иначе заблокируется вся таблица на время поиска.
* Пакетное удаление/обновление (Chunking): Удаление 10 000 000 строк одним запросом DELETE FROM logs WHERE created_at < '2025-01-01' положит базу данных: забьется RAM, заблокируются таблицы, раздуется WAL-лог.
* Решение: Удалять небольшими порциями (пакетами по 5 000–10 000 строк) в цикле с небольшим таймаутом.
* Влияние индексов: Чем больше индексов на таблице, тем медленнее выполняются UPDATE и DELETE, так как СУБД обязана удалить/переписать указатели во всех деревьях индексов.

------------------------------
##### 9. Практические примеры (До и После)## Пример 1: Оптимизация WHERE (Отключение индекса функцией)

* До (Медленно): СУБД вычисляет функцию для каждой строки таблицы.

```sql
SELECT * FROM orders WHERE DATE(created_at) = '2026-06-01';
 -- План: Seq Scan (Полное сканирование)
``` 

* После (Быстро): Превращаем в диапазон, задействуя индекс по created_at.

```sql
SELECT * FROM orders WHERE created_at >= '2026-06-01 00:00:00' AND created_at <= '2026-06-01 23:59:59';-- План: Index Scan
```

##### Пример 2: Оптимизация JOIN и Сортировки

Есть таблицы users и articles. Задача — получить последние статьи активных пользователей.

* До (Медленно): Сортировка происходит после объединения огромного массива данных.

SELECT a.title, u.name FROM articles a JOIN users u ON a.user_id = u.id WHERE u.status = 'active' ORDER BY a.created_at DESC LIMIT 10;-- План: Hash Join + Heavy FileSort в памяти

* После (Быстро): Создаем составной индекс на таблицу статей (user_id, created_at DESC). Оптимизатор сможет использовать Nested Loop, забирая из индекса уже отсортированные статьи порциями, пока не наберет 10 штук, соответствующих активным пользователям.

##### Пример 3: Пакетный DELETE (Chunking на бэкенде)

* До (Антипаттерн):

```php
// Блокирует базу, может упасть по таймауту
DB::table('logs')->where('created_at', '<', now()->subMonths(6))->delete();
```

* После (Оптимально):

```php
// Удаляем порциями по 5000 записей, не нагружая транзакции
do {
    $deleted = DB::table('logs')
        ->where('created_at', '<', now()->subMonths(6))
        ->limit(5000)
        ->delete();

    usleep(50000); // Пауза 50мс между батчами для "отдыха" БД
} while ($deleted > 0);
```

## Frontend Optimization

##### База знаний: Оптимизация фронтенда (Frontend Optimization)

Frontend Optimization (Оптимизация фронтенда) — это процесс уменьшения времени загрузки веб-страницы и повышения плавности её интерфейса. Главная цель — улучшить пользовательский опыт (UX) и показатели Core Web Vitals (метрики Google для ранжирования сайтов):

* LCP (Largest Contentful Paint): Скорость загрузки основного контента.
* INP (Interaction to Next Paint): Скорость отклика интерфейса на клики и нажатия.
* CLS (Cumulative Layout Shift): Стабильность верстки (отсутствие внезапных сдвигов элементов).

------------------------------
##### 1. Сборщики: Laravel Mix и Vite

Современный фронтенд не отдается в браузер "как есть". Он проходит через сборщики, которые автоматизируют оптимизацию.

##### Laravel Mix (Устаревший подход)

Инструмент поверх Webpack. Работает по принципу предварительной сборки.

* Как работает: При любом изменении кода Webpack пересобирает весь проект (или его крупную часть) в один или несколько файлов и сохраняет их на диск. На больших проектах hot-reload (обновление в браузере) мог занимать по 5–10 секунд.

##### Vite (Современный стандарт)

Новое поколение сборщиков, пришедшее на смену Mix в Laravel 9+.

* Как работает: В режиме разработки Vite использует Native ESM (ES-модули) браузера. Браузер сам запрашивает только те файлы, которые нужны для текущей страницы. Вместо пересборки всего бандла Vite мгновенно заменяет измененный модуль (HMR — Hot Module Replacement).
* В продакшене: Для финальной сборки Vite использует сверхбыстрый сборщик Rollup, который максимально агрессивно оптимизирует код.

------------------------------
##### 2. Минификация и сжатие данных

Процесс уменьшения физического веса текстовых файлов (HTML, CSS, JS), передаваемых по сети.

##### Минификация (Minification)

Удаление из кода всех символов, ненужных для работы браузера: пробелов, переносов строк, комментариев. Также применяется Uglify/Mangle — переименование длинных переменных и функций в короткие (например, function calculateTotalPrice() превращается в function c()).

* Инструменты: Выполняется автоматически сборщиками (Terser, esbuild) при запуске npm run build.

##### Сжатие на сервере (Gzip / Brotli)

Архивация файлов на стороне веб-сервера (Nginx/Apache) перед отправкой клиенту. Браузер на лету распаковывает эти архивы.

* Gzip: Стандарт де-факто. Сжимает текстовые файлы на 60–80%.
* Brotli: Более современный алгоритм от Google. Сжимает данные еще на 15–25% эффективнее, чем Gzip, и быстрее распаковывается в браузере.

------------------------------
##### 3. Оптимизация изображений

Картинки составляют до 60-70% веса среднестатистического сайта.

##### Основные методы оптимизации

* Современные форматы: Замена старых форматов (JPEG, PNG) на WebP и AVIF. Они обеспечивают аналогичное качество при весе на 30–50% меньше.
* Сжатие (Compression): Удаление метаданных (информация о камере, геолокация) и легкое ухудшение качества, незаметное для человеческого глаза (качество 75–80%).
* Адаптивные изображения: Отдача разных разрешений картинки в зависимости от экрана устройства (мобильный или 4K-монитор) с помощью тега <picture> или атрибута srcset.
* Ленивая загрузка (Lazy Loading): Атрибут loading="lazy". Браузер не скачивает картинку, пока пользователь не доскроллит до неё.

------------------------------
##### 4. CDN (Content Delivery Network)

CDN (Сеть доставки контента) — это географически распределенная сеть серверов (Cloudflare, Akamai, Keycdn), которая ускоряет доставку статических файлов (картинок, JS, CSS, шрифтов) до конечного пользователя.

```
[Пользователь в Токио] ---> (Ближайший сервер CDN в Токио) ---> Мгновенный ответ
                                   | (Если файла нет в кеше CDN)
                                   v
                      [Оригинальный сервер в Москве]

```

##### Принцип работы

   1. Оригинальный сервер (Origin) находится, например, в Москве.
   2. Пользователь из Токио запрашивает сайт.
   3. CDN перехватывает запрос и отдает статику со своего сервера в Токио (Edge-сервер).
   4. Результат: Сетевая задержка (RTT) снижается с 300 мс до 10 мс. CDN также снижает нагрузку и трафик на ваш основной сервер.

------------------------------
##### 5. Кеширование фронтенда (HTTP-кеширование)

Позволяет браузеру сохранять файлы локально на устройстве пользователя, чтобы не скачивать их повторно при переходе между страницами.

##### Ключевые заголовки ответа (Headers)

* Cache-Control: max-age=31536000, public: Инструкция браузеру закэшировать файл "намертво" на 1 год. Применяется для сборных файлов (app.js, style.css), у которых в названии есть хэш.
* Etag / Last-Modified: Валидаторы кэша. Если файл может измениться, браузер делает легкий запрос к серверу: "У меня есть файл с Etag 'abc', он изменился?". Если нет, сервер возвращает статус 304 Not Modified без тела файла.

##### Инвалидация кэша (Cache Busting)

Если файл закэширован на год, как раскатить обновление? Сборщики добавляют хэш контента в имя файла: app.a8f2b3.js. Если код изменился, хэш станет другим (app.991cde.js), HTML-документ сошлется на новое имя, и браузер скачает свежую версию.

------------------------------
##### 6. Code Splitting (Разделение кода)

По умолчанию сборщики пытаются объединить весь JS-код проекта в один гигантский файл. Code Splitting разбивает этот монолит на мелкие части (чанки), которые загружаются только тогда, когда они действительно нужны.

##### Стратегии разделения

* По маршрутам (Route-based): Код административной панели не должен загружаться, когда пользователь зашел на главную страницу сайта. Каждому роуту — свой JS-файл.
* Динамический импорт: Загрузка тяжелых библиотек (например, графиков или текстовых редакторов типа TinyMCE) только в момент клика на кнопку "Редактировать".

## PHP Optimization

##### База знаний: Оптимизация PHP и Laravel (Backend Optimization)

PHP Optimization — это комплекс мер по повышению скорости выполнения PHP-скриптов, снижению потребления оперативной памяти (RAM) процессом и увеличению пропускной способности (RPS — запросов в секунду) веб-сервера.
В отличие от компилируемых языков (Go, Rust), PHP по умолчанию работает по принципу «умереть после каждого запроса» (Shared-Nothing Architecture). Оптимизация PHP направлена на то, чтобы минимизировать накладные расходы на инициализацию окружения, парсинг кода и нерациональное использование памяти при обработке каждого HTTP-запроса.

------------------------------
##### 1. OPcache (Кеширование байт-кода)

По умолчанию при каждом HTTP-запросе PHP считывает исходный .php файл, анализирует его (лексический и синтаксический анализ) и компилирует в промежуточный байт-код (opcode), который затем исполняется виртуальной машиной Zend VM.

##### Как работает OPcache

OPcache — это встроенное расширение, которое компилирует PHP-скрипт в байт-код ровно один раз и сохраняет его в общую оперативную память (Shared Memory). При последующих запросах PHP полностью пропускает стадии чтения диска и компиляции, мгновенно выполняя готовый байт-код из RAM.

##### Ключевые директивы php.ini для Production

* opcache.enable=1: Включение расширения.
* opcache.memory_consumption=256: Объем выделяемой RAM в мегабайтах (подбирается под размер кодовой базы проекта вместе с вендорами).
* opcache.max_accelerated_files=20000: Максимальное количество файлов в кеше. Для крупных фреймворков (Laravel) значение должно быть выше реального числа файлов в проекте.
* opcache.validate_timestamps=0: Критично для продакшена. Если выставлен 0, PHP вообще не проверяет, изменился ли файл на диске. Это дает максимальную скорость. Инвалидация кеша происходит при деплое через вызов opcache_reset().

------------------------------
##### 2. Настройки PHP-FPM (FastCGI Process Manager)

PHP-FPM — это менеджер процессов, который принимает запросы от веб-сервера (Nginx) и распределяет их по рабочим процессам (worker processes).

##### Режимы работы (pm)

Настраиваются в конфигурационном файле пула (например, www.conf):

   1. pm = dynamic: Количество процессов гибко меняется в зависимости от нагрузки. Подходит для серверов с ограниченными ресурсами, где крутятся разные сервисы.
   2. pm = static: Рекомендуется для высоконагруженного Production. Фиксированное количество воркеров запускается сразу и постоянно висит в памяти. Нет накладных расходов на создание и уничтожение процессов ОС.

##### Важные параметры для pm = static

* pm.max_children: Количество одновременно работающих процессов.
* Формула расчета: (Доступная RAM сервера - RAM для БД/Redis) / Средний размер одного PHP-процесса. Если у вас свободно 4 ГБ RAM, а один процесс Laravel под нагрузкой занимает 50 МБ, то max_children = 4000 / 50 = 80.
* pm.max_requests = 1000: Количество запросов, которое обрабатывает один воркер перед тем, как перезапуститься. Защищает систему от постепенных утечек памяти (Memory Leaks) в сторонних библиотеках.

------------------------------
##### 3. Управление памятью (Memory Management)

PHP использует автоматический сборщик мусора (Garbage Collector) с подсчетом ссылок.

* memory_limit: Ограничение памяти на один скрипт. На продакшене для стандартных веб-запросов хорошей практикой считается 128M или 256M. Ставить -1 (безлимит) для веб-запросов запрещено (один «тяжелый» запрос может вызвать Out-Of-Memory и уронить сервер). Безлимит допускается только в CLI-скриптах и очередях.
* Утечки памяти (Memory Leaks): В долгоживущих процессах (очереди Laravel Horizon, RoadRunner, FrankenPHP) память может не освобождаться, если объекты остаются связанными циклическими ссылками или бесконечно наполняют глобальные массивы (static $cache = []).
* Освобождение ресурсов: Использование встроенного метода gc_collect_cycles() позволяет принудительно запустить сборщик мусора, если скрипт обрабатывает огромный массив данных в цикле.

------------------------------
##### 4. Оптимизации в PHP 8.x

Версии PHP 8.x принесли революционные изменения в производительность.

* JIT (Just-In-Time) компиляция: Компилятор, который превращает часто используемый байт-код напрямую в машинный код процессора на лету. В веб-приложениях (где узкое горло — это I/O, запросы к БД и сети) прирост минимален, но в вычислительных задачах (математика, обработка текста, изображений) JIT дает ускорение в разы.
* Улучшения в Zend VM: Оптимизирована работа со строками, хэш-таблицами и внутренними структурами данных.
* Constructor Property Promotion: Синтаксический сахар, который также немного снижает затраты на парсинг и создание свойств объекта.

------------------------------
##### 5. Оптимизация кода (Кодинг-паттерны)

* Ленивая инициализация (Lazy Initialization): Не создавайте объекты и не подключайте тяжелые классы, если они не понадобятся в текущей ветке выполнения (используйте WeakRef или анонимные фабрики, если необходимо).
* Работа со строками: Используйте `str_pos` / `str_starts_with` вместо тяжелых регулярных выражений (preg_match), если задача тривиальна.
* Итераторы и Генераторы (yield): При чтении больших файлов или выборок из БД генераторы позволяют обрабатывать данные построчно, не загружая весь гигабайтный файл или миллион строк таблицы в RAM.
* Строгая типизация (`declare(strict_types=1);`): Помогает движку Zend VM оптимизировать выполнение функций, так как ему не нужно тратить ресурсы на неявное приведение типов на лету.

------------------------------
##### 6. Laravel-специфичные оптимизации

Laravel — мощный, но «тяжелый» фреймворк. Большинство накладных расходов связано с парсингом сервис-провайдеров, сервис-контейнера и конфигурационных файлов при каждом запросе.

##### Кеширование на Production (Обязательный шаг при деплое)

* php artisan config:cache: Объединяет сотни конфигурационных файлов проекта и пакетов в один плоский PHP-массив, который моментально считывается через OPcache.
* php artisan route:cache: Компилирует все роуты приложения в один быстрый метод поиска. Особенно критично, если в проекте больше 50–100 роутов.
* php artisan view:cache: Компилирует все Blade-шаблоны в чистый PHP-код заранее, чтобы не делать этого во время запроса пользователя.

##### Другие архитектурные приемы

* Смена драйверов: Переключение сессий (SESSION_DRIVER) и кеша (CACHE_DRIVER) с медленного file или database на сверхбыстрый redis.
* Асинхронные очереди (Queue Workers): Вынос всех долгих задач (отправка писем, генерация PDF, вызовы внешних API) из синхронного HTTP-потока в фоновые воркеры (Laravel Horizon). Пользователь мгновенно получает ответ 202 Accepted, пока сервер отправляет письмо в фоне.

------------------------------
##### 7. Профилирование бэкенда (Profiling)

Профилирование — это детальный замер ресурсов (время CPU, вызовы функций, выделение памяти), затраченных на выполнение кода.

##### Основные инструменты

* Xdebug: Отличный инструмент для локальной отладки, но категорически запрещен на Production, так как сильно замедляет работу интерпретатора (на 200–500%).
* XHprof / Blackfire.io: Легковесные профилировщики, разработанные специально для Production-окружения. Они собирают граф вызовов функций с минимальным оверхедом (~1–3%). Позволяют увидеть, какая именно функция или метод класса «съедает» больше всего процессорного времени (Wall Time).

------------------------------
##### 8. Практические примеры (До и После)## Пример 1: Использование Генератора для экономии памяти

Задача: Сгенерировать и выгрузить CSV-отчет на 500 000 строк.

* До (Память переполнится, скрипт упадет с Fatal Error):

```php

public function export() {
    $data = [];
    foreach (User::all() as $user) {
        $data[] = [$user->id, $user->email]; // Все 500к записей оседают в RAM
    }
    return $data;
}
```
* После (Потребление памяти стабильно низкое — около 15-20 МБ):

```php
public function export() {
    // Метод cursor() в Laravel использует генераторы (LazyCollection)
    foreach (User::cursor() as $user) {
        yield [$user->id, $user->email]; // В памяти в один момент времени находится только 1 строка
    }
}
```


##### Пример 2: Оптимизация работы с коллекциями Laravel

Задача: Проверить, есть ли в коллекции хотя бы один пользователь с верифицированным email.

* До (Медленно): Фильтрует всю коллекцию в памяти, создает новый объект коллекции и считает элементы.

```php
$hasVerified = $users->filter(fn($u) => $u->is_verified)->count() > 0;
```

* После (Быстро): Прерывает выполнение сразу же при обнаружении первого совпадения.

```php
$hasVerified = $users->contains(fn($u) => $u->is_verified);
```

## Scaling & Load Balancing

Scaling (Масштабирование) и Load Balancing (Балансировка нагрузки) — это системные подходы, направленные на обеспечение высокой доступности (High Availability), отказоустойчивости (Fault Tolerance) и способности приложения обрабатывать возрастающее количество запросов (RPS) без деградации производительности. 

------------------------------
##### 1. Vertical vs Horizontal Scaling

Когда один сервер перестает справляться с текущей нагрузкой, систему масштабируют одним из двух способов. 

| Критерий [3, 4, 5, 6, 7] | Vertical Scaling (Scale Up / Вертикальное) | Horizontal Scaling (Scale Out / Горизонтальное) |
|---|---|---|
| Суть | Увеличение мощности одного существующего сервера (апгрейд CPU, RAM, дисков). | Добавление новых серверов (нод, экземпляров) в общую систему. |
| Пределы | Имеет жесткий физический и экономический потолок (лимиты железа/материнской платы). | Практически безгранично. Можно добавлять сотни типовых серверов. |
| Отказоустойчивость | Низкая. Сервер остается единой точкой отказа (Single Point of Failure). | Высокая. При выходе из строя одного сервера нагрузка перераспределится на остальные. |
| Изменения в коде | Не требуются. Приложение просто получает больше ресурсов. | Требуются (Stateless-архитектура). Код должен быть готов работать параллельно. |

------------------------------
##### 2. Load Balancer (Балансировщик нагрузки)

Load Balancer (LB) — это выделенный компонент (программный или аппаратный), который принимает входящий трафик от пользователей и распределяет его между группой бэкэнд-серверов (серверной фермой / upstream). 

##### Основные уровни работы LB (по модели OSI)

* L4 (Транспортный уровень): Балансировка на основе IP-адресов и TCP/UDP-портов (например, HAProxy в режиме TCP, AWS Network Load Balancer). Работает очень быстро, так как не вникает в содержимое пакетов.
* L7 (Прикладной уровень): Балансировка на основе протокола HTTP/HTTPS (Nginx, HAProxy, AWS ALB). Может распределять трафик в зависимости от URL (например, /api отправлять на одни сервера, а /images — на другие), кук или заголовков. [7] 

##### Алгоритмы балансировки

* Round Robin: Запросы передаются серверам последовательно по кругу (1-й, 2-й, 3-й, затем снова 1-й).
* Least Connections: Запрос направляется серверу, у которого сейчас меньше всего активных соединений. Идеально для тяжелых запросов.
* IP Hash: Хэшируется IP-адрес клиента. Закрепляет конкретного пользователя за определенным сервером (подходит для легаси-систем с локальными сессиями).

------------------------------
##### 3. Session Management (Управление сессиями при масштабировании)

В классическом веб-приложении сессии пользователей хранятся в виде файлов на диске сервера. При добавлении второго сервера возникает проблема: пользователь авторизовался на Server 1, но следующий его запрос LB отправил на Server 2, где сессии нет — пользователя «разлогинивает».

##### Способы решения

* Sticky Sessions (Session Affinity): Настройка балансировщика, заставляющая его всегда направлять запросы конкретного пользователя на один и тот же сервер (через куку или IP). Минус: ломает идею равномерной балансировки и отказоустойчивости.
* Централизованное хранилище сессий (Рекомендуется): Перевод сессий на общий быстрый in-memory сервер БД, доступный всем нодам приложения (обычно Redis или Memcached).
* Stateless JWT: Отказ от сессий на сервере. Вся информация о пользователе шифруется в токен (JWT) и хранится в браузере клиента. Сервера лишь валидируют токен с помощью секретного ключа. [7] 

------------------------------
##### 4. Cache Synchronization (Синхронизация кеша)

При горизонтальном масштабировании локальный кеш приложения (например, кеш в памяти PHP-процесса или файлы на диске ноды) становится неактуальным на других серверах при обновлении данных.

##### Стратегии синхронизации

* Единый удаленный кеш: Использование централизованного кластера Redis/Memcached для всех инстансов приложения вместо локального кэширования нод.
* Паттерн Pub/Sub (Издатель/Подписчик): При изменении сущности Server 1 обновляет базу данных и публикует событие в канал (через Redis/RabbitMQ). Остальные сервера, подписанные на этот канал, получают уведомление и инвалидируют/очищают свой локальный кеш.

------------------------------
##### 5. File Storage Synchronization (Синхронизация файлового хранилища)

Если пользователь загрузил аватарку на Server 1, она сохранится на локальном диске этой ноды. Пользователи, чьи запросы попадут на Server 2, картинку не увидят.

##### Способы решения

* Сетевые файловые системы (NFS / GlusterFS / Ceph): Монтирование общей папки по сети ко всем серверам. Минус: Сетевой оверхед, низкая скорость при работе с миллионами мелких файлов.
* Облачное объектное хранилище (Рекомендуется): Полный отказ от хранения медиа-файлов на серверах приложения. Все файлы загружаются напрямую или через бэкэнд в S3-совместимые хранилища (AWS S3, MinIO, Yandex Object Storage). Сервера хранят в базе только URL-ссылки на эти файлы.

------------------------------
##### 6. Database Scaling (Масштабирование баз данных)

Базу данных масштабировать сложнее всего, так как она обязана сохранять консистентность состояния.

* Репликация (Read Replicas): Разделение на Master (принимает все операции записи — INSERT, UPDATE, DELETE) и одну или несколько реплик Slave (только чтение — SELECT). Приложение отправляет тяжелую аналитику и чтение контента на реплики, разгружая мастер-ноду.
* Шардирование (Sharding): Горизонтальное разделение данных одной таблицы по разным физическим серверам БД на основе ключа шардирования (Sharding Key). Например, пользователи с ID от 1 до 1 000 000 хранятся на DB-1, а от 1 000 001 — на DB-2. [7] 

------------------------------
##### 7. Queue Workers Scaling (Масштабирование обработчиков очередей)

Обработчики очередей (воркеры) идеально подходят для горизонтального масштабирования, так как они изначально проектируются в рамках асинхронной архитектуры.

* Параллельная обработка: Если очередь (например, в Redis, RabbitMQ, SQS) переполняется задачами, достаточно запустить дополнительные процессы воркеров на текущем сервере или ввести новые выделенные сервера под воркеры.
* Конкурентный потребитель (Competing Consumers): Брокер очередей гарантирует, что одну задачу из очереди получит и обработает только один воркер, исключая дублирование выполнения.
* Автомасштабирование (KEDA в Kubernetes): Автоматическое увеличение или уменьшение количества подов с воркерами в зависимости от длины очереди (количества необработанных задач в очереди).

##### Пример 1: Stateless-конфигурация Nginx для балансировки трех Laravel-нод

Балансировщик распределяет HTTP-трафик, а бэкэнды обращаются к единым сервисам данных.

# Конфигурация на сервере Балансировщика (L7 Nginx)

```php

upstream laravel_app {
    server 10.0.0.11:80 weight=3; # Более мощный сервер (получит 3 запроса из 5)
    server 10.0.0.12:80;          # Обычный сервер
    server 10.0.0.13:80;          # Обычный сервер
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://laravel_app;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Host $host;
    }
}

```

##### 9. CDN для масштабирования (Scaling)

CDN (Content Delivery Network) — это первая линия обороны при масштабировании.

* Снятие нагрузки с серверов (Offloading): Перенос отдачи всей статики (картинки, шрифты, скомпилированные JS/CSS-бандлы) на Edge-сервера CDN позволяет освободить до 70–80% мощностей бэкэнд-серверов. Бэкэнд начинает обрабатывать исключительно динамические API-запросы.
* Кеширование динамики (Edge Caching): Позволяет кэшировать редкие или неперсонализированные GET-запросы к API прямо на стороне CDN (например, Cloudflare) на короткое время (например, на 1 минуту). При резком наплыве трафика (Slashdot-эффект) миллионы запросов закроет CDN, а до вашего сервера дойдет лишь 1 запрос в минуту.

##### 10. Monitoring при масштабировании

При наличии десятков серверов классический мониторинг одного сервера становится неэффективным. Требуется агрегация метрик.

* Централизованный сбор метрик (Prometheus + Grafana): На каждой ноде разворачивается легковесный агент (Node Exporter), который отправляет данные в единое хранилище Prometheus. Grafana строит графики по всему кластеру серверов.
* Агрегированный сбор логов (ELK / Vector / Loki): Запрещено заходить на каждый сервер по SSH для чтения файлов логов (tail -f laravel.log). Логи со всех нод в реальном времени стримятся в единую систему (например, Elasticsearch + Kibana или Grafana Loki), где их можно фильтровать по ID запроса (Correlation ID).
* Метрики для автоскейлинга (Autoscaling Metrics): Облачные платформы (AWS EC2 Auto Scaling) или Kubernetes (HPA) отслеживают утилизацию CPU/RAM пула серверов. Если средняя нагрузка по всему кластеру превышает 70% в течение 3 минут, система автоматически разворачивает новые ноды приложения.
