
## MVC (Model-View-Controller)


##### База знаний: Архитектурный паттерн MVC (Model-View-Controller)

MVC (Model-View-Controller) — это фундаментальный архитектурный паттерн, который разделяет бизнес-логику приложения, пользовательский интерфейс и управляющую логику на три отдельных компонента. Основная цель паттерна — изолировать данные от их визуального представления, обеспечивая слабую связанность элементов (Low Coupling) и возможность их независимой модификации.

------------------------------
##### 1. Компоненты MVC

* Model (Модель): Отвечает за хранение, выборку и модификацию данных, а также за выполнение бизнес-логики. Она «не знает», как данные будут отображаться на экране (ей безразличен HTML, JSON или консольный вывод). Модель лишь гарантирует валидность и целостность данных.
* View (Представление / Вид): Отвечает исключительно за отображение данных, полученных от Модели, конечному пользователю (UI). В веб-приложениях это чаще всего HTML-шаблоны (Blade, Twig), JSON-ответы API или XML. В представлении не должно быть сложной логики — только вывод переменных и простейшие циклы.
* Controller (Контроллер): Выступает в роли посредника (Indirection). Он принимает входящий запрос от пользователя (например, клик по ссылке или отправку формы), интерпретирует его, при необходимости вызывает методы Модели для изменения данных и передает результат в нужное Представление.

------------------------------
##### 2. Как это работает (Жизненный цикл запроса в Web-MVC)

В контексте веб-приложений (например, на Laravel или Django) классический цикл обработки запроса выглядит следующим образом:

```

[Пользователь] --(1. HTTP GET /posts/5)--> [ Роутер / Контроллер ]
                                                    |
             +--------------------------------------+ (2. Дай мне пост №5)
             |                                      v
             |                              [     Модель     ] (Получает данные из БД)

             |                                      |
             | <---(3. Возвращает объект данных)----+
             v
 [  Контроллер  ] --(4. Передает данные в шаблон)--> [ Представление ]
                                                             |
[Пользователь] <--(5. Возвращает готовый HTML-ответ)---------+

```

   1. Запрос: Пользователь вводит URL или отправляет форму, формируя HTTP-запрос.
   2. Маршрутизация: Фреймворк направляет запрос на конкретный метод Контроллера.
   3. Работа с данными: Контроллер обращается к Модели: «Найди мне товар с ID 15».
   4. Бизнес-логика: Модель делает SQL-запрос, применяет правила (например, рассчитывает скидку) и возвращает объект с данными Контроллеру.
   5. Рендеринг: Контроллер берет эти данные и «скармливает» их Представлению (шаблону).
   6. Ответ: Представление генерирует итоговый HTML/JSON и отправляет его обратно браузеру пользователя.

------------------------------
##### 3. Когда использовать MVC## Использовать:

* В веб-приложениях с классическим серверным рендерингом (Server-Side Rendering — SSR).
* При проектировании REST API (в роли View здесь выступают JSON-трансформеры или ресурсы).
* В крупных проектах, где над кодом параллельно работают разные специалисты (бекэндер может менять логику в Модели, не мешая верстальщику править HTML во View).

##### Когда паттерн может меняться/уступать место другим:

* В современных SPA-приложениях (React, Vue) классический MVC на бэкэнде часто трансформируется в схему API-Менеджер -> Сервис -> Репозиторий, а логика отображения полностью уходит на фронтенд.
* В клиентских интерфейсах часто применяются модификации паттерна: MVVM (Model-View-ViewModel в Vue/Angular) или MVP (Model-View-Presenter).

## Repository Pattern

Repository Pattern (Паттерн Репозиторий) — это архитектурный паттерн, который создает слой абстракции между бизнес-логикой приложения (Сервисами/Контроллерами) и слоем доступа к данным (СУБД, ORM, внешними API).
По определению Эрика Эванса (в рамках Domain-Driven Design), Репозиторий имитирует коллекцию объектов, находящуюся целиком в оперативной памяти. Он позволяет коду бизнес-логики работать с объектами, полностью игнорируя технические нюансы выполнения SQL-запросов, методов ORM или путей к файлам.

------------------------------
##### 1. Зачем он нужен (Решаемые проблемы)

В классическом MVC (особенно при использовании паттерна Active Record, как в Laravel Eloquent) контроллеры или сервисы часто перенасыщаются методами построения SQL-запросов (например: User::where('status', 1)->orderBy('created_at')->take(10)->get()).
Это приводит к следующим проблемам:

* Жесткая связность (Tight Coupling): Код бизнес-логики намертво привязан к конкретной ORM. Если вы решите заменить базу данных или перейти на другую ORM, вам придется переписать весь проект.
* Нарушение DRY: Однотипные сложные выборки (например, «получить топ-10 активных пользователей из определенного региона») начинают копироваться и дублироваться в разных контроллерах, джобах и консольных командах.
* Сложность тестирования (Mocking): Модели Active Record тяжело тестировать в изоляции от реальной базы данных, так как они совмещают в себе и структуру данных, и методы подключения к СУБД.

------------------------------
##### 2. Как устроен паттерн

Репозиторий состоит из двух ключевых элементов, реализующих принцип инверсии зависимостей (Dependency Inversion Principle из SOLID) и обеспечивающих слабую связанность (Low Coupling):

   1. Интерфейс (Contract): Объявляет набор методов для работы с данными, понятных бизнес-логике (например, getUserById(), getPremiumUsers()). Он ничего не знает о SQL.
   2. Конкретная реализация (Implementation): Класс, который имплементирует интерфейс и содержит внутри себя специфический код для работы с текущей технологией (Eloquent, Query Builder, Doctrine, Mongo Client или cURL-запросы к внешнему REST API).

------------------------------
##### 3. Пример реализации в Laravel## Шаг 1: Создание интерфейса (Контракта)

Этот интерфейс размещается в слое ядра приложения. Бизнес-логика будет зависеть только от него.

```php
namespace App\Repositories\Contracts;

use App\Models\User;
use Illuminate\Support\Collection;

interface UserRepositoryInterface
{
    public function findById(int $id): ?User;
    
    public function getActivePremiumUsers(): Collection;
}
```

##### Шаг 2: Создание конкретной реализации (через Eloquent)

Этот класс скрывает в себе всю специфику Laravel Eloquent.

```php

namespace App\Repositories\Eloquent;

use App\Models\User;
use App\Repositories\Contracts\UserRepositoryInterface;
use Illuminate\Support\Collection;

class EloquentUserRepository implements UserRepositoryInterface
{
    public function findById(int $id): ?User
    {
        return User::find($id);
    }

    public function getActivePremiumUsers(): Collection
    {
        // Вся тяжелая Eloquent-выборка инкапсулирована здесь
        return User::where('status', 'active')
            ->where('is_premium', true)
            ->orderBy('rating', 'desc')
            ->get();
    }
}

```

##### Шаг 3: Регистрация в Service Container (AppServiceProvider.php)

Связываем интерфейс и реализацию, чтобы Laravel знал, какой класс подставить при внедрении зависимости.

```php

namespace App\Providers;

use Illuminate\Support\ServiceProvider;
use App\Repositories\Contracts\UserRepositoryInterface;
use App\Repositories\Eloquent\EloquentUserRepository;

class AppServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        // Если код просит UserRepositoryInterface, отдаем EloquentUserRepository
        $this->app->bind(UserRepositoryInterface::class, EloquentUserRepository::class);
    }
}
```

##### Шаг 4: Использование в Контроллере/Сервисе

Контроллер чист от SQL и методов ORM. Он слабо связан с базой данных.

```php

namespace App\Http\Controllers;

use App\Repositories\Contracts\UserRepositoryInterface;

class DashboardController extends Controller
{
    // Внедряем интерфейс через конструктор (Dependency Injection)
    public function __construct(
        protected UserRepositoryInterface $userRepository
    ) {}

    public function index()
    {
        // Контроллер просто просит данные, не зная, откуда они берутся
        $premiumUsers = $this->userRepository->getActivePremiumUsers();

        return view('dashboard', compact('premiumUsers'));
    }
}

```

------------------------------
##### 4. Когда использовать, а когда это антипаттерн

Паттерн «Репозиторий» — один из самых обсуждаемых в экосистеме Laravel, так как сама ORM Eloquent уже является реализацией паттернов Data Mapper/Active Record и содержит в себе базовые методы абстракции данных.

##### Когда Репозиторий необходим:

* В крупных проектах со сложной доменной логикой (DDD — Domain-Driven Design).
* Когда приложение использует несколько источников данных для одной сущности (например, старые статьи берутся из архива в MongoDB, новые — из PostgreSQL, а частые запросы кэшируются в Redis). Реализация репозитория скроет это разделение от остального кода.
* При написании строгих Unit-тестов, когда нужно протестировать логику контроллера без фактического обращения к тестовой БД (интерфейс репозитория легко подменяется заглушкой (Mock) прямо в тесте).

##### Когда это антипаттерн (Ошибки применения):

* В простых CRUD-проектах: Создавать интерфейс и класс-репозиторий для таблицы, где выполняются только методы `User::all()` и `User::create()`, — это избыточное усложнение (нарушение принципов KISS и YAGNI).
* Репозиторий, возвращающий Query Builder: Если метод репозитория возвращает не готовую коллекцию/объект, а недостроенный запрос (return User::where('id', $id);), паттерн полностью теряет свой смысл. Бизнес-логика снаружи по-прежнему остается жестко привязанной к ORM.

## Service Layer

Service Layer (Слой сервисов) — это архитектурный паттерн, который выделяет логику бизнес-процессов приложения в отдельный, независимый слой. Он располагается строго между точками входа (Контроллеры, Консольные команды, API-эндпоинты) и слоем доступа к данным (Модели, Репозитории).
Главная задача Сервиса — инкапсулировать в себе «как именно работает бизнес» (бизнес-правила, транзакции, интеграции), полностью изолируя эту логику от особенностей HTTP-протокола или консольного окружения.

------------------------------
##### 1. Зачем он нужен (Решаемые проблемы)

Без слоя сервисов бизнес-логику обычно размазывают либо по контроллерам (Fat Controllers), либо по моделям (Fat Models). Это приводит к ряду проблем:

* Нарушение DRY и дублирование кода: Если логика регистрации пользователя (создание записи, начисление бонусов, отправка email) написана в AuthController, вы не сможете легко переиспользовать её, когда понадобится регистрировать пользователей через консольную команду (Seed-файлы, импорт из CSV) или через REST API для мобильного приложения.
* Сложные, нечитаемые контроллеры: Контроллер начинает отвечать за валидацию, логику СУБД, отправку уведомлений, работу со сторонними API и формирование ответа. Протестировать или изменить такой код становится крайне тяжело.
* Сложность тестирования (Unit Testing): Чтобы протестировать бизнес-логику в "толстом" контроллере, вам приходится симулировать полноценный HTTP-запрос (Feature-тесты), что сильно замедляет прогон тестов.

------------------------------
##### 2. Как работает Слой сервисов

Сервисы проектируются как Stateless (без сохранения состояния) компоненты. Это значит, что они не хранят внутри себя данные конкретного запроса между вызовами методов, а получают все необходимые параметры через аргументы функций.

##### Взаимодействие в архитектуре:

   1. Контроллер принимает HTTP-запрос, валидирует входные данные (например, через Form Request в Laravel) и передает их в метод Сервиса.
   2. Сервис координирует выполнение задачи:
   * Вызывает нужные Репозитории или Модели для работы с БД.
      * Взаимодействует с другими сервисами (например, PaymentService).
      * Вызывает сторонние API (отправка СМС, логистика).
      * Инициирует системные события (Events).
   3. Контроллер получает результат от Сервиса и решает, в каком виде отдать его клиенту (отобразить HTML-вид или вернуть JSON-ответ).

------------------------------
##### 3. Когда использовать

##### Использовать:

* В любых приложениях, где логика выполнения действия состоит более чем из одного шага (не просто CRUD «сохрани строку в БД», а «сохрани, отправь письмо, обнови баланс, логируй действие»).
* Когда одно и то же действие должно запускаться из разных мест приложения (Web, API, CLI, Очереди).
* В проектах, разрабатываемых по методологии TDD/Unit-тестирования, так как чистые PHP-классы сервисов легко покрывать быстрыми изолированными тестами.

##### Не использовать (или использовать с осторожностью):

* В простых CRUD-проектах (визитки, простые панели администрирования). Если метод вашего сервиса просто вызывает $repository->create($data), вы создаете бессмысленную обертку, нарушая принципы KISS и YAGNI. Контроллер может вызвать модель/репозиторий напрямую.

------------------------------
##### 4. Практический пример из жизни (Laravel)

Реализуем бизнес-процесс: Оформление заказа в интернет-магазине. Нам нужно списать деньги, создать заказ, уменьшить остаток товара на складе и отправить уведомление.

##### Шаг 1: Создание сервисного класса (OrderProcessingService.php)

Вся бизнес-логика и координация процессов инкапсулирована здесь.

```php

namespace App\Services;

use App\Models\User;
use App\Repositories\Contracts\OrderRepositoryInterface;
use App\Repositories\Contracts\ProductRepositoryInterface;
use App\Services\Payment\PaymentGatewayInterface;
use App\Events\OrderPlaced;
use Illuminate\Support\Facades\DB;
use Exception;

class OrderProcessingService
{
    // Внедряем необходимые зависимости (Репозитории, другие Сервисы)
    public function __construct(
        protected OrderRepositoryInterface $orderRepository,
        protected ProductRepositoryInterface $productRepository,
        protected PaymentGatewayInterface $paymentGateway
    ) {}

    public function placeOrder(User $user, array $cartItems): mixed
    {
        // Используем БД-транзакцию для атомарности процесса
        return DB::transaction(function () use ($user, $cartItems) {
            
            // 1. Считаем общую стоимость заказа
            $totalPrice = 0;
            foreach ($cartItems as $item) {
                $product = $this->productRepository->findById($item['product_id']);
                
                if ($product->stock < $item['quantity']) {
                    throw new Exception("Товара {$product->name} недостаточно на складе.");
                }
                
                $totalPrice += $product->price * $item['quantity'];
            }

            // 2. Списываем деньги через платежный шлюз
            $this->paymentGateway->charge($user, $totalPrice);

            // 3. Создаем запись заказа через репозиторий
            $order = $this->orderRepository->create([
                'user_id' => $user->id,
                'total' => $totalPrice,
                'status' => 'paid'
            ]);

            // 4. Обновляем остатки на складе
            foreach ($cartItems as $item) {
                $this->productRepository->decrementStock($item['product_id'], $item['quantity']);
                $this->orderRepository->attachItem($order->id, $item);
            }

            // 5. Вызываем событие (Indirection). Конверты/СМС уходят в фон
            event(new OrderPlaced($order));

            return $order;
        });
    }
}

```

##### Шаг 2: Использование в контроллере (OrderController.php)

Контроллер остается максимально тонким. Его задача — только маршрутизация, валидация и формирование ответа.

```php

namespace App\Http\Controllers;

use App\Http\Requests\PlaceOrderRequest;
use App\Services\OrderProcessingService;
use Illuminate\Http\JsonResponse;

class OrderController extends Controller
{
    // Внедряем сервис через конструктор
    public function __construct(
        protected OrderProcessingService $orderService
    ) {}

    public function store(PlaceOrderRequest $request): JsonResponse
    {
        try {
            // Передаем валидированные данные в сервис
            $order = $this->orderService->placeOrder(
                auth()->user(),
                $request->validated('items')
            );

            return response()->json([
                'success' => true,
                'order_id' => $order->id
            ], 201);

        } catch (\Exception $e) {
            // Контроллер решает, как вернуть ошибку клиенту
            return response()->json([
                'success' => false,
                'message' => $e->getMessage()
            ], 422);
        }
    }
}

```

Мы полностью укомплектовали архитектурный блок вашей базы знаний для собеседований! Теперь вы можете аргументированно объяснить взаимосвязь цепочки: Controller (маршрутизация/HTTP) -> Service (бизнес-процессы/транзакции) -> Repository (выборка данных) -> Model/ORM (сущность БД).
Чтобы закрепить эту структуру перед реальным созвоном, я могу предложить:

* Сформировать итоговый Mind Map (текстовую карту связей) всех изученных тем от Баз Данных до Архитектуры, чтобы вы могли окинуть всю картину взглядом за минуту до интервью.

## SOLID

SOLID — это аббревиатура пяти основных принципов объектно-ориентированного проектирования и программирования, сформулированных Робертом Мартином (Дядей Бобом).
Главная цель SOLID — сделать программную систему гибкой, понятной, тестируемой и устойчивой к изменениям. Применение этих принципов предотвращает «гниение кода» (Code Rot), помогая избежать жесткой зависимости компонентов друг от друга.

------------------------------
##### 1. S — Single Responsibility Principle (Принцип единственной ответственности)

* Суть: У класса должна быть только одна причина для изменения. Это означает, что класс должен отвечать за выполнение строго одной бизнес-функции или решать одну узкую техническую задачу.
* Зачем: Повышает зацепление (High Cohesion). Изменение логики в одной сфере не ломает код в другой.
* Антипаттерн: «Божественный класс» (God Object), который одновременно валидирует входные данные, делает SQL-запрос к БД, рассчитывает налоги и генерирует HTML-ответ.
* Пример из практики (Laravel): Вынос логики валидации из контроллера в кастомный класс Form Request (OrderStoreRequest). Контроллер теперь отвечает только за координацию запроса, а Request — только за правила валидации.

------------------------------
##### 2. O — Open/Closed Principle (Принцип открытости/закрытости)

* Суть: Программные сущности (классы, модули, функции) должны быть открыты для расширения, но закрыты для модификации. Вы должны иметь возможность добавлять новое поведение в систему, не переписывая уже существующий и протестированный код.
* Зачем: Предотвращает появление регрессионных багов в работающем коде при добавлении новых фич.
* Антипаттерн: Функция с бесконечной конструкцией switch($type) или цепочкой if/else, в которую приходится дописывать новые блоки каждый раз, когда у бизнеса появляется новый тариф, способ оплаты или тип доставки.
* Пример из практики: Использование интерфейсов. Вместо жесткого ветвления внутри сервиса создается интерфейс DeliveryInterface и классы DHLDelivery, FedExDelivery. При добавлении UPSDelivery существующий код сервиса вообще не меняется.

------------------------------
##### 3. L — Liskov Substitution Principle (Принцип подстановки Барбары Лисков)

* Суть: Объекты в программе должны быть заменяемыми на экземпляры их подтипов (наследников) без изменения правильности работы программы. Класс-наследник должен расширять, а не ломать или сужать поведение базового класса.
* Зачем: Гарантирует предсказуемость кода. Вы можете использовать полиморфизм и быть уверенными, что любой наследник ведет себя в рамках контракта.
* Правила принципа:
1. Наследник не должен вырезать или отключать методы родителя (например, бросать Exception("Метод не поддерживается")).
   2. Наследник не должен усиливать предусловия (требовать на вход более строгие типы данных).
   3. Наследник не должен ослаблять постусловия (возвращать тип данных, который родительский метод возвращать не обещал).
* Антипаттерн: Базовый класс Bird имеет метод fly(). Мы создаем наследника Ostrich (Страус), который летать не умеет, и внутри метода fly() пишем throw new Exception(). Код, работающий с коллекцией птиц, упадет на страусе.

------------------------------
##### 4. I — Interface Segregation Principle (Принцип разделения интерфейса)

* Суть: Клиенты не должны зависеть от методов, которые они не используют. Много специализированных интерфейсов лучше, чем один универсальный.
* Зачем: Избавляет классы от необходимости реализовывать «заглушки» для методов, которые им не нужны по смыслу, и снижает связность кода.
* Антипаттерн: Гигантский интерфейс WorkerInterface, содержащий методы code(), test(), manage(). Если мы создадим класс JuniorDeveloper, ему придется реализовывать пустой метод manage(), так как управлять командой он не должен.
* Пример из практики: Вместо одного толстого интерфейса мы разделяем его на три маленьких: CodableInterface, TestableInterface, ManageableInterface. Класс TeamLead имплементирует все три, а Developer — только первые два.

------------------------------
##### 5. D — Dependency Inversion Principle (Принцип инверсии зависимостей)

* Суть: Модули верхних уровней (бизнес-логика) не должны зависеть от модулей нижних уровней (конкретных технических реализаций: СУБД, библиотек, кэша). Оба уровня должны зависеть от абстракций. Кроме того, абстракции не должны зависеть от деталей, а детали — от абстракций.
* Зачем: Обеспечивает максимальную слабую связанность (Low Coupling). Позволяет безболезненно менять технологический стек (например, сменить драйвер очереди с Redis на RabbitMQ).
* Антипаттерн: Контроллер или Сервис внутри себя жестко создает экземпляр класса через new StripeClient(). Бизнес-логика становится заложником библиотеки Stripe.
* Пример из практики: Сервис зависит от интерфейса PaymentGatewayInterface. Конкретный класс StripePaymentGateway имплементирует этот интерфейс. Laravel с помощью Service Container автоматически подставляет реализацию в конструктор сервиса на основе конфигов.

------------------------------

```php
class OrderProcessor {
    // Нарушение DIP: жесткая привязка к конкретному банку (MySQL и Stripe)
    // Нарушение SRP: класс считает цену, списывает деньги и пишет в базу
    public function process(Order $order) {
        $total = 0;
        foreach ($order->items as $item) {
            $total += $item->price;
        }

        $stripe = new StripePayment();
        $stripe->charge($total); // Если Stripe упадет или мы сменим банк — код переписывать целиком

    }
}

```

##### После (Соблюдение SOLID):

```php

// 1. Создаем абстракцию для платежей (Соблюдаем DIP и OCP)
interface PaymentGatewayInterface {
    public function charge(float $amount): void;
}

// 2. Реализуем конкретные шлюзы. Они взаимозаменяемы (Соблюдаем LSP)
class StripeGateway implements PaymentGatewayInterface {
    public function charge(float $amount): void { /* Логика Stripe */ }
}

class PayPalGateway implements PaymentGatewayInterface {
    public function charge(float $amount): void { /* Логика PayPal */ }
}

// 3. Выделяем логику подсчета в модель-эксперт (Соблюдаем SRP)
class Order extends Model {
    public function calculateTotal(): float {
        return $this->items->sum('price');
    }
}

// 4. Процессор теперь чист, гибок и зависит только от интерфейса
class OrderProcessor {
    // Внедряем зависимость через интерфейс (DIP)
    public function __construct(
        protected PaymentGatewayInterface $paymentGateway,
        protected OrderRepositoryInterface $orderRepository
    ) {}

    public function process(Order $order): void {
        // Каждый класс делает свою работу (SRP)
        $total = $order->calculateTotal();
        
        // Мы можем передать сюда Stripe или PayPal — код процессора не изменится (OCP)
        $this->paymentGateway->charge($total); 
        
        $this->orderRepository->updateStatus($order->id, 'paid');
    }
}

```

## DDD (Domain-Driven Design) — Проектирование на основе предметной области

DDD (Domain-Driven Design) — это подход к проектированию сложного программного обеспечения, который ставит в центр внимания предметную область (Domain) бизнеса и её логику, а не технологии (БД, фреймворки, API).
Главная цель DDD — создать общую и понятную ментальную модель приложения, в которой разработчики и эксперты со стороны бизнеса (аналитики, продукт-менеджеры) говорят на одном едином языке (Ubiquitous Language).

------------------------------
##### 1. Ключевые концепции и строительные блоки (Strategic & Tactical)

DDD делится на два уровня: стратегический (разделение большой системы на автономные части — Bounded Contexts) и тактический (конкретные шаблоны кода внутри одного контекста).

##### Тактические блоки DDD:

* Entity (Сущность): Объект, который имеет уникальный идентификатор (ID), сохраняющийся на протяжении всей его жизни. Два объекта Entity с одинаковым ID — это один и тот же объект, даже если остальные их свойства различаются (пример: User, Order).
* Value Object (Объект-значение): Объект, не имеющий ID. Он полностью определяется своими свойствами. Если свойства двух объектов-значений совпадают, они считаются идентичными. Объекты-значения всегда неизменяемы (Immutable) (пример: Money {amount: 100, currency: 'USD'}, Address).
* Aggregate (Агрегат): Кластер связанных объектов (Entities + Value Objects), которые воспринимаются системой как единое целое с точки зрения изменения данных.
* У каждого агрегата есть Aggregate Root (Корень агрегата) — единственная сущность, через которую внешний код имеет право взаимодействовать с внутренностями агрегата. Внешний мир не может напрямую менять объекты внутри агрегата.
* Repository (Репозиторий): Предоставляет интерфейс для сохранения и извлечения корней агрегатов из постоянного хранилища.
* Domain Events (Доменные события): Факт того, что в бизнес-логике произошло что-то важное (в прошедшем времени: OrderPaid, UserRegistered). Они используются для уведомления других агрегатов или контекстов о подписке на изменения.

------------------------------
##### 2. Трехуровневая архитектура DDD (Layered Architecture)

DDD изолирует бизнес-логику от технических деталей с помощью жесткого разделения на слои:

   1. Domain Layer (Доменный слой): Ядро системы. Содержит Entities, Value Objects, Агрегаты, Доменные события и интерфейсы Репозиториев. Этот слой абсолютно чист — он не зависит от Laravel, баз данных или сторонних библиотек.
   2. Application Layer (Прикладной слой / Use Cases): Управляет потоком приложения. Он принимает команды из внешнего мира, достает Агрегат из Репозитория, вызывает метод Агрегата для выполнения бизнес-логики и сохраняет изменения обратно. Здесь нет SQL или HTML — только сценарии использования (например, RegisterUserHandler).
   3. Infrastructure Layer (Инфраструктурный слой): Техническая реализация. Сюда уходят конкретные SQL-запросы, Eloquent-модели, отправка почты, интеграции с CRM и вызовы внешних API. Этот слой имплементирует интерфейсы, объявленные в Доменном слое (реализуя принцип DIP из SOLID).

------------------------------
##### 3. Когда использовать DDD## Использовать:

* В крупных enterprise-проектах со сложной, запутанной и постоянно меняющейся бизнес-логикой.
* Когда над проектом работают несколько команд и важно разделить систему на изолированные контексты (Bounded Contexts), которые затем могут стать микросервисами.
* Когда бизнес-эксперты активно участвуют в формировании требований, и технический лексикон программистов начинает вредить пониманию задач.

##### НЕ использовать (Антипаттерн):

* В простых CRUD-приложениях (сайты-визитки, простые каталоги, админки). Написание десятков классов (Value Objects, Агрегаты, Команды) там, где достаточно стандартной модели Eloquent, — это грубейшее нарушение принципов KISS и YAGNI. DDD в таких условиях создает огромный оверхед (Overengineering).

------------------------------
##### 4. Практический пример на Laravel (с элементами DDD)

Реализуем доменное правило: Пользователь может сменить тарифный план, но только если у него достаточно средств на балансе. При смене тарифа генерируется событие.

##### Шаг 1: Доменный слой (Чистый PHP)

```php

// app/Domain/User/ValueObjects/Money.php
// Value Object: неизменяем, инкапсулирует валидацию денег
readonly class Money {
    public function __construct(public float $amount, public string $currency) {
        if ($amount < 0) throw new InvalidArgumentException("Баланс не может быть отрицательным");
    }
    public function canAfford(Money $cost): bool {
        return $this->amount >= $cost->amount && $this->currency === $cost->currency;
    }
}

// app/Domain/User/Entities/User.php
// Aggregate Root: управляет своим состоянием сам
class User {
    private int $id;
    private Money $balance;
    private string $tier;
    private array $recordedEvents = [];

    public function changeTier(string $newTier, Money $cost): void {
        if (!$this->balance->canAfford($cost)) {
            throw new DomainException("Недостаточно средств для перехода на тариф {$newTier}");
        }
        $this->tier = $newTier;
        // Записываем доменное событие
        $this->recordedEvents[] = new UserTierChanged($this->id, $newTier);
    }
    
    public function pullEvents(): array {
        $events = $this->recordedEvents;
        $this->recordedEvents = [];
        return $events;
    }
}

```

##### Шаг 2: Прикладной слой (Application Use Case)

```php

// app/Application/User/Commands/ChangeTierHandler.php
class ChangeTierHandler {
    public function __construct(
        protected UserRepositoryInterface $userRepository,
        protected EventDispatcherInterface $dispatcher
    ) {}

    public function handle(ChangeTierCommand $command): void {
        // Извлекаем агрегат через интерфейс репозитория
        $user = $this->userRepository->findById($command->userId);
        
        $cost = new Money($command->costAmount, 'USD');
        
        // Вызываем бизнес-логику внутри агрегата
        $user->changeTier($command->newTier, $cost);
        
        // Сохраняем измененный агрегат обратно
        $this->userRepository->save($user);
        
        // Публикуем доменные события во внешний мир
        foreach ($user->pullEvents() as $event) {
            $this->dispatcher->dispatch($event);
        }
    }
}

```

##### Шаг 3: Инфраструктурный слой (Laravel-специфика)

```php

    public function findById(int $id): User {

            new Money($eloquentUser->balance_amount, $eloquentUser->currency),

    }

    public function save(User $user): void {
        $eloquentUser = UserModel::find($user->getId());
        $eloquentUser->tier = $user->getTier();
        $eloquentUser->save();
    }
}
```

## CQRS (Command Query Responsibility Segregation)

CQRS (Command Query Responsibility Segregation / Разделение ответственности на команды и запросы) — это архитектурный паттерн, который предписывает использовать принципиально разные модели для изменения данных (Запись / Команды) и для извлечения данных (Чтение / Запросы).
Паттерн вырос из принципа проектирования CQS (Command-Query Separation) Бертрана Мейера, который гласит: «Метод должен либо изменять состояние объекта, либо возвращать данные, но не делать и то, и другое одновременно». CQRS переносит этот принцип с уровня отдельных методов на уровень всей архитектуры приложения.

------------------------------
##### 1. Ключевые концепции и модели CQRS

В традиционных CRUD-архитектурах одна и та же модель данных (например, модель User в Eloquent) используется как для сложных бизнес-мутаций, так и для вывода списков на UI. При росте нагрузки это создает конфликт интересов. CQRS разделяет приложение на два независимых потока:

##### 1. Поток записи (Commands)

* Задача: Выполнение бизнес-логики, проверка доменных правил и изменение состояния системы.
* Модель записи (Write Model): Строится вокруг доменных сущностей (Entities/Aggregates в DDD). Оптимизирована для обеспечения консистентности и валидности данных.
* Правило: Команда инициирует действие (CreateOrder, CancelSubscription), выполняет его и не возвращает бизнес-данные (допускается возврат ID созданной сущности или статус выполнения).

##### 2. Поток чтения (Queries)

* Задача: Максимально быстрый и легкий вывод данных пользователю.
* Модель чтения (Read Model / View Model): Чистые структуры данных (DTO), лишенные какой-либо бизнес-логики. Оптимизированы под конкретные экраны интерфейса или API-ответы.
* Правило: Запрос (GetOrderById, GetActiveUsers) не имеет права изменять состояние системы (никаких сайд-эффектов), он только считывает и возвращает данные.

------------------------------
##### 2. Уровни реализации CQRS (Как это работает)

CQRS можно внедрять постепенно, в зависимости от масштаба проблем с производительностью:

   1. Логический CQRS (в рамках одной БД):
   Код разделяется на классы команд и запросов. Они используют одну и ту же реляционную базу данных, но запросы (Queries) могут обходить тяжелые ORM-модели, выполняя плоские и быстрые SQL-выборки напрямую в DTO через Query Builder.
   2. Физический CQRS (Раздельные базы данных):
   * Write DB: Реляционная СУБД (PostgreSQL, MySQL), нормализованная по 3NF для исключения дублирования и обеспечения безопасных транзакций.
      * Read DB: Денормализованная база данных (NoSQL вроде MongoDB, Redis или ElasticSearch). Данные в ней хранятся именно в том виде, в каком их требует UI (готовые JSON-документы), что исключает необходимость в тяжелых операциях JOIN и GROUP BY.
      * Синхронизация: При изменении данных в Write DB, система генерирует событие, которое асинхронно обновляет Read DB. Здесь возникает концепция Eventual Consistency (согласованность в конечном счете) — данные на чтение могут обновиться с задержкой в несколько миллисекунд.
   
------------------------------
##### 3. Когда использовать CQRS## Использовать:

* В высоконагруженных системах (Highload), где количество операций чтения в разы или порядки превосходит количество операций записи.
* Когда требования интерфейса к отображению данных кардинально отличаются от структуры данных, необходимой для выполнения бизнес-правил.
* В проектах, где разработка ведется параллельно разными командами: одна команда может оптимизировать сложную доменную логику записи, а другая — независимо верстать быстрые API-отчеты на чтение.

##### НЕ использовать (Антипаттерн):

* В классических CRUD-системах со слабой или средней нагрузкой. Если ваше приложение просто сохраняет форму из 5 полей в базу и выводит её обратно на экран, CQRS превратится в Overengineering (избыточное усложнение). Вы получите удвоение количества классов и кода (Command, Query, Handlers, DTO), не приобретя никакой выгоды.

------------------------------
##### 4. Практический пример на Laravel (Логический CQRS)

Реализуем задачу: Просмотр списка заказов (Query) и отмена заказа (Command).

##### 📊 Часть 1: Поток чтения (Query)

Запрос полностью игнорирует тяжелую модель Eloцент со всеми её связями и логикой, работая напрямую с быстрыми структурами данных.

```php

// app/Queries/GetCustomerOrdersQuery.php
// Простой DTO для параметров запроса
readonly class GetCustomerOrdersQuery {
    public function __construct(public int $customerId) {}
}

// app/Queries/Handlers/GetCustomerOrdersHandler.php
// Обработчик запроса: делает плоскую выборку без ORM оверхеда
class GetCustomerOrdersHandler {
    public function handle(GetCustomerOrdersQuery $query): array {
```

## Паттерн «Event Sourcing» (Источник событий)

Event Sourcing (Источник событий) — это архитектурный паттерн, при котором текущее состояние сущности (агрегата) не сохраняется в виде одной строки в базе данных (как при классическом CRUD). Вместо этого система фиксирует и сохраняет полную последовательность атомарных событий, произошедших с этой сущностью.
Текущее состояние приложения не считывается напрямую со страницы БД, а вычисляется (восстанавливается) путем последовательного воспроизведения («накатывания») всех исторических событий от самого первого до последнего.

------------------------------
##### 1. Как это работает (Ключевые механизмы)

В традиционных системах при обновлении баланса пользователя с 100$ до 150$ старое значение 100 стирается, и поверх него записывается 150. История изменений теряется (если нет кастомных логов).
В Event Sourcing этот процесс выглядит принципиально иначе:

```
[ Событие 1: КошелекСоздан ] ---> [ Событие 2: БалансПополнен (+100$) ] ---> [ Событие 3: БалансСписан (-20$) ]
                                                                                   |
                                                                                   v
[ Текущее состояние: 80$ ] <--- ( Воспроизведение всех событий по порядку ) <------+
```

##### Основные компоненты:

* Event Store (Хранилище событий): Специализированная база данных (например, EventStoreDB, Apache Kafka или оптимизированная реляционная таблица), которая работает в режиме Append-Only (данные можно только дописывать, операции UPDATE и DELETE запрещены). События неизменяемы.
* Rehydration (Восстановление состояния): Процесс, при котором пустой объект бизнес-логики (Aggregate) последовательно принимает исторические события из Event Store, меняя свои внутренние поля.
* Snapshot (Снапшот / Снимок состояния): Оптимизация для долгоживущих сущностей. Если у пользователя накопилось 100 000 событий, восстанавливать его каждый раз с нуля будет слишком медленно. Система периодически (например, каждые 500 событий) сохраняет текущий слепок состояния в кэш. При запросе СУБД берет последний снапшот и докручивает только те события, которые произошли после него.

------------------------------
##### 2. Когда использовать Event Sourcing

##### Использовать:

* В финансовых и банковских системах (биллинг, кошельки, транзакции), где критически важен детальный аудит, и контролирующие органы могут спросить: «Как именно баланс этого пользователя стал равен X на дату Y?».
* В логистических системах (трекинг посылок, перемещение грузов).
* В системах со сложной совместной работой над данными, где важна история отмены действий (например, Google Docs, CRM-системы, история изменения статусов задач).
* Когда бизнес-аналитикам требуется строить ретроспективные отчеты — запускать новые гипотезы на основе реального поведения пользователей в прошлом.

##### НЕ использовать (Антипаттерн):

* В подавляющем большинстве стандартных приложений (интернет-магазины без жестких требований к аудиту, блоги, новостные порталы, простые SaaS-сервисы).
* Event Sourcing значительно усложняет код: становится тяжело менять структуру старых событий (проблема миграции событий — Schema Evolution), усложняется отладка, а разработчикам приходится полностью перестраивать мышление с CRUD на событийную модель. Внедрение паттерна ради "моды" — это классический Overengineering.

------------------------------
##### 3. Синергия с CQRS

Event Sourcing практически никогда не используется самостоятельно. Выборка списков данных с помощью JOIN по Append-only таблице событий физически невозможна.
Поэтому Event Sourcing всегда работает в связке с CQRS:

   1. Поток записи (Command): Изменяет данные, генерирует события и пишет их в Event Store.
   2. Синхронизация: Специальный обработчик (Projector) перехватывает эти события и асинхронно обновляет плоскую реляционную или NoSQL базу данных для чтения (Read Model).
   3. Поток чтения (Query): Быстро отдает данные пользователю из денормализованной Read Model.

------------------------------
##### 4. Практический пример на чистом PHP (с логикой Laravel)

Реализуем доменный агрегат банковского счета (BankAccount), который умеет открываться, пополняться и восстанавливать себя из истории.

##### Шаг 1: Определение неизменяемых событий

```php

// События пишутся в прошедшем времени
readonly class AccountOpened {
    public function __construct(public string $accountId, public string $owner) {}
}

readonly class MoneyDeposited {
    public function __construct(public string $accountId, public float $amount) {}
}

```

##### Шаг 2: Доменный Агрегат

Класс умеет применять новые события, а также восстанавливаться из старых через метод apply.

```php

class BankAccount 
{
    private string $id;
    private float $balance = 0;
    private array $uncommittedEvents = []; // Новые события, еще не записанные в БД

    // Метод создания (Команда)
    public static function open(string $id, string $owner): self {
        $account = new self();
        $account->recordThat(new AccountOpened($id, $owner));
        return $account;
    }

    // Метод пополнения (Команда)
    public function deposit(float $amount): void {
        if ($amount <= 0) throw new InvalidArgumentException("Сумма должна быть больше нуля");
        $account->recordThat(new MoneyDeposited($this->id, $amount));
    }

    // Восстановление агрегата из истории событий (Rehydration)
    public static function reconstituteFromHistory(array $events): self {
        $account = new self();
        foreach ($events as $event) {
            $account->apply($event);
        }
        return $account;
    }

    // Внутренний маппинг событий на изменение полей
    private function apply(object $event): void {
        if ($event instanceof AccountOpened) {
            $this->id = $event->accountId;
            $this->balance = 0;
        }
        if ($event instanceof MoneyDeposited) {
            $this->balance += $event->amount;
        }
    }

    private function recordThat(object $event): void {
        $this->apply($event);
        $this->uncommittedEvents[] = $event;
    }

    public function getUncommittedEvents(): array { return $this->uncommittedEvents; }
}

```

##### Шаг 3: Использование в Прикладном слое (Service/Handler)

```php

class DepositMoneyHandler 
{
    public function __construct(protected EventStoreRepository $eventStore) {}

    public function handle(string $accountId, float $amount): void 
    {
        // 1. Извлекаем ВСЕ прошлые события этого аккаунта из базы Event Store
        $history = $this->eventStore->getEventsFor($accountId);
        
        // 2. Восстанавливаем текущее состояние объекта (Rehydration)
        $account = BankAccount::reconstituteFromHistory($history);
        
        // 3. Вызываем доменную логику
        $account->deposit($amount);
        
        // 4. Сохраняем ТОЛЬКО новые события, сгенерированные в ходе шага 3
        $this->eventStore->append(
            $accountId, 
            $account->getUncommittedEvents()
        );
    }
}

```

## Dependency Injection — DI

Dependency Injection (Внедрение зависимостей) — это паттерн проектирования, который реализует принцип инверсии управления (Inversion of Control — IoC). Его суть заключается в том, что класс не должен самостоятельно создавать (через оператор new) другие объекты, от которых зависит его работа (зависимости). Вместо этого все необходимые внешние объекты должны передаваться (внедряться) в класс извне.
Внедрение зависимостей является главным техническим инструментом для достижения слабой связанности компонентов системы (Low Coupling).

------------------------------
##### 1. Как это работает

В классическом подходе класс жестко контролирует создание своих зависимостей. При использовании DI контроль инвертируется: класс становится пассивным, он лишь декларирует, какие инструменты ему нужны для работы, а сборку и передачу этих инструментов берет на себя внешняя система — IoC-контейнер (в Laravel это Service Container).

```php

[ Классический подход (Hardcoding) ]
Класс А ---> создает внутри себя ---> Класс Б (Жесткая связь)

[ Подход с Dependency Injection ]
[ IoC-Контейнер ] ---> создает Класс Б ---> внедряет его в ---> Контроллер/Класс А

```

------------------------------
##### 2. Типы инъекций (Dependency Injection Types)

Существует три основных способа передать зависимость в класс:

##### 1. Constructor Injection (Внедрение через конструктор)

Самый распространенный и рекомендуемый способ. Зависимости передаются в качестве параметров конструктора при создании экземпляра класса.

* Когда использовать: Для обязательных зависимостей, без которых класс физически не может выполнять свою работу.
* Плюс: Объект гарантированно создается в валидном состоянии.

##### 2. Setter Injection (Внедрение через сеттер / свойства)

Зависимость передается через специальный публичный метод (мутатор) после того, как объект уже был создан.

* Когда использовать: Для опциональных зависимостей, которые имеют дефолтную реализацию или могут быть изменены / донастроены в процессе работы скрипта.

##### 3. Method Injection (Внедрение в метод)

Зависимость передается непосредственно в аргументы конкретного вызываемого метода.

* Когда использовать: Когда зависимость нужна только для выполнения одной специфической операции и ее бессмысленно хранить в состоянии класса на протяжении всей его жизни.

------------------------------
##### 3. Когда использовать

##### Использовать:

* В любых классах бизнес-логики (Контроллеры, Сервисы, Репозитории, Джобы).
* Когда код должен быть покрыт изолированными Unit-тестами. Благодаря DI вы можете легко подменить реальный класс отправки СМС на фейковую заглушку (Mock), чтобы тесты не слали реальные сообщения.
* Когда проект зависит от сторонних API, провайдеров или технологий, которые могут измениться в будущем.

##### НЕ использовать (Антипаттерн):

* Для простых объектов-значений (Value Objects) и DTO (например, класс Money или Address). Они создаются через new напрямую, так как не содержат логики внешних инфраструктурных сервисов.
* В легаси-проектах без IoC-контейнера попытка внедрять всё вручную приведет к созданию «ада сборки», где для вызова одного контроллера вам придется вручную написать: new Controller(new Service(new Repository(new Database(...)))).

------------------------------
##### 4. Практический пример в Laravel

## Паттерн «Фабрика» (Factory Pattern)

Factory Pattern (Паттерн Фабрика) — это группа порождающих паттернов проектирования, которые инкапсулируют (скрывают) логику создания экземпляров классов. Вместо прямого вызова оператора new клиентский код делегирует создание объектов специальной сущности — фабрике.
Это позволяет изолировать процесс инициализации сложных объектов, управлять их конфигурацией в одном месте и соблюдать принцип открытости/закрытости (OCP из SOLID).

------------------------------
##### 1. Типы фабрик и как они работают

В зависимости от сложности архитектурной задачи паттерн разделяют на три основных вида:

##### 1. Simple Factory (Простая фабрика)

Не является официальным паттерном GoF (Gang of Four), а считается популярным идиоматическим подходом. Представляет собой один класс с методом (часто статическим), который на основе входных параметров (например, строки или enum) решает, экземпляр какого класса создать.

* Как работает: Клиент передает параметр type, фабрика внутри switch/match конструкции создает и возвращает нужный объект.

##### 2. Factory Method (Фабричный метод)

Классический паттерн GoF. Определяет интерфейс для создания объекта, но оставляет подклассам (наследникам) право решать, какой именно класс инициализировать.

* Как работает: Базовый абстрактный класс объявляет абстрактный метод createProduct(). Подклассы переопределяют этот метод, возвращая специфические реализации продукта. Логика создания делегируется через наследование.

##### 3. Abstract Factory (Абстрактная фабрика)

Предоставляет интерфейс для создания семейств взаимосвязанных или зависимых объектов (например, семейство UI-элементов в стиле Windows: кнопка + чекбокс, и семейство в стиле Mac: кнопка + чекбокс), не привязываясь к конкретным классам.

* Как работает: Комбинирует несколько фабричных методов в один интерфейс для комплексной сборки связанных сущностей.

------------------------------
##### 2. Когда использовать## Использовать:

* Когда процесс создания объекта содержит сложную логику, требует передачи множества конфигурационных параметров или обращения к внешним сервисам/БД.
* Когда тип создаваемого объекта должен динамически определяться в процессе работы приложения на основе входных данных.
* Для снижения связности (Low Coupling): чтобы бизнес-логика зависеть от интерфейсов продуктов, а не от их конкретных реализаций.

##### НЕ использовать (Антипаттерн):

* Для создания простых объектов без внутренней логики. Написание UserFactory::create() вместо стандартного new User() для обычного DTO-класса — это бессмысленный оверхед (нарушение принципов KISS и YAGNI).

------------------------------
##### 3. Практический пример на Laravel (Simple Factory)

Реализуем бизнес-задачу: Отправка уведомлений пользователю через разные каналы (SMS, Email, Push). Выбор канала зависит от настроек в профиле пользователя.

##### Шаг 1: Определение интерфейса и продуктов

```php

namespace App\Services\Notifications;

interface NotificationSenderInterface {
    public function send(string $recipient, string $message): void;
}

class EmailSender implements NotificationSenderInterface {
    public function send(string $recipient, string $message): void { /* Код отправки через SMTP/Mailgun */ }
}

class SmsSender implements NotificationSenderInterface {
    public function send(string $recipient, string $message): void { /* Код отправки через Twilio */ }
}

class PushSender implements NotificationSenderInterface {
    public function send(string $recipient, string $message): void { /* Код отправки через Firebase (FCM) */ }
}

```

##### Шаг 2: Создание Фабрики (NotificationFactory.php)

Фабрика берет на себя всю грязную логику выбора класса и его начальной конфигурации.

```php

namespace App\Services\Notifications;

use InvalidArgumentException;

class NotificationFactory
{
    // Метод динамически создает нужный класс (Simple Factory)
    public static function make(string $channel): NotificationSenderInterface
    {
        return match ($channel) {
            'email' => new EmailSender(config('mail.from.address')), // Передаем настройки из конфига
            'sms'   => new SmsSender(config('services.twilio.sid'), config('services.twilio.token')),
            'push'  => new PushSender(config('services.firebase.credentials')),
            default => throw new InvalidArgumentException("Канал уведомлений [{$channel}] не поддерживается."),
        };
    }
}

```

##### Шаг 3: Использование в контроллере/сервисе

Клиентский код полностью изолирован от создания конкретных сендеров. Если завтра добавится Telegram, код контроллера не изменится (Соблюдение OCP).

```php

namespace App\Http\Controllers;

use App\Models\User;
use App\Services\Notifications\NotificationFactory;
use Illuminate\Http\JsonResponse;

class NotificationController extends Controller
{
    public function sendAlert(int $userId): JsonResponse
    {
        $user = User::findOrFail($userId);
        
        // Например, у пользователя в БД в поле preferred_channel записано 'sms'
        $channel = $user->preferred_channel; 
        
        // Делегируем создание объекта фабрике
        $sender = NotificationFactory::make($channel);
        
        // Работаем исключительно через общий интерфейс
        $sender->send($user->contact_destination, "Ваш заказ успешно оформлен!");

        return response()->json(['status' => 'Notification sent']);
    }
}

```

##  Паттерн «Наблюдатель» (Observer Pattern)

Observer Pattern (Паттерн Наблюдатель) — это поведенческий паттерн проектирования, который реализует отношение «один ко многим». Он позволяет одним объектам (наблюдателям/подписчикам) автоматически получать уведомления и реагировать на изменения состояния другого объекта (субъекта/издателя) без жесткой привязки к его классу.
Паттерн является главным инструментом для реализации событийно-ориентированной архитектуры (Event-Driven Architecture) и обеспечения принципа слабой связанности (Low Coupling).


##### 1. Как это работает (Компоненты паттерна)

Паттерн состоит из трех ключевых участников:

   1. Subject (Субъект / Издатель): Объект, обладающий важным состоянием. Он хранит список подписанных на него наблюдателей и предоставляет методы для их добавления (attach) или удаления (detach). При изменении своего состояния он обходит список и вызывает у каждого метод уведомления (notify).
   2. Observer (Наблюдатель / Подписчик): Интерфейс, объявляющий метод обновления (обычно update()), который вызывается издателем.
   3. Event (Событие): Объект-контейнер (DTO), который передается от издателя подписчикам и содержит информацию о произошедшем изменении.

------------------------------
##### 2. Реализация в Laravel: Events и Listeners

В Laravel классический паттерн Observer разделен на связку Events (События) и Listeners (Слушатели) для большей гибкости.

* Event (Событие) — это простой чистый PHP-класс (DTO), который констатирует факт: «что-то произошло» (например, OrderPlaced). Он содержит только данные (объект заказа), но не содержит логики.
* Listener (Слушатель) — это класс, содержащий логику реакции на событие (метод handle()). На одно событие можно подписать десятки независимых слушателей (например, SendOrderEmail, DecreaseStock, NotifyLogistics).

##### Асинхронность (ShouldQueue)

По умолчанию Laravel выполняет слушателей синхронно в рамках того же HTTP-запроса. Если слушатель должен отправить email или вызвать внешнее API, это замедлит ответ пользователю.
Добавление интерфейса ShouldQueue в класс Listener автоматически отправляет выполнение этой задачи в фоновую очередь (Queue Workers). Пользователь мгновенно получает ответ, а тяжелая работа выполняется сервером асинхронно.

------------------------------
##### 3. Контекст Laravel: Model Observers

В Laravel есть специализированная реализация паттерна — Model Observers (app/Observers). Они жестко привязаны к жизненному циклу моделей Eloquent и слушают системные события самой базы данных (creating, created, updating, updated, deleting, deleted).

* Когда использовать: Когда логика реакции касается исключительно этой модели (например: при удалении пользователя автоматически удалить его аватарку с диска S3, или при генерации статьи автоматически заполнить поле slug из title).

------------------------------
##### 4. Когда использовать

##### Использовать:

* Когда изменение состояния одного объекта требует изменения других объектов, и вы заранее не знаете, сколько именно объектов нужно обновить.
* Когда компоненты системы должны быть изолированы друг от друга: издатель не должен знать, кто именно обрабатывает его события.
* Для соблюдения принципа Single Responsibility (SRP): контроллер или сервис просто фиксирует факт действия, а специфические побочные задачи (аналитика, СМС, логи) уходят в отдельных слушателей.

##### НЕ использовать (Антипаттерн):

* Для сквозной последовательной цепочки бизнес-логики, где шаг 2 строго зависит от успешного возврата данных из шага 1. События работают по принципу "выстрелил и забыл" (Fire and Forget).
* Если чрезмерно увлечься событиями в простом проекте, код превратится в "спагетти", где логику приложения невозможно отследить по шагам (теряется явная причинно-следственная связь выполнения).

##### 5. Практический пример из жизни (Laravel Events + Listeners)

Задача: Пользователь зарегистрировался на сайте. Нужно отправить ему приветственное письмо, начислить 100 приветственных бонусов и передать данные в CRM-систему (RetailRocket).

##### Шаг 1: Создание События (UserRegistered.php)

```php

namespace App\Events;

use App\Models\User;
use Illuminate\Foundation\Events\Dispatchable;
use Illuminate\Queue\InteractsWithSockets;

class UserRegistered
{
    use Dispatchable, InteractsWithSockets;

    // Событие — это просто контейнер для данных
    public function __construct(public User $user) {}
}

```

##### Шаг 2: Создание Асинхронного Слушателя (SendWelcomeEmail.php)

```php

namespace App\Listeners;

use App\Events\UserRegistered;
use Illuminate\Contracts\Queue\ShouldQueue; // Критично: отправляем в очередь!
use App\Mail\WelcomeMail;
use Mail;

class SendWelcomeEmail implements ShouldQueue
{
    public function handle(UserRegistered $event): void
    {
        // Доступ к пользователю получаем из объекта события
        Mail::to($event->user->email)->send(new WelcomeMail($event->user));
    }
}
```

##### Шаг 3: Регистрация связки в EventServiceProvider.php

```php

protected $listen = [
    \App\Events\UserRegistered::class => [
        \App\Listeners\SendWelcomeEmail::class,
        \App\Listeners\AwardWelcomeBonus::class, // Другой независимый слушатель
        \App\Listeners\SyncWithCrm::class,       // Третий независимый слушатель
    ],
];

```

##### Шаг 4: Вызов события в Сервисе или Контроллере

Код регистрации остается чистым, коротким и сфокусированным (High Cohesion).

```php

namespace App\Http\Controllers;

use App\Models\User;
use App\Events\UserRegistered;
use Illuminate\Http\Request;

class RegisterController extends Controller

    {
        $user = User::create($request->all());

        // Оповещаем систему. Логика отправки писем и CRM сюда не вмешивается
        event(new UserRegistered($user));

        return response()->json(['message' => 'Success'], 201);
    }
}

```
