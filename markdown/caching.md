
## Стратегии кэширования

Стратегии кэширования — это набор архитектурных паттернов и правил, определяющих, как приложение взаимодействует с быстрым временным хранилищем данных (кэшем) и основным постоянным хранилищем (базой данных).
На собеседованиях эта тема является обязательной для разработчиков любого уровня. Она проверяет умение снижать нагрузку на СУБД, проектировать отказоустойчивые системы и балансировать между скоростью ответа API и актуальностью данных.

------------------------------
##### 1. Типы кэширования (Где можно кэшировать?)

Кэширование может применяться на разных уровнях архитектуры приложения:

   1. Client-Side (Браузерное кэширование): Использование HTTP-заголовков (Cache-Control, ETag, Last-Modified). Данные сохраняются прямо на устройстве пользователя, полностью избавляя сервер от запросов.
   2. CDN (Content Delivery Network): Кэширование статики (картинки, JS, CSS) и тяжелых API-ответов на географически распределенных серверах (Cloudflare, Akamai). Запрос закрывается на подлете к вашему серверу.
   3. Reverse Proxy / Gateway (Nginx, Varnish): Кэширование готовых HTML-страниц или JSON-ответов целиком прямо на веб-сервере.
   4. Application-Side (In-Memory кэш приложения): Использование сверхбыстрых хранилищ типа Redis или Memcached в коде бэкенда для сохранения результатов тяжелых SQL-запросов, сессий или объектов.
   5. Database-Side (Внутренний кэш СУБД): Буферные пулы самой базы данных (shared_buffers в Postgres, innodb_buffer_pool в MySQL), удерживающие частые страницы данных в RAM.

------------------------------

##### 2. Пять ключевых стратегий кэширования

На интервью важно четко различать, кто отвечает за чтение и запись: приложение или сам кэш.

##### 1. Cache-Aside (Lazy Loading / Ленивая загрузка)

Самая популярная стратегия в веб-разработке (стандарт для Laravel). Приложение само управляет процессом.

* Чтение: Приложение проверяет кэш. Если данные есть (Cache Hit) — возвращает их. Если данных нет (Cache Miss) — читает из БД, записывает в кэш и возвращает пользователю.
* Запись: Данные пишутся напрямую в БД. Кэш для этого ключа просто удаляется (инвалидируется).
* Плюсы: Кэшируется только то, что реально запрашивают пользователи. Сбой кэша не ломает приложение (оно просто пойдет в БД).
* Минусы: Первый запрос всегда медленный (Cache Miss).

##### 2. Read-Through

Приложение всегда обращается только к кэшу. Если в кэше данных нет, слой кэширования (библиотека/плагин) сам идет в БД, затягивает данные, сохраняет у себя и отдает приложению.

* Плюсы: Код приложения чист от логики работы с БД.
* Минусы: Сложно реализовать, если данные требуют сложной трансформации перед сохранением.

##### 3. Write-Through

Данные пишутся одновременно и в кэш, и в базу данных. Транзакция записи считается успешной только после сохранения в обоих местах.

* Плюсы: Данные в кэше всегда на 100% актуальны. При следующем чтении гарантирован быстрый Cache Hit.
* Минусы: Высокая задержка (Latency) на операциях записи.

##### 4. Write-Behind (Write-Back)

Приложение пишет данные только в быстрый кэш. Кэш моментально подтверждает запись. Затем специальный фоновый процесс асинхронно (пачками) переносит эти данные из кэша в постоянную БД.

* Плюсы: Экстремально быстрая запись. Идеально для Write-Heavy систем (сбор метрик, лайки, просмотры).
* Минусы: Риск потери данных. Если сервер с кэшем упадет до того, как фоновый процесс перенес данные в БД, они исчезнут навсегда.

##### 5. Refresh-Ahead

Кэш автоматически обновляет данные до того, как истечет их срок жизни (TTL), если прогнозирует, что к ним скоро обратятся.

##### 3. Cache Invalidation (Инвалидация кэша)

"В IT есть две сложные проблемы: инвалидация кэша и придумывание названий для переменных" — Фил Карлтон.

Инвалидация — это процесс удаления или обновления устаревших данных в кэше.

##### Подходы к инвалидации:

   1. TTL (Time-To-Live): Каждому ключу задается жесткое время жизни (например, 600 секунд). По истечении времени кэш удаляет ключ сам. Самый простой и безопасный подход, но данные могут быть «протухшими» в течение этого времени.

   2. Событийная инвалидация (Event-driven): Код приложения явно удаляет кэш при изменении сущности. В Laravel это реализуется через Observers или Events.
   
   ```php
   // Внутри ProductObserver
   public function updated(Product $product) {
       Cache::forget("product:{$product->id}");
   }
   ```
   
   3. Кэш-теги (Cache Tags): Группировка ключей. Позволяет удалить весь кэш, связанный, например, с конкретным пользователем, одной командой: `Cache::tags(['user_15', 'reports'])->flush();`. (Не поддерживается в Memcached, отлично работает в Redis).

------------------------------
##### 4. Проблема Thundering Herd (Гонка за кэшем / Cache Stampede)

Thundering Herd Problem (Проблема «грохочущего стада») — это ситуация, когда у высоконагруженного ключа (например, главная страница крупного СМИ с 10 000 RPS) истекает срок жизни (TTL) [1, 2].

* Что происходит: Ключ исчезает из кэша. В эту же миллисекунду 5000 параллельных запросов от пользователей получают Cache Miss. Все эти 5000 потоков одновременно бегут в базу данных и начинают выполнять один и тот же тяжелый SQL-запрос.
* Результат: База данных моментально ложится от перегрузки процессора (CPU) и диска, а веб-сервер начинает сыпать ошибками 502 Bad Gateway [1, 2].

##### Способы решения:

   1. Мьютексы / Блокировки (Mutex / Locking): Первый поток, поймавший Cache Miss, захватывает эксклюзивную атомарную блокировку в Redis (например, через Cache::lock()). Только этот один поток идет в БД за данными. Остальные 4999 потоков ждут секунду или получают старое значение, пока первый поток не обновит кэш [1].
   2. Раннее обновление (XFetch алгоритм): Вероятностный подход. Когда до истечения TTL остается, например, 10% времени, приложение с определенной вероятностью запускает фоновый пересчет кэша в асинхронной джобе. Кэш обновляется незаметно для пользователей.
   3. Вечный кэш с фоновым обновлением: Задавать TTL = бесконечность. Обновлять данные в кэше строго по расписанию через Cron или по событиям INSERT/UPDATE базовых таблиц.

------------------------------
##### 5. Cache Warming (Прогрев кэша)

Прогрев кэша — это принудительное заполнение кэша популярными данными до того, как к ним обратятся реальные пользователи.

* Когда нужен:
* Сразу после деплоя новой версии приложения (когда кэш был полностью очищен).
   * Перед проведением крупных акций (Черная Пятница в интернет-магазине) — в кэш заранее закидывают топ-1000 акционных товаров.
* Реализация: Пишется консольный скрипт (в Laravel — Artisan Command), который вычитывает из БД самые посещаемые страницы/товары и принудительно делает Cache::put(). Скрипт запускается в процессе CI/CD пайплайна или по расписанию за полчаса до распродажи.

------------------------------
##### 6. Best Practices (Лучшие практики)

   1. Защищайтесь от Cache Penetration (Пробитие кэша): Ситуация, когда злоумышленник запрашивает несуществующие ID (например, /api/user/999999999). База возвращает null, кэш не записывается, каждый следующий запрос летит в БД.
   * Решение: Кэшируйте пустые ответы (Cache::put("user:999999999", null, $ttl)) на короткое время или используйте Фильтры Блума (Bloom Filters).
   2. Никогда не завязывайте бизнес-логику на кэш как на постоянное хранилище: Память в Redis может закончиться, и он начнет удалять старые ключи по алгоритму вытеснения (LRU / LFU). Данные в кэше могут исчезнуть в любой момент.
   3. Используйте разные инстансы Redis: Держите сессии пользователей на одном сервере Redis (с политикой noeviction — не удалять при нехватке памяти, лучше упасть с ошибкой), а кэш данных — на другом (с политикой allkeys-lru — автоматически удалять старый кэш при заполнении памяти).
   4. Следите за размером сериализованных объектов: Не пишите в кэш гигантские Eloquent-коллекции со всеми связями (relations). Преобразуйте их в простые легковесные массивы (DTO) перед записью в Redis для экономии RAM.

## Redis Cache

Redis (Remote Dictionary Server) — это быстрая open-source СУБД класса NoSQL, работающая по принципу «ключ-значение» (Key-Value) непосредственно в оперативной памяти (In-Memory). Из-за своей скорости (запросы выполняются за микросекунды) Redis является стандартом индустрии для кэширования, хранения сессий, брокеров сообщений и очередей.
Ниже представлен детальный разбор всех заявленных тем для подготовки к собеседованию.

------------------------------
##### 1. Установка Redis

В продакшене Redis разворачивается на Linux-серверах или в Docker-контейнерах. Для локальной разработки проще всего использовать Docker.

* Запуск через Docker:

   `docker run --name redis-local -p 6379:6379 -d redis:latest`

* Подключение к CLI внутри контейнера:

   `docker exec -it redis-local redis-cli`

* Основные CLI-команды для проверки:

    ```
    PING            # Ответ: PONG (Проверка связи)
    SET user:1 "Alex" # Запись
    GET user:1      # Чтение -> "Alex"
    FLUSHALL        # Полная очистка всех баз данных (Опасная команда!)
    ```

------------------------------
##### 2. Базовое использование в Laravel

Laravel имеет нативную интеграцию с Redis через библиотеки predis или phpredis (последняя является C-расширением для PHP и работает быстрее).

* Настройка (.env):

    ```
    CACHE_STORE=redis
    REDIS_HOST=127.0.0.1
    REDIS_PASSWORD=null
    REDIS_PORT=6379
    ```
* Использование через Фасад Cache (Уровень абстракции фреймворка):

    ```php
    use Illuminate\Support\Facades\Cache;

    // Запись с TTL (временем жизни) на 10 минут
    Cache::put('popular_products', $products, now()->addMinutes(10));

    // Чтение с замыканием (Паттерн Cache-Aside)
    $users = Cache::remember('active_users', 3600, function () {
        return User::where('is_active', true)->get();
    });

    // Удаление
    Cache::forget('active_users');
    ```
* Использование через Фасад Redis (Прямые команды в Redis):

```php
    use Illuminate\Support\Facades\Redis;

    Redis::set('stats:clicks', 10);
    Redis::incr('stats:clicks'); // Атомарный инкремент (+1)
```

------------------------------

##### 3. Структуры данных Redis (Data Structures)

Это самый популярный вопрос на собеседовании. Redis называют «сервером структур данных», потому что значениями ключей могут быть не только строки.

   1. Strings (Строки): Базовый тип данных. Может хранить текст, сериализованные объекты, JSON или бинарные данные (до 512 МБ).
   * Команды: SET, GET, INCR, DECR.
      * Кейс: Обычный кэш страниц, счетчики просмотров.
   2. Lists (Списки): Связные списки строк, упорядоченные по порядку вставки. Добавление элемента в начало или конец происходит за постоянное время $O(1)$.
   * Команды: LPUSH, RPUSH, LPOP, RPOP, LRANGE.
      * Кейс: Простые очереди задач, лента последних событий.
   3. Sets (Множества): Неупорядоченная коллекция уникальных строк. Позволяет мгновенно проверять наличие элемента за $O(1)$, а также пересекать множества.
   * Команды: SADD, SREM, SISMEMBER, SINTER (пересечение).
      * Кейс: Список уникальных IP-адресов, посетивших сайт; общие друзья пользователей.
   4. Hashes (Хэши / Словари): Объекты, состоящие из пар «поле-значение». Идеально для хранения плоских моделей.
   * Команды: HSET, HGET, HGETALL, HINCRBY.
      * Кейс: Профиль пользователя (user:100 -> name: "John", age: 30), чтобы не сериализовать весь объект в String, а обновлять поля точечно.
   5. Sorted Sets (ZSETS / Упорядоченные множества): Уникальные множества, где каждый элемент связан со специальным числовым коэффициентом (score). Данные автоматически сортируются по этому коэффициенту.
   * Команды: ZADD, ZRANGE, ZREVRANGE (обратный порядок).
      * Кейс: Лидерборды в играх, рейтинги популярных товаров, очереди с приоритетом.
   
------------------------------
##### 4. Pipeline (Конвейеризация)

По умолчанию каждый запрос приложения к Redis — это блокирующая сетевая операция (Round Trip Time — RTT): отправили команду -> дождались ответа. Если нужно отправить 10 000 команд SET, приложение сделает 10 000 сетевых запросов, что вызовет жесткие задержки.
Pipeline позволяет упаковать массив команд в один сетевой пакет, отправить его на сервер Redis, выполнить их там последовательно и вернуть массив ответов за один раз.

```php
use Illuminate\Support\Facades\Redis;

// Выполняется ОДИН сетевой запрос вместо 1000
$responses = Redis::pipeline(function ($pipe) {
    for ($i = 0; $i < 1000; $i++) {
        $pipe->set("user:score:$i", rand(1, 100));
    }
});
```

Нюанс для интервью: Pipeline не гарантирует атомарность. Если в процессе выполнения 500-й команды произойдет сбой, предыдущие 499 команд останутся примененными. Это инструмент оптимизации сети, а не транзакция.

------------------------------
##### 5. Transactions (Транзакции)

Redis поддерживает транзакции, но они сильно отличаются от транзакций в ACID-реляционных СУБД. Транзакции в Redis строятся на блоке команд MULTI, EXEC, DISCARD и WATCH.

```
Redis::multi();
Redis::set('account:1:balance', 100);
Redis::set('account:2:balance', 200);
Redis::exec(); // Только в этот момент ВСЕ команды будут выполнены атомарно

```
##### Особенности транзакций в Redis (Важно для интервью):

   1. Нет отката (No Rollback): Если внутри блока транзакции одна из команд завершится ошибкой синтаксиса или типов (например, попытка вызвать INCR на буквенной строке), Redis все равно выполнит остальные команды. Отката к предыдущему состоянию не произойдет.
   2. Оптимистичная блокировка (WATCH): Позволяет следить за ключом. Если кто-то другой изменил ключ в промежутке между WATCH и EXEC, вся транзакция автоматически отменится. Это защищает от Race Conditions.

------------------------------
##### 6. Cache Patterns (Паттерны кэширования в Redis)

Помимо базового Cache-Aside, на Redis строятся продвинутые паттерны:

* Cache Tags (Тегирование кэша): Позволяет метить разные ключи общим тегом, чтобы потом удалить их одной командой. Под капотом Laravel реализует это, создавая в Redis структуры Set, где хранит списки ключей, привязанных к тегу.
* Distributed Lock (Распределенная блокировка): Защита от одновременного выполнения процесса разными воркерами/серверами. Реализуется через команду SET key value NX PX 10000 (флаг NX означает «записать, только если ключа еще нет»). В Laravel это обернуто в удобный метод:

```php
$lock = Cache::lock('processing_report', 10); // Блокировка на 10 секунд
if ($lock->get()) {
    // Выполняем тяжелую задачу, другие сервера сюда не зайдут
    $lock->release();
}
```

------------------------------
##### 7. Best Practices (Лучшие практики работы с Redis)

   1. Выбирайте правильную политику вытеснения данных (Eviction Policy): Когда оперативная память сервера заполнится, Redis начнет удалять старые ключи. Настраивается в redis.conf:
   * allkeys-lru — удаляет наименее используемые ключи (Идеально для обычного кэша).
      * noeviction — возвращает ошибку при попытке записи (Используется, если Redis хранит критичные данные, например сессии или очереди).
   2. Избегайте команды KEYS * на продакшене: Команда KEYS осуществляет полный перебор всех ключей в базе за $O(N)$. Поскольку Redis однопоточный (Single-threaded), тяжелый поиск KEYS * заблокирует сервер на несколько секунд, парализовав весь продакшен. Используйте вместо неё курсорную команду SCAN.
   3. Используйте неймспейсы (Разделители :): Группируйте ключи по иерархии, например: app:users:123:profile. Современные GUI-клиенты для Redis автоматически сгруппируют такие ключи в удобные папки.
   4. Следите за фрагментацией памяти: Из-за особенностей аллокатора памяти (Jemalloc) Redis может не отдавать операционной системе освободившуюся память сразу. Мониторьте показатель mem_fragmentation_ratio.

    ✓ Отдельные Redis databases для разных целей (cache, sessions, queues)
    ✓ Namespace keys (user:1:name)
    ✓ TTL для всех cache keys
    ✓ Pipeline для batch operations
    ✓ Lua scripts для atomic operations
    ✓ Monitor memory usage (eviction policy)
    ✓ Persistence для критичных данных
    ✓ Redis Sentinel/Cluster для HA
    ✓ НЕ хранить огромные values (< 1MB)

"Redis — это однопоточная In-Memory СУБД, работающая со скоростью оперативной памяти. Она предлагает богатый набор структур данных: Strings, Lists, Sets, Hashes и Sorted Sets. При проектировании кэша на Redis важно помнить, что он однопоточный, поэтому команды вроде KEYS * запрещены, а для управления памятью критически важно правильно выбрать политику Eviction (например, Volatile-LRU или Allkeys-LRU)".

## Memcached

Memcached — это высокопроизводительное, распределенное хранилище данных в оперативной памяти (In-Memory Key-Value store), предназначенное для кэширования строк и объектов. Это один из старейших и наиболее проверенных инструментов для снижения нагрузки на базы данных, созданный еще в 2003 году для проекта LiveJournal.
На собеседованиях Memcached чаще всего всплывает в паре с Redis. Главная цель интервьерщика в этой теме — проверить, понимает ли кандидат разницу между простым, узкоспециализированным кэшем (Memcached) и полноценной In-Memory СУБД (Redis).

------------------------------
##### 1. Что это такое и ключевая архитектура

Memcached устроен максимально просто: это гигантская хэш-таблица в оперативной памяти.

* Тип данных: Поддерживает только один тип данных — строки (Strings). Любые объекты, массивы или коллекции перед сохранением в Memcached приложение обязано сериализовать (например, в JSON или через PHP Serialize).
* Многопоточность (Мультипоточность): В отличие от Redis, который является однопоточным, Memcached многопоточен (Multi-threaded). Он может утилизировать все ядра процессора сервера на полную мощность.
* Управление памятью (Slab Allocation): Чтобы избежать фрагментации памяти, Memcached при старте разбивает RAM на фиксированные блоки (Slabs) разного размера (например, под данные размером 1 КБ, 2 КБ, 4 КБ). Он не выделяет память под каждый ключ динамически.

------------------------------
##### 2. Когда использовать Memcached (и почему не Redis)?

На вопрос: "Почему бы просто всегда не использовать Redis?", правильный ответ кроется в следующих кейсах:

   1. Кэширование очень больших объемов простых данных: Если вам нужно кэшировать терабайты готовых HTML-страниц, JSON-ответов API или сессий. За счет многопоточности на тяжелом сетевом I/O Memcached может обходить Redis по пропускной способности.
   2. Ограниченность ресурсов (Низкое потребление RAM/CPU): Memcached практически не имеет метаданных и накладных расходов. Он занимает фиксированный объем памяти и никогда не выйдет за его пределы.
   3. Вам точно не нужны продвинутые фичи: Если приложению не требуются подписки (Pub/Sub), транзакции, сохранение данных на диск (Persistence) и структуры данных (списки, хэши, сортированные множества).

------------------------------
##### 3. Установка Memcached

Локально для разработки и тестирования проще всего поднять официальный Docker-контейнер.

* Запуск через Docker:

`docker run --name memcached-local -p 11211:11211 -d memcached:latest`

* Проверка через Telnet (Протокол Memcached текстовый и очень простой):

telnet localhost 11211# Вводим команду ручной записи: ` set <key> <flags> <exptime> <bytes>set my_key 0 60 5`
`hello# Ответ: STORED`

* Чтение:

`get my_key# Ответ: VALUE my_key 0 5 \n hello \n END`

------------------------------
##### 4. Базовое использование в Laravel

Laravel предоставляет встроенный драйвер для Memcached. Для работы на сервере должно быть установлено PHP-расширение ext-memcached.

* Настройка конфигурации (.env):

```
CACHE_STORE=memcached
MEMCACHED_HOST=127.0.0.1
MEMCACHED_PORT=11211
```

* Использование через Фасад Cache:
Интерфейс Laravel абстрагирует работу, поэтому код выглядит точно так же, как для Redis или файлового кэша.

```php
use Illuminate\Support\Facades\Cache;

// Запись объекта (Laravel автоматически сериализует его в строку)
Cache::put('user_profile_1', $userDto, 3600);

// Чтение
$userDto = Cache::get('user_profile_1');

// Инкремент счетчика (Memcached поддерживает атомарные инкременты строк-чисел)
Cache::increment('page_views_count');

```

Важное ограничение в Laravel: При использовании Memcached вы не можете использовать метод `Cache::tags()`. Memcached не поддерживает тегирование кэша из коробки, и попытка вызвать этот метод приведет к исключению (Runtime Exception).

------------------------------
##### 5. Горизонтальное масштабирование: Multi-Server Setup

Одно из главных преимуществ Memcached — его распределенность. Сам сервер Memcached ничего не знает о других серверах Memcached (у него нет концепции кластера или репликации на уровне СУБД). Масштабированием управляет клиент (драйвер в приложении).

```php
// Конфигурация config/cache.php для нескольких серверов
'memcached' => [
    'servers' => [
        ['host' => '10.0.0.1', 'port' => 11211, 'weight' => 100],
        ['host' => '10.0.0.2', 'port' => 11211, 'weight' => 100],
        ['host' => '10.0.0.3', 'port' => 11211, 'weight' => 100],
    ],
],
```

##### Алгоритм Consistent Hashing (Последовательное хэширование)

На собеседовании обязательно спросят: "Как клиент понимает, на какой из трех серверов записать ключ user:5?".

   1. Драйвер приложения (например, php-memcached) берет хэш от ключа user:5 (например, через MD5 или CRC32).
   2. Используя алгоритм Consistent Hashing (Консистентное хэширование), драйвер отображает этот хэш на виртуальное кольцо серверов и определяет, какому конкретно серверу принадлежит этот диапазон хэшей.
   3. Запрос отправляется напрямую на выбранный сервер (например, 10.0.0.2).
   4. Плюс схемы: Если сервер 10.0.0.3 упадет, из строя выйдет только $1/3$ кэша. Остальные серверы продолжат работать, а ключи упавшего сервера равномерно распределятся между оставшимися без полной инвалидации всего кэша.

------------------------------
##### 6. Stats & Monitoring (Статистика и мониторинг)

Мониторинг Memcached выполняется через отправку команды stats в сокет СУБД.

`echo "stats" | nc localhost 11211`

Ключевые метрики для контроля:

* `bytes`: Количество байт, занятых данными в RAM в данный момент. Помогает понять, близок ли лимит памяти.
* `limit_maxbytes`: Максимальный объем памяти, выделенный Memcached (параметр -m при запуске).
* `get_hits` и `get_misses`: Самая важная метрика эффективности кэширования (Cache Hit Ratio). Вычисляется по формуле: $hits / (hits + misses) * 100\%$. Если показатель ниже 80%, кэш работает неэффективно.
* `evictions`: Количество ключей, которые Memcached был вынужден принудительно удалить из памяти до истечения их TTL, потому что ему физически не хватило выделенной RAM для записи новых ключей. Рост этой метрики означает, что серверу нужно срочно выделить больше оперативной памяти.

------------------------------
##### 7. Best Practices (Лучшие практики)

   1. Ограничение на размер ключа и значения:
   * Максимальная длина ключа в Memcached — 250 байт.
      * Максимальный размер одного значения (объекта) по умолчанию — 1 МБ. Если вы попытаетесь записать туда гигантский JSON-ответ на 5 МБ, Memcached вернет ошибку.
   2. Помните про алгоритм LRU (Least Recently Used): Memcached всегда работает в режиме жесткого вытеснения данных. Если память заполнена, он без предупреждения удалит старые ключи (даже если их TTL еще не истек), чтобы освободить место для новых. Никогда не храните там критически важные данные (например, сессии корзины покупок без дублирования в БД).
   3. Не используйте длинные TTL без необходимости: Из-за структуры Slab Allocation место, освобожденное из-под старого ключа, может быть эффективно утилизировано только ключом похожего размера. Разнообразие размеров элементов может приводить к внутренней неэффективности утилизации памяти.
   4. Защищайте сетевой порт: Исторически Memcached не имеет встроенной аутентификации (паролей) по умолчанию. Оставленный наружу порт 11211 — классическая причина уязвимостей и участия серверов в DDoS-атаках (Memcached reflection amplification). Закрывайте порт файрволом (iptables/ufw) внутри приватной сети.


    ✓ Multi-server для high availability
    ✓ Consistent hashing для распределения
    ✓ Prefix для namespacing
    ✓ Monitoring: hit rate, memory usage
    ✓ TTL для всех keys
    ✓ Не хранить критичные данные (может быть evicted)
    ✓ Cache только read-heavy данные
    ✓ Для персистентности используй Redis

| Критерий | Memcached | Redis |
|---|---|---|
| Архитектура потоков | Многопоточный (Multi-threaded) | Однопоточный (Single-threaded) |
| Типы данных | Только строки (простой Key-Value) | Строки, Списки, Хэши, Множества, Геоданные |

## HTTP Cache

HTTP-кэширование (HTTP Cache) — это механизм повторного использования ранее загруженных веб-ресурсов (HTML-страниц, картинок, файлов стилей, скриптов, ответов API) без необходимости их повторной скачки с сервера. Это один из главных способов ускорить загрузку сайта для пользователя и одновременно снизить нагрузку на бэкенд и сеть.
На собеседованиях эта тема проверяет понимание кандидатом сетевых протоколов, клиент-серверного взаимодействия и умение оптимизировать веб-приложения на уровне инфраструктуры.

------------------------------
##### 1. Что это такое и где живет HTTP-кэш?

HTTP-кэширование работает на основе специальных метаданных — HTTP-заголовков (HTTP Headers), которые сервер отправляет в ответе клиенту.
По месту хранения HTTP-кэш делится на два основных типа:

   1. Приватный кэш (Browser Cache): Находится на устройстве конкретного пользователя (в браузере). Предназначен только для него (например, страница личного кабинета, аватарка).
   2. Публичный кэш (Shared / Proxy Cache): Находится между сервером и браузером. Это могут быть прокси-серверы провайдеров или CDN (Content Delivery Network). Здесь кэшируются общие для всех ресурсы (логотип сайта, файлы app.js, CSS-стили).

------------------------------
##### 2. Главный заголовок: Cache-Control

Cache-Control — это основной заголовок `HTTP/1.1`, который задает правила кэширования ресурса. Он состоит из набора директив, разделенных запятыми.

##### Ключевые директивы:

* `private` — ресурс может быть закэширован только браузером пользователя. Промежуточные CDN-серверы не имеют права его сохранять.
* `public` — ресурс может быть закэширован любым агентом, включая CDN и прокси-серверы.
* `max-age=<seconds>` — максимальное время (в секундах), в течение которого ресурс считается свежим (fresh). Браузер будет брать его из локального кэша, вообще не совершая запросов к серверу.
* `s-maxage=<seconds>` — аналогично max-age, но применяется только к публичным кэшам (CDN). Браузер эту директиву игнорирует.
* `no-cache` — Важная ловушка на интервью! Это не означает «не кэшировать». Это означает: кэшировать можно, но браузер обязан перед каждым использованием ресурса отправить запрос на сервер и проверить, не изменился ли он (валидация).
* `no-store` — Вот это означает «абсолютно не кэшировать». Нигде и никогда. Браузер и CDN обязаны каждый раз скачивать ресурс заново. Используется для банковских данных, токенов авторизации и критически личной информации.

------------------------------
##### 3. Практические примеры использования Cache-Control

На собеседованиях часто просят привести примеры заголовков для конкретных типов файлов.

##### Пример 1: Статика со сборщика (Vite / Webpack)

Файлы скриптов и стилей содержат хэш в названии (например, app.a8f9b2.js). Если код изменится, имя файла тоже изменится. Поэтому их можно кэшировать «навечно».

`Cache-Control: public, max-age=31536000, immutable`

(31536000 секунд — это 1 год. Директива immutable говорит браузеру, что файл гарантированно никогда не изменится, и даже при перезагрузке страницы (F5) не нужно слать запросы проверки).

##### Пример 2: Динамический JSON-ответ API (например, список товаров)

Данные могут обновиться в любой момент, но мы хотим снизить нагрузку. Кэшируем в браузере на 10 минут, но запрещаем CDN хранить эти данные, так как они зависят от сессии пользователя.

`Cache-Control: private, max-age=600`

##### Пример 3: Главная страница HTML (index.html)

Мы хотим, чтобы пользователи моментально получали изменения на сайте, но если изменений нет — брали файл из кэша.

`Cache-Control: no-cache`

(Браузер сохранит файл, но перед отображением сделает быстрый микро-запрос на сервер для проверки свежести).

------------------------------
##### 4. Валидация кэша: ETag и Last-Modified

Когда срок «свежести» ресурса (max-age) истек, или если используется директива no-cache, браузер должен проверить у сервера: "Эй, у меня есть старая копия, она еще актуальна?". Это называется условным запросом (Conditional Request). Для этого используются два механизма.

##### Механизм 1: Last-Modified / If-Modified-Since (По времени)

   1. Сервер отдает файл и прикрепляет заголовок: Last-Modified: Wed, 21 Jan 2026 12:00:00 GMT.
   2. При следующем запросе этого файла браузер берет эту дату и отправляет серверу заголовок: If-Modified-Since: Wed, 21 Jan 2026 12:00:00 GMT.
   3. Сервер проверяет дату изменения файла на диске:
   * Если файл не менялся, сервер возвращает пустой ответ со статусом 304 Not Modified. Браузер берет файл из своего кэша. Сетевой трафик сэкономлен.
      * Если файл изменился, сервер возвращает статус 200 OK и прикрепляет новое тело файла.
   
##### Механизм 2: ETag / If-None-Match (По хэш-сумме — более надежный)

Проблема Last-Modified в том, что время измеряется с точностью до секунды. Если файл меняется чаще или дата перезаписалась без изменения содержимого, механизм сбоит. ETag лишен этого недостатка.

   1. Сервер вычисляет хэш-сумму (контрольную сумму) файла и отдает ее: `ETag: "w/3a8b9f2c"`.
   2. При повторном запросе браузер отправляет этот хэш обратно: If-None-Match: `"w/3a8b9f2c"`.
   3. Сервер на лету сравнивает хэши:
   * Хэши совпали -> Ответ 304 Not Modified.
      * Хэши не совпали -> Ответ 200 OK с новым файлом и новым ETag.
   
------------------------------
##### 5. CDN Cache (Кэширование на стороне сетей доставки контента)

CDN (Cloudflare, Fastly, Akamai) — это сеть серверов (Edge-серверов), разбросанных по всему миру.

* Как работает: Когда пользователь из Владивостока запрашивает картинку с сервера в Москве, запрос сначала прилетает на Edge-сервер CDN во Владивостоке. Если там есть копия (CDN Cache Hit), картинка отдается за 5 мс, а сервер в Москве даже не узнает о запросе.
* Управление через заголовки: CDN в первую очередь смотрят на директиву s-maxage в Cache-Control. Если её нет — смотрят на max-age.
* Сброс кэша (Purge): Если вы обновили картинку на основном сервере (Origin), а на CDN она закэширована на месяц, вам нужно зайти в панель CDN (или через API из кода Laravel) и вызвать команду Purge Cache для конкретного URL, чтобы Edge-серверы стерли старую копию.

------------------------------
##### 6. Browser Cache Busting (Инвалидация кэша в браузере)

Если браузер сохранил файл со стратегией `max-age=31536000` (на год), вы физически не можете залезть в компьютер пользователя и очистить его кэш. Если вы выкатите багфикс в JS-коде, пользователь его не увидит, пока вручную не нажмет Ctrl + F5.
Для решения этой проблемы применяется Cache Busting (Сброс кэша путем изменения имени ресурса):

   1. Query String (Устаревший способ): Добавление версии в параметры: script.js?v=1.0.2.
   * Минус: Некоторые старые прокси-серверы и CDN принципиально отказываются кэшировать файлы, содержащие знак ? в URL.
   2. Хэширование имен (Современный стандарт): Сборщики проектов (Vite, Webpack) переименовывают файлы на основе их контента: style.eff45a.css.
   * Как это работает: Когда вы меняете одну строчку кода в стилях, сборщик генерирует файл с новым хэшем: style.99bc12.css. В HTML-коде ссылка меняется. Для браузера это совершенно новый файл, он скачивает его заново (200 OK), а старый файл просто мирно умирает в кэше по истечении TTL.
   
------------------------------
##### 7. Best Practices (Лучшие практики)

   1. Для динамического API разделяйте кэш: Никогда не отдавайте Cache-Control: public для страниц, содержащих личные данные (имя пользователя, корзина). Иначе CDN закэширует профиль первого зашедшего юзера и покажет его всем остальным. Используйте строго private.
   2. Всегда настраивайте ETag: На бэкенде (например, в Laravel через Middleware) генерация ETag экономит колоссальные объемы исходящего трафика, заменяя тяжелые ответы на легкие 304 Not Modified.
   3. Кэшируйте index.html правильно: Главный HTML-файл приложения (особенно в SPA-приложениях) должен иметь заголовок Cache-Control: no-cache или max-age=0, must-revalidate. Если закэшировать сам index.html на год, пользователи никогда не узнают, что у вас вышли новые версии скриптов с новыми хэшами.
   4. Используйте заголовок Vary: Если ваш API отдает данные в формате JSON для мобилок и HTML для браузеров по одному URL (в зависимости от заголовка Accept), используйте заголовок Vary: Accept. Это скажет кэшу, что нужно сохранять разные копии ответа для разных типов клиентов.

    ✓ Static assets: public, max-age=31536000, immutable
    ✓ Dynamic HTML: private, no-cache
    ✓ Public pages: public, max-age=3600
    ✓ API: public, max-age=300
    ✓ ETag или Last-Modified для revalidation
    ✓ Hash в URL для cache busting (Laravel Mix/Vite)
    ✓ CDN для static assets
    ✓ Vary header для разных версий (language, user-agent)
    ✓ Monitoring: cache hit rate
    ✓ Purge CDN после deploy

------------------------------
##### 💡 Шпаргалка для ответа на интервью:

"HTTP-кэширование управляется сервером через заголовки ответа. Cache-Control задает стратегию свежести: max-age указывает время жизни кэша без запросов к серверу, no-cache требует обязательной валидации, а no-store полностью запрещает кэширование. Для валидации используются заголовки Last-Modified (по времени) и ETag (по контрольной сумме файла), которые при совпадении возвращают статус 304 Not Modified. Сброс кэша «навечно» сохраненных файлов в браузерах реализуется через механизм Cache Busting — изменение хэш-суммы прямо в имени файла при сборке проекта".

## OPcache

OPcache (PHP Opcode Cache) — это встроенный в PHP механизм кэширования, который радикально повышает производительность приложений. На собеседованиях middle/senior-уровня вопросы по OPcache проверяют понимание того, как интерпретируемый язык PHP работает на низком уровне и как правильно конфигурировать инфраструктуру под Production-нагрузки.
Ниже представлен детальный разбор темы с акцентом на архитектуру, конфигурацию и подводные камни.

------------------------------
##### 1. Что такое OPcache и как он работает?

PHP — это интерпретируемый язык. При каждом HTTP-запросе (например, когда пользователь заходит на сайт) интерпретатор PHP должен выполнить неизменный цикл для каждого подключаемого скрипта:

```
[Исходный код PHP (.php)] 
       │
       ▼ (1. Лексический и синтаксический анализ)
[Токены / Абстрактное синтаксическое дерево (AST)]
       │
       ▼ (2. Компиляция)
[ Байт-код / Опкоды (Opcodes) ] <-- Это инструкции, понятные движку Zend Engine
       │
       ▼ (3. Выполнение)
[ Результат (HTML / JSON) ]
```

Если в проекте (например, на Laravel) при каждом запросе подключается более 1000 файлов фреймворка, процесс парсинга и компиляции выжигает процессорное время (CPU) сервера.
OPcache решает эту проблему: он компилирует PHP-скрипты в байт-код (опкоды) ровно один раз и сохраняет этот байт-код в общей оперативной памяти (Shared Memory). При всех последующих запросах PHP полностью пропускает этапы анализа и компиляции, мгновенно считывая готовые опкоды из RAM и отправляя их на выполнение. Это дает прирост производительности в 2–5 раз.

------------------------------
##### 2. Установка и базовая проверка

Начиная с PHP 5.5, OPcache поставляется в ядре PHP. Его нужно просто включить.

* Установка на Linux (Ubuntu/Debian):

`sudo apt install php8.3-opcache`

* Включение в php.ini:

```ini
zend_extension=opcache.so
opcache.enable=1
opcache.enable_cli=1 ; Включает OPcache для CLI-скриптов (полезно для Laravel Swoole/Octane)
```

* Проверка из терминала:

`php -v# В выводе должно быть: with Zend OPcache v8.3..., Copyright (c) ... `

------------------------------
##### 3. Настройки для Production (Production Settings)

В продакшене конфигурация должна выжимать максимум скорости и запрещать PHP проверять изменения в файлах на диске.

```ini
; Идеальный конфиг для Production (php.ini)
opcache.enable=1
opcache.enable_cli=1

; 1. Объем памяти под опкоды (в мегабайтах)
; Для средних проектов на Laravel ставьте от 128 до 256. Если памяти мало, OPcache начнет сбрасывать кэш.
opcache.memory_consumption=256

; 2. Объем памяти под строки (Interned Strings)
; PHP кэширует имена переменных, методов и классов. Ставьте 16 или 32.
opcache.interned_strings_buffer=16

; 3. Максимальное количество файлов в кэше
; Число ДОЛЖНО быть простым (prime number) из заложенного списка (например: 7963, 16229, 32531, 65407).
; Запустите `find . -type f -name "*.php" | wc -l` в корне проекта, чтобы узнать сколько у вас файлов.
opcache.max_accelerated_files=32531

; 4. ГЛАВНОЕ: Отключение валидации файлов (Прирост производительности)
; PHP больше никогда не будет делать системный вызов stat() для проверки, изменился ли файл.
opcache.validate_timestamps=0

; 5. Сохранение комментариев в байт-коде
; Обязательно для многих ORM и DI-контейнеров, которые парсят аннотации/атрибуты PHP.
opcache.save_comments=1

; 6. Быстрое освобождение памяти (Ускоряет завершение процессов)
opcache.fast_shutdown=1
```
------------------------------
##### 4. Настройки для разработки (Development Settings)

На локальной машине разработчика настройка opcache.validate_timestamps=0 станет кошмаром: вы измените код в IDE, обновите страницу в браузере, но код не поменяется, так как PHP выполняет старые опкоды из памяти.

```ini
; Идеальный конфиг для Локальной разработки (Local/Dev)
opcache.enable=1
opcache.validate_timestamps=1
; Как часто (в секундах) проверять файлы на диске. 0 означает "проверять при каждом запросе".
opcache.revalidate_freq=0
```

------------------------------
##### 5. Очистка OPcache (Сброс кэша при деплое)

Если на прод-сервере установлена директива opcache.validate_timestamps=0, то после выполнения git pull или деплоя новой версии кода приложение продолжит выполнять старый код, застрявший в памяти. Кэш необходимо принудительно сбросить.

##### Способы очистки кэша:

   1. Перезапуск веб-сервера / обработчика процессов:
   Самый надежный способ, который полностью очищает Shared Memory.
   
   ```
   sudo systemctl restart php8.3-fpm# или если используется Nginx + Apache:
   sudo systemctl restart apache2
   ```
   
   2. Использование встроенной функции opcache_reset():
   Ловушка на интервью: Если вы запустите php -r "opcache_reset();" из консоли (CLI), вы очистите OPcache только для консольного режима. Кэш веб-сервера (PHP-FPM) останется нетронутым, так как у них разные области памяти.
   * Как решить: Чтобы сбросить кэш PHP-FPM без перезапуска сервиса, нужно выполнить скрипт через веб-сервер. Для этого используют утилиты автоматизации деплоя, например cachetool.
   
   `cachetool opcache:reset --fcgi=/var/run/php/php8.3-fpm.sock`
   
   
------------------------------
##### 6. Preloading (Предзагрузка) — Начиная с PHP 7.4

Preloading — это шаг вперед по сравнению с обычным OPcache.

* Как это работает: При старте сервера (до того, как прилетит первый HTTP-запрос) PHP запускает специальный скрипт предзагрузки. Этот скрипт компилирует выбранные файлы (например, ядро Laravel) и загружает их опкоды в память навсегда.
* В чем отличие от OPcache: Файлы, попавшие в Preloading, становятся частью самого ядра PHP. Они доступны всем последующим запросам мгновенно без единого вызова внутренних механизмов проверки наличия кэша.
* Главный минус: Эти файлы невозможно выгрузить из памяти через opcache_reset(). Единственный способ обновить код, попавший в Preload — это физический перезапуск PHP-FPM.

##### Настройка в php.ini:

opcache.preload=/var/www/my-app/preload.php
opcache.preload_user=www-data ; Имя пользователя, от которого работает PHP-FPM

В Laravel для генерации этого скрипта используется встроенный компилятор (начиная с Laravel 8+ автоматизировано в инфраструктурных пакетах).

------------------------------
##### 7. Мониторинг OPcache

Для контроля за состоянием памяти используются встроенные функции PHP. Выводить их наружу в продакшене без авторизации строго запрещено.

* opcache_get_status() — возвращает массив со статистикой (объем свободной памяти, количество закэшированных файлов, процент попадания в кэш).
* opcache_get_configuration() — возвращает текущие директивы конфигурации.

##### 8. Best Practices (Лучшие практики)

   1. Контролируйте blacklist_filename: Вы можете указать файл со списком скриптов, которые OPcache не должен кэшировать ни при каких условиях (например, динамически генерируемые PHP-файлы конфигураций).
   2. Следите за переполнением кэша (OOM): Если в логах PHP-FPM появляется предупреждение “Zend OPcache shifts into restart mode”, это значит, что у вас закончилось выделенное место (opcache.memory_consumption), и OPcache вынужден полностью сбрасывать кэш на лету, что вызывает кратковременные жесткие просадки по RPS сайта. Срочно увеличивайте объем памяти.
   3. Используйте JIT (Just-In-Time) компиляцию (PHP 8.0+): JIT работает поверх OPcache. Он берет наиболее часто вызываемые опкоды (hot-команды) и компилирует их напрямую в машинный код процессора (CPU x86/ARM), минуя виртуальную машину Zend. Настраивается через opcache.jit_buffer_size=64M. (Очень эффективно для математических расчетов и фоновых воркеров, малоэффективно для классических CRUD API).

На Production критически важно выставлять opcache.validate_timestamps=0 для отключения дисковых проверок, но это требует обязательного сброса кэша через перезапуск PHP-FPM или утилиту cachetool при каждом деплое.
