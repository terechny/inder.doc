# Docker

## Docker Basics

Docker — это платформа для контейнеризации приложений. Она позволяет упаковать код приложения, системные зависимости, библиотеки, настройки и среду выполнения в единый изолированный объект — контейнер.

* Главная цель: Решить проблему «На моей машине всё работает, а на сервере падает». Docker гарантирует, что приложение будет работать абсолютно одинаково на локальном компьютере разработчика (Windows/Mac/Linux), стейджинге и продакшн-сервере.
* Контейнер vs Виртуальная машина (ВМ): Виртуальная машина (например, VirtualBox) эмулирует полноценное «железо» и запускает внутри себя тяжелую гостевую операционную систему со своим ядром. Контейнер Docker изолирован, но использует ядро хостовой ОС, не эмулирует виртуальное железо и запускает процессы напрямую. Поэтому контейнеры легковесные, запускаются за доли секунды и потребляют минимум памяти.

##### Основные понятия Docker

* Dockerfile: Простой текстовый файл с пошаговыми инструкциями (командами), по которым собирается образ вашего приложения (например: обновить ОС, поставить PHP, скопировать код, запустить сервер).
* Образ (Image): Неизменяемый снимок (шаблон), готовый к запуску. Это «чертеж» приложения, содержащий в себе код и окружение. Образы скачиваются из репозиториев (например, Docker Hub).
* Контейнер (Container): Живой, запущенный экземпляр образа. Если образ — это класс, то контейнер — это объект этого класса. В одном проекте может быть запущено много одинаковых контейнеров из одного образа.

##### Основные команды Docker CLI

* `docker build -t my-app .` — собрать образ из Dockerfile в текущей директории и дать ему тег (имя) my-app.
* `docker run -p 8080:80 my-app` — запустить контейнер из образа my-app и пробросить порт 80 из контейнера на порт 8080 вашего компьютера.
* `docker run -d my-app` — запустить контейнер в фоновом режиме (detached mode, терминал останется свободным).
* `docker ps` — показать список всех запущенных в данный момент контейнеров (флаг -a покажет и остановленные).
* `docker stop <container_id>` — корректно остановить запущенный контейнер.
* `docker rm <container_id>` — удалить остановленный контейнер.
* `docker rmi <image_id>` — удалить образ с локального компьютера.
* `docker logs <container_id>` — посмотреть логи (вывод в консоль) конкретного контейнера.
* `docker exec -it <container_id> bash` — подключиться к терминалу (командной строке) внутри запущенного контейнера в интерактивном режиме.

##### Docker для PHP и Laravel

Классическое монолитное Laravel-приложение требует для работы связку из нескольких технологий (стек). В Docker-архитектуре действует правило: один процесс (сервис) — один контейнер. Поэтому проект разбивается на инфраструктурные слои с помощью утилиты Docker Compose (инструмент для управления многоконтейнерными приложениями через файл docker-compose.yml).
Типичная схема контейнеров для Laravel:

   1. Контейнер nginx: Принимает веб-трафик и отдает статичные файлы.
   2. Контейнер php-fpm: Выполняет PHP-код приложения, общаясь с Nginx по протоколу FastCGI.
   3. Контейнер mysql / postgres: База данных.
   4. Контейнер redis: Используется для кэша, сессий и очередей Laravel.

В современном Laravel стандартом для локальной разработки является инструмент Laravel Sail — это официальная, преднастроенная Docker-инфраструктура, управляемая через лаконичный интерфейс (команда ./vendor/bin/sail up).

##### Сети в Docker (Networks)

Контейнеры по умолчанию полностью изолированы друг от друга и от внешнего мира. Чтобы контейнер с PHP-кодом мог отправить SQL-запрос в контейнер с базой данных MySQL, они должны находиться в одной сети.

* Как это работает: Docker создает виртуальную сеть. Внутри этой сети контейнеры могут общаться друг с другом по своим именам сервисов (используется внутренний DNS-сервер Docker).
* Пример: В Laravel-файле .env в качестве DB_HOST вместо IP-адреса 127.0.0.1 вы просто указываете имя сервиса из docker-compose.yml — DB_HOST=mysql. Docker сам поймет, на какой контейнер перенаправить запрос.
* Основные драйверы сетей:
* bridge (мост) — дефолтный драйвер. Создает изолированную сеть на одном компьютере (хосте). Идеален для связки PHP + MySQL.
   * host — убирает изоляцию между контейнером и вашим компьютером. Контейнер использует сеть хоста напрямую (порты не нужно пробрасывать).
   * overlay — используется для соединения контейнеров, запущенных на разных физических серверах (в кластерах Docker Swarm).

##### Хранилища данных в Docker (Volumes)

Контейнеры по своей природе эфемерны (stateless). Это значит, что любые файлы, созданные или измененные внутри контейнера во время его работы, будут безвозвратно удалены, если контейнер остановить и пересоздать (docker-compose down). Для сохранения данных (базы данных, загруженные пользователями картинки) используются Volumes (Тома).
Тома позволяют «примонтировать» папку с вашего компьютера внутрь изолированного контейнера. Выделяют два основных типа монтирования:

##### 1. Named Volumes (Именованные тома)

Управляются самим Docker. Он создает скрытую изолированную папку на вашем компьютере и жестко привязывает её к контейнеру.

* Для чего используется: Для баз данных (MySQL, Redis). Вам не нужно видеть эти файлы на своем компьютере, но они должны гарантированно выжить при перезапуске контейнеров.

##### Пример в docker-compose.ymlservices:

```yaml

  mysql:
    image: mysql:8.0
    volumes:
      - db_data:/var/lib/mysql # Данные БД сохранятся на вашем ПК
volumes:
  db_data: # Объявление именованного тома

```

##### 2. Bind Mounts (Монтирование директорий)

Вы явно указываете конкретный путь к папке на вашем компьютере, которая должна отобразиться внутри контейнера.

* Для чего используется: Для исходного кода PHP в режиме разработки. Вы меняете код в своей любимой IDE (PhpStorm/VS Code) на локальном компьютере, и эти изменения мгновенно появляются внутри контейнера php-fpm, без необходимости заново собирать образ приложения.

```yaml

services:
  laravel_app:
    image: my-php-image
    volumes:
      - ./:/var/www/html # Пробрасывает текущую папку с кодом в контейнер
```

##### 🔥 Вопросы на собеседовании по Docker

   1. Какая разница между командами `CMD` и `RUN` в Dockerfile?
   * Ответ: Инструкция RUN выполняется в процессе сборки образа (Build time). Она используется для установки пакетов, компиляции кода (например, `RUN apt-get update && apt-get install -y php`). Результат выполнения фиксируется в слое образа. Инструкция CMD задает команду, которая выполнится только в момент старта контейнера (Runtime). В Dockerfile может быть только одна рабочая команда CMD (например, CMD ["php-fpm"]).
   2. Почему контейнер с MySQL сразу падает после запуска, если не указать переменную `MYSQL_ROOT_PASSWORD`?
   * Ответ: Разработчики официального образа MySQL заложили в логику стартового скрипта проверку безопасности. Если при запуске контейнера через переменные окружения (ENV) не передан пароль суперпользователя, процесс инициализации аварийно завершается с ошибкой безопасности, и контейнер останавливается.
   3. В чем разница между docker-compose up и docker-compose start?
   * Ответ: `docker-compose up` проверяет наличие конфигурации, сетей, томов, при необходимости собирает или скачивает образы, создает контейнеры с нуля и запускает их. Команда docker-compose start может запустить только уже существующие, но ранее остановленные контейнеры, не пересоздавая их структуру.
   


##  Dockerfile

Dockerfile — это обычный текстовый файл, содержащий последовательный список инструкций (команд), по которым Docker автоматически собирает готовый образ (Image) [INDEX].
Каждая строчка в Dockerfile представляет собой отдельную команду. Docker выполняет их сверху вниз. Большинство инструкций создают новый слой (layer) в образе. Слои кэшируются, что позволяет Docker не пересобирать весь проект с нуля, если вы изменили всего одну строчку кода.

##### 📜 Основные инструкции Dockerfile

* `FROM` — Задает базовый (родительский) образ, на основе которого строится ваш образ (например, чистый Linux Ubuntu или преднастроенный PHP). С этой инструкции обязан начинаться любой Dockerfile.
* `RUN` — Выполняет команду внутри контейнера на этапе сборки образа (например, установка пакетов через apt-get или запуск composer install).
* `COPY` / `ADD` — Копируют файлы с вашего локального компьютера внутрь образа. `COPY` просто переносит файлы, а `ADD` умеет автоматически распаковывать .tar архивы и скачивать файлы по URL. Рекомендуется всегда использовать COPY.
* `WORKDIR` — Устанавливает рабочую (текущую) директорию для всех последующих инструкций (RUN, CMD, COPY). Аналог команды cd в Linux. Если папки не существует, Docker создаст её автоматически.
* `ENV` — Устанавливает переменные окружения, которые будут доступны как во время сборки, так и внутри запущенного контейнера.
* `EXPOSE` — Информационная инструкция. Указывает, какой сетевой порт контейнер будет слушать во время работы. Реально порт наружу она не пробрасывает (это делает docker run -p).
* `CMD` — Задает команду по умолчанию, которая выполнится только в момент запуска контейнера. В Dockerfile может быть только одна инструкция CMD.

##### 📑 Базовый Dockerfile (Минимальный пример для PHP)

Пример простого Dockerfile, который собирает образ для запуска изолированного PHP-скрипта:

- 1. Берем официальный легковесный образ PHP на базе дистрибутива Alpine LinuxFROM php:8.3-alpine
- 2. Создаем рабочую папку внутри образаWORKDIR /app
- 3. Копируем файл скрипта с нашего компьютера в рабочую папку образаCOPY script.php .
- 4. Указываем команду, которая выполнится при старте контейнераCMD ["php", "script.php"]

##### 🐘 Реальный Dockerfile для Laravel (Production-ready)

Для запуска Laravel в продакшене бэкенд-контейнеру (сервису php-fpm) требуются системные расширения для работы с БД, кэшем, а также пакетный менеджер Composer.

- Используем официальный образ PHP с установленным FPMFROM php:8.3-fpm-alpine
- Устанавливаем рабочую директорию веб-сервераWORKDIR /var/www/html
- Устанавливаем системные зависимости и PHP-расширения (например, для MySQL, строк и архивов)

```bash
RUN apk add --no-cache \
    bash \
    curl \
    libpng-dev \
    libzip-dev \
    zip \
    unzip \
    && docker-php-ext-install pdo_mysql gd zip
```

- Копируем системный Composer из официального образа Composer внутрь нашего образаCOPY --from=composer:latest /usr/bin/composer /usr/bin/composer
- Копируем файлы конфигурации зависимостей (сначала только их, чтобы кэшировать слой)COPY composer.json composer.lock ./
- Устанавливаем PHP-зависимости без запуска скриптов Laravel и без dev-пакетовRUN composer install --no-scripts --no-autoloader --no-dev
- Копируем весь остальной код проекта в контейнерCOPY . .
- Генерируем финальный оптимизированный автозагрузчик классов ComposerRUN composer dump-autoload --optimize --no-dev
- Меняем владельца файлов на системного пользователя www-data, чтобы у Laravel были права на запись в storageRUN chown -R www-data:www-data /var/www/html/storage /var/www/html/bootstrap/cache
- Открываем порт 9000 для общения с NginxEXPOSE 9000
- Запускаем сервер PHP-FPMCMD ["php-fpm"]


##### 🧊 Многоэтапная сборка (Multi-stage build)
Multi-stage build — это продвинутая техника оптимизации Dockerfile, которая позволяет использовать несколько инструкций FROM в одном файле.

* Зачем это нужно: Чтобы собрать фронтенд для Laravel (скомпилировать JS/CSS через Vite/NPM), вам в контейнере нужны Node.js и npm. Но тащить тяжелую Node.js на продакшн-сервер в финальный PHP-контейнер — грубая ошибка (это раздует размер образа и снизит безопасность).
* Как это работает: Мы создаем временный этап («стейдж») сборки, компилируем там ассеты, а затем копируем только итоговый результат в финальный чистый PHP-образ.

##### Пример Multi-stage Dockerfile для Laravel:

- --- ЭТАП 1: Сборка фронтенда (Называем этап как "frontend") ---FROM node:20-alpine AS frontendWORKDIR /appCOPY package.json package-lock.json vite.config.js ./COPY resources/ ./resources/RUN npm ci && npm run build # На выходе получаем готовую статику в public/build
- --- ЭТАП 2: Финальный продакшн образ ---FROM php:8.3-fpm-alpineWORKDIR /var/www/html
- Устанавливаем PHP зависимости...COPY --from=composer:latest /usr/bin/composer /usr/bin/composerCOPY composer.json composer.lock ./RUN composer install --no-scripts --no-autoloader --no-dev
- Копируем код бэкендаCOPY . .
- КЛЮЧЕВОЙ МОМЕНТ: Забираем скомпилированный фронтенд из первого этапаCOPY --from=frontend /app/public/build ./public/build
RUN composer dump-autoload --optimize --no-devCMD ["php-fpm"]

Результат: Финальный образ весит в 3-4 раза меньше, так как в нем нет кэша NPM, папки node_modules и бинарников Node.js.

##### 🙈 Файл .dockerignore

Когда выполняется инструкция COPY . ., Docker берет всю текущую папку проекта и отправляет её на сборку (в контекст Docker-демона). Если в проекте лежат тяжелые или секретные файлы, это сильно замедлит сборку и раздует образ.
`.dockerignore` — это файл-исключение (аналог `.gitignore`), в котором прописываются файлы и папки, которые категорически нельзя копировать внутрь Docker-образа.

##### Типичный .dockerignore для Laravel:

```

.git
.env
/node_modules
/vendor
/storage/framework/cache/data/*
/storage/framework/sessions/*
/storage/framework/views/*
/storage/logs/*
.phpunit.result.cache
Dockerfile
docker-compose.yml

```

Почему это важно: Мы не должны копировать локальную папку `/vendor` или `/node_modules`, так как зависимости должны устанавливаться заново внутри самого контейнера под нужную операционную систему (Linux Alpine). Файл `.env` нельзя копировать, чтобы не зашить секретные пароли локальной машины прямо внутрь публичного образа.


##### Практические советы

- Кеширование слоёв:

```Dockerfile
# ❌ ПЛОХО: весь код копируется, при изменении кеш сбрасывается
COPY . .
RUN composer install

# ✅ ХОРОШО: сначала composer файлы, потом код
COPY composer.json composer.lock ./
RUN composer install
COPY . .
# Composer кеш сохранится если composer.json не менялся

```

##### 🔥 Вопросы на собеседовании по Dockerfile

   1. Почему важно располагать инструкцию COPY composer.json ... и RUN composer install ДО инструкции COPY . .?
   * Ответ: Из-за механизма кэширования слоев Docker. Если слой с файлами не изменился, Docker берет его из кэша. Если мы сначала сделаем COPY . ., то любое минимальное изменение в коде контроллера сбросит кэш для всех последующих строк. Из-за этого при каждом изменении кода Docker будет заново скачивать и устанавливать все пакеты через Composer, что займет много времени. Разделяя копирование зависимостей и кода, мы заставляем composer install запускаться только тогда, когда реально изменился файл composer.json.
   2. В чем разница между инструкциями CMD и ENTRYPOINT?
   * Ответ: `ENTRYPOINT` задает неизменяемую команду, которая всегда выполнится при старте контейнера. CMD в таком случае превращается в список дефолтных аргументов для ENTRYPOINT. Главное отличие: параметры из CMD можно легко переопределить при запуске контейнера через консоль (docker run my-image php artisan migrate), а команду ENTRYPOINT переопределить гораздо сложнее (требуется флаг --entrypoint).
   3. Зачем использовать дистрибутивы alpine в качестве базовых образов FROM?
   * Ответ: Alpine Linux — это сверхлегковесный дистрибутив Linux (базовый образ весит всего около 5 МБ). Использование alpine-образов вместо тяжелых Ubuntu/Debian позволяет уменьшить финальный размер Docker-образа проекта на сотни мегабайт, ускоряет его скачивание на сервер и уменьшает количество уязвимостей в системе, так как в нем нет лишнего софта.
   

 
## Docker Compose

##### 1. Теория и архитектура: Что под капотом?

На собеседовании важно подчеркнуть: Docker Compose — это клиентский инструмент оркестрации (не демон). Он транслирует декларативный YAML-файл в императивные запросы к Docker Engine API.

##### Сетевая изоляция (Networks)

По умолчанию Compose создает единую изолированную bridge-сеть для всех сервисов в файле.

* Важная фишка (Service Discovery): Контейнеры общаются друг с другом напрямую по именам сервисов из YAML, а не по IP-адресам (например, DB_HOST=mysql). Docker под капотом использует встроенный DNS-сервер для резолвинга.
* Продвинутый уровень: В production-подобных средах сети разделяют. Например, frontend-network (Nginx + PHP) и backend-network (PHP + MySQL + Redis). Это ограничивает доступ: Nginx физически не сможет «достучаться» до базы данных напрямую.

##### Постоянство данных (Volumes) и разница с Bind Mounts

На вопрос «Где хранить данные БД?» нужно четко разделять два механизма:

   1. Named Volumes (Управляемые тома): Инициализируются в секции volumes. Docker сам управляет директорией на хосте (/var/lib/docker/volumes/...). Это production-ready подход, так как он гарантирует высокую производительность ввода-вывода (I/O) и изолированность файловой системы.
   2. Bind Mounts (Привязка папок): Прямое прокидывание папки с хост-машины в контейнер (например, - .:/var/www/html). Используется строго для разработки, чтобы код обновлялся в контейнере налету («hot-reload»). На продакшене bind-mount ухудшает производительность и нарушает безопасность.


##### 2. Развернутый Laravel-стек: Взаимодействие компонентов

Типичная ошибка на собеседованиях — сказать, что всё работает в «одном контейнере». Правильный ответ: Один процесс — один контейнер (философия Docker).
Архитектура Laravel-стека в docker-compose.yml выглядит так:

```yaml

version: '3.8'
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - .:/var/www/html
    depends_on:
      php:
        condition: service_healthy # Ждем полной готовности PHP, а не просто старта

  php:
    build:
      context: .
      dockerfile: Dockerfile
    volumes:
      - .:/var/www/html
    environment:
      - DB_HOST=mysql
      - REDIS_HOST=redis

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: laravel
      MYSQL_ROOT_PASSWORD: root
    volumes:
      - db_data:/var/lib/mysql
    healthcheck: # Критично для автоматизации
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:alpine

  queue-worker:
    build:
      context: .
    command: php artisan queue:work
    volumes:
      - .:/var/www/html
    depends_on:
      mysql:
        condition: service_healthy
volumes:
  db_data:

```

##### Как они взаимодействуют (Фактура для ответа):

   1. Nginx слушает внешний порт 80. Получая запрос на .php файл, он не обрабатывает его сам, а перенаправляет по протоколу FastCGI на сервис php:9000.
   2. PHP-FPM выполняет код, общаясь с MySQL и Redis по их именам сервисов внутри изолированной сети.
   3. Queue-Worker использует тот же самый образ (Image) и код, что и основной PHP-контейнер, но Compose переопределяет его стартовую команду на php artisan queue:work. Это позволяет не дублировать Dockerfile.

##### 3. «Подводные камни» и Best Practices на собеседованиях## Проблема «Ложного Старта» (Race Condition)

Вопрос интерviewer'а: «У вас в depends_on указана база данных, но при запуске проекта php artisan migrate падает с ошибкой Connection Refused. Почему?»
Ответ: По умолчанию depends_on: - mysql ждет только момента, когда процесс базы данных запустился ОС в контейнере. Но MySQL в первые 5-10 секунд проводит внутреннюю инициализацию и не принимает сетевые соединения.
Решение: Использовать Healthchecks в связке с расширенным depends_on:

```yaml
depends_on:
  mysql:
    condition: service_healthy
```

В самом сервисе mysql прописывается команда тестирования (например, mysqladmin ping). 
Compose не запустит PHP/миграции, пока тест не вернет exit code 0.

##### Разница между ports и expose

* ports (Порты наружу): Открывает порт на хост-машине (например, 80:80 у Nginx). Теперь веб-сайт доступен из внешнего мира.
* expose (Внутренние порты): Документирует порт, делая его доступным только для других контейнеров внутри сети Compose. Для MySQL (3306) или Redis (6379) нужно использовать expose (или вообще ничего не писать, если они открыты в самом Dockerfile), но не ports, чтобы никто из интернета не смог подключиться к вашей БД.

##### 4. Командный интерфейс (CLI) в деталях

На собеседовании покажите знание флагов:

* `docker compose up -d` — Запуск в фоне (detached). Если изменился код в Dockerfile, Compose по умолчанию использует старый образ. Чтобы принудительно пересобрать, нужно говорить: docker compose up -d --build.
* `docker compose down -v` — Флаг -v (volumes) критически важен. Без него down просто удалит контейнеры. Если вы хотите полностью очистить базу данных (сбросить состояние до девственно чистого), флаг -v удалит именованные тома (volumes).
* `docker compose exec php artisan migrate` — Выполняет команду в уже запущенном контейнере.
* Уточнение для Senior-ответа: В чем разница с docker compose run? run создает новый, одноразовый контейнер для выполнения команды и может нарушить связи, а exec подключается к существующему рабочему окружению.

```bash
# Запустить 3 экземпляра queue worker
docker-compose up -d --scale queue=3
```

## CI/CD

Вот развернутый, глубокий теоретический материал с технической фактурой и архитектурными нюансами для ответов на собеседованиях уровня Middle/Senior.

##### Комплексный гайд по CI/CD, Автоматизации и Docker для Backend-разработчика## 1. CI/CD: Что это на самом деле под капотом?

На собеседовании важно уйти от банального перевода аббревиатур и показать понимание бизнес-процесса и технических метрик.

* CI (Continuous Integration / Непрерывная интеграция): Процесс автоматического слияния кода всех разработчиков в общую ветку (обычно main или develop).
* Фактура для ответа: CI запускается на каждый Pull Request / Push. Его задача — как можно раньше обнаружить конфликты и баги. Включает в себя: статический анализ (линтеры, PHPStan), проверку кодстайла (PHP CS Fixer) и обязательный запуск пайплайна unit- и интеграционных тестов (PHPUnit).
* CD (Continuous Delivery / Непрерывная доставка): Автоматическая сборка готового к релизу артефакта (например, Docker-образа) после успешного прохождения CI. Артефакт автоматически деплоится на тестовый контур (staging), а деплой в прод происходит вручную по кнопке (после одобрения QA/менеджером).
* CD (Continuous Deployment / Непрерывное развертывание): Полностью автоматический процесс. Изменения из ветки main, прошедшие все тесты, без участия человека за пару минут долетают до конечных пользователей на Production.

##### 2. Сравнительный анализ: GitHub Actions vs GitLab CI

Интервьюеры любят спрашивать, почему для одного проекта выбрали один инструмент, а для другого — альтернативный.

##### GitHub Actions (Декларативный подход на базе событий)

* Архитектура: Построен вокруг концепции Events (событий): push, pull_request, release, issue.
* Ключевая фишка (Marketplace): Огромная экосистема готовых блоков (Actions). Вам не нужно писать bash-скрипт для настройки PHP или авторизации в AWS, вы берете готовое: uses: shivamathur/setup-php@v2 или uses: aws-actions/configure-aws-credentials@v2.
* Фактура для ответа: Идеально подходит для Open Source и проектов, которые изначально хостятся на GitHub. Конфигурация пишется в .github/workflows/*.yml.

##### GitLab CI (Мощный enterprise-инструмент на базе стейджей)

* Архитектура: Строится на четком разделении на Stages (этапы, например: build -> test -> deploy) и Jobs (конкретные задачи внутри этапа). Задачи внутри одного стейджа выполняются параллельно.
* Ключевая фишка (GitLab Runners): Невероятно гибкая система раннеров. Вы можете легко установить собственный агент (раннер) на любой свой сервер, изолировать его и гонять тяжелые тесты на своем железе бесплатно.
* Продвинутые фичи: extends (наследование конфигураций), include (переиспользование YAML-файлов из других репозиториев) и мощный встроенный Docker Registry. Конфигурация: .gitlab-ci.yml.

##### 3. Эволюция деплоя: Deploy Script vs Laravel Envoy

Вопрос на собеседовании: «Как вы доставляете код на сервер, если нет Kubernetes?»

##### Самописный Deploy Script (Bash)

Это классический Bash-скрипт (deploy.sh), который лежит в корне проекта или пишется прямо внутри CI/CD-пайплайна.

* Как это работает: Скрипт по SSH подключается к целевому серверу, делает git pull, выполняет composer install --no-dev, php artisan migrate, php artisan config:cache и перезапускает очереди/веб-сервер.
* Минусы (Мишени для критического вопроса): Отсутствие атомарности. Если на этапе composer install упадет сеть или не хватит памяти, ваш рабочий сайт «ляжет» прямо в процессе деплоя (Downtime). Скрипты сложно масштабировать на 3–4 сервера.

##### Laravel Envoy (SSH-орекстратор на Blade-синтаксисе)

Инструмент от создателей Laravel для запуска SSH-задач на удаленных серверах. Он использует привычный синтаксис Blade (конфиг Envoy.blade.php).

* Архитектура и фишки:
* Позволяет описывать деплой в виде макросов (групп задач).
   * Поддерживает параллельный деплой на множество серверов (@servers(['web1' => '...', 'web2' => '...']) с флагом parallel).
   * Позволяет делать локальные проверки перед отправкой команд на удаленный хост.
* Реализация Zero-Downtime (Бесперебойного деплоя):
На Envoy легко реализуется стратегия деплоя без простоя (как в инструментах Capistrano или Deployer). На сервере создается структура:
* /var/www/app/releases/ (папки со временем деплоя: 20260605120000, 20260605130000).
   * /var/www/app/current — символическая ссылка (symlink), которая указывает на последний успешный релиз. Nginx/PHP-FPM смотрят на current.
   Скрипт Envoy стягивает код в новую папку релиза, ставит зависимости, прогоняет миграции, и только если всё прошло успешно, одной атомарной командой перенаправляет симлинк current на новую папку. Если что-то упало — старый релиз продолжает работать без сбоев.

##### 4. Docker в CI/CD: Архитектура сборки и доставки

Использование Docker в пайплайнах кардинально меняет подход к релизам. Главный принцип: «Мы собираем артефакт один раз, а тестируем и деплоим его везде» (Build once, run anywhere).

##### Этапы Docker-пайплайна (Сквозной процесс)

   1. Build (Сборка): CI-раннер собирает Docker-образ вашего Laravel-приложения на основе Dockerfile.
   * Важно для Senior: Здесь применяется Multi-stage build. На первом этапе (Stage) ставится Composer, Node.js, собираются фронтенд-ассеты и вендоры. На втором этапе чистый продакшн-образ (где есть только PHP-FPM) копирует из первого этапа готовые файлы /vendor и public/build. Это уменьшает размер итогового образа с 1 ГБ до 150 МБ и исключает лишние инструменты разработки из продакшена.
   2. Push (Публикация): Готовый образ тегируется (например, по SHA-коммита: my-app:a1b2c3d4) и отправляется в приватный реестр — Docker Registry (GitLab Container Registry, GitHub Packages или AWS ECR).
   3. Test (Тестирование): Тесты запускаются внутри этого же собранного образа, гарантируя, что среда тестирования на 100% идентична будущему продакшену.
   4. Deploy (Развертывание): CI-пайплайн заходит на прод-сервер по SSH и выполняет:
   
   `docker login ...`
   `docker compose pull` # Скачивает новые образы из Registry
   `docker compose up -d --remove-orphans` # Обновляет контейнеры без остановки остальных
   `docker compose exec -T php php artisan migrate --force` # Запуск миграций
   
   
##### Разница между Docker в Dev и в CI/CD (Частый вопрос на собеседовании)

| Критерий | Разработка (Local Dev) | CI/CD и Production |
|---|---|---|
| Исходный код | Прокидывается через Bind Mount (- .:/var/www), чтобы код менялся на хосте и сразу работал в контейнере. | Код полностью впекается внутрь образа (COPY . /var/www) на этапе сборки. Контейнер автономен. |
| Зависимости (vendor) | Устанавливаются локально или через контейнер в процессе работы. | Устанавливаются строго внутри Dockerfile во время сборки образа. |
| Окружение (.env) | Монтируется локальный .env файл со всеми доступами к dev-базам. | Секреты (пароли, ключи) прокидываются через переменные окружения CI/CD (Secrets) в рантайме. |

## Deployment Strategies

Вот развернутый, глубокий теоретический материал по стратегиям деплоя и управлению базами данных в production-средах. Этот блок продемонстрирует интервьюеру ваше понимание высоконагруженных систем (High Availability) и умение минимизировать риски при релизах.

##### Стратегии развертывания (Deployment Strategies) и миграции БД в Production## 1. Deployment Strategies: Что это такое?

На собеседовании важно дать четкое определение: Стратегия деплоя — это технический сценарий переключения трафика со старой версии приложения (v1) на новую (v2).
Главная цель выбора правильной стратегии — минимизировать или полностью исключить Downtime (время простоя сервиса) и снизить риски для пользователей в случае обнаружения критических багов в новом релизе.

##### 2. Глубокий разбор стратегий развертывания## Recreate (Пересоздание)

Самая простая, но агрессивная стратегия.

* Как работает: Все старые поды/контейнеры с версией v1 полностью останавливаются (down), и только после их полной остановки запускаются новые контейнеры v2.
* Плюсы: Идеально, если новая версия приложения принципиально не может работать одновременно со старой версией (например, из-за несовместимых изменений в схеме БД). Нет накладных расходов на ресурсы.
* Минусы: Гарантированный Downtime. В промежутке времени, пока v1 выключился, а v2 еще загружается (прогревает кэш, стартует фреймворк), пользователи будут получать ошибку 502 Bad Gateway.

##### Rolling Deployment (Поэтапное обновление)

Стандартная стратегия по умолчанию в Kubernetes.

* Как работает: Контейнеры обновляются постепенно, один за другим. Балансировщик трафика плавно перенаправляет пользователей.
* Допустим, у нас есть 3 контейнера v1.
   * Запускается первый контейнер v2. Как только он успешно проходит Readiness Probe (проверку готовности), балансировщик пускает на него часть трафика, а один контейнер v1 уничтожается.
   * Процесс повторяется, пока все контейнеры не станут v2.
* Плюсы: Zero-Downtime. Сайт доступен непрерывно. Не требует двойного объема серверов/ресурсов (нужен лишь небольшой запас для запуска одного нового контейнера).
* Минусы: В процессе деплоя в системе одновременно живут две версии кода (v1 и v2). Если v2 пишет в БД новые поля, а v1 про них ничего не знает, приложение может упасть. Код должен быть строго обратно совместим.

##### Blue-Green Deployment (Сине-зеленый деплой)

Стратегия с полным дублированием инфраструктуры.

* Как работает: В рантайме существуют два абсолютно идентичных продакшн-контура. Контур Blue активен и принимает 100% реального трафика пользователей (v1). Контур Green — это копия, куда деплоится новая версия v2. На Green прогоняются тесты и smoke-проверки. Если всё работает идеально, балансировщик (Nginx, HAProxy, Route53) мгновенно (за одну транзакцию) переключает весь трафик с Blue на Green. Blue контур становится резервным для следующего релиза.
* Плюсы: Абсолютный Zero-Downtime и мгновенный откат (Rollback). Если в v2 обнаружился баг, балансировщик за секунду переключает трафик обратно на Blue.
* Минусы: Двойная стоимость инфраструктуры (нужно содержать в 2 раза больше серверов/баз).

##### Canary Deployment (Канареечный деплой)

Выкатка изменений на ограниченную аудиторию. Название пошло от канареек, которых шахтеры брали под землю для детекции газа.

* Как работает: Новая версия v2 выкатывается на минимальное количество серверов (например, 1 из 10). Балансировщик настраивается так, чтобы направлять туда всего 1–5% случайных пользователей (или пользователей из определенного региона/тест-группы). Разработчики мониторят логи, метрики (Prometheus/Grafana) и бизнес-показатели (Sentry, количество покупок). Если аномалий и всплеска 500-х ошибок нет, процент трафика на v2 постепенно увеличивают до 100%.
* Плюсы: Максимальная безопасность. Если в коде критический баг, пострадает лишь крошечный процент пользователей, а не весь бизнес.
* Минусы: Сложная настройка балансировки и мониторинга. Также требует параллельного сосуществования v1 и v2 в одной БД.

##### 3. Database Migrations в Production

Самый популярный вопрос на собеседовании для Middle+/Senior: «Как выкатывать миграции на продакшене без простоя, если у нас Rolling или Canary деплой, и одновременно работают старый и новый код?»

##### Главное правило Production-миграций

Никогда не делать Breaking Changes (несовместимых изменений) за один релиз. Изменение схемы БД должно быть растянуто на несколько этапов (шагов деплоя). Этот подход называется Expand and Contract (Расширение и Сужение).

##### Кейс 1: Удаление колонки (или переименование)

Если вы просто удалите колонку phone из таблицы, старый код (v1), который еще работает в соседнем контейнере во время Rolling-деплоя, упадет с ошибкой при первой же попытке сделать SELECT * или INSERT.
Правильный алгоритм (3 релиза):

   1. Релиз №1 (Expand): Выкатываем код, который перестает использовать колонку phone (все обращения к ней в Laravel-моделях удалены). Сама колонка в БД остается нетронутой. Ждем, пока деплой завершится и v1 полностью исчезнет из системы.
   2. Релиз №2: Пишем отдельную миграцию, которая удаляет колонку (или делает ее nullable). Теперь это безопасно, так как активный код v2 к ней уже не обращается.

##### Кейс 2: Изменение типа данных или разделение колонки (например, name -> first_name + last_name)

Правильный алгоритм (4 релиза):

   1. Релиз №1 (Expand): Создаем новые колонки first_name и last_name. Код v2 пишется так, чтобы записывать данные и в старую колонку, и в новые одновременно (Dual Writing), но читать пока только из старой name.
   2. Релиз №2 (Data Migration): Запускаем консольный скрипт (фоновое консольное задание), которое аккуратно, небольшими пачками (chunks), чтобы не заблокировать таблицу, переносит старые исторические данные из name в новые поля.
   3. Релиз №3: Переключаем код приложения на чтение из новых колонок first_name / last_name. Запись в name прекращается.
   4. Релиз №4 (Contract): Удаляем старую колонку name из базы данных.

##### Специфика Laravel на Production:

* Команда `php artisan migrate --force` должна запускаться строго до того, как новые контейнеры начнут принимать трафик (в CI/CD пайплайне или на этапе инициализации первого нового контейнера). Флаг --force обязателен, чтобы подавить интерактивные вопросы в консоли.
* При работе с большими таблицами (миллионы строк) стандартные миграции Laravel (`$table->string('...')->default(...)`) могут наглухо заблокировать таблицу на запись (Table Lock) на несколько минут. На собеседовании блесните знанием концепции Online Schema Change и инструментов вроде пакета shuttle или утилит от Percona (pt-online-schema-change), которые меняют структуру таблицы в фоне через триггеры, не мешая пользователям работать с сайтом.

##### Мониторинг, Логирование и Observability в Enterprise-приложениях

Вот развернутый теоретический материал с глубокой технической фактурой для ответов на собеседованиях уровня Middle/Senior. Этот блок покажет интервьерам, что вы умеете не просто писать код, но и поддерживать его стабильность в production-средах (навыки Observability).

------------------------------

##### 1. Monitoring vs Logging: В чем принципиальная разница?

На собеседовании важно четко разделить эти два понятия, которые часто путают:

* Logging (Логирование): Отвечает на вопрос «Что конкретно произошло и почему?». Это текстовые или структурированные записи о конкретных дискретных событиях в системе (например, «Пользователь X не смог оплатить заказ, так как API банка вернул 402»). Логи дискретны, детальны и объемны.
* Monitoring (Мониторинг): Отвечает на вопрос «Как система чувствует себя прямо сейчас и есть ли проблема?». Это непрерывный сбор числовых метрик в реальном времени (например, «Загрузка CPU — 85%», «Количество 500-х ошибок за минуту — 42»). Мониторинг агрегирован и служит триггером для алертов.

------------------------------

##### 2. Глубокое погружение в Логирование## Laravel Logging

Laravel под капотом использует мощную библиотеку Monolog. Вся конфигурация живет в config/logging.php.

* Каналы (Channels): Laravel поддерживает запись в разные каналы (single, daily с ротацией файлов, syslog, slack, stderr).
* Фактура для Senior-ответа: На продакшене в Docker-окружении категорически нельзя писать логи в файлы (single или daily). На это есть две причины:
1. Контейнеры эфемерны — при перезапуске контейнера файлы логов сотрутся.
   2. Философия Docker (12-Factor App) требует писать логи строго в стандартный поток вывода (stdout / stderr). Поэтому в продакшн .env всегда выставляется LOG_CHANNEL=stderr. Дальше эти логи подхватывает сам Docker Daemon.

##### Structured Logging (Структурированное логирование)

Классический лог — это сырой текст: `[2026-06-05] production.ERROR: User 145 failed to buy product 99`. Такой лог невозможно эффективно фильтровать и анализировать на миллионах запросов.

* Что это: Это запись логов в машиночитаемом формате, почти всегда в JSON.
* Пример в Laravel: Использование JSON-форматтера для Monolog.
* Почему это критично: В JSON каждое свойство становится отдельным индексируемым полем. Вы можете мгновенно сделать запрос в поисковый движок: «Найди мне все логи с уровнем ERROR, где user_id = 145 и environment = production». В Laravel это реализуется через передачу контекста:

```php

Log::error('Order payment failed', [
    'user_id' => $user->id,
    'order_id' => $order->id,
    'error_code' => $e->getCode()
]);

```

##### Centralized Logging (Централизованное логирование)

Когда у вас 10 контейнеров с PHP на 3 разных серверах, заходить по SSH на каждый сервер и читать логи через tail -f физически невозможно. Для этого выстраивают шину централизованного сбора логов.

* Классический стек (ELK / EFK): Elasticsearch/OpenSearch (база данных для полнотекстового поиска логов), Logstash/Fluentd/Vector (агенты-сборщики, которые парсят логи из Docker и отправляют их в базу), Kibana (веб-интерфейс для графиков и поиска).
* Современная альтернатива: Grafana Loki (более легковесный стек, который индексирует только метаданные логов, экономя терабайты дискового пространства) или облачные решения (Datadog, New Relic).

##### 3. Метрики, Мониторинг и APM## Metrics и Monitoring (Сбор числовых данных)

Метрики — это числовые показатели, которые собираются через определенные интервалы времени (Time-Series данные).

* Индустриальный стандарт: Стек Prometheus (база данных для метрик) + Grafana (визуализация, дашборды).
* Архитектура сбора: Prometheus работает по модели Pull. Он раз в N секунд сам опрашивает ваши сервисы по специальному эндпоинту (например, https://my-app.com). Так как PHP-FPM умирает после каждого запроса и не может хранить метрики в памяти, в Laravel для интеграции с Prometheus используют промежуточный демон Redis или Prometheus Pushgateway.
* Ключевые метрики для Laravel (Four Golden Signals):
1. Latency (Задержка): Время ответа сервера (в миллисекундах).
   2. Traffic (Трафик): Количество запросов в секунду (RPS).
   3. Errors (Ошибки): Рейт 5xx и 4xx ответов.
   4. Saturation (Насыщенность): Насколько заполнена память, CPU, пулы соединений с БД.

##### Application Performance Monitoring (APM)

APM — это «высшая лига» мониторинга. Обычные метрики показывают, что сервер тормозит. APM показывает, какая конкретно строчка кода или какой SQL-запрос вызывает тормоза.

* Популярные инструменты: Datadog, New Relic, OpenTelemetry (открытый стандарт), Сентри (Sentry — больше для ошибок, но теперь и APM).
* Как это работает (Трейсинг): APM внедряется как расширение в PHP (например, через расширение dd-trace или Sentry SDK). Он замеряет время выполнения каждой операции внутри одного HTTP-запроса:
* Контроллер стартовал: 0ms
   * Запрос SELECT * FROM users WHERE...: 120ms 🔴 (Бутылочное горлышко!)
   * Запрос в Redis: 2ms
   * Рендеринг ответа: 10ms
* Фишка (Distributed Tracing): Если ваш Laravel вызывает сторонний микросервис на Go, APM прокидывает специальный заголовок X-Trace-Id сквозь все системы. Вы видите сквозной граф вызовов одного запроса по всей сети вашей компании.

##### 4. Health Checks, Uptime и Alerting## Health Checks (Проверки работоспособности)

Оркестраторы (Kubernetes) или балансировщики (Nginx/AWS ELB) должны знать, «жив» ли контейнер, чтобы не отправлять на него клиентов. Для этого создается роут GET /health или /up (в Laravel 11 он встроен по умолчанию).

* `Liveness Probe`: Жив ли процесс? Если /health не отвечает, оркестратор убьет и перезапустит контейнер.
* `Readiness Probe`: Готов ли контейнер принимать трафик?
* `Фактура для ответа`: Мало просто вернуть 200 OK. Настоящий production-healthcheck внутри кода должен проверить: есть ли успешное соединение с БД, доступен ли Redis, не переполнена ли очередь задач. Если база «легла», healthcheck должен вернуть 503 Service Unavailable, и балансировщик временно исключит этот сервер из раздачи трафика.

##### Uptime Monitoring

Внешний независимый мониторинг (Pingdom, UptimeRobot, Better Stack), который с периодичностью раз в 1-5 минут делает запросы к вашему сайту из разных точек мира. Его задача — зафиксировать полную недоступность сервиса (например, если упал DNS, просрочился SSL-сертификат или упал дата-центр).

##### Alerting (Оповещение о проблемах)

Система алертинга (например, Prometheus Alertmanager, встроенные алерты в Grafana или Sentry) агрегирует аномалии и отправляет уведомления дежурным инженерам (в Slack, Telegram, PagerDuty, Opsgenie).

* Правило хорошего тона (Anti-pattern "Alert Fatigue"): Нельзя настраивать оповещения на каждое мелкое событие (например, «CPU подскочил до 91% на 5 секунд» или «произошла одна 500-я ошибка»). Это приводит к выгоранию от алертов — инженеры начинают их игнорировать.
* Правильный подход: Алерты должны настраиваться на бизнес-метрики и пользовательский опыт (SLA/SLO). Сирена должна включаться, только если:
* Рейт 500-х ошибок превысил 1% от всего трафика за последние 5 минут.
   * Среднее время ответа (p99 latency) выросло выше 2 секунд.
   * В очереди Laravel скопилось более 10 000 необработанных задач (завис воркер).
