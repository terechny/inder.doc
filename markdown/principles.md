
## KISS, DRY, YAGNI

Эти три концепции составляют фундамент чистого, поддерживаемого и расширяемого кода. Их главная цель — борьба с главной проблемой разработки: неконтролируемым ростом сложности.

------------------------------
##### 1. KISS (Keep It Simple, Stupid)## Что это такое

Принцип предписывает делать код, архитектуру и технические решения максимально простыми и понятными. Код должен быть написан так, чтобы джуниор-разработчик или вы сами через полгода могли мгновенно понять его логику без чтения километров документации. Сложный код сложнее тестировать, оптимизировать и поддерживать.

##### Примеры KISS

* Антипаттерн: Использование сложных тернарных операторов, вложенных друг в друга, или тяжелых регулярных выражений там, где достаточно обычного if/else или простых строковых функций.
* Пример кода (До): Попытка написать «красивый» однострочник.

```php
// Слишком заумно, сложно читать при отладке
$status = ($user->age > 18) ? (($user->balance > 100) ? 'premium' : 'active') : 'restricted';
```

* Пример кода (После): Простая и понятная структура.

```php
if ($user->age <= 18) {
    return 'restricted';
}

return ($user->balance > 100) ? 'premium' : 'active';
```

------------------------------
##### 2. DRY (Don't Repeat Yourself)## Что это такое

Принцип запрещает дублирование знаний, логики или бизнес-правил в системе. Каждая часть информации или бизнес-логики должна иметь единственное, однозначное и авторитетное представление в коде. Дублирование приводит к тому, что при изменении одного правила разработчику приходится искать и править его в десяти разных местах, что неизбежно ведет к багам.

##### Примеры DRY

* Антипаттерн: Копирование кода валидации номера телефона или формулы расчета налога в пяти разных контроллерах.
* Пример кода (До): Дублирование расчета цены со скидкой.

```php

// В CartController.php
$finalPrice = $item->price - ($item->price * ($user->discount / 100));

// В OrderController.php
$total = $product->price - ($product->price * ($user->discount / 100));
```
* Пример кода (После): Вынос логики в единственное место (в модель или сервис).

```php
// В модели Product.php / Item.php
public function getPriceForUser(User $user): float 
{
    return $this->price * (1 - $user->discount / 100);
}

```

------------------------------
##### 3. YAGNI (You Aren't Gonna Need It)## Что это такое

Принцип переводится как "Вам это не понадобится". Он запрещает писать код, создавать классы, добавлять фичи или закладывать архитектурные абстракции на основе предположений в духе: "А вдруг в будущем нам это пригодится?". Программируйте только то, что требуется прямо сейчас по техническому заданию.

##### Примеры YAGNI

* Антипаттерн: Создание сложной системы плагинов, кастомных движков уведомлений или пяти уровней абстракции (интерфейсов и репозиториев) для простой CRUD-таблицы, у которой по ТЗ никогда не изменится источник данных.
* Пример (До): Потрачено 3 дня на написание интерфейсов, фабрик и драйверов для поддержки SMS-провайдеров Twilio, Infobip и SMS.ru, хотя бизнес сейчас работает только с Twilio.
* Пример (После): Написан один простой сервис TwilioService. Когда через год бизнес решит сменить провайдера, код будет отрефакторен под паттерн «Стратегия».

------------------------------
##### 4. Когда намеренно нарушать эти принципы

Инженерия — это всегда компромисс (trade-off). Слепое следование догмам часто вредит проекту.

##### Когда нарушать DRY

   1. Случайное дублирование (Accidental Duplication): Если два куска кода сейчас выглядят одинаково, но представляют разные бизнес-концепции, их нельзя объединять. Например, валидация возраста для покупки алкоголя и валидация возраста для регистрации на авиарейс. Объединив их в один метод ради DRY, вы сломаете систему, когда изменятся правила только для одной из этих сфер.
   2. Изоляция микросервисов: Попытка сделать одну общую библиотеку кода для всех микросервисов ради DRY создает жесткую связность (Tight Coupling). Изменение в библиотеке может сломать все сервисы сразу. В микросервисах часто проще скопировать простую структуру данных (DTO), сохраняя автономность команд.

##### Когда нарушать KISS

   1. Оптимизация производительности: Написание высокоэффективных алгоритмов, работа со сложными структурами данных (B-Tree, графы) или битовыми масками часто выглядит зубодробительно и сложно. Но если это узкое горло системы, простоту приносят в жертву производительности, закрывая код подробными комментариями и юнит-тестами.

------------------------------
##### 5. Комбинация принципов (Связь между ними)

В реальной разработке эти три принципа работают в синергии, балансируя друг друга.

* YAGNI защищает от нарушения KISS: Разработчики часто усложняют код (нарушают KISS), пытаясь заложить гибкость на будущее (нарушая YAGNI). Помня о YAGNI, вы автоматически сохраняете код простым (KISS).
* Слепой DRY может убить KISS: Если вы попытаетесь вынести в абстрактный класс абсолютно все мелкие совпадения в коде, вы получите огромную, запутанную иерархию наследования. Код станет настолько сложным и хрупким, что нарушит принцип KISS. В среде разработчиков есть правило: "Лучше небольшое дублирование, чем плохая абстракция" (A little duplication is far cheaper than the wrong abstraction).
* Идеальный баланс: Пишите код максимально просто под текущую задачу (KISS + YAGNI). Если в процессе реализации или последующих задач вы видите явное дублирование бизнес-логики в трех и более местах — проведите рефакторинг и примените DRY.

## GRASP (General Responsibility Assignment Software Patterns)

GRASP — это набор из 9 фундаментальных принципов, которые помогают разработчику правильно распределять обязанности (отвечать на вопросы: «Какой класс должен выполнять этот метод?» или «Кто должен создавать этот объект?») между классами и объектами при объектно-ориентированном проектировании.
В отличие от SOLID (которые являются более строгими архитектурными правилами), GRASP — это скорее карточки-подсказки для ментального моделирования связей между объектами.

------------------------------
##### 1. Information Expert (Информационный эксперт)

* Суть: Обязанность должна быть возложена на тот класс, который владеет максимумом информации, необходимой для её выполнения.
* Применение в Laravel: Модели Eloquent. Модель Order (Заказ) содержит в себе коллекцию связанных элементов OrderItems. Метод подсчета полной стоимости заказа должен находиться внутри модели Order, так как она является экспертом по своим данным.
* Пример кода:

```php
class Order extends Model {
    public function items() { return $this->hasMany(OrderItem::class); }

    // Класс сам считает сумму, так как знает всё о своих айтемах
    public function getTotalPrice(): float {
        return $this->items->sum(fn($item) => $item->price * $item->quantity);
    }
}
```

##### 2. Creator (Создатель)

* Суть: Определяет, какой класс должен создавать экземпляры другого класса. Класс A должен создавать класс B, если: A содержит/агрегирует B, A записывает B, A тесно использует B или имеет данные для инициализации B.
* Применение в Laravel: Отношения Eloquent. Создание связанных записей через родительскую модель (например, $user->posts()->create(...)). Также фабрики (Post::factory()) инкапсулируют логику создания объектов.

##### 3. Controller (Контроллер)

* Суть: Первый не-UI объект, который принимает системное событие от интерфейса и координирует выполнение бизнес-логики, не выполняя саму работу напрямую.
* Применение в Laravel: Стандартные HTTP-контроллеры (app/Http/Controllers). Они получают HTTP-запрос, валидируют его и делегируют работу Сервисам, Моделям или Очередям, возвращая клиенту ответ.

##### 4. Low Coupling (Слабая связанность)

* Суть: Классы должны зависеть друг от друга как можно меньше. Это делает код изолированным: изменение в одном классе не ломает другие классы, а сами компоненты легко тестировать.
* Применение в Laravel: Использование Интерфейсов и Service Container. Вместо привязки к конкретной реализации платежного шлюза StripeService, контроллер зависит от интерфейса PaymentGatewayInterface.

##### 5. High Cohesion (Высокое зацепление / Сплоченность)

* Суть: Класс должен быть сфокусирован на выполнении одной узкой задачи и иметь мало методов. Если класс делает «всё подряд» (например, и логирует, и шлет почту, и считает налоги), у него низкое зацепление (это плохо). Резонирует с принципом SRP из SOLID.
* Применение в Laravel: Form Requests (php artisan make:request). Вынос логики валидации входящих HTTP-данных из контроллера в отдельный специализированный класс позволяет сохранить контроллер сплоченным.

##### 6. Polymorphism (Полиморфизм)

* Суть: Если поведение системы зависит от типа объекта, ответственность за выбор поведения возлагается на сам объект с помощью полиморфных операций (интерфейсов/наследования), избавляя код от громоздких конструкций if/else или switch.
* Применение в Laravel: Полиморфные связи Eloquent (Polymorphic Relations). Например, таблица comments может принадлежать как к Post, так и к Video. Laravel на основе полиморфизма автоматически определяет, какую связь подгрузить через колонки commentable_type и commentable_id.

##### 7. Pure Fabrication (Чистая выдумка)

* Суть: Если возложение обязанности на Информационного Эксперта нарушает Low Coupling или High Cohesion, нужно придумать искусственный класс (которого нет в предметной области бизнеса), чтобы спасти чистоту архитектуры.
* Применение в Laravel: Сервисные классы (app/Services), Репозитории (app/Repositories), Джобы (app/Jobs). В реальном мире нет сущности «Загрузчик файлов», но мы создаем FileUploaderService, чтобы разгрузить модель User от несвойственной ей технической логики.

##### 8. Indirection (Посредник)

* Суть: Чтобы избежать прямой связи между двумя компонентами (Low Coupling), между ними создается промежуточный объект-посредник.
* Применение в Laravel: Middlewares (Посредники) и Events/Listeners. Вместо того чтобы контроллер напрямую вызывал код проверки авторизации или код отправки аналитики, эти задачи берет на себя Middleware, стоящий между роутом и контроллером, или события (event(new UserRegistered($user))), изолирующие компоненты друг от друга.

##### 9. Protected Variations (Защищенные изменения)

* Суть: Защита системы от изменений в ее внешних элементах или зависимостях путем создания стабильного интерфейса (точки перегиба), который изолирует остальную систему от возможных потрясений и рефакторинга в нестабильном модуле.
* Применение в Laravel: Слои абстракции над сторонними библиотеками. Система Socialite предоставляет единый стабильный интерфейс для авторизации через соцсети. Если Facebook завтра радикально обновит свой API, изменится только внутренний драйвер Laravel Socialite, а код вашего контроллера останется нетронутым.

------------------------------
##### 💡 Практический пример: Архитектурный синергизм в Laravel

Представим задачу: Пользователь совершает покупку на сайте. Посмотрим, как паттерны GRASP работают вместе в рамках одного процесса в Laravel:

```php

// 3. CONTROLLER: Принимает запрос, но сам ничего не считает и не отправляет
class OrderController extends Controller 
{
    // 4. LOW COUPLING: Зависим от абстракции интерфейса, а не от конкретного банка
    public function __construct(
        protected PaymentGatewayInterface $paymentGateway 
    ) {}

    // 5. HIGH COHESION: Валидация унесена в CheckoutRequest, контроллер чист
    public function checkout(CheckoutRequest $request, CartService $cart) 
    {
        $user = auth()->user();
        
        // 7. PURE FABRICATION: Создаем сервисный класс для проведения бизнес-транзакции
        $orderProcessor = app(OrderProcessorService::class);
        $order = $orderProcessor->createFromCart($user, $cart);

        // 6. POLYMORPHISM: СУБД/Eloquent под капотом вызовет нужный шлюз (Stripe/PayPal)
        $this->paymentGateway->charge($order->getTotalPrice());

        // 8. INDIRECTION: Извещаем систему через событие. Контроллер не знает, кто и как шлет письма
        event(new OrderPlaced($order));

        return response()->json(['status' => 'success']);
    }
}

```
