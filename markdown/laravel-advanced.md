
# Laravel Продвинутое

## Eloquent Relationships

##### ⛓️ Типы связей (Eloquent Relationships)

Eloquent позволяет связывать таблицы через простые методы в классах моделей. Главное правило: имя метода должно отражать количество связанных записей (в единственном или множественном числе).

Типы связей:

- One to One (1:1) — hasOne / belongsTo
- One to Many (1:N) — hasMany / belongsTo
- Many to Many (N:N) — belongsToMany
- Has Many Through — hasManyThrough
- Polymorphic — morphTo / morphMany


* Один к одному (One to One): Связь, где одной записи в таблице А соответствует ровно одна запись в таблице Б.
* `hasOne()` — объявляется в главной модели (например, у User есть один Profile).
   * `belongsTo()` — объявляется в зависимой модели, которая физически содержит внешний ключ user_id (у Profile есть владелец User).
* Один ко многим (One to Many): Самый частый тип связи. Одной записи соответствует несколько записей из другой таблицы.
* `hasMany()` — объявляется в главной модели (например, у Post есть много Comment).
   * `belongsTo()` — объявляется в модели комментария (каждый Comment принадлежит конкретному Post).
* Многие ко многим (Many to Many): Связь, требующая промежуточную таблицу (pivot). Например, у User может быть много Role, а у Role — много пользователей.
* `belongsToMany()` — объявляется в обеих моделях. Laravel под капотом ищет промежуточную таблицу role_user.
* Полиморфные связи (Polymorphic): Позволяют одной модели принадлежать более чем к одной другой модели с помощью одного набора полей. Например, модель Comment может принадлежать одновременно и к Post, и к Video. Используются методы `morphTo()`, `morphMany()` и другие.

##### ⚡ Eager Loading (Жадная загрузка)

По умолчанию Eloquent использует Lazy Loading (Ленивую загрузку). Это значит, что связанные данные не запрашиваются из базы до тех пор, пока вы явно к ним не обратитесь.
Проблема N+1: Если вы выберете 100 постов, а затем в цикле начнете выводить автора каждого поста (`$post->user->name`), Eloquent сделает 101 запрос к базе данных (1 запрос на получение постов + 100 запросов на каждого автора). Это намертво положит сервер при росте трафика.
Решение — метод `with()`: Eager Loading решает эту проблему, загружая все связанные данные сразу с помощью всего 2 запросов, используя SQL-оператор IN.

```php

// Проблема N+1 решена. Будет выполнено всего 2 SQL-запроса:
// 1. SELECT * FROM posts;
// 2. SELECT * FROM users WHERE id IN (1, 2, 3...);
$posts = Post::with('user')->get();

foreach ($posts as $post) {
    echo $post->user->name; // Данные берутся уже из памяти, без новых запросов!
}

// Можно загружать несколько связей или вложенные связи:
$posts = Post::with(['user', 'comments.author'])->get();

```
##### 🔍 Has / WhereHas запросы (Фильтрация по связям)

Иногда нужно выбрать записи, основываясь на факте наличия или условиях в их связях.

* `has('relationship')` — выбирает записи, у которых просто есть указанная связь.
* `whereHas('relationship', callback)` — позволяет наложить дополнительные условия SQL на связанные таблицы через анонимную функцию.

```php

// Выбрать только те посты, у которых есть хотя бы один комментарий
$posts = Post::has('comments')->get();

// Выбрать посты, у которых есть комментарии, содержащие слово "PHP"
$posts = Post::whereHas('comments', function ($query) {
    $query->where('text', 'like', '%PHP%');
})->get();

// Противоположные методы (выбрать записи, у которых НЕТ связей):
$posts = Post::doesntHave('comments')->get();
$posts = Post::whereDoesntHave('comments', function ($query) { ... });

```
##### 📊 WithCount (Подсчет связанных записей)

Если вам нужно просто узнать количество связанных элементов (например, сколько комментариев у каждого поста), использовать `with('comments')` и затем считать их через `count()` в PHP — грубая ошибка, так как вы загрузите в память тысячи ненужных объектов.
Метод `withCount()` делает эффективный подзапрос `COUNT(*)` прямо на уровне базы данных. Он автоматически добавляет к вашей модели новое свойство в формате `{relation}_count`.

```php

// Делает один оптимальный SQL-запрос с COUNT
$posts = Post::withCount('comments')->get();

foreach ($posts as $post) {
    // Доступно динамическое свойство comments_count
    echo "Пост: {$post->title}, Комментариев: {$post->comments_count}";
}

// Можно комбинировать с условиями с помощью массива:
$posts = Post::withCount(['comments as approved_comments_count' => function ($query) {
    $query->where('is_approved', true);
}])->get();

```

## Query Builder

Вот средне-краткий разбор инструмента Query Builder (Конструктор запросов) в Laravel.

##### ⚙️ Query Builder. Что это

Query Builder (Конструктор запросов) — это встроенный в Laravel низкоуровневый инструмент, который предоставляет удобный, объектно-ориентированный интерфейс для создания и выполнения SQL-запросов к базе данных.
В отличие от Eloquent, Query Builder работает не с объектами моделей, а с «сырыми» данными (возвращает массивы или коллекции стандартных PHP-объектов stdClass). Под капотом он использует механизм PDO binding, что автоматически защищает ваше приложение от SQL-инъекций. Доступ к нему осуществляется через фасад DB.


##### 🔍 Базовые запросы

Позволяют гибко выбирать данные из базы с помощью цепочки методов:

* Выбор таблицы: `DB::table('users')` задает целевую таблицу.
* Условия (`where`): Фильтруют записи. Можно строить цепочки или передавать массивы условий.
* Сортировка и лимиты: Настраиваются методами orderBy, skip (смещение) и take (лимит).
* Получение результата: Метод `get()` возвращает коллекцию, а `first()` — одну первую запись.

```php

// Выбрать имя и почту активных пользователей, отсортированных по дате
$users = DB::table('users')
    ->select('name', 'email')
    ->where('status', 'active')
    ->where('age', '>', 18)
    ->orderBy('created_at', 'desc')
    ->limit(10)
    ->get();

```

```php 

use Illuminate\Support\Facades\DB;

// SELECT
$users = DB::table('users')->get();
$user = DB::table('users')->where('id', 1)->first();
$email = DB::table('users')->where('id', 1)->value('email');

// SELECT с условиями
$users = DB::table('users')
    ->where('active', true)
    ->where('age', '>', 18)
    ->get();

// OR условие
$users = DB::table('users')
    ->where('role', 'admin')
    ->orWhere('role', 'moderator')
    ->get();

// WHERE IN
$users = DB::table('users')
    ->whereIn('id', [1, 2, 3])
    ->get();

// WHERE BETWEEN
$users = DB::table('users')
    ->whereBetween('age', [18, 65])
    ->get();

// WHERE NULL
$users = DB::table('users')
    ->whereNull('deleted_at')
    ->get();

// WHERE LIKE
$users = DB::table('users')
    ->where('name', 'like', '%John%')
    ->get();

```

##### 💾 INSERT, UPDATE, DELETE

Операции модификации данных выполняются максимально лаконично:

* `insert()` — принимает массив (или массив массивов для множественной вставки) полей и значений. Метод `insertGetId()` дополнительно возвращает ID созданной записи.
* `update()` — обновляет указанные поля у записей, попавших под условие where.
* `delete()` — удаляет строки из таблицы на основе условий. Для полной очистки таблицы используют `truncate()`.

```php

// Вставка
DB::table('users')->insert([
    'email' => 'kaylee@example.com',
    'votes' => 0
]);

// Обновление
DB::table('users')
    ->where('id', 1)
    ->update(['votes' => 1]);

// Удаление
DB::table('users')
    ->where('votes', '<', 10)
    ->delete();

```

##### ⛓️ JOIN (Объединение таблиц)

Query Builder позволяет связывать таблицы с помощью классических SQL-соединений: join (INNER JOIN), leftJoin, rightJoin и crossJoin. В аргументах передается имя связываемой таблицы, а затем условия, по которым они соединяются.

```php

$users = DB::table('users')
    ->join('contacts', 'users.id', '=', 'contacts.user_id')
    ->join('orders', 'users.id', '=', 'orders.user_id')
    ->select('users.name', 'contacts.phone', 'orders.price')
    ->get();
```

##### 📊 Aggregates (Агрегатные функции)

Конструктор запросов содержит прямые методы для выполнения математических и статистических вычислений прямо на стороне сервера базы данных. Они возвращают не коллекции, а конкретное числовое значение:

* `count()` — подсчет количества строк.
* `max()` / min() — поиск максимального или минимального значения в колонке.
* `avg()` — вычисление среднего арифметического.
* `sum()` — подсчет суммы всех значений.

```php

$count = DB::table('users')->where('status', 'active')->count();

$maxPrice = DB::table('orders')->max('price');

```

##### 🛡️ Транзакции (Transactions)

Транзакции гарантируют атомарность данных: либо вся группа запросов выполнится успешно, либо (если произойдет сбой) база данных откатится к исходному состоянию, будто ничего не происходило. Это критически важно, например, при переводе денег с одного счета на другой.
В Laravel транзакции можно запускать двумя способами:
1. Автоматический (через замыкание): Если внутри блока произойдет исключение (Exception), Laravel сам сделает откат (Rollback).

```php

DB::transaction(function () {
    DB::table('users')->update(['votes' => 1]);
    DB::table('posts')->delete();
});

2. Ручной (для полного контроля):

DB::beginTransaction(); // Старт

try {
    DB::table('accounts')->where('id', 1)->decrement('balance', 100);
    DB::table('accounts')->where('id', 2)->increment('balance', 100);
    
    DB::commit(); // Всё прошло успешно, сохраняем изменения
} catch (\Exception $e) {
    DB::rollBack(); // Ошибка! Отменяем всё, что успело сделаться
}

```

##### 🎯 Когда использовать Query Builder

Поскольку в Laravel есть более продвинутый Eloquent ORM, использовать Query Builder нужно осознанно.

##### Выбирайте Query Builder, если:

* Важна максимальная производительность: Eloquent тратит много ресурсов и памяти на создание объектов моделей для каждой строки базы. Для тяжелых выборок, отчетов или аналитики Query Builder работает в разы быстрее и потребляет меньше RAM.
* Сложные SQL-запросы: Большое количество вложенных JOIN, группировок (groupBy), подзапросов или использование специфических функций конкретной БД (например, PostgreSQL или MySQL).
* Массовая обработка данных: Импорт, экспорт или миграция сотен тысяч записей, где не нужна сложная логика моделей (события, геттеры/сеттеры).

##### Когда лучше остаться на Eloquent ORM:

* Для классического CRUD (создание, чтение, редактирование, удаление записей).
* Когда проекту необходимы удобные связи между таблицами (`with`, `hasMany` и т.д.).
* Если вы активно используете глобальные/локальные области видимости (Scopes), наблюдатели (Observers) и мутаторы полей.


## Migrations & Seeders


##### ⚙️ Migrations & Seeders. Что это

* Миграции (Migrations) — это система контроля версий для вашей базы данных. Вместо того чтобы вручную создавать таблицы через графические интерфейсы (phpMyAdmin, DataGrip) или делиться SQL-дампами, структура описывается в виде PHP-кода. Это позволяет развернуть базу данных со всей структурой на любом компьютере одной командой в консоли.
* Сидеры (Seeders) — это инструмент для наполнения базы данных начальными или тестовыми данными. Они автоматизируют заполнение таблиц информацией сразу после того, как миграции создали структуру.


##### 🔨 Создание миграций

Миграции создаются через консоль Artisan:

php artisan make:migration create_posts_table

Файл миграции появляется в папке `database/migrations/`. Класс миграции содержит два обязательных метода:

* `up()` — описывает, какие таблицы и колонки нужно создать или изменить [].
* `down()` — описывает, как полностью отменить действия метода up() (обычно это удаление таблицы) [].

```PHP

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration {
    public function up(): void {
        Schema::create('posts', function (Blueprint $table) {
            $table->id(); // Первичный ключ auto_increment
            $table->string('title'); // VARCHAR
            $table->timestamps(); // Создает колонки created_at и updated_at
        });
    }

    public function down(): void {
        Schema::dropIfExists('posts');
    }
};

```

Главные консольные команды:

* `php artisan migrate` — запускает все новые миграции [].
* `php artisan migrate:rollback` — отменяет последнюю партию миграций [].
* `php artisan migrate:fresh` — полностью удаляет все таблицы и запускает все миграции с чистого листа (осторожно, стирает данные!).

##### 📋 Типы колонок

Внутри замыкания `Schema::create` объект `$table` предоставляет десятки методов для создания любых типов данных SQL. К ним можно применять модификаторы (например, `nullable()` или `default()`):

* `$table->string('name', 100)`; — строка VARCHAR (можно указать длину).
* `$table->text('description')`; — длинный текст TEXT.
* `$table->integer('votes')`; — целое число INT.
* `$table->bigInteger('count')`; — большое число BIGINT.
* `$table->boolean('is_active')->default(true)`; — логический тип TINYINT(1) со значением по умолчанию.
* `$table->decimal('price', 8, 2)`; — точное число с плавающей точкой (8 знаков всего, 2 после запятой).
* `$table->json('options')->nullable()`; — колонка JSON (может оставаться пустой).

```php

Schema::create('products', function (Blueprint $table) {
    // Numeric
    $table->id();  // BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
    $table->bigInteger('views');
    $table->integer('stock');
    $table->tinyInteger('status');
    $table->decimal('price', 8, 2);  // 8 знаков, 2 после запятой
    $table->float('rating', 8, 2);

    // String
    $table->string('name', 255);  // VARCHAR(255)
    $table->text('description');  // TEXT
    $table->longText('content');  // LONGTEXT

    // Date/Time
    $table->date('birth_date');  // DATE
    $table->dateTime('published_at');  // DATETIME
    $table->timestamp('verified_at');  // TIMESTAMP
    $table->timestamps();  // created_at, updated_at
    $table->softDeletes();  // deleted_at

    // Boolean
    $table->boolean('is_active')->default(true);

    // JSON
    $table->json('metadata');

    // Foreign Key
    $table->foreignId('user_id')->constrained()->onDelete('cascade');

    // Enum
    $table->enum('status', ['pending', 'approved', 'rejected']);
});

```

Модификаторы колонок:

```php

$table->string('email')->nullable();  // NULL
$table->string('role')->default('user');  // DEFAULT
$table->string('name')->unique();  // UNIQUE
$table->string('description')->comment('Product description');  // COMMENT
$table->integer('order')->unsigned();  // UNSIGNED
$table->timestamp('created_at')->useCurrent();  // DEFAULT CURRENT_TIMESTAMP
$table->timestamp('updated_at')->useCurrentOnUpdate();  // ON UPDATE CURRENT_TIMESTAMP

```

Индексы:

```php

Schema::table('users', function (Blueprint $table) {
    $table->string('email')->unique();  // UNIQUE
    $table->index('email');  // INDEX
    $table->index(['user_id', 'created_at']);  // Composite index

    // Именованный индекс
    $table->index('email', 'idx_users_email');

    // Удалить индекс
    $table->dropIndex('idx_users_email');
    $table->dropUnique(['email']);
});

```

Выполнение миграций:

```bash

# Выполнить все миграции
php artisan migrate

# Откатить последний batch
php artisan migrate:rollback

# Откатить все миграции
php artisan migrate:reset

# Откатить и заново выполнить
php artisan migrate:refresh

# Откатить, выполнить и засеять
php artisan migrate:refresh --seed

# Удалить все таблицы и заново создать
php artisan migrate:fresh

# Статус миграций
php artisan migrate:status

```

##### ⛓️ Foreign Keys (Внешние ключи)

Внешние ключи связывают таблицы между собой и обеспечивают целостность данных на уровне СУБД. В Laravel есть два способа их объявления:

1. Современный и лаконичный (рекомендуемый):

Метод `foreignId()` создает колонку типа BIGINT UNSIGNED, а метод `constrained()` автоматически понимает, что она связана с таблицей users по полю id.

`$table->foreignId('user_id')->constrained()->onDelete('cascade')`;

2. Классический способ:

`$table->unsignedBigInteger('user_id')`;
`$table->foreign('user_id')->references('id')->on('users')->onDelete('cascade')`;

Модификатор onDelete('cascade') (каскадное удаление) означает: если удалить пользователя из таблицы users, база данных автоматически удалит все связанные с ним посты.

##### 🌱 Seeders (Сидеры)

Сидеры используются для вызова логики наполнения БД. Создаются командой `php artisan make:seeder PostSeeder`. Все они лежат в `database/seeders/`.
Главный файл `DatabaseSeeder.php` запускает все остальные. Внутри метода `run()` вы можете использовать Query Builder или Eloquent для создания записей:

```php

class PostSeeder extends Seeder {
    public function run(): void {
        DB::table('posts')->insert([
            'title' => 'Первый демонстрационный пост',
            'created_at' => now(),
        ]);
    }
}

```

Запуск всех сидеров: `php artisan db:seed`.
Запуск полной очистки БД с последующим сидингом: `php artisan migrate:fresh --seed`.


##### 🏭 Factories (Фабрики)

Фабрики переводят генерацию тестовых данных на новый уровень, используя библиотеку Faker для создания реалистичных случайных имён, адресов, текстов и дат. Фабрика привязывается к конкретной модели Eloquent.
Создание фабрики: `php artisan make:factory PostFactory`. Они лежат в `database/factories/`.

```php
namespace Database\Factories;

use Illuminate\Database\Eloquent\Factories\Factory;

class PostFactory extends Factory {
    public function definition(): array {
        return [
            'title' => $this->faker->sentence(5), // Случайное предложение из 5 слов
            'body' => $this->faker->paragraph(3),   // 3 случайных абзаца текста
            'user_id' => \App\Models\User::factory(), // Автоматически создаст и привяжет пользователя!
        ];
    }
}
```

Использование фабрик внутри Сидеров:
Вместо ручного прописывания массивов, вы можете генерировать данные тысячами одной строкой кода:

```php
// Создать 50 случайных постов в базе данных
Post::factory()->count(50)->create();
```

##### 🎯 Когда использовать

* Миграции: Используйте всегда для любого изменения структуры БД (создание таблиц, добавление новой колонки, удаление индекса). Никогда не меняйте базу данных вручную на "живом" сервере — это приведет к рассинхронизации структуры у вашей команды.
* Сидеры: Используйте для наполнения базы справочниками, константами и системными данными, без которых сайт физически не сможет работать на продакшене. Примеры: список стран, валют, языков сайта или первая учетная запись супер-администратора.
* Фабрики + Тестовые Сидеры: Используйте исключительно на этапе локальной разработки и тестирования. Они незаменимы, чтобы мгновенно наполнить пустую копию сайта 10 000 товарами и проверить, как работает пагинация, поиск, фильтры и не падает ли сервер от нагрузки.


## Events & Listeners


Это механизм, реализующий классический паттерн проектирования «Наблюдатель» (Observer), который помогает добиться слабой связанности (loose coupling) кода.

* Событие (Event) — это простой объект-информатор (DTO). Он просто фиксирует факт: "В системе что-то произошло" (например, UserRegistered, OrderPaid) и хранит сопутствующие данные (например, объект пользователя или заказа). Сам по себе ивент никакой логики не выполняет.
* Слушатель (Listener) — это класс, который «ждет» возникновения конкретного события и выполняет реакцию на него. На одно событие можно повесить сколько угодно слушателей.

Зачем это нужно: Вместо того чтобы в контроллере заказа писать код отправки СМС, начисления бонусов, генерации чека и отправки CRM-аналитики (что превратит код в кашу), вы просто генерируете одно событие: «Заказ оплачен». Контроллер мгновенно возвращает ответ клиенту, а все остальные задачи изолированно выполняются своими слушателями.


##### 🔨 Создание Events / Listeners

В современных версиях Laravel проще всего создавать их парой через консоль:

`php artisan make:event OrderPaid`
`php artisan make:listener SendOrderInvoice --event=OrderPaid`

Класс события (`app/Events/OrderPaid.php`) содержит только конструктор:

```php

class OrderPaid
{
    use Dispatchable, InteractsWithSockets, SerializesModels;

    // Передаем модель, с которой связано событие
    public function __construct(public Order $order) {}
}

```

Класс слушателя (`app/Listeners/SendOrderInvoice.php`) содержит метод `handle()`, куда аргументом автоматически передается объект события:

```php

class SendOrderInvoice
{
    public function handle(OrderPaid $event): void
    {
        // Доступ к данным заказа через объект события
        $order = $event->order;
        // Логика отправки инвойса...
    }
}

```

#####  📝 Регистрация

Начиная с Laravel 11, фреймворк поддерживает автоматическое обнаружение событий (Event Discovery). Если ваши классы лежат в стандартных папках, Laravel сам сопоставит слушателей с событиями на основе тайпхинта в методе `handle()`.
Если вам нужна ручная регистрация, её делают в файле `app/Providers/AppServiceProvider.php`` внутри метода `boot()` с помощью фасада `Event`:

```php

use App\Events\OrderPaid;
use App\Listeners\SendOrderInvoice;
use App\Listeners\UpdateBonusAccount;
use Illuminate\Support\Facades\Event;

public function boot(): void
{
    Event::listen(
        OrderPaid::class,
        SendOrderInvoice::class
    );
    
    // Можно вешать несколько слушателей на одно событие
    Event::listen(OrderPaid::class, UpdateBonusAccount::class);
}

```

##### 🚀 Dispatch событий (Вызов)

Чтобы запустить событие в коде (в контроллере, сервисе или консольной команде), используется статический метод `dispatch()` самого класса события или глобальный хелпер `event()`.

```php

public function checkout(Request $request)
{
    $order = Order::create($request->all());

    // Вызываем событие. Все привязанные слушатели сработают автоматически.
    OrderPaid::dispatch($order);

    return response()->json(['message' => 'Order processed!']);
}

```

##### 🔄 Model Events / Observers (События моделей)

Модели Eloquent имеют собственный встроенный жизненный цикл и генерируют события при совершении любых действий с базой данных: retrieved, creating, created, updating, updated, saving, saved, deleting, deleted.

##### Обсерверы (Observers)

Если вам нужно сгруппировать логику реакций на события конкретной модели, создается класс-наблюдатель:

`php artisan make:observer UserObserver --model=User`

Он появляется в `app/Observers/UserObserver.php` и содержит готовые методы:


```php
class UserObserver
{
    // Сработает ДО сохранения в базу данных
    public function creating(User $user): void
    {
        $user->uuid = Str::uuid(); // Генерируем UUID автоматически
    }

    // Сработает ПОСЛЕ создания записи в базе данных
    public function created(User $user): void
    {
        // Отправляем приветственное письмо
    }
}
```

Регистрация обсервера в AppServiceProvider.php:

```php

public function boot(): void
{
    User::observe(UserObserver::class);
}

```

##### ⏳ Queued Listeners (Очереди для слушателей)

По умолчанию события и слушатели выполняются синхронно (в рамках того же HTTP-запроса пользователя). Если слушатель отправляет email через внешнее API, пользователь будет покорно ждать 3-5 секунд, пока загружается страница.
Чтобы сделать слушатель асинхронным, достаточно добавить к его классу интерфейс ShouldQueue []. Laravel сам поймет это, перехватит выполнение метода handle(), упакует его в задачу и отправит в фоновую очередь (Redis, Database и др.) []. Пользователь получит ответ мгновенно, а тяжелая задача выполнится в фоне [].

```php
use Illuminate\Contracts\Queue\ShouldQueue;

// Просто добавляем implements ShouldQueue, и магия работает
class SendOrderInvoice implements ShouldQueue
{
    // Можно настроить параметры очереди прямо здесь
    public $connection = 'redis';
    public $queue = 'emails';

    public function handle(OrderPaid $event): void { ... }
}
```

##### 🎯 Когда использовать## Выбирайте связку Events / Listeners, если:

* Действие не влияет на текущий ответ пользователю: Отправка писем, СМС, пуш-уведомлений, выгрузка данных в CRM, пересчет статистики.
* Нужно соблюдать принцип Single Responsibility (SRP): Метод контроллера или сервиса должен делать только одну задачу (например, регистрировать пользователя). Всё остальное должно быть вынесено в слушатели.
* Вам нужно легко включать/отключать функционал: Например, если временно нужно отключить начисление кэшбэка при оплате, вы просто комментируете одну строку регистрации слушателя, не трогая основной код корзины.

##### Выбирайте Model Events / Observers, если:

* Логика жестко привязана к состоянию данных в БД: Автоматическая генерация слагов (slug) для URL из названия статьи перед сохранением, очистка связанных файлов на диске (аватарок) при удалении профиля пользователя, логирование истории изменений полей модели (аудит).


## Jobs & Queues


##### ⚙️ Laravel Jobs & Queues. Что это

* Задача (Job) — это изолированный класс, содержащий конкретный кусок бизнес-логики, который можно выполнить отложено. По сути, это инструкция: «что именно нужно сделать» (например, сгенерировать тяжелый Excel-отчет, обработать видео или сделать массовую рассылку).
* Очередь (Queue) — это буфер или «лист ожидания» (обычно развернутый в Redis, базе данных или SQS), куда Laravel складывает сериализованные (упакованные) объекты задач.
* Воркер (Worker) — это фоновый PHP-процесс, который постоянно запущен на сервере. Он непрерывно опрашивает очередь, достает оттуда задачи по одной и выполняет их в фоновом режиме.

##### 🔄 Как это работает (Жизненный цикл)

Работа с очередями в Laravel состоит из четырех основных шагов:

   1. Настройка драйвера: В файле конфигурации или `.env` указывается, где именно хранить задачи. На локальном компьютере часто используют драйвер `database` (задачи пишутся в таблицу), а в продакшене — сверхбыстрый `redis`.
   2. Создание задачи: Через консоль создается класс задачи: `php artisan make:job ProcessPodcast`. Он автоматически получает интерфейс ShouldQueue. Внутри метода handle() пишется вся тяжелая логика.
   3. Отправка в очередь (Dispatch): В коде контроллера вместо выполнения тяжелого действия вызывается метод `dispatch()`. Laravel берет переданные данные, превращает их в строку (сериализует) и мгновенно сохраняет в хранилище очереди. Контроллер сразу же продолжает работу и выдает ответ пользователю.
   4. Фоновое выполнение: На сервере работает команда `php artisan queue:work` (воркер). Она перехватывает задачу из Redis/БД, восстанавливает объект (десериализует) и запускает метод handle(). Пользователь в этот момент уже давно закрыл страницу или пользуется сайтом дальше.

```php

// 1. Так выглядит вызов задачи в контроллере:
public function storeReport(Request $request) {
    // Отправляем задачу в очередь и мгновенно возвращаем ответ
    GenerateReportJob::dispatch($request->user(), $request->date_range);

    return response()->json(['message' => 'Отчет генерируется, мы пришлем его на почту!']);
}

// 2. А так выглядит сама задача (app/Jobs/GenerateReportJob.php):
class GenerateReportJob implements ShouldQueue {
    use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;

    public function __construct(public User $user, public array $range) {}

    public function handle(ReportService $service): void {
        // Здесь выполняется тяжелый процесс, который может идти 5 минут
        $pdf = $service->build($this->range);
        $this->user->notify(new ReportReadyNotification($pdf));
    }
}

```

##### 🎯 Когда использовать

Очереди необходимы везде, где действие занимает больше нескольких десятков миллисекунд, чтобы интерфейс сайта оставался моментальным для пользователя.

##### Идеальные сценарии для использования Jobs & Queues:

* Интеграция со сторонними API (Сеть): Отправка СМС через шлюзы, пуш-уведомлений, взаимодействие со службами доставки, CRM-системами или платежными шлюзами. Если чужой сервер «зависнет» на 10 секунд, ваш сайт не упадет, так как воркер подождет в фоне, а при сбое — автоматически попробует выполнить задачу позже (tries).
* Тяжелая обработка файлов: Изменение размера картинок при загрузке, обрезка видео (FFmpeg), генерация больших PDF или Excel документов, парсинг гигантских CSV-логов.
* Массовые операции: Рассылка еженедельного дайджеста по базе из 50 000 подписчиков, начисление кэшбэка всем клиентам в конце месяца, синхронизация каталога товаров с остатками на складе.
* Цепочки сложных зависимых процессов: С помощью встроенных инструментов Laravel (Bus::chain, Bus::batch) можно выстраивать логику: «Сначала выполни эти 10 задач параллельно, а когда они завершатся — запусти финальную задачу-отчет».

##### Когда использовать НЕ нужно:

* Если результат действия нужен пользователю прямо сейчас на этой же странице для продолжения работы. Например, проверка правильности пароля при авторизации или получение текущей корзины товаров. Такие вещи должны обрабатываться строго синхронно.

## Notifications

Notifications (Уведомления) — это встроенный в Laravel инструмент для отправки коротких информационных сообщений пользователям через различные каналы связи.
Главная фишка этого компонента — абстракция от каналов доставки. Вы описываете логику уведомления (например, «Ваш заказ отправлен») в одном единственном классе, но отправить его можете одновременно на Email, по СМС, в Telegram, в Slack и сохранить в базу данных на сайте. Если завтра вам понадобится добавить новый канал связи, вам не придется переписывать логику приложения — достаточно будет дописать один метод в класс уведомления.


##### 🔄 Как это работает

Система состоит из трех ключевых элементов: класс Уведомления, модель Получателя и Каналы.

##### 1. Подготовка модели (Получателя)

Чтобы модель (например, User) могла получать уведомления, к ней нужно добавить встроенный трейт Notifiable. Он добавляет модели методы для отправки (например, `$user->notify()`

```php
use Illuminate\Notifications\Notifiable;

class User extends Authenticatable {
    use Notifiable; // Магия уведомлений активирована
}
```

##### 2. Создание класса уведомления

Класс создается командой `php artisan make:notification OrderShipped` и сохраняется в папку `app/Notifications/`.
Внутри класса есть метод `via()`, который определяет, какими каналами слать сообщение, и отдельные методы для настройки каждого канала (`toMail()`, `toDatabase()`, `toTelegram()` и т.д.):

```php

class OrderShipped extends Notification 
{
    use Queueable;

    public function __construct(public Order $order) {}

    // Определяем каналы доставки (можно динамически на основе настроек юзера)
    public function via(object $notifiable): array {
        return ['mail', 'database']; 
    }

    // Формируем красивое HTML-письмо
    public function toMail(object $notifiable): MailMessage {
        return (new MailMessage)
            ->subject('Ваш заказ отправлен!')
            ->line('Уважаемый ' . $notifiable->name)
            ->line('Ваш заказ #' . $this->order->id . ' уже в пути.')
            ->action('Отследить заказ', url('/orders/' . $this->order->id));
    }

    // Формируем массив для сохранения уведомления в БД сайта
    public function toArray(object $notifiable): array {
        return [
            'order_id' => $this->order->id,
            'message' => 'Заказ успешно отправлен со склада.'
        ];
    }
}

## 3. Отправка уведомления
Отправить уведомление можно конкретному пользователю или сразу целой коллекции (через Фасад):

// Вариант 1: Точечно пользователю
$user->notify(new OrderShipped($order));

// Вариант 2: Массово через Фасад Notification
Notification::send($usersCollection, new OrderShipped($order));

```

##### ⏳ Продвинутая фишка: Уведомления в Очередь

Отправка email через внешние SMTP-серверы или запросы к API СМС-провайдеров занимают время. Чтобы пользователь не ждал загрузки страницы, отправку уведомлений можно легко перенести в фон.
Для этого достаточно добавить интерфейс ShouldQueue к классу вашего уведомления []:

```php
use Illuminate\Contracts\Queue\ShouldQueue;

class OrderShipped extends Notification implements ShouldQueue {
    use Queueable;
    // ...
}
```

Laravel автоматически превратит отправку этого уведомления в фоновую задачу (Job) и передаст её воркеру.


##### 🎯 Когда использовать

Компонент Notifications идеален для транзакционных и сервисных сообщений, завязанных на действия конкретных пользователей.

##### Идеальные сценарии:

* Системные действия: Ссылка на сброс пароля, подтверждение email при регистрации, одноразовый код для двухфакторной аутентификации (2FA).
* Статусы процессов: «Ваш счет оплачен», «Товар снова в наличии», «Заявка принята в работу».
* Интерактивность (Социальные фичи): Оповещения в колокольчик на сайте (через канал database): «Пользователь Иван прокомментировал ваш пост», «Вас упомянули в чате».
* Уведомления для команды (Логи/Алерты): Отправка важных технических алертов в рабочий Slack или Telegram-канал разработчиков, например: «База данных переполнена» или «Получена крупная оплата от VIP-клиента».

##### Когда использовать НЕ нужно:

* Маркетинговые рассылки: Для отправки рекламных писем или спама по базе в 100 000 человек Notifications не подходят. Для массового email-маркетинга лучше использовать специализированные сервисы (Mailchimp, SendPulse) и специализированные пакеты, так как управление отписками, аналитика открываемости и верстка сложных писем выходят за рамки этого компонента.


## API Resources

API Resources — это специализированный слой в Laravel, который выполняет роль трансформера данных между вашими моделями Eloquent и JSON-ответами, которые ожидает клиент (фронтенд-приложение, мобильное приложение или сторонний интегратор).
По сути, это реализация паттерна Presenter или Data Transformer для создания REST API.
Зачем они нужны: Напрямую возвращать модель из контроллера в формате return $user; — грубая ошибка. Во-первых, вы рискуете случайно слить наружу конфиденциальные данные (например, password_hash или remember_token). Во-вторых, формат данных в базе (например, is_active как 1 или 0) часто не совпадает с тем, что хочет видеть фронтенд (true/false). API-ресурсы изолируют структуру базы данных от структуры API.

##### 🔄 Как это работает

В Laravel есть два основных типа ресурсов: JsonResource (для одного объекта/модели) и ResourceCollection (для списка объектов/коллекций).

##### 1. Создание ресурса

Ресурс создается через консоль Artisan:

`php artisan make:resource UserResource`

Файл создается в папке `app/Http/Resources/UserResource.php`. Внутри метода `toArray()` вы вручную описываете карту JSON-ответа. Для обращения к свойствам модели используется `$this`:

```php

namespace App\Http\Resources;

use Illuminate\Http\Request;
use Illuminate\Http\Resources\Json\JsonResource;

class UserResource extends JsonResource
{
    public function toArray(Request $request): array
    {
        return [
            'id' => $this->id,
            'name' => $this->name,
            'email' => $this->email,
            // Трансформируем данные: объединяем строки
            'full_address' => "{$this->city}, {$this->street}",
            // Приводим к правильному типу данных
            'is_verified' => (bool) $this->email_verified_at,
            // Форматируем даты в понятный для фронтенда вид
            'registered_at' => $this->created_at->toIso8601String(),
        ];
    }
}

```

##### 2. Использование в Контроллере

В контроллере вы просто оборачиваете модель или коллекцию моделей в созданный ресурс. Laravel автоматически применит трансформацию и выставит JSON-заголовки:

```php

// Вариант 1: Возврат одного объекта
public function show(User $user)
{
    return new UserResource($user);
}

// Вариант 2: Возврат списка объектов (используется статический метод collection())
public function index()
{
    $users = User::paginate(10);
    
    // Магия: метод collection автоматически сохранит пагинацию (links, meta) в JSON!
    return UserResource::collection($users);
}

```

##### ⚡ Продвинутые фичи API-ресурсов## 1. Условное включение связей (Conditional Relationships)

Одна из лучших фич. Если вы хотите, чтобы ресурс выводил связанные комментарии поста только тогда, когда они были предварительно загружены через Eager Loading (with('comments')), используйте метод whenLoaded(). Это полностью защищает проект от проблемы N+1 внутри ресурсов.

```php

return [
    'id' => $this->id,
    'title' => $this->title,
    // Появится в JSON только если была вызвана жадная загрузка ->with('comments')
    'comments' => CommentResource::collection($this->whenLoaded('comments')),
];

```

##### 2. Условное включение атрибутов (Conditional Attributes)

Вы можете скрывать или показывать конкретные поля в зависимости от условий (например, от прав текущего авторизованного пользователя).

```php

return [
    'id' => $this->id,
    'name' => $this->name,
    // Поле секретного ключа увидят только администраторы
    'secret_token' => $this->when($request->user()?->isAdmin(), $this->token),
];

```

##### 🎯 Когда использовать## Выбирайте API Resources, если:

* Вы разрабатываете бэкенд исключительно как REST API для SPA-приложений (Vue, React), мобильных приложений (iOS/Android) или публичного API для внешних разработчиков.
* Вам нужно жестко контролировать и стандартизировать структуру JSON (например, оборачивать все ответы в единый корень 'data' => [...]).
* Формат хранения данных в БД отличается от формата, удобного клиенту (нужно менять названия ключей, мапить статусы, переводить таймстампы).

##### Когда использовать НЕ нужно:

* Если вы пишете классический монолит на шаблонизаторе Blade или используете Livewire. В этом случае данные передаются напрямую в HTML-шаблоны, и слой API-ресурсов просто не задействован.
* Если вы используете связку Laravel + Inertia.js. В Inertia контроллеры возвращают данные напрямую в JS-компоненты (React/Vue). Для фильтрации полей в Inertia обычно передают обычные массивы через метод only() или специализированные DTO.



## Validation


Валидация в Laravel — это процесс проверки соответствия входящих данных (из POST/GET запросов, API или файлов) заданным бизнес-правилам. 
Если данные проходят проверку, скрипт выполняется дальше. Если данные не проходят валидацию:

* В контексте обычных веб-страниц: Laravel автоматически перенаправляет пользователя назад на предыдущую страницу, сохраняя старый ввод в сессии и генерируя переменную ошибок $errors для вывода в HTML.
* В контексте API: Laravel мгновенно прерывает запрос и возвращает JSON-ответ со статусом 422 Unprocessable Entity и подробным массивом ошибок. 


##### 📥 Способы написания валидации

Laravel предоставляет три основных способа валидации данных.

##### 1. Валидация прямо в Контроллере (Для простых форм)

Используется метод validate() объекта $request. Самый быстрый, но наименее чистый способ, так как он засоряет код контроллера.

```php

public function store(Request $request)
{
    // Если правила нарушены, код ниже не выполнится
    $validatedData = $request->validate([
        'title' => 'required|unique:posts|max:255',
        'body' => 'required|string|min:10',
    ]);

    // В $validatedData лежат только проверенные поля
    Post::create($validatedData);
}

```

##### 2. Выделенные классы Form Requests (Лучшая практика) 

Вся логика валидации и проверки прав переносится в отдельный изолированный класс. Создается командой: `php artisan make:request StorePostRequest`. 

Класс создается в папке `app/Http/Requests/` and содержит два метода:

```php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;

class StorePostRequest extends FormRequest
{
    // 1. Проверяем, имеет ли право текущий юзер совершать это действие
    public function authorize(): bool
    {
        return $this->user()->can('create-posts');
    }

    // 2. Описываем массив правил
    public function rules(): array
    {
        return [
            'email' => 'required|email|unique:users,email',
            'password' => 'required|string|min:8|confirmed', // Ищет поле password_confirmation
            'avatar' => 'nullable|image|max:2048', // Картинка до 2 МБ
        ];
    }
}

```

В контроллере достаточно указать этот класс в качестве тайпхинта. Валидация сработает автоматически до выполнения тела метода:

```
public function store(StorePostRequest $request)
{
    // Сюда код попадет только если данные идеальны
    User::create($request->validated()); 
}
```

##### 3. Ручная валидация через Фасад Validator

Используется, если нужно провалидировать произвольный массив данных (например, информацию, пришедшую из стороннего API или консольной команды).

```php

use Illuminate\Support\Facades\Validator;

$validator = Validator::make($dataArray, [
    'sku' => 'required|alpha_num',
]);

if ($validator->fails()) {
    // Логика обработки ошибок вручную
}

```

##### 📋 Популярные правила валидации (Validation Rules) 

Laravel содержит более 90 встроенных правил. Их можно писать через вертикальную черту | или в виде элементов массива (рекомендуется для сложных правил). 

* required / nullable — поле обязательно к заполнению / поле может быть пустым.
* email — проверка на корректный формат адреса электронной почты.
* unique:table,column — проверка, что такого значения еще нет в указанной таблице базы данных.
* exists:table,column — проверка, что переданный ID (или другое поле) физически существует в базе данных.
* confirmed — требует, чтобы в запросе было парное поле со словом _confirmation (используется для паролей).
* image / mimes:pdf,doc — проверка файлов на формат изображения или конкретные расширения.
* sometimes — запускает валидацию поля только в том случае, если это поле вообще присутствует во входящем массиве данных (идеально для PATCH-обновлений). [23, 24, 25, 26, 27] 

##### 🛠️ Кастомизация сообщений и кастомные правила## Изменение текста ошибок

Вы можете переписать стандартные английские сообщения на свои в методе `messages()` внутри вашего Form Request: 

```php

public function messages(): array
{
    return [
        'email.required' => 'Нам обязательно нужен ваш email адрес!',
        'email.unique' => 'Пользователь с таким email уже зарегистрирован.',
    ];
}

```

##### Создание собственных правил (Custom Rules)

Если стандартных правил не хватает (например, нужна сложная проверка ИНН или номера телефона), создается кастомное правило: php artisan make:rule Uppercase. [30, 31, 32] 
В сгенерированном классе в методе validate() пишется логика:

```php

public function validate(string $attribute, mixed $value, Closure $fail): void
{
    if (strtoupper($value) !== $value) {
        $fail('Поле :attribute должно быть написано заглавными буквами.');
    }
}

```
------------------------------

##### 🎯 Когда использовать

* Всегда используйте валидацию для любых данных, приходящих извне (от пользователей, фронтенда, мобильных приложений или внешних вебхуков). Главное правило веб-разработки: «Никогда не доверяй входящим данным».
* Используйте Form Requests для 95% задач в контроллерах. Это разгружает код, делает его тестируемым и позволяет повторно использовать правила (например, одни и те же правила для создания и редактирования товара).
* Используйте Фасад Validator только в сервисах, консольных командах (Artisan commands) или фоновых задачах (Jobs), где нет прямого HTTP-объекта $request.
