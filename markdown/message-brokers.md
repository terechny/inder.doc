## RabbitMQ

[RabbitMQ](https://www.rabbitmq.com/) — это распределенный брокер сообщений (Message Broker) с открытым исходным кодом, который используется для организации асинхронной коммуникации, сглаживания пиковых нагрузок (Rate Limiting) и построения слабосвязанных микросервисных архитектур. В отличие от Redis, который является In-Memory хранилищем, RabbitMQ изначально спроектирован по спецификации промышленного сетевого протокола AMQP 0-9-1 (Advanced Message Queuing Protocol).

------------------------------
##### 1. Базовая архитектура и компоненты

В RabbitMQ отправитель никогда не отправляет сообщение напрямую в очередь. Архитектура состоит из четырех сущностей:

```
[ Producer ] --(Сообщение + Routing Key)--> [ Exchange ]
                                                |
                                        (Правила / Binding)
                                                |
                                                v
[ Consumer ] <--- (Подписка / Fetch) --- [ Queue ]
```

* Producer (Издатель): Приложение, которое генерирует и отправляет сообщение.
* Exchange (Точка обмена): Маршрутизатор. Принимает сообщения от Producer и на основе правил (Bindings) распределяет их по очередям.
* Queue (Очередь): Буфер, хранящий сообщения в памяти или на диске до тех пор, пока их не заберет воркер.
* Consumer (Потребитель / Воркер): Приложение, которое подписывается на очередь, забирает сообщения и обрабатывает их.

------------------------------
##### 2. Типы Точек Обмена (Exchanges)

Способ, которым Exchange распределяет сообщения в очереди, зависит от его типа:

   1. Direct Exchange (Прямой): Направляет сообщение в очереди на основе строгого совпадения ключа маршрутизации (Routing Key) и ключа связывания (Binding Key). Идеален для точечной отправки задач (например, routing_key: "orders.create").
   2. Fanout Exchange (Вещательный): Игнорирует Routing Key. Дублирует и рассылает полученное сообщение во все очереди, которые к нему привязаны. Используется для паттерна Publish/Subscribe (например, уведомление всех сервисов о том, что товар обновился).
   3. Topic Exchange (Тематический): Маршрутизирует сообщения на основе шаблонов (паттернов) совпадения Routing Key. Ключ состоит из слов, разделенных точками (например, eu.orders.shipping). Поддерживает маски:
   * * (звездочка) — заменяет ровно один уровень (слово). Шаблон *.orders.* поймает eu.orders.shipping.
      * `#` (решетка) — заменяет ноль или более уровней (слов). Шаблон eu.# поймает и eu.orders, и eu.orders.shipping.delivery.
   4. Headers Exchange (Заголовки): Игнорирует Routing Key. Маршрутизация строится на основе сопоставления аргументов (заголовков) в словаре headers, передаваемом вместе с сообщением.

##### 3. Практические примеры реализации

##### Вариант А: Чистый PHP (без фреймворков) с использованием официальной библиотеки php-amqplib

##### 1. Код Отправителя (producer.php)

```php

require_once __once 'vendor/autoload.php';
use PhpAmqpLib\Connection\AMQPStreamConnection;
use PhpAmqpLib\Message\AMQPMessage;

// 1. Устанавливаем TCP-соединение
$connection = new AMQPStreamConnection('localhost', 5672, 'guest', 'guest');
$channel = $connection->channel();

// 2. Объявляем Direct Exchange и Очередь
$channel->exchange_declare('sms_exchange', 'direct', false, true, false);
$channel->queue_declare('sms_queue', false, true, false, false);
$channel->queue_bind('sms_queue', 'sms_exchange', 'send_sms');

// 3. Создаем сообщение с флагом персистентности (сохранение на диск)
$payload = json_encode(['phone' => '79991112233', 'text' => 'Код: 1234']);
$msg = new AMQPMessage($payload, ['delivery_mode' => AMQPMessage::DELIVERY_MODE_PERSISTENT]);

// 4. Публикуем
$channel->basic_publish($msg, 'sms_exchange', 'send_sms');

echo " [x] Сообщение отправлено\n";
$channel->close();
$connection->close();

```

##### 2. Код Обработчика (consumer.php)

```php

require_once __once 'vendor/autoload.php';
use PhpAmqpLib\Connection\AMQPStreamConnection;

$connection = new AMQPStreamConnection('localhost', 5672, 'guest', 'guest');
$channel = $connection->channel();

$channel->queue_declare('sms_queue', false, true, false, false);

echo " [*] Ожидание сообщений. Нажмите CTRL+C для выхода\n";

$callback = function ($msg) {
    $data = json_decode($msg->body, true);
    echo " [x] Отправляем СМС на номер: " . $data['phone'] . "\n";
    
    // Ручное подтверждение успешной обработки
    $msg->ack();
};

// basic_qos задает prefetch_count (не давать воркеру больше 1 задачи одновременно)
$channel->basic_qos(null, 1, false);
$channel->basic_consume('sms_queue', '', false, false, false, false, $callback);

while ($channel->is_consuming()) {
    $channel->wait();
}

```

##### 4. Гарантии доставки и отказоустойчивость

RabbitMQ предоставляет инструменты для реализации модели At-least-once (Доставка как минимум один раз):

* Acks (Acknowledgements / Подтверждения):
* Auto-Ack: Сообщение удаляется из очереди сразу, как только RabbitMQ передал его воркеру по TCP-сокету. Если воркер упадет в процессе выполнения (Out of Memory), сообщение будет потеряно безвозвратно.
   * Manual-Ack (Рекомендуется): Воркер обрабатывает задачу и явно шлет ответ $msg->ack(). Если воркер умирает без отправки подтверждения, RabbitMQ возвращает сообщение в очередь (nack или reject с флагом requeue=true), и его забирает другой воркер.
* Durable Queues & Persistent Messages: Чтобы сообщения не пропали при перезагрузке самого сервера RabbitMQ, очередь должна быть объявлена как durable = true, а сообщения отправляться с флагом delivery_mode = 2 (Persistent) для принудительного логирования на диск.
* Publisher Confirms: Механизм, при котором RabbitMQ шлет асинхронный ответ издателю (Producer) о том, что сообщение успешно принято и записано на диск.

------------------------------
##### 5. Продвинутые фичи (DLX, Priority)

##### Dead Letter Exchange (DLX / Точка обмена "мертвых" писем)

Если сообщение не может быть обработано воркером (например, упало с ошибкой бизнес-логики 5 раз подряд, или у него истекло время жизни TTL), его нельзя бесконечно возвращать в основную очередь (requeue), иначе зациклится вся система.

* Решение: Очередь настраивается с аргументом x-dead-letter-exchange. "Протухшие" или отклоненные через `$msg->reject(false)` сообщения автоматически пересылаются в этот вспомогательный Exchange, а оттуда — в специальную очередь failed_jobs_queue для ручного разбора разработчиками.

##### Priority Queues (Очереди с приоритетом)

Очередь объявляется с аргументом x-max-priority. При публикации сообщения в заголовках указывается его приоритет (например, от 1 до 10). Сообщения с приоритетом 10 будут продвигаться в начало очереди и обрабатываться воркерами первыми, даже если они поступили позже остальных (важно для разделения VIP и бесплатных пользователей).

------------------------------
##### 6. Мониторинг и Best Practices

##### Инструменты мониторинга

* [RabbitMQ Management Plugin](https://www.rabbitmq.com/docs/management): Встроенная веб-панель (по умолчанию на порту 15672), показывающая графики RPS, количество активных воркеров, скорость накопления сообщений и объемы свободной памяти ноды.
* Prometheus + Grafana: Экспорт метрик (rabbitmq_queue_messages_ready) для настройки алертов (например, если очередь выросла более чем на 5000 задач — слать уведомление в чат команды).

##### Best Practices (Лучшие практики)

* Держите очереди короткими: Очереди в RabbitMQ спроектированы для быстрой прокачки данных. Миллионные залежи непрочитанных сообщений начинают нагружать оперативную память и сбрасываться во временные файлы на диск, критически замедляя брокер.
* Используйте Prefetch Count (basic_qos): Всегда ограничивайте количество сообщений, выдаваемых одному воркеру за раз (ставьте prefetch=1 или prefetch=10). Без этого RabbitMQ отдаст первому свободному воркеру по принципу Round-Robin половину очереди, нагрузив его, пока остальные воркеры будут простаивать.
* Идемпотентность воркеров: В распределенных системах из-за сетевых сбоев подтверждение ack может потеряться, и RabbitMQ пришлет сообщение воркеру повторно. Воркер обязан проверять, обрабатывал ли он уже эту задачу (например, по уникальному UUID транзакции в Redis/БД).

## Apache Kafka

[Apache Kafka](https://kafka.apache.org/) — это распределенная горизонтально масштабируемая платформа потоковой передачи данных (Distributed Streaming Platform). В отличие от классических брокеров сообщений (RabbitMQ), Kafka спроектирована не как буфер задач, а как распределенный, отказоустойчивый и реплицируемый журнал фиксации изменений (Commit Log), способный обрабатывать триллионы событий в секунду и постоянно хранить их на диске.

------------------------------
##### 1. Архитектура: Topics, Partitions и Consumer Groups

В архитектуре Kafka логика хранения данных принципиально отличается от традиционных очередей.

```
[ Producer ] 
     |
     v
[ Topic: orders ] 
     |---> [ Partition 0 ] ( Записи: 0, 1, 2, 3... ) ---> [ Consumer A ] (Группа 1)
     |---> [ Partition 1 ] ( Записи: 0, 1, 2...    ) ---> [ Consumer B ] (Группа 1)
     |---> [ Partition 2 ] ( Записи: 0, 1, 2, 3... ) ---> [ Consumer C ] (Группа 2)
```

##### Основные компоненты

* Topic (Тема): Логический поток сообщений определенного типа (аналог таблицы в БД).
* Partition (Партиция / Раздел): Физический лог-файл на диске одного из серверов кластера (Broker). Каждый топик разделен на одну или несколько партиций.
* Важно: Строгий порядок сообщений гарантируется только в рамках одной партиции, но не всего топика.
* Producer (Издатель): Отправляет сообщения в топик. По умолчанию сообщения распределяются по партициям циклично (Round-Robin), но если передать Message Key, то Kafka хэширует ключ и гарантирует, что все сообщения с одинаковым ключом (например, user_id: 42) всегда попадут в одну и ту же партицию.
* Consumer (Потребитель): Читает сообщения из партиций.
* Consumer Group (Группа потребителей): Объединение воркеров для параллельного чтения. Каждая партиция внутри группы может читаться только одним консьюмером. Если у вас 3 партиции и 4 воркера в одной группе, один воркер будет простаивать. Это обеспечивает масштабируемость чтения.

------------------------------
##### 2. Управление чтением (Offset Management) и Хранение (Retention)

##### Offset (Смещение)

В Kafka сообщения не удаляются из партиции сразу после того, как их прочитал воркер. Каждое сообщение имеет монотонно растущий порядковый номер — Offset.
Воркер сам хранит и двигает указатель своей текущей позиции чтения (коммитит офсет в специальный внутренний топик [Kafka](https://ru.wikipedia.org/wiki/Apache_Kafka)).
Гарантии доставки:

* At-least-once (По умолчанию): Воркер прочитал пачку сообщений, успешно обработал, записал в БД и закоммитил офсет. Если упадет в процессе — перечитает пачку заново.
* Exactly-once (Ровно один раз): Достигается за счет использования транзакционного API Kafka (Kafka Transactions), связывающего запись в Kafka и обновление состояния воркера.

##### Retention (Политика хранения)

Данные хранятся на диске фиксированное время, независимо от того, прочитали их или нет.

* retention.ms: Время хранения (например, 7 дней). По истечении срока Kafka автоматически удаляет старые сегменты файлов.
* retention.bytes: Ограничение по объему (например, удалять старое, если топик раздулся до 50 ГБ).
* Log Compaction (Уплотнение лога): Режим, при котором Kafka удаляет старые дубликаты записей, оставляя в истории только последнее актуальное значение для каждого уникального ключа (идеально для паттерна Event Sourcing).

------------------------------
##### 3. Репликация и отказоустойчивость

Kafka спроектирована для работы в кластере из десятков брокеров. Каждая партиция дублируется на несколько серверов на основе фактора репликации (replication.factor).

* Leader (Лидер): Один из брокеров назначается лидером конкретной партиции. Все операции записи (Produce) и чтения (Consume) идут строго через него.
* Follower (Реплика/Следователь): Остальные брокеры асинхронно копируют логи лидера к себе. Если Лидер падает, кластер автоматически выбирает нового Лидера из числа синхронизированных реплик (ISR — In-Sync Replicas).

------------------------------
##### 4. Практический пример интеграции в Laravel

В PHP экосистеме для работы с Kafka используется расширение php-rdkafka (обертка над производительной Си-библиотекой librdkafka). На уровне [Laravel](https://ru.wikipedia.org/wiki/Laravel) популярностью пользуется пакет mateusjunges/laravel-kafka.

## 2. Обработка потока (Consumer в Laravel Command)
Обычно запускается как долгоживущий демон через консольную команду Laravel:

use Junges\Kafka\Facades\Kafka;

Kafka::consume('localhost:9092')
    ->withConsumerGroupId('analytics_service_group')
    ->subscribe('orders_stream')
    ->withHandler(function(\Junges\Kafka\Contracts\KafkaConsumerMessage $message) {
        $data = $message->getBody();
        // Логика аналитики в реальном времени
        \Log::info("Обработан заказ: " . $data['order_id']);
    })
    ->build()
    ->consume();

```

##### 5. Сравнение: Apache Kafka vs RabbitMQ

| Критерий | RabbitMQ | Apache Kafka |
|---|---|---|
| Архитектурный тип | Классический брокер сообщений (Smart broker, dumb consumer). | Распределенный журнал логов (Dumb broker, smart consumer). |
| Жизненный цикл данных | Сообщение удаляется сразу после подтверждения (ACK). | Сообщения хранятся на диске согласно политике Retention. |
| Модель чтения | Push (Брокер сам выталкивает задачи свободным воркерам). | Pull (Воркеры сами запрашивают пачки данных по своему офсету). |
| Повторное чтение | Невозможно (данные стерты). | Возможен сброс офсета назад для перечитывания истории. |
| Идеальные Use Cases | Фоновые задачи, СМС, биллинг, сложные правила маршрутизации. | Сбор логов, аналитика Big Data, Event Sourcing, телеметрия (IoT). |

------------------------------
##### 6. Основные Use Cases (Сценарии использования)

* Event Streaming (Потоковая передача событий): Связующий слой в событийно-ориентированных микросервисах, где события стримятся непрерывно.
* CDC (Change Data Capture): Стриминг изменений из БД. Инструменты (Debezium) читают бинарный лог репликации PostgreSQL/MySQL и транслируют каждую изменившуюся строку как событие в Kafka.
* Агрегация логов и метрик: Сбор гигабайтных системных логов со всех серверов компании (через Vector или FluentBit) в единую шину Kafka для последующей заливки в ClickHouse или Elasticsearch.

------------------------------
##### 7. Мониторинг и метрики

Для контроля здоровья кластера Kafka отслеживают специфические инфраструктурные метрики:

* Consumer Lag (Отставание потребителя): Главная метрика. Показывает разницу между последним записанным офсетом в партиции и офсетом, который успел прочитать воркер. Если Lag растет — воркеры не справляются с нагрузкой, нужно добавлять новые партиции и масштабировать консьюмеров.
* Under-Replicated Partitions: Количество партиций, у которых упали или не успевают синхронизироваться реплики. Значение больше 0 сигнализирует об аппаратных проблемах с брокерами.
* Инструменты: Prometheus JMX Exporter + Grafana, AKHQ или Confluent Control Center.

## Redis Pub/Sub и Real-time в экосистеме Laravel

Redis Pub/Sub (Publish/Subscribe) — это легковесный поведенческий паттерн обмена сообщениями, встроенный в СУБД Redis. Он работает по принципу Fire-and-Forget (Выстрелил и забыл): издатель (Publisher) отправляет сообщение в именованный канал (Channel), а все подписчики (Subscribers), находящиеся в этот момент в сети (online), мгновенно его получают.
Главная особенность Redis Pub/Sub — абсолютное отсутствие постоянного хранения (No Persistence). Сообщения не сохраняются ни в оперативной памяти, ни на диске. Если подписчик в момент отправки сообщения был оффлайн хотя бы одну миллисекунду, он никогда не узнает об этом сообщении.

------------------------------
##### 1. Основы и механизмы работы

Архитектура паттерна максимально упрощена для обеспечения ультра-низкой задержки (Sub-millisecond latency):

```
[ Publisher ] --( PUBLISH "chat.room1" "Hello" )--> [ Redis Server ]

##### 3. Laravel Broadcasting: Real-time для веб-интерфейсов

В Laravel механизм Broadcasting (Вещание) позволяет транслировать серверные события Laravel (Events) напрямую в клиентский браузер по протоколу WebSockets.

##### 5. Ограничения и Сравнение: Pub/Sub vs Streams

##### Главные ограничения Redis Pub/Sub:

* Нет буферизации: Если воркер или WebSocket-сервер перезагружался во время деплоя, все сообщения, отправленные в этот момент, теряются навсегда.
* Нагрузка на память при медленных клиентах: Если подписчик не успевает вычитывать данные из сокета, Redis начинает накапливать сообщения в буфере вывода конкретного клиента (client-output-buffer-limit). Если лимит превышен, Redis принудительно обрывает соединение с этим подписчиком.

##### Сравнение: Redis Pub/Sub vs Redis Streams

| Критерий | Redis Pub/Sub | Redis Streams |
|---|---|---|
| Хранение данных | Нет (Fire-and-forget). | Да (Append-only лог на диске/в RAM, как в Kafka). |
| Повторное чтение | Невозможно. | Возможно с любого места истории или по ID. |
| Группы потребителей | Нет (сообщение получают все подписчики). | Да (Consumer Groups — балансировка задач между воркерами). |
| Сценарии использования | WebSockets, чаты, живые нотификации UI, инвалидация кэша в ОЗУ. | Очереди задач, финансовые транзакции, логирование, замена RabbitMQ. |

## Сравнение Message Brokers

Выбор правильного инструмента для асинхронного взаимодействия — ключевое решение при проектировании распределенных систем. Разные системы предлагают принципиально разные абстракции данных (очередь, лог или канал вещания), что напрямую влияет на их производительность, надежность и сценарии использования.

------------------------------
##### 1. Сравнительная матрица (Сводный анализ)

| Критерий | Redis Pub/Sub | RabbitMQ | Apache Kafka |
|---|---|---|---|
| Основной тип | Канал вещания (Fire-and-Forget). | Брокер сообщений (Task Queue). | Платформа распределенного лога (Commit Log). |
| Хранение (Persistence) | Нет (Строго в рантайме). | Да (В памяти / На диске, стирается после ACK). | Да (Постоянно на диске согласно политике Retention). |
| Модель чтения | Push (Брокер рассылает всем онлайн). | Push (Брокер выталкивает свободным воркерам). | Pull (Консьюмеры сами вытягивают пачки по своему Offset). |
| Маршрутизация | Простая (по имени канала / маске). | Очень гибкая (Direct, Fanout, Topic, Headers Exchanges). | Жесткая (По топикам и партициям). |
| Повторное чтение | Невозможно. | Невозможно (удаляется после обработки). | Да (Сброс офсета назад на любую глубину истории). |

------------------------------
##### 2. Глубокое сравнение по инженерным критериям## Гарантии доставки (Delivery Guarantees)

* Redis Pub/Sub: At-most-once (Доставка максимум один раз). Нет никаких гарантий и подтверждений. Если подписчик моргнул по сети, сообщение потеряно.
* RabbitMQ: At-least-once (Доставка как минимум один раз) за счет механизмов Manual ACK, Durable Queues и Publisher Confirms. При падении воркера задача вернется в очередь.
* Apache Kafka: At-least-once по умолчанию (через коммит офсетов воркером). Также из коробки поддерживает Exactly-once (Ровно один раз) в рамках транзакционного API (Kafka Transactions), исключая дублирование данных в downstream-системах.

##### Производительность и Пропускная способность (Performance)

* Redis Pub/Sub: Ультра-низкая задержка (Sub-millisecond). Работает со скоростью оперативной памяти. Ограничен мощностью процессора (Redis однопоточен) и объемом RAM для клиентских буферов вывода.
* RabbitMQ: Высокая скорость (~десятки тысяч сообщений в секунду). Скорость сильно деградирует, если очереди начинают расти (накапливаться) и сбрасываться брокером на диск.
* Apache Kafka: Экстремальная пропускная способность (Миллионы событий в секунду). За счет линейной записи на диск (Sequential I/O), использования механизма Zero-Copy (данные из дискового кэша ОС летят в сетевой сокет в обход приложения) и агрессивного батчинга (пакетирования задач).

##### Порядок сообщений (Message Ordering)

* Redis Pub/Sub: Гарантирует порядок доставки, но только для тех, кто прямо сейчас онлайн.
* RabbitMQ: Гарантирует строгий порядок First-In-First-Out (FIFO) только в рамках одного потока (одного воркера). Если на одну очередь подписано 10 параллельных воркеров, порядок обработки гарантировать невозможно из-за разной скорости выполнения задач.
* Apache Kafka: Гарантирует абсолютный строгий порядок в рамках конкретной партиции (Partition). Передавая Message Key (например, order_id), разработчик на 100% уверен, что все события по этому заказу обработаются строго в хронологическом порядке, сколько бы воркеров ни было в системе.

##### Масштабирование (Scaling)

* Redis Pub/Sub: Вертикальное. Горизонтально масштабируется только через кластеризацию Redis (Sharding), дублируя вещание на ноды.
* RabbitMQ: Горизонтальное масштабирование через кластеры (классические очереди или современные отказоустойчивые Quorum Queues). Но масштабирование имеет предел из-за накладных расходов на координацию состояния очередей между нодами кластера.
* Apache Kafka: Практически безграничное горизонтальное масштабирование. Новые брокеры добавляются в кластер на лету, а партиции топиков перераспределяются по серверам, обеспечивая линейный рост производительности как на запись, так и на чтение.

##### 3. Сценарии использования (Use Cases)

##### Выбирайте Redis Pub/Sub, если вам нужны:

* Real-time интерфейсы, веб-сокеты (Laravel Reverb / Pusher-like системы, чаты, живые лайки/комментарии).
* Мгновенные push-нотификации в браузер или мобильное приложение.
* Межпроцессное уведомление (например, инвалидация локального кэша в ОЗУ на веб-серверах).

##### Выбирайте RabbitMQ, если вам нужны:

* Классические фоновые задачи (Background Jobs) в веб-приложениях (отправка писем, генерация PDF, тяжелый ресайз картинок).
* Сложные бизнес-сценарии с хитрой маршрутизацией (например, слать уведомления на основе ГЕО, тегов, приоритета или типа клиента одновременно).
* Системы с гарантией ручного разбора ошибок через Dead Letter Exchange (DLX).

##### Выбирайте Apache Kafka, если вам нужны:

* Потоковая аналитика в реальном времени (Real-time Analytics, Stream Processing).
* Логирование Big Data и агрегация логов / метрик (метрики со 100 серверов стекаются в один топик, а оттуда льются в ClickHouse).
* Реализация архитектурных паттернов Event Sourcing и Change Data Capture (CDC), где лог сообщений — это и есть истинный первоисточник состояния системы.

##### 4. Комбинирование инструментов в одной системе

В крупных распределенных системах эти инструменты часто работают вместе, решая комплементарные задачи:

```
[ Пользователь ] ---> ( Laravel API ) ---> [ Публикация в RabbitMQ ] ---> [ Воркер обработки заказа ]

                             |                                                       |
                    ( Подписка по WS )                                         (Успех! Пишем в БД)
                             ^                                                       |
                             |                                                       v
                   [ Laravel Reverb ] <--- ( Мгновенный сигнал ) <--- [ Redis Pub/Sub "order.paid" ]

```

   1. RabbitMQ берет на себя надежную обработку транзакции: списывает деньги, проверяет склад, гарантирует Retry в случае сбоя.
   2. Как только воркер RabbitMQ завершает работу, он отправляет легкий сигнал в Redis Pub/Sub.
   3. Laravel Reverb, слушая Redis, мгновенно доставляет зеленую галочку «Заказ оплачен» в браузер пользователя по WebSocket.

##### 5. Иерархическое дерево принятия решений (Decision Tree)

Чтобы быстро определить целевой инструмент на архитектурном комитете или собеседовании, используйте следующий алгоритм:

   1. Критична ли потеря данных при перезагрузке воркера/сервера?
   * Нет (данные нужны только тем, кто онлайн прямо сейчас) -> Выбираем Redis Pub/Sub.
      * Да (данные должны дождаться воркера в любом случае) -> Переходим к шагу 2.
   2. Каков характер и объем этих данных?
   * Это высоконагруженный непрерывный поток метрик, логов, трекинга или Event Sourcing (High Throughput) -> Выбираем Apache Kafka.
      * Это дискретные, атомарные бизнес-задачи (фоновые джобы, транзакции, API-интеграции) -> Переходим к шагу 3.
   3. Нужна ли гибкая маршрутизация, приоритеты и гарантированный DLX для ручного разбора ошибок?
   * Да -> Выбираем RabbitMQ.
      * Нет (нужен просто надежный постоянный лог задач в памяти с минимальным оверхедом) -> Рассматриваем Redis Streams.
