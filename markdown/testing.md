
#  Тестирование

## Unit тесты

* Определение: Юнит-тесты (Unit Tests / Модульные тесты) — это автоматизированные тесты, которые проверяют корректность работы минимально изолированной единицы кода (обычно одного конкретного метода или функции) в полном отрыве от внешней среды.
* Главный принцип (Изоляция): Юнит-тест проверяет исключительно чистую логику кода. Он не должен общаться с реальной базой данных, ходить по сети в сторонние API, читать файлы с диска или использовать текущее системное время. Все внешние зависимости подменяются заглушками.
* Философия (Первый шаг в пирамиде): Юнит-тесты находятся в самом низу пирамиды тестирования. Их должно быть больше всего, они выполняются за миллисекунды, и их цель — мгновенная обратная связь для разработчика: «Сломал ли мой последний коммит логику функции?».

##### 2. Как это работает (Структура и Инструменты)

На собеседовании идеальный ответ должен содержать три ключевых столпа: структуру теста AAA, использование моков и базовые инструменты.

##### Структура теста: Паттерн AAA (Arrange, Act, Assert)

Каждый правильно написанный юнит-тест делится на три последовательных шага:

   1. Arrange (Подготовка): Инициализация объектов, создание тестовых данных, настройка заглушек (моков).
   2. Act (Действие): Вызов тестируемого метода с подготовленными аргументами.
   3. Assert (Проверка): Сравнение полученного результата от метода с ожидаемым эталоном.

Пример (на концептуальном PHP/PHPUnit):

```php
public function test_it_calculates_order_total_with_discount()
{
    // 1. Arrange
    $calculator = new OrderCalculator();
    $price = 100;
    $discount = 20;

    // 2. Act
    $result = $calculator->calculate($price, $discount);

    // 3. Assert
    $this->assertEquals(80, $result);
}
```
##### Изоляция через Тестовые заглушки (Test Doubles)

Когда тестируемый класс зависит от других сервисов (например, отправляет Email или списывает деньги через эквайринг), эти зависимости подменяют. Интервьюеры любят спрашивать разницу между ними:

* Dummy (Пустышка): Объект, который передается в метод просто для заполнения аргумента, но внутри никак не используется.
* Stub (Стаб): Жестко запрограммированная заглушка, которая на любой вызов метода всегда возвращает один и тот же заранее определенный ответ (например, всегда возвращает true на проверку баланса).
* Mock (Мок): Умная заглушка, которая проверяет, как именно её вызвали. Тест упадет, если тестируемый метод не вызвал мок, вызвал его с неправильными параметрами или нарушил очередность вызовов.

##### Инструменты в PHP/Laravel экосистеме

* PHPUnit — классический, мощный стандарт де-факто для тестирования в мире PHP.
* Pest — современный, элегантный тестовый фреймворк поверх PHPUnit с фокусом на читаемость кода (через Fluent API) [INDEX].
* Mockery — библиотека для удобного создания моков (интегрирована в Laravel).

##### 3. Когда использовать (И когда они бесполезны)## Когда юнит-тесты НЕОБХОДИМЫ:

* Сложная бизнес-логика (Доменные сервисы): Калькуляторы скидок, парсеры строк, валидаторы сложных паролей, хелперы трансформации данных.
* Разработка через тестирование (TDD — Test Driven Development): Подход, когда вы сначала пишете падающий юнит-тест, фиксируя требования к функции, затем пишете минимальный код для его прохождения, а в конце делаете рефакторинг.
* Проекты с долгим жизненным циклом: Где легаси-код постоянно дорабатывается. Юнит-тесты дают уверенность, что при обновлении версии фреймворка или PHP ничего не «отвалится».

##### Когда юнит-тесты НЕ НУЖНЫ (или неэффективны):

* Простой CRUD-код: Тестировать юнит-тестом метод контроллера, который просто делает User::create($request->all()), бессмысленно. Вы будете тестировать не свой код, а внутренности самого фреймворка Laravel. Здесь нужны функциональные / интеграционные тесты (Feature tests), которые делают реальный HTTP-запрос и проверяют запись в тестовой БД.
* UI и визуальное отображение: Шаблоны Blade или фронтенд-верстку проверять юнит-тестами неэффективно, для этого существуют E2E-тесты (например, Playwright или Laravel Dusk).

## Feature-тесты

В экосистеме PHP и особенно в Laravel понятие Feature-тестов возведено в абсолют. Из коробки Laravel предлагает мощнейшую и очень удобную среду для их написания, из-за чего в PHP-сообществе часто отдают приоритет именно Feature-тестированию (TDD на уровне фич), а не Unit.
Ниже расширенная и адаптированная под PHP/Laravel шпаргалка для базы знаний.

##### Feature-тестирование в PHP и Laravel## 🐘 Специфика PHP-стека

Исторически в PHP-мире долгое время правил инструмент PHPUnit, а сценарии фич описывали с помощью BDD-фреймворков вроде Behat (с синтаксисом Gherkin).
Сегодня стандартом де-факто в Laravel стал Pest (минималистичная обертка над PHPUnit) или классический PHPUnit с трейтами Laravel. Laravel упростил Feature-тесты до такой степени, что отпала необходимость в сложных сторонних BDD-инструментах.

##### 🚀 Архитектура Feature-теста в Laravel

В Laravel Feature-тест имитирует полноценный жизненный цикл HTTP-запроса (Request Lifecycle) без запуска реального веб-сервера (Nginx/Apache).
Когда вы вызываете `$this->post()`, Laravel:

   1. Создает виртуальный HTTP-запрос.
   2. Прогоняет его через глобальные и роутинговые Middleware (проверка CSRF, аутентификация).
   3. Направляет в Controller, где срабатывает FormRequest (валидация).
   4. Выполняет бизнес-логику и делает реальные запросы к БД через Eloquent ORM.
   5. Возвращает объект TestResponse для удобных проверок (Asserts).

##### 🛠 Ключевой инструментарий Laravel для Feature-тестов## 1. Работа с базой данных

* Трейт RefreshDatabase / UseDatabaseTransactions: Важнейшая деталь. Накатывает миграции один раз и оборачивает каждый тест в транзакцию. После завершения теста происходит Rollback — база данных всегда остается чистой, а тесты не влияют друг на друга.
* Элегантные Фабрики (Model Factories): Позволяют мгновенно создавать реалистичные тестовые данные.

```php
$user = User::factory()->create(); // Создаст и сохранит в тестовую БД
$order = Order::factory()->for($user)->create(['status' => 'pending']);
```

##### 2. Фасады-заглушки (Fakes)

В Laravel встроена идеальная система моканья (Mocking) внутренних компонентов без использования сложных конструкций PHPUnit. Вы просто вызываете метод fake() на нужном Фасаде:

* `Event::fake()`; — блокирует отправку реальных событий.
* `Queue::fake()`; — не дает задачам уходить в реальный Redis/Database воркер, позволяя проверить сам факт отправки в очередь.
* `Mail::fake()`; и `Notification::fake()`; — предотвращают отправку писем клиентам.
* `Http::fake()`; — перехватывает внешние API-запросы (например, к платежному шлюзу) и возвращает фейковый JSON.

##### 3. Аутентификация одной строчкой
Вместо ручного заполнения сессий или генерации JWT-токенов:

`$this->actingAs($user)->get('/dashboard')`;

##### 💻 Пример реального Feature-теста (PHPUnit)
Сценарий: Пользователь пытается оформить заказ, но у него недостаточно денег на балансе.

```php

<?php

namespace Tests\Feature;

use App\Models\User;
use App\Models\Product;
use Illuminate\Foundation\Testing\RefreshDatabase;
use Illuminate\Support\Facades\Notification;
use Tests\TestCase;

class OrderCheckoutTest extends TestCase
{
    use RefreshDatabase; // Очищаем БД после каждого теста

    /** @test */
    public function user_cannot_checkout_with_insufficient_balance(): void
    {
        // 1. Arrange (Given) — Подготовка окружения
        Notification::fake(); // Заглушаем уведомления, чтобы не слать реальные email
        
        $user = User::factory()->create(['balance' => 100]); // В кармане 100$
        $product = Product::factory()->create(['price' => 250]); // Товар стоит 250$

        // 2. Act (When) — Выполняем действие от лица пользователя
        $response = $this->actingAs($user)
            ->postJson(route('orders.store'), [
                'product_id' => $product->id,
            ]);

        // 3. Assert (Then) — Проверяем результаты
        // Проверяем HTTP статус (Ожидаем ошибку валидации/бизнес-логики)
        $response->assertStatus(422); 
        
        // Проверяем структуру и текст ошибки в JSON
        $response->assertJsonValidationErrors(['balance']);

        // Проверяем, что в БД НЕ создался заказ
        $this->assertDatabaseMissing('orders', [
            'user_id' => $user->id,
        ]);

        // Проверяем, что уведомление об успешной покупке НЕ было отправлено
        Notification::assertNothingSent();
    }
}
```

##### 🔥 Топ вопросов на собеседовании по Laravel Feature-тестам

   1. В чем разница между RefreshDatabase и DatabaseMigrations?
   * Ответ: DatabaseMigrations полностью пересоздает (дропает и накатывает) все таблицы перед каждым тестом, что очень медленно. RefreshDatabase накатывает миграции только один раз при первом тесте, а последующие тесты просто запускает внутри быстрых БД-транзакций.
   2. Как протестировать отправку файла (например, аватарки) в Laravel?
   * Ответ: С помощью класса Illuminate\Http\UploadedFile. Используется метод UploadedFile::fake()->image('avatar.jpg'), который создает виртуальный файл в памяти, а затем проверяется через Storage::fake().
   3. Почему не стоит использовать Http::fake() для модульных (Unit) тестов?
   * Ответ: Unit-тесты не должны инициализировать ядро Laravel и контейнер зависимостей (Service Container). Http::fake() завязан на механизм фасадов Laravel, поэтому его место исключительно в Feature-тестах. В Unit-тестах внешние зависимости мокаются через нативный PHPUnit/Mockery.
   

##### TDD (Test-Driven Development)

TDD (Test-Driven Development / Разработка через тестирование) — это методология разработки программного обеспечения, при которой сначала пишутся тесты, и только затем — код, который эти тесты удовлетворяет.
В PHP-сообществе (особенно в Laravel с инструментами Pest и PHPUnit) TDD часто применяется на уровне Feature-тестов: вы сначала описываете желаемое поведение API или фичи, а затем реализуете контроллеры и модели.

##### 🔄 Цикл TDD: Red-Green-Refactor

Разработка по TDD идет строгими короткими итерациями:

   1. 🔴 Red (Красный): Вы пишете тест на еще не существующую фичу (или баг). Запускаете его — тест обязательно должен упасть (выдать ошибку, что класса или метода не существует, либо вернулся не тот статус). Если тест сразу зеленый, значит, вы проверяете то, что уже работает, или тест написан неверно.
   2. 🟢 Green (Зеленый): Вы пишете минимально необходимый код, чтобы тест просто прошел. На этом этапе плевать на красоту кода, архитектуру и оптимизацию. Главное — перевести тест из красного статуса в зеленый.
   3. 🔵 Refactor (Рефакторинг): Теперь, когда у вас есть «страховочная сетка» в виде зеленого теста, вы улучшаете написанный код. Вы убираете дублирование, делите большие методы, оптимизируете SQL-запросы через Eloquent. После каждого изменения вы запускаете тест — он должен оставаться зеленым.

##### 👍 Когда TDD ИДЕАЛЬНО подходит (When to use)

* Четкие и стабильные требования: Когда бизнес-логика понятна до начала кодинга (например: «Если у пользователя VIP-статус, начислить 15% кэшбека, иначе 5%»). Написать тест на такую логику очень просто.
* Исправление багов (Bug Fixing): Лучшая практика в PHP. Нашелся баг? Сначала напишите Feature-тест, который воспроизводит этот баг (тест упадет). Затем исправьте код, пока тест не позеленеет. Этот баг больше никогда не вернется (regression protection).
* Чистая бизнес-логика и алгоритмы: Создание калькуляторов тарифов, систем лояльности, валидаторов сложных форм, интеграция формул. Здесь легко перебрать все пограничные случаи (Edge Cases) в секции DataProvider в PHPUnit/Pest.
* Разработка REST API: В Laravel очень удобно писать TDD для API endpoints. Вы описываете в тесте: ->postJson('/api/v1/posts'), передаете структуру, проверяете assertJsonStructure. Кода еще нет, но контракт API уже зафиксирован в тесте.
* Проектирование архитектуры: TDD заставляет вас быть первым пользователем собственного кода. Если вам неудобно писать тест для своего класса, значит, у класса плохой интерфейс, слишком много зависимостей или нарушен принцип SOLID. TDD пресекает создание «переусложненной» архитектуры.


##### 👎 Когда TDD НЕ стоит использовать (When NOT to use)

* R&D, стартапы на стадии MVP и жесткий лаг по времени: Когда требования меняются три раза в день, а код пишется методом «научного тыка», чтобы просто проверить гипотезу. Написание тестов вперед кода замедлит процесс в 2–3 раза. Сначала найдите Product-Market Fit, потом покройте тестами.
* Сложный UI/UX и фронтенд: Тестировать интерфейсы (Livewire, Inertia.js, Vue components) через TDD на бэкенде — это мучение. Поведение пользователя, анимации и верстку проще сначала сверстать и пощупать руками.
* Интеграция с нестабильными внешними системами: Если внешнее API плохо задокументировано, вы не сможете написать правильный Mock/Fake до того, как вручную «пощупаете» живые запросы через Postman/Tinker и поймете, что вообще прилетает в ответ.
* Простой CRUD: Если задача — просто выводить список пользователей из базы данных без какой-либо логики. Написание теста займет больше времени, чем генерация контроллера через php artisan make:controller --api. Здесь тесты лучше написать после (или ограничиться одним базовым Smoke-тестом).
* Низкий навык тестирования у команды: Если разработчики плохо умеют мокать зависимости, изолировать базу данных и путаются в синтаксисе Pest/PHPUnit, TDD превратится в пытку. Сначала нужно научиться писать обычные тесты после кода, а уже потом переходить на TDD.

##### 💡 Шпаргалка для ответа на собеседовании

«TDD — это не про тестирование, это про проектирование. Оно смещает фокус с вопроса „Как это закодить?“ на вопрос „Какую бизнес-задачу мы решаем и какие входные/выходные данные нам нужны?“. Я использую TDD, когда пишу сложную доменную логику, API-контракты или фикшу баги, но избегаю его на этапе создания прототипов (MVP), где скорость проверки продуктовых гипотез важнее стабильности архитектуры».

## Mocking и Stubbing

В тестировании на PHP и Laravel понятия Mocking (макетирование) и Stubbing (создание заглушек) относятся к категории Test Doubles (двойники тестирования). Их главная цель — изолировать тестируемый код от внешних зависимостей (сторонних API, отправки писем, системных вызовов времени или тяжелых классов), чтобы тест оставался быстрым, стабильным и предсказуемым [1].
Хотя эти термины часто используют как синонимы, между ними есть принципиальная разница в фокусе проверки.


##### 🎭 Разница между Stub и Mock (Кратко для собеседования)

* Stub (Заглушка) — отвечает на вопрос «Что возвращать?». Он просто заменяет реальный объект и возвращает заранее подготовленные данные (филонит вместо реальной работы). Тест не проверяет, сколько раз и с какими параметрами вызывался Stub.
* Mock (Макет) — отвечает на вопрос «Как меня использовали?». Он не просто возвращает данные, но и контролирует поведение. Тест проверяет ожидания (expectations): был ли вызван метод, сколько раз, и какие именно аргументы в него передали. Если метод не вызвался, Mock завалит тест.

Аналогия:

* Stub — это манекен в магазине. Он просто стоит и изображает человека.
* Mock — это датчик сигнализации. Он ждет конкретного действия и бьет тревогу (валит тест), если действие не произошло или произошло не так.

##### 🐘 Нативный подход в PHP (PHPUnit и Mockery)

Исторически в PHP для создания заглушек использовался встроенный функционал PHPUnit или популярная библиотека Mockery.

##### Пример Stub (PHPUnit)

Мы тестируем сервис заказа. Нам нужно подменить калькулятор налогов, чтобы он всегда возвращал 10, независимо от логики:

```php
$taxServiceStub = $this->createStub(TaxServiceInterface::class);
// Говорим стабу всегда возвращать 10 при вызове метода calculate
$taxServiceStub->method('calculate')->willReturn(10); 

$orderService = new OrderService($taxServiceStub);
$this->assertEquals(110, $orderService->total(100)); // 100 + 10 = 110

```

##### Пример Mock (Mockery)

Мы тестируем, что при отмене заказа система обязательно отправляет лог в специальный сервис мониторинга ровно 1 раз:

```php

$loggerMock = Mockery::mock(LoggerInterface::class);

// Устанавливаем жесткое ожидание (Assertion встроен внутрь мока)
$loggerMock->shouldReceive('logAlert')
    ->once() // Должен вызваться ровно один раз
    ->with('Order cancelled', ['order_id' => 42]); // Именно с этими параметрами

$orderService = new OrderService($loggerMock);
$orderService->cancel(42);
// Если cancel() не вызовет logAlert, тест упадет автоматически
```

##### 🚀 Подход Laravel: Фасады-заглушки (Fakes)

Laravel кардинально изменил подход к Mocking/Stubbing через свои Фасады. Вместо сложного создания моков через PHPUnit, Laravel предоставляет лаконичный метод fake(). Под капотом Laravel сам подменяет реальный класс в контейнере зависимостей (Service Container) на удобный шпионский объект.
В Laravel разделение на Stub и Mock на уровне кода фасадов выглядит невероятно элегантно.

##### 1. Использование Laravel Fakes как Stub (Задаем поведение)

Когда мы тестируем отправку запроса к стороннему API (например, погоды), нам нужно, чтобы API не дергалось в реальности, а просто вернуло нужный нам статус. Это Stubbing:

```php

use Illuminate\Support\Facades\Http;

// Отрезаем реальный мир и заставляем HTTP-клиент возвращать фейковый JSON
Http::fake([
    '://weather.com*' => Http::response(['temp' => 25], 200),
]);

// Act: Наш код выполняет логику
$weather = $this->weatherService->getForCity('London');

// Assert: Проверяем результат работы НАШЕГО кода
$this->assertEquals(25, $weather->temperature);

```

##### 2. Использование Laravel Fakes как Mock (Проверяем поведение)

Когда нам важно убедиться, что система совершила побочный эффект (отправила email, положила задачу в очередь) — это Mocking. Проверки (asserts) в Laravel делаются в самом конце теста:

```php

use Illuminate\Support\Facades\Queue;
use App\Jobs\ProcessOrderJob;

// 1. Изолируем очередь
Queue::fake();

// 2. Act: Выполняем экшен (например, оплата заказа)
$this->postJson('/api/orders/1/pay');

// 3. Assert: Проверяем, что фасад зафиксировал вызов нужной задачи
Queue::assertPushed(ProcessOrderJob::class, function ($job) {
    return $job->orderId === 1; // Проверяем правильность переданных в джобу данных
});

```

##### ⚠️ Опасности и антипаттерны (О чем спросят на собеседовании)

   1. Не мокайте то, чем вы не владеете (Don't mock third-party libraries): Не стоит писать моки для классов чужих SDK (например, Stripe\StripeClient). Чужой код может измениться при обновлении пакета, и ваши тесты останутся зелеными, но приложение упадет в продакшене. Решение: Оберните чужую библиотеку в свой интерфейс (паттерн Адаптер) и мокайте уже свой собственный интерфейс. В случае с HTTP-запросами — используйте Http::fake().
   2. Переизбыток моков (Over-mocking): Если в вашем Feature-тесте замокано абсолютно всё (база данных через репозитории, кэш, все сервисы), то ваш Feature-тест превращается в кучу плохих Unit-тестов. Теряется весь смысл фича-тестирования — проверка интеграции. Мокайте только внешние границы системы (сеть, СМС-шлюзы, очереди) и оставляйте реальную БД.
   3. Хрупкость тестов из-за жестких моков: Если вы детально описываете через Mockery цепочки вызовов внутренних методов (->shouldReceive('stepOne')->andReturn()->shouldReceive('stepTwo')), то любой минимальный рефакторинг внутренностей класса сломает тест, даже если фича работает исправно. Отдавайте предпочтение Stubs или высокоуровневым Laravel Fakes.


## PHPUnit

PHPUnit — это официальный стандарт де-факто и самый популярный фреймворк для автоматизированного тестирования в PHP-экосистеме [1]. На нем строятся как изолированные модульные (Unit), так и сквозные функциональные (Feature) тесты в большинстве современных фреймворков, включая Laravel и Symfony [1].
В контексте Laravel PHPUnit работает «под капотом»: базовый класс Tests\TestCase, от которого наследуются все тесты, расширяет именно стандартный класс PHPUnit.

##### ⚙️ Анатомия PHPUnit: Основные компоненты## 1. Структура тестового класса

Каждый тестовый класс в PHPUnit должен наследоваться от PHPUnit\Framework\TestCase (в Laravel — от Tests\TestCase) [1].
Чтобы фреймворк распознал метод как тест, его имя должно начинаться с префикса test или содержать аннотацию/атрибут /** @test */ [1]:

```php

use Tests\TestCase;

class UserTest extends TestCase
{
    // Вариант 1: Префикс в имени метода
    public function testUserHasProfile(): void { ... }

    // Вариант 2: Использование атрибута PHP (начиная с PHPUnit 10+)
    #[\PHPUnit\Framework\Attributes\Test]
    public function user_can_be_activated(): void { ... }
}

```

##### 2. Жизненный цикл теста (Fixtures / Hooks)

PHPUnit предоставляет методы для выполнения кода до и после запуска тестов [1]. Это критически важно для настройки окружения (Setup) и его очистки (Teardown) [1]:

* `setUpBeforeClass()` / `tearDownAfterClass()` — выполняются один раз перед запуском всего класса (методы должны быть static).
* setUp() — выполняется перед каждым отдельным методом-тестом [1]. Идеально для инициализации объектов или фейков.
* tearDown() — выполняется после каждого метода-теста [1]. Используется для очистки памяти или закрытия соединений.

##### 3. Утверждения (Assertions)

Это функции, которые проверяют, совпадает ли ожидаемый результат работы кода с фактическим [1]. Если утверждение не выполняется, тест считается упавшим.

* $this->assertEquals($expected, $actual) — проверка на равенство значений [1].
* $this->assertSame($expected, $actual) — строгая проверка (эквивалент ===, проверяет типы и ссылки на объекты) [1].
* $this->assertTrue($condition) / $this->assertFalse($condition) — проверка булевых значений.
* $this->assertCount(3, $array) — проверка количества элементов в массиве или коллекции.
* $this->assertDatabaseHas(...) — специфичное для Laravel утверждение для проверки данных в БД.

##### 🔥 Продвинутые фичи PHPUnit (Любимые темы на собеседованиях)## 1. Поставщики данных (Data Providers)

Позволяют запустить один и тот же тест многократно с разными наборами входных данных, избегая дублирования кода (Don't Repeat Yourself).

```php

use PHPUnit\Framework\Attributes\DataProvider;
use Tests\TestCase;

class PasswordValidatorTest extends TestCase
{
    public static function passwordProvider(): array
    {
        return [
            'too short'  => ['123', false],
            'no numbers' => ['qwertyuiop', false],
            'valid pass' => ['SecurePass123!', true],
        ];
    }

    #[DataProvider('passwordProvider')]
    public function test_password_validation(string $password, bool $isValid): void
    {
        $validator = new PasswordValidator();
        $this->assertEquals($isValid, $validator->validate($password));
    }
}

```

##### 2. Тестирование исключений (Exceptions)

Правильный код должен не только возвращать верные данные, но и корректно падать при ошибках. PHPUnit проверяет это через метод expectException:

```php
public function test_it_throws_exception_when_dividing_by_zero(): void
{
    $calculator = new Calculator();

    // Ожидание нужно объявлять ДО совершения ошибочного действия
    $this->expectException(\InvalidArgumentException::class);
    $this->expectExceptionMessage('Cannot divide by zero');

    $calculator->divide(10, 0); 
}
```

##### 🛠 Конфигурация: Файл phpunit.xml

В корне любого PHP-проекта с тестами лежит файл настроек phpunit.xml. В нем определяются:

* Test Suites: Логическое разделение тестов (например, отдельно папка Feature, отдельно Unit).
* Переменные окружения (Environment Variables): Для тестов задаются отдельные доступы (например, <env name="DB_DATABASE" value="testing"/>), чтобы тесты случайно не стерли локальную или продакшн базу данных.
* Покрытие кода (Code Coverage): Настройки генерации отчетов о том, какой процент вашего исходного кода покрыт тестами.

##### 🆚 PHPUnit vs Pest (Современный тренд в PHP)

На собеседованиях часто спрашивают про Pest.

* Pest — это современная обертка над PHPUnit, созданная Нуно Мадуро (команда Laravel).
* Он использует функциональный синтаксис вместо объектно-ориентированного (очень похож на Jest в JavaScript).
* Под капотом Pest полностью выполняет код через PHPUnit. Вы можете бесшовно использовать старые PHPUnit-тесты внутри Pest-проекта.

Сравнение синтаксиса:

```php
// PHPUnit
public function test_user_is_adult(): void {
    $this->assertTrue($this->user->isAdult());
}

// Pest
it('checks if user is adult', function () {
    expect($this->user->isAdult())->toBeTrue();
});
```

##### 💡 Как уверенно ответить на собеседовании?

«PHPUnit — это фундамент тестирования в PHP. Даже если в проекте используется Pest, под капотом всё равно работает ядро PHPUnit. В своей практике я активно использую его фикстуры (setUp), Data Providers для тестирования граничных значений и встроенные механизмы проверки Exception'ов. В Feature-тестах Laravel PHPUnit расширяется кастомными ассертами для работы с HTTP-ответами и базой данных, что делает его мощным инструментом автоматизации».

## Pest

Pest — это современный, элегантный фреймворк для тестирования в PHP, созданный Нуно Мадуро (Nuno Maduro) из команды Laravel. Он построен поверх PHPUnit (использует его ядро под капотом), но предлагает совершенно другой, минималистичный и декларативный синтаксис, вдохновленный миром JavaScript (в частности, Jest).
Сегодня Pest является стандартом де-факто для новых проектов на Laravel и активно вытесняет классический PHPUnit благодаря высокой скорости написания кода и отличной читаемости.


##### 🎨 Ключевые особенности Pest

   1. Функциональный стиль (Expression-based): Вместо создания тяжелых классов и методов, тесты пишутся как глобальные функции it() или test().
   2. Цепочки ожиданий (Expectations API): Вместо сотен методов $this->assert..., Pest использует интуитивно понятную функцию expect(), к которой можно применять цепочки проверок.
   3. Полная совместимость с PHPUnit: Вы можете запускать старые PHPUnit-тесты внутри Pest без изменения кода. Они работают вместе «из коробки».
   4. Потрясающий CLI (Консольный интерфейс): Вывод ошибок в Pest невероятно красив и информативен — он подсвечивает синтаксис кода прямо в терминале и точно указывает на упавшую строку.

##### 💻 Сравнение синтаксиса: PHPUnit vs Pest
Одно и то же тестирование бизнес-логики:
Классический PHPUnit:

```php
namespace Tests\Unit;

use Tests\TestCase;
use App\Models\User;

class UserTest extends TestCase
{
    /** @test */
    public function it_can_determine_if_user_is_an_admin(): void
    {
        $user = User::factory()->make(['role' => 'admin']);
        
        $this->assertTrue($user->isAdmin());
        $this->assertStringContainsString('admin', $user->role);
    }
}
```

Современный Pest:

```php

use App\Models\User;

it('can determine if user is an admin', function () {
    $user = User::factory()->make(['role' => 'admin']);

    expect($user->isAdmin())->toBeTrue()
        ->and($user->role)->toContain('admin');
});

```

## 🚀 Работа с Feature-тестами и Laravel в Pest
Pest идеально интегрирован с Laravel через плагин pest-plugin-laravel. Он позволяет использовать все фичи Laravel в лаконичном стиле.

```php

use App\Models\User;
use function Pest\Laravel\{actingAs, postJson, assertDatabaseHas};

// Подключаем трейт очистки БД для этого файла
uses(\Illuminate\Foundation\Testing\RefreshDatabase::class);

it('allows authenticated user to create a post', function () {
    // Arrange
    $user = User::factory()->create();

    // Act & Assert в одну цепочку (fluent API)
    actingAs($user)
        ->postJson('/api/posts', ['title' => 'New Post'])
        ->assertStatus(201)
        ->assertJsonPath('data.title', 'New Post');

    // Проверка базы данных через хелпер Pest
    assertDatabaseHas('posts', ['title' => 'New Post']);
});

```

##### 🛠 Продвинутые фичи Pest (Полезно для собеседования)## 1. Высшие функции (Higher Order Expectations)

Вы можете вызывать проверки прямо на свойствах или методах тестируемого объекта, сокращая код до одной строки:

```php

it('has a valid structure', function () {
    expect(User::factory()->make())
        ->isAdmin()->toBeFalse()
        ->email->toBeString()
        ->role->not->toBeNull();
});

```

##### 2. Замена Data Providers (Datasets)

Вместо создания громоздких статических методов, как в PHPUnit, Pest предлагает изящные датасеты:

```php

it('validates secure passwords', function ($password, $expected) {
    expect(Validator::validate($password))->toBe($expected);
})->with([
    'too short'  => ['123', false],
    'valid pass' => ['Secure123!', true],
]);

```

##### 3. Архитектурное тестирование (Arch Testing)

Уникальная фича Pest (начиная со 2-й версии). Позволяет тестировать правила архитектуры вашего приложения прямо в тестах. Например, можно запретить разработчикам использовать dd() или заставить контроллеры наследоваться от определенного класса:

```php

it('does not leave debug statements in the code')
    ->expect(['dd', 'dump', 'ray'])
    ->not->toBeUsed();

it('ensures models extend BaseModel')
    ->expect('App\Models')
    ->toExtend('App\Models\BaseModel');

```

##### ⚖️ Плюсы и минусы Pest

Плюсы:

* Скорость написания: Кода меньше на 30–40% за счет отсутствия бойлерплейта (классов, пространств имен).
* Порог входа: Разработчикам с опытом в JS/TS (Jest) или Ruby (RSpec) синтаксис понятен мгновенно.
* Архитектурные тесты: Заменяют сложные сторонние утилиты вроде PHPStan/Deptrac для базовых проверок структуры кода.

Минусы:

* Магия под капотом: Из-за глобальных функций и кастомного парсера сложнее дебажить сам тестовый фреймворк, если что-то пошло не так.
* Поддержка IDE: Хотя PHPStorm и VS Code отлично работают с Pest, для старых версий или специфичных плагинов автодополнение цепочек expect() иногда может сбоить (требуется генерация метафайлов .phpstorm.meta.php).

##### 💡 Как ответить на собеседовании?

«Pest — это современный стандарт в Laravel-разработке. Это не замена PHPUnit, а надстройка (обертка) над ним. Я предпочитаю Pest для новых проектов, потому что его синтаксис Expectations API и функциональный стиль ускоряют написание тестов и делают их более читаемыми для бизнеса. Особенно ценю в Pest встроенные датасеты и механизм Arch Testing, который позволяет автоматически контролировать чистоту архитектуры приложения на CI/CD».

## Test Coverage

Test Coverage — процент кода, покрытого тестами. Метрики: line, function, class, branch coverage.

Запуск: php artisan test --coverage. Требует Xdebug или PCOV. HTML отчёт: --coverage-html coverage/.

Важно: Целевые значения: критичная логика 90%+, общий проект 70-80%. 100% coverage НЕ гарантирует отсутствие багов. Mutation testing (Infection) проверяет качество тестов.

Что это: Test Coverage — процент кода, покрытого тестами. Показывает, какие строки/функции/классы протестированы.

Метрики:

- Line Coverage — покрытие строк
- Function Coverage — покрытие функций
- Class Coverage — покрытие классов
- Branch Coverage — покрытие веток (if/else)
