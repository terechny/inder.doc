
## Архитектура Laravel

##### ⚙️ Laravel. Как работает

Laravel — MVC фреймворк с архитектурой, построенной на Service Container и Service Providers. Основные компоненты: Router → Middleware → Controller → Model → View.

Основные принципы:

- MVC паттерн (Model-View-Controller)
- Service Container (IoC контейнер для DI)
- Facades (статический интерфейс к сервисам)
- Request Lifecycle (жизненный цикл запроса)

Laravel — это самый популярный фреймворк для PHP, построенный на архитектуре MVC (Model-View-Controller) и использующий модульные компоненты (многие из которых взяты из фреймворка Symfony).

Жизненный цикл типичного запроса в Laravel устроен следующим образом:

   1. Точка входа (public/index.php): Все запросы от пользователей веб-сервер (Nginx/Apache) направляет в один-единственный файл.
   2. Загрузка окружения и Ядра (HTTP Kernel): Загружаются конфигурационные файлы, переменные окружения (.env) и базовые системные классы.
   3. Сервис-провайдеры (Service Providers): Главное сердце настройки Laravel. Здесь регистрируются и запускаются все ключевые компоненты системы: подключение к базе данных, системы авторизации, очереди задач и кэширование.
   4. Маршрутизация (Routing): Запрос передается в роутер (routes/web.php или routes/api.php). Он сопоставляет URL-адрес запроса с конкретным действием.
   5. Посредники (Middleware): Перед тем как попасть к логике приложения, запрос проходит сквозь «слои» фильтров. Они проверяют, авторизован ли пользователь, защищают от CSRF-атак и проверяют права доступа.
   6. Контроллер (Controller): Мозг конкретной страницы. Он принимает входные данные пользователя, обращается к Модели (Eloquent ORM) для получения или сохранения данных в БД, выполняет бизнес-логику и подготавливает ответ.
   7. Ответ (Response): Контроллер либо возвращает JSON-данные (для API), либо компилирует HTML-страницу с помощью шаблонизатора Blade (View) и отдает её обратно пользователю.


```
1. public/index.php (точка входа)
   ↓
2. Bootstrap (загрузка фреймворка)
   ↓
3. Kernel (HTTP Kernel)
   ↓
4. Service Providers (регистрация сервисов)
   ↓
5. Router (маршрутизация)
   ↓
6. Middleware (обработка запроса)
   ↓
7. Controller (бизнес-логика)
   ↓
8. Model (работа с данными)
   ↓
9. View (отображение)
   ↓
10. Response (ответ клиенту)

```

##### 🎯 Когда использовать Laravel

Laravel называют фреймворком «для веб-ремесленников», потому что он берет на себя всю рутину и позволяет сразу писать бизнес-логику проекта.

##### Кому и для чего он идеально подходит:

* Традиционные веб-сайты и SaaS-сервисы: Блоги, CRM-системы, ERP-системы корпораций, маркетплейсы, платформы онлайн-обучения, социальные сети. Благодаря встроенной авторизации, работе с почтой и платежными шлюзами такие проекты собираются очень быстро.
* Разработка RESTful API: Отличная основа для бэкенда мобильных приложений или SPA-приложений (на Vue.js, React, Angular).
* Проекты со сложной фоновой логикой: Системы, где нужно обрабатывать тяжелые задачи в фоне (генерация PDF, массовые рассылки писем, импорт гигантских прайс-листов), эффективно используют встроенную систему очередей (Queues) и планировщик задач (Task Scheduler).
* Проекты для быстрого старта (MVP): Когда стартапу нужно проверить гипотезу за 2-3 недели. Экосистема готовых пакетов (Breeze, Jetstream, Nova) позволяет развернуть готовую админку и авторизацию за пару команд в консоли.

##### Когда Laravel использовать НЕ стоит:

* Простые одностраничные сайты (Landing Page) или визитки: Полноценный фреймворк слишком тяжел для статичной страницы. Здесь проще использовать чистый HTML или WordPress.
* Высоконагруженные микросервисы со сверхнизким пингом (Low Latency): Если вам нужно обрабатывать сотни тысяч запросов в секунду с задержкой в 1-2 миллисекунды (например, чаты реального времени или обработка биржевых котировок). Для таких задач лучше подходят Node.js, Go или специализированные асинхронные PHP-инструменты (RoadRunner, Swoole).
* Простые скрипты автоматизации: Для парсинга одной страницы или переноса папки на сервере фреймворк не нужен — достаточно обычного PHP-скрипта.

Используй Laravel когда:

- Нужен полнофункциональный фреймворк (не микрофреймворк)
- Проект средний/большой размер
- Важна скорость разработки
- Нужен ORM (Eloquent), роутинг, middleware из коробки
- Команда знает Laravel

НЕ используй когда:

- Микросервис с минимальными зависимостями (Lumen, Slim)
- Высоконагруженный проект требует максимальной производительности (Symfony components, RoadRunner)
- Legacy проект на другом фреймворке

## ⚙️ Service Container. Как работает
Service Container (Сервис-контейнер) — это мощный инструмент для управления зависимостями классов и выполнения Dependency Injection (внедрения зависимостей). Простыми словами, это большой «черный ящик» или реестр, где Laravel хранит инструкции по созданию всех объектов вашего приложения.
Работа контейнера строится на двух главных этапах:

   1. Регистрация (Binding): Вы объясняете контейнеру, как нужно создавать объект. Это делается в Сервис-провайдерах (Service Providers). Вы можете привязать класс как синглтон (чтобы он создался один раз на весь запрос) или как обычную связь (каждый раз создавать новый экземпляр).
   2. Разрешение зависимостей (Resolving): Когда вам нужен объект, вы просите его у контейнера. Самая большая магия контейнера Laravel — автоматическое внедрение (Auto-wiring) через рефлексию. Если вы укажете нужный вам класс в аргументах конструктора или метода контроллера, контейнер сам заглянет внутрь этого класса, поймет, какие ему нужны зависимости, создаст их и передаст вам готовый объект в собранном виде.


```php

// Пример автоматической магии Laravel:
class OrderController extends Controller {
    // Контейнер сам создаст объект BillingService и передаст его в конструктор
    public function __construct(
        protected BillingService $billing
    ) {}

    public function checkout() {
        $this->billing->charge();
    }
}
```

## Service Container

Service Container (IoC контейнер) — ядро Laravel для управления зависимостями. Автоматически создаёт объекты и внедряет их зависимости.

Основное:

- Регистрация сервисов (bind, singleton)
- Разрешение зависимостей (автоматическая инъекция)
- Dependency Injection через конструктор


##### 🎯 Когда использовать Service Container

В Laravel вы используете контейнер постоянно, даже не замечая этого (когда пишете тайпхинты в контроллерах). Однако осознанно настраивать и связывать классы в контейнере нужно в следующих сценариях:

#####  1. Работа с интерфейсами (Слабая связанность)
Если ваш код зависит от абстракции (интерфейса), а не от конкретной реализации. Вы регистрируете в контейнере связь: "когда системе нужен PaymentInterface, дай ей класс StripePaymentProcessor". Если завтра вы решите перейти на PayPal, вам нужно будет изменить всего одну строку кода в сервис-провайдере, а контроллеры останутся нетронутыми.

#####  2. Настройка сторонних библиотек и сложных сервисов
Если для создания объекта класса требуется передать в конструктор конфигурационные данные, API-ключи из .env файла или другие сервисы. Вы описываете логику создания один раз в контейнере, и вам больше не нужно вручную писать new ComplexService($config, $http, $logger) по всему проекту.

#####  3. Создание Синглтонов (Singletons)
Когда в приложении должен существовать строго один экземпляр класса за весь жизненный цикл запроса. Например, подключение к внешней базе данных, кэш-клиент или объект текущего авторизованного приложения. Контейнер создаст его при первом вызове, а при последующих будет отдавать тот же самый объект.

##### 4. Упрощение тестирования (Mocking)
Если класс зарегистрирован в контейнере, во время тестов вы можете легко подменить его на "заглушку" (Mock). Например, при тестировании отправки СМС вы можете временно подменить реальный сервис отправки на фейковый, чтобы тесты не тратили реальные деньги и не отправляли сообщения пользователям.


#### Как работает

Регистрация сервисов:

```php

// app/Providers/AppServiceProvider.php
public function register(): void
{
    // bind — каждый раз новый экземпляр
    $this->app->bind(PaymentGateway::class, StripeGateway::class);

    // singleton — один экземпляр на весь запрос
    $this->app->singleton(CacheService::class, function ($app) {
        return new CacheService(
            $app->make('cache.store')
        );
    });

    // instance — использовать уже существующий объект
    $logger = new Logger('app');
    $this->app->instance(Logger::class, $logger);
}

```

Разрешение зависимостей:

```php

// 1. Через конструктор (автоматически)
class OrderController extends Controller
{
    // Laravel автоматически создаст OrderService
    public function __construct(
        private OrderService $orderService
    ) {}
}

// 2. Через app() helper
$service = app(OrderService::class);

// 3. Через resolve()
$service = resolve(OrderService::class);

// 4. Через make()
$service = app()->make(OrderService::class);

// 5. Через фасад
use Illuminate\Support\Facades\App;
$service = App::make(OrderService::class);

```

Contextual Binding (контекстная привязка):

```php

// Разные реализации для разных классов
public function register(): void
{
    // OrderService получит StripeGateway
    $this->app->when(OrderService::class)
        ->needs(PaymentGateway::class)
        ->give(StripeGateway::class);

    // RefundService получит PayPalGateway
    $this->app->when(RefundService::class)
        ->needs(PaymentGateway::class)
        ->give(PayPalGateway::class);
}


```


Binding Interfaces:

```php

// Интерфейс
interface PaymentGateway
{
    public function charge(int $amount): bool;
}

// Реализация
class StripeGateway implements PaymentGateway
{
    public function charge(int $amount): bool
    {
        // Stripe API
    }
}

// Регистрация
public function register(): void
{
    $this->app->bind(
        PaymentGateway::class,
        StripeGateway::class
    );
}

// Использование
class OrderService
{
    // Автоматически получит StripeGateway
    public function __construct(
        private PaymentGateway $gateway
    ) {}
}

```

##### Когда использовать

Используй когда:

- Нужна подмена реализаций (интерфейсы)
- Тестирование (mock зависимостей)
- Singleton сервисы (например, логгер)
- Сложная инициализация объектов

Не используй когда:

- Простые value objects без зависимостей
- DTO (Data Transfer Objects)
- Eloquent модели (они сами создаются)


Service Container — IoC контейнер для DI в Laravel. `bind()` создаёт новый экземпляр каждый раз, `singleton()` — один на запрос. Автоматическая инъекция через конструктор. Регистрирую в Service Providers. Интерфейсы связываю с реализациями через bind(`Interface::class`, `Implementation::class`). Contextual binding для разных реализаций. В тестах подменяю через `app()->instance()`.


## Service Providers


##### ⚙️ Что такое Service Provider и как он работает


Service Provider — место регистрации сервисов в Service Container. Все сервисы Laravel регистрируются через providers.

Основные методы:

- register() — регистрация привязок в контейнере
- boot() — выполняется после регистрации всех providers

Service Provider (Сервис-провайдер) — это центральный узел начальной загрузки (bootstrapping) всего приложения Laravel. Если Сервис-контейнер — это склад готовых объектов, то Сервис-провайдеры — это грузчики и курьеры, которые этот склад наполняют, настраивают и подготавливают к работе при каждом HTTP-запросе [].
Когда Laravel запускается, он первым делом проходит по списку провайдеров (который в современных версиях автоматически регистрируется в файле bootstrap/providers.php или config/app.php) и поочередно вызывает в них два метода:

   1. `register()` — этот метод служит исключительно для привязки сервисов в Сервис-контейнер через $this->app->bind() или `$this->app->singleton()` []. Здесь нельзя использовать другие функции фреймворка (например, роуты или события), так как они могут быть еще не загружены в этот момент [].
   2. boot() — вызывается после того, как метод register отработал во всех существующих провайдерах []. Это значит, что все зависимости уже зарегистрированы на «складе». В boot() вы можете свободно писать любую логику: регистрировать слушатели событий, настраивать макросы, правила валидации, роуты или представления (Blade) [, ].


##### 🎯 Когда нужно создавать свой Service Provider

Хотя для мелких проектов хватает стандартного AppServiceProvider, по мере роста приложения логику регистрации нужно разделять по собственным провайдерам. Их создают через консоль: php artisan make:provider MyCustomServiceProvider.
Вот основные сценарии, когда вам необходим собственный Service Provider:

* Создание крупного независимого модуля (сервиса): Если вы пишете сложную интеграцию (например, с ERP-системой 1С или платежным шлюзом). Вы выносите всю логику её настройки и привязки интерфейсов в отдельный BillingServiceProvider, чтобы не захламлять AppServiceProvider.
* Разработка собственных пакетов (Packages): Если вы создаете библиотеку для Laravel, которую планируете повторно использовать в других проектах. Через метод boot() вашего провайдера пакет «скажет» Laravel, где искать свои роуты, конфигурации, миграции и Blade-шаблоны.
* Настройка роутинга и кастомных middleware: Когда стандартной логики роутера не хватает, и вам нужно динамически загружать разные файлы маршрутов в зависимости от поддомена или языковой версии сайта.
* Группировка по смыслу: Например, EventServiceProvider используется исключительно для связывания Событий (Events) и их Слушателей (Listeners), а AuthServiceProvider — для регистрации политик доступа (Gates и Policies).


##### ⚡ Что такое Отложенные провайдеры (Deferrable Providers)

По умолчанию Laravel загружает абсолютно все сервис-провайдеры при каждом запросе пользователя []. Если ваш провайдер регистрирует тяжелый сервис, который нужен только на одной-двух страницах (например, генератор PDF-отчетов), постоянная загрузка будет впустую тратить оперативную память и процессорное время [].
Чтобы решить эту проблему, провайдер можно сделать отложенным. Для этого класс должен реализовать интерфейс DeferrableProvider [].

```php

use Illuminate\Contracts\Support\DeferrableProvider;
use Illuminate\Support\ServiceProvider;

class HeavyReportServiceProvider extends ServiceProvider implements DeferrableProvider
{
    public function register(): void
    {
        $this->app->singleton(PdfGenerator::class, function ($app) {
            return new PdfGenerator();
        });
    }

    // Указываем, какие именно классы предоставляет этот провайдер
    public function provides(): array
    {
        return [PdfGenerator::class];
    }
}

```

Как это работает: Laravel проигнорирует этот провайдер при старте []. Но как только в каком-нибудь контроллере код запросит класс `PdfGenerator`, Laravel мгновенно «вспомнит» про этот провайдер, запустит его метод `register()` и выдаст нужный объект []. Это существенно экономит ресурсы сервера.


## Facades

Facades — статический интерфейс к классам в Service Container. Выглядят как статические методы, но работают через контейнер.

Основное:

- Cache::get() вместо app('cache')->get()
- Статический синтаксис, динамическое связывание
- Testable (можно mock-ать)

Вот средне-краткий разбор концепции Facades (Фасадов) в Laravel — одного из самых узнаваемых и обсуждаемых инструментов фреймворка.


##### ⚙️ Что такое Facade и как он работает

Facade (Фасад) — это инструмент, который предоставляет статическую обертку над классами, зарегистрированными в Сервис-контейнере. Они позволяют вызывать методы обычных объектов так, будто это статические методы класса: `Cache::get('key')` вместо `$container->make('cache')->get('key')`.

Многие разработчики путают Фасады с обычными статическими классами, но архитектурно это принципиально разные вещи. Под капотом Фасады работают на магии динамического вызова методов PHP:

   1. Все фасады наследуют базовый класс `Illuminate\Support\Facades\Facade`.
   2. Внутри конкретного фасада (например, Cache) переопределяется всего один защищенный метод — `getFacadeAccessor()`. Он просто возвращает строковую строку-ключ, под которой реальный сервис зарегистрирован в контейнере (например, return 'cache';).
   3. Когда вы вызываете `Cache::get('name')`, этот статический метод физически не существует. Срабатывает магический метод `__callStatic()`, встроенный в базовый класс Facade.
   4. Метод `__callStatic()` берет ключ 'cache', обращается к Сервис-контейнеру, достает оттуда реальный объект `Repository` и вызывает у него обычный нестатический метод `get('name')`.

```php
// Как это выглядит для разработчика:
use Illuminate\Support\Facades\Cache;

$value = Cache::get('user_count');

// Что происходит под капотом (упрощенно):
$realObject = app()->make('cache'); 
$value = $realObject->get('user_count');
```

##### 🎯 Зачем они нужны (Плюсы)

* Лаконичность кода: Код становится коротким, чистым и легко читаемым. Не нужно внедрять зависимости через конструктор ради одного простого вызова.
* Удобство тестирования: Несмотря на «статический» вид, Фасады легко подменяются на заглушки (Mock) во время юнит-тестов. Если вы напишете Cache::shouldReceive('get')->once()->andReturn('mocked-value'), Laravel подменит реальный сервис в контейнере фейковым. Это главное отличие от классических жестких статических классов PHP, которые невозможно протестировать в изоляции.
* Быстрый доступ к экосистеме: Вы получаете мгновенный доступ к логированию (`Log::info()`), очередям (`Queue::push()`), базе данных (`DB::table()`) и конфигурации (`Config::get()`) из любой точки приложения.

##### ⚠️ Когда использовать НЕ стоит (Минусы)

Критики часто называют Фасады антипаттерном из-за следующих побочных эффектов:

* Скрытие реальных зависимостей (Scooter Code): Если вы внедряете зависимости через конструктор, появление 10 аргументов сразу намекнет вам, что класс перегружен (нарушение Single Responsibility Principle). С фасадами вы можете использовать 15 разных сервисов внутри одного метода, и визуально код будет казаться «чистым», хотя архитектурно он превратится в монолитную путаницу.
* Отсутствие автодополнения в старых IDE: Если не использовать специальные плагины (например, laravel-ide-helper), среда разработки не понимает, какие методы есть у Фасада, так как они вызываются динамически через __callStatic().
* Сложность при выносе кода в пакеты: Если вы пишете независимый PHP-пакет, использование фасадов жестко привяжет ваш код к глобальному состоянию Laravel, сделав невозможным его использование вне фреймворка.

Рекомендация по использованию: Фасады идеально подходят для контроллеров, роутов и верхнеуровневой логики приложения, где важна скорость написания кода. В глубокой бизнес-логике, сервисах обработки данных и репозиториях лучше использовать явное внедрение зависимостей (Dependency Injection) через конструктор.

* Продемонстрировать пример тестирования (Mocking) кода, в котором используется Фасад.

## Маршрутизация (Routing)


##### ⚙️ Как устроена маршрутизация

Маршрутизация — это механизм, который принимает входящий URL-запрос от пользователя (например, GET /products) и определяет, какой именно код (контроллер или анонимная функция) должен его обработать.
Все основные файлы маршрутов находятся в папке routes/:

* web.php — для обычных страниц сайта. К этим роутам автоматически применяется сессия, куки и защита от CSRF-атак.
* api.php — для API-эндпоинтов. Они не используют сессии (stateless) и обычно возвращают JSON. В современных версиях Laravel этот файл активируется отдельно через команду php artisan install:api.

Простейший роут выглядит так:

```php
use App\Http\Controllers\UserController;
use Illuminate\Support\Facades\Route;

// Роут, возвращающий анонимную функцию
Route::get('/welcome', function () {
    return 'Привет, мир!';
});

// Современный синтаксис привязки к Контроллеру
Route::get('/user/{id}', [UserController::class, 'show']);

```

##### 🧩 Параметры маршрутов

Laravel позволяет легко вырезать динамические данные прямо из URL-адреса:

* Обязательные параметры: Оборачиваются в фигурные скобки {id}. Переменная автоматически передается аргументом в функцию или метод контроллера.
* Необязательные параметры: Помечаются знаком вопроса {name?}. В аргументах функции для такой переменной обязательно нужно указать значение по умолчанию.
* Регулярные выражения (Валидация): С помощью метода where можно жестко ограничить формат параметра (например, разрешить только цифры).


```php

// ID должен быть строго числом, а имя — необязательное
Route::get('/user/{id}/{name?}', function (int $id, string $name = 'Гость') {
    return "User: {$id}, Name: {$name}";
})->where(['id' => '[0-9]+', 'name' => '[a-zA-Z]+']);

```

##### 🏷️ Именованные маршруты

Вы можете присвоить любому роуту уникальное имя с помощью метода name(). Это критически важно для того, чтобы не «прибивать гвоздями» ссылки к HTML-шаблонам.
Если завтра вы решите изменить URL-адрес личного кабинета с `/profile-dashboard-v2` на `/my-profile`, вам не придется переписывать сотни ссылок на сайте. Достаточно изменить URL в файле роутов, а имя оставить прежним.

```php

// Объявление роута с именем
Route::get('/user/profile', [UserProfileController::class, 'show'])->name('profile');

// Генерация ссылки в коде или Blade-шаблоне:
$url = route('profile'); // Вернет: http://your-site.com

// Редирект по имени:
return redirect()->route('profile');

```

##### 👥 Группировка маршрутов (Route Groups)

Чтобы не дублировать настройки для десятков похожих роутов, их объединяют в группы с помощью метода `Route::group()` или цепочки методов:

* Middleware: Применяет посредников (например, проверку авторизации auth) ко всей группе.
* Префиксы (Prefix): Добавляет общую часть к URL (например, все роуты внутри группы будут начинаться с /admin/...).
* Пространства имен или Имена (As): Добавляет префикс к именам всех роутов в группе (например, admin.dashboard).

```php

Route::middleware(['auth'])->prefix('admin')->name('admin.')->group(function () {
    // URL: /admin/users  |  Имя: admin.users
    Route::get('/users', [AdminUserController::class, 'index'])->name('users');
    
    // URL: /admin/posts  |  Имя: admin.posts
    Route::get('/posts', [AdminPostController::class, 'index'])->name('posts');
});

```

##### ⚡ Route Model Binding (Связывание моделей)

Это одна из самых удобных фич Laravel. Если имя параметра в роуте {user} совпадает с именем переменной в контроллере, и у переменной указан тип модели User $user, Laravel автоматически сделает запрос в базу данных (`User::findOrFail($id)`) и передаст в контроллер уже готовый объект. Если запись в БД не найдена, фреймворк сам вернет пользователю страницу «404 Not Found».

```php
// Роут:
Route::get('/users/{user}', [UserController::class, 'show']);

// Контроллер:
public function show(User $user) {
    // В $user уже лежит готовый объект пользователя из базы!
    return view('user.profile', compact('user'));
}
```

##### Middleware

Middleware (Посредник) — это механизм фильтрации HTTP-запросов, который стоит между входящим запросом пользователя и конечной точкой (роутом или контроллером). Это слои луковицы, через которые запрос должен пройти внутрь приложения, а затем ответ (Response) должен пройти обратно наружу.
Работа типичного middleware устроена по следующему алгоритму:

   1. Запрос поступает в систему.
   2. Посредник проверяет входящие данные (например, авторизован ли пользователь, валиден ли токен).
   3. Если проверка не прошла — посредник обрывает цепочку и возвращает ответ (например, редирект на страницу логина или ошибку 401 Unauthorized).
   4. Если проверка прошла успешно — посредник вызывает функцию $next($request), передавая запрос следующему слою или контроллеру.

```php
// Пример базовой структуры посредника:
namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;

class CheckAge
{
    public function handle(Request $request, Closure $next)
    {
        // 1. Действие ДО выполнения запроса
        if ($request->age <= 18) {
            return redirect('home'); // Обрыв цепочки
        }

        // 2. Передача запроса дальше
        return $next($request);
    }
}
```

## 📁 Классификация Middleware в Laravel

Все посредники делятся на три основные категории в зависимости от области их применения:

   1. Глобальные (Global Middleware): Запускаются при каждом HTTP-запросе к приложению. Сюда входят базовые системные проверки: проверка конфигурации прокси, отлов исключений, очистка строк от лишних пробелов (TrimStrings).
   2. Группы посредников (Middleware Groups): Наборы посредников, сгруппированные под одним ключом (обычно web и api).
   * Группа web отвечает за сессии, куки, старт аутентификации и защиту от CSRF-атак.
      * Группа api нацелена на работу без состояния (stateless) и включает ограничение частоты запросов (ThrottleRequests).
   3. Назначенные роутам (Route Middleware): Посредники, которые вы вешаете точечно на конкретные маршруты или группы маршрутов (например, auth, can).

(Примечание: Начиная с Laravel 11, регистрация и управление кастомными middleware перенесены в файл bootstrap/app.php).


##### 🎯 Когда использовать Middleware

Посредники незаменимы в сценариях, когда логику нужно применить к группе страниц, не дублируя код в десятках контроллеров:

* Аутентификация и авторизация (auth): Проверка, вошел ли пользователь в систему, и есть ли у него роль Администратора для доступа к панели управления.
* Защита от спама и парсинга (throttle): Ограничение количества запросов к серверу (например, разрешить отправку формы восстановления пароля не чаще 3 раз в минуту).
* Логирование и аналитика: Сбор статистики посещений, запись подозрительных запросов или замер времени выполнения скрипта (выполняется как After Middleware после метода $next).
* Модификация данных на лету: Изменение формата ответа (например, приведение всех JSON-ключей к формату camelCase) или локализация приложения (установка языка сайта на основе куки или заголовка Accept-Language).
* Технические работы (Maintenance Mode): Встроенный посредник PreventRequestsDuringMaintenance проверяет, не запущена ли команда php artisan down, и если да — отдает заглушку "Сайт на обслуживании".

## Контроллеры (Controllers)


##### ⚙️ Что такое Контроллер и как он работает

Контроллер — это компонент архитектуры MVC, который группирует логику обработки связанных HTTP-запросов в одном классе. Вместо того чтобы писать всю бизнес-логику сайта внутри анонимных функций в файлах роутов (routes/web.php), её выносят в методы контроллеров. Это делает код структурированным, чистым и удобным для тестирования.
Контроллеры создаются через консоль: `php artisan make:controller UserController`. Они располагаются в папке `app/Http/Controllers/`.
Типичный контроллер принимает объект запроса (Request), обращается к моделям или сервисам за данными и возвращает ответ (Response в виде HTML-представления или JSON):

```php

namespace App\Http\Controllers;

use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\View\View;

class UserController extends Controller
{
    // Метод контроллера (Action)
    public function show(int $id): View
    {
        $user = User::findOrFail($id); // Получаем данные
        
        return view('user.profile', ['user' => $user]); // Возвращаем шаблон
    }
}

```

##### 📦 Виды контроллеров в Laravel

В зависимости от задач, Laravel предлагает три удобных формата контроллеров:

##### 1. Классический контроллер

Содержит произвольный набор публичных методов (index, show, store, customAction), каждый из которых привязывается к своему роуту вручную.

##### 2. Одноцелевой контроллер (Single Action / Invokable)

Если контроллер выполняет ровно одно сложное действие (например, обработка регистрации или проведение платежа), в нем объявляют всего один магический метод `__invoke()`. При привязке в роутере имя метода указывать не нужно.

```php

// Создание: php artisan make:controller RegisterController --invokable
class RegisterController extends Controller {
    public function __invoke(Request $request) {
        // Логика регистрации
    }
}
// В роутере: Route::post('/register', RegisterController::class);

```

##### 3. Ресурсный контроллер (Resource Controller)

Идеальный инструмент для управления сущностями (CRUD: создание, чтение, обновление, удаление). Команда `php artisan make:controller PostController --resource` автоматически создаст класс с 7 готовыми методами (index, create, store, show, edit, update, destroy).
В файле роутов вместо прописывания 7 строк достаточно вызвать один метод:

`Route::resource('posts', PostController::class)`;

##### 🎯 Лучшие практики: Принцип "Тонких контроллеров"

Главная ошибка начинающих разработчиков — превращать контроллеры в монолиты (писать там SQL-запросы, отправку почты, валидацию и расчеты). В профессиональной разработке принято следовать правилу Skinny Controllers, Fat Models/Services (Тонкие контроллеры, толстые модели/сервисы).
Что ДОЛЖЕН делать контроллер:

* Принять HTTP-запрос.
* Передать данные на валидацию (лучше всего использовать Form Requests).
* Вызвать нужный метод бизнес-логики (передать задачу в Сервис или Модель).
* Получить результат и вернуть пользователю ответ (JSON, Редирект или Blade-вид).

Чего в контроллере быть НЕ должно:

* Сложных математических расчетов и алгоритмов.
* Прямых тяжелых SQL-запросов (для этого есть репозитории или Eloquent скоупы).
* Кода отправки СМС, генерации PDF, общения со сторонними API (выносите это в Service-классы).

##### Request / Response

Request — объект HTTP запроса с данными формы, файлами, заголовками, cookies.

Response — объект ответа, возвращает view, json, redirect, download, stream.

Важно: Валидация через $request->validate(), работа с файлами через file(), макросы для расширения, JSON для API.

* Request — входные данные ($request->input(), $request->file())
* Response — возврат данных (view, json, download)

Request:

- $request->input() — форма данные
- $request->query() — query параметры
- $request->file() — загрузка файлов
- $request->header() — заголовки

Response:

- response()->json() — API ответы
- view() — HTML страницы
- redirect() — перенаправления
- response()->download() — скачивание файлов

##### 📥 Входящий запрос (Request)

Объект Illuminate\Http\Request инкапсулирует в себе абсолютно всю информацию о текущем HTTP-запросе: от GET-параметров URL и отправленных форм до cookies и HTTP-заголовков. Laravel автоматически внедряет этот объект в методы контроллеров с помощью тайпхинтинга.

##### Основные методы работы с данными:

* `$request->all()` — возвращает массив всех переданных параметров (из формы и URL) [].
* `$request->input('key', 'default')` — безопасно достает конкретное значение []. Если ключа нет, вернет значение по умолчанию [].
* `$request->only(['username', 'email'])` / `except()` — возвращает только указанные поля (или все, кроме указанных) []. Защищает от передачи лишних данных.
* `$request->has('key')` / `filled()` — проверяет, присутствует ли ключ в запросе, а `filled()` дополнительно проверяет, что поле не пустое [].
* `$request->file('avatar')` — возвращает объект загруженного файла (UploadedFile), предоставляя методы для его валидации и сохранения на сервере [].

##### Проверка метаданных:

* `$request->path()` / `url()` — возвращает путь (например, api/users) или полный URL-адрес [].
* `$request->isMethod('post')` — проверяет HTTP-метод [].
* `$request->ajax()` / `wantsJson()` — определяет, является ли запрос AJAX-вызовом или ожидает ли клиент ответ в формате JSON (удобно для разделения логики Web/API).

##### 🛡️ Продвинутая обработка: Form Requests

Для того чтобы не загромождать контроллер кодом валидации, в Laravel используются Form Requests. Это кастомные классы запросов, которые создаются командой `php artisan make:request StorePostRequest`.
Они инкапсулируют логику проверки прав и валидации полей еще до того, как запрос попадет в метод контроллера:

```php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;

class StorePostRequest extends FormRequest
{
    // 1. Проверяем, имеет ли право пользователь совершить это действие
    public function authorize(): bool
    {
        return $this->user()->can('create-posts');
    }

    // 2. Задаем правила валидации
    public function rules(): array
    {
        return [
            'title' => 'required|string|unique:posts|max:255',
            'body' => 'required|string|min:10',
        ];
    }
}

```

Если валидация провалилась, Laravel автоматически вернет пользователя назад с ошибками (для веб-страниц) или отдаст 422 Unprocessable Entity JSON-ответ (для API).

##### 📤 Исходящий ответ (Response)

Все методы контроллеров должны возвращать ответ. Это может быть обычная строка (Laravel сам превратит её в HTTP-ответ), но чаще всего используются специализированные объекты ответов с помощью глобального хелпера response() или фасада Response.

##### Популярные типы ответов:

1. Возврат HTML-представления (Blade View)

`return view('posts.show', compact('post'))`;

2. JSON-ответы (Идеально для API)
Автоматически превращает массивы или коллекции в JSON-строку и выставляет правильный заголовок 

```php

Content-Type: application/json.

return response()->json([
    'success' => true,
    'data' => $user
], 200); // 200 — HTTP-статус успешного запроса

```

3. Перенаправления (Redirects)
Отправляют пользователя на другой URL, именованный маршрут или метод контроллера. Метод `with()` позволяет передать одноразовые данные сессии (Flash-сообщения).

`return redirect()->route('posts.index')->with('status', 'Пост успешно создан!')`;

4. Скачивание и отображение файлов

* `download()` — принудительно запускает скачивание файла у пользователя.
* `file()` — отображает файл (например, картинку или PDF) прямо в браузве за счет правильных заголовков.

`return response()->download(storage_path('exports/report.xlsx'))`;

5. Кастомизация заголовков и статус-кодов
Вы можете гибко управлять HTTP-статусами и заголовками, выстраивая методы в цепочку.

```php

return response('Контент не найден', 404)
            ->header('Content-Type', 'text/plain')
            ->header('X-Custom-Header', 'Value');

```
