
## Пространства имён (Namespaces)

##### Что такое namespace

Namespace (пространство имен) — это виртуальная папка для кода. 
Она решает проблему конфликта имен, позволяя создавать классы, 
функции и константы с одинаковыми названиями в разных модулях.

##### Объявление namespace

* Пишется в самом начале файла через ключевое слово namespace.
* Допускается только инструкция `declare` перед ним.
* Разделяется обратным слэшем `\`.

```php
<?php
namespace App\Modules\User;

class Profile {}
```

##### use, as, группировка

- `use`: импортирует класс, функцию или константу в текущий файл, чтобы не писать полный путь.
- `as`: создает псевдоним (алиас), если имена импортируемых классов совпадают.
- Группировка: позволяет импортировать несколько элементов из одного пространства в одной строке.

```php
// Импорт и алиас
use App\Models\User as Account;

// Группировка
use App\Services\{Logger, Auth, Transport\Http};
```
##### Глобальный namespace

- Это корневое пространство (все встроенные функции PHP: `strlen`, `array_merge`, классы `Exception` и т.д.).
- Если вы находитесь внутри namespace, для вызова глобального класса перед ним нужно ставить обратный слэш `\`.

```php
namespace App;

// Без слэша PHP будет искать App\Exception
throw new \Exception('Error'); 
```
##### namespace и автозагрузка (PSR-4)

- `PSR-4` — стандарт, связывающий структуру `namespace` с реальной структурой папок на диске.
- Префикс пространства (например, `App\`) привязывается к конкретной папке (например, `src/`).
- Дальнейший путь в `namespace` в точности повторяет подпапки и имя файла, 
  что позволяет `Composer` автоматически загружать нужные классы.

##### NAMESPACE константа

- Магическая константа.
- Возвращает строку с именем текущего пространства имен.
- Если код находится в глобальном пространстве, возвращает пустую строку "".

```php
namespace App\Tools;

echo __NAMESPACE__; // Выведет: App\Tools
```

##### namespace_alias для функций и констант

- В PHP нет прямой функции `namespace_alias`, но есть `class_alias` для классов.
- Для функций и констант используется конструкция `use function` и `use const` с возможностью переименования через `as`.

```php
use function App\Math\calculate as runCalc;
use const App\Math\PI as MATH_PI;
```

## Автозагрузка (Composer, PSR-4)


- Aвтозагрузка - aвтоматическая загрузка классов без `require/include`.
  Автозагрузка автоматически подключает классы без `require`. `Composer` генерирует `autoloader` по `PSR-4`. 
  Достаточно `require vendor/autoload.php` один раз. `Composer` связывает `namespace` со структурой папок.

```php
// БЕЗ автозагрузки (старый способ)
require_once 'app/Models/User.php';
require_once 'app/Services/UserService.php';
require_once 'app/Repositories/UserRepository.php';

$user = new App\Models\User();
$service = new App\Services\UserService();

// С автозагрузкой
// require_once 'vendor/autoload.php';  // Только один раз

$user = new App\Models\User();  // Автоматически загружает app/Models/User.php
$service = new App\Services\UserService();  // Автоматически загружает app/Services/UserService.php
```

##### Базовый механизм PHP

* `spl_autoload_register` — встроенная функция PHP.
   Она регистрирует вашу собственную функцию-загрузчик. 
   Когда код обращается к неизвестному классу, 
   PHP автоматически вызывает этот загрузчик, 
   чтобы найти и подключить нужный файл через `include` или `require`.

##### Стандарты и Composer

* `PSR-4` стандарт — современное правило именования файлов и папок.
 Стандарт автозагрузки: `namespace` соответствует структуре папок.
 Он связывает пространство имен (namespace) класса с конкретной папкой на диске. 
 Например, класс `App\Service\User` должен лежать в файле `src/Service/User.php`.

* `composer dump-autoload` — команда Composer, которая перезапускает процесс сборки карты классов. 
Команда для регенерации файлов автозагрузки. Ее запускают, когда добавляют новые 
классы (для classmap) или меняют настройки автозагрузки в `composer.json`.

```php
// 1. Добавили новый namespace в composer.json
{
    "autoload": {
        "psr-4": {
            "App\\": "app/",
            "MyPackage\\": "packages/my-package/src/"  // Новый namespace
        }
    }
}

// 2. Регенерировать autoload
composer dump-autoload

// 3. Теперь можно использовать
use MyPackage\Services\MyService;
$service = new MyService();

// Для продакшена (быстрее)
composer dump-autoload --optimize

// Laravel Artisan (обёртка)
php artisan optimize  # Включает composer dump-autoload -o

// CI/CD pipeline
composer install --no-dev --optimize-autoloader
```

##### Режимы автозагрузки в Composer

* `classmap` автозагрузка — режим, при котором `Composer` сканирует указанные папки 
   и создает точную карту: "Имя класса => Путь к файлу". 
   Работает быстрее всего, но карту нужно обновлять вручную через `dump-autoload` при создании каждого нового класса.

* `files` автозагрузка — режим для принудительного подключения файлов при каждом запросе. 
   Обычно используется для загрузки глобальных функций (хелперов), которые не обернуты в классы.

* `autoload-dev` — секция в composer.json для автозагрузки файлов, нужных только при разработке. 
   Сюда прописывают пути к тестам (например, PHPUnit).
   В продакшене эти файлы отключаются командой `composer install --no-dev`.


## Исключения и ошибки

##### try-catch-finally

Конструкция для перехвата и обработки исключений (ошибок), 
которая позволяет программе продолжать работу, даже если что-то пошло не так.

* `try` — блок, в котором выполняется потенциально опасный код. 
   Если внутри него происходит ошибка, выполнение блока прерывается.

* `catch` — блок, который ловит исключение определенного типа. 
   Внутри него прописывают логику спасения: запись в лог, вывод сообщения или откат транзакции. 
   Можно использовать несколько блоков `catch` для разных типов ошибок.

* `finally` — блок, который выполняется всегда, независимо от того, 
   произошла ошибка или нет (даже если внутри `try` или `catch` сработал `return`). 
   Обычно здесь закрывают соединения с базой данных или удаляют временные файлы.

```php
try {
    $db->connect();
    // потенциальная ошибка
} catch (Exception $e) {

    echo "Ошибка: " . $e->getMessage();

} finally {

    $db->close(); // выполнится в любом случае
}
```
------------------------------
##### throw (выброс исключения)

Оператор `throw` используется для принудительного прерывания нормального хода программы и создания ошибки. 
Выбрасывать можно только объекты, которые реализуют интерфейс `Throwable` (классы `Exception`, `Error` и их наследники).

Когда срабатывает `throw`, PHP мгновенно прекращает выполнение текущей функции 
и ищет ближайший блок `catch`. Если его нет, программа завершается с критической ошибкой.

```php
function divide($a, $b) {
    if ($b === 0) {
        throw new InvalidArgumentException("Деление на ноль невозможно");
    }
    return $a / $b;
}
```

##### Кастомные исключения

Это собственные классы ошибок, которые создаются для разделения логики. 
Например, для ошибок валидации можно создать `ValidationException`, 
а для ошибок оплаты — `PaymentException`.

Чтобы создать кастомное исключение, достаточно унаследоваться от базового класса Exception. 
Это позволяет точечно перехватывать конкретные проблемы в блоках `catch`, не затрагивая другие системные ошибки.

```php
class AccountBlockedException extends Exception {}

// Использование
try {

    if ($user->isBlocked()) {
        throw new AccountBlockedException("Доступ запрещен");
    }

} catch (AccountBlockedException $e) {
    // обрабатываем только блокировку аккаунта
}
```

##### Exception методы

Базовый класс Exception предоставляет встроенные методы для получения полной информации о том, где и почему произошел сбой:

* `getMessage()` — возвращает текст сообщения об ошибке, переданный при создании.
* `getCode()` — возвращает числовой код ошибки (если он был задан).
* `getFile()` — возвращает абсолютный путь к файлу, в котором произошел выброс исключения.
* `getLine()` — возвращает номер строки кода, на которой сработал throw.
* `getTrace()` — возвращает массив с подробной историей вызовов функций (стек вызовов) до точки ошибки.
* `getTraceAsString()` — возвращает тот же стек вызовов, но в виде готовой строки для вывода на экран или в лог.

##### Error (PHP 7.0+)

До версии PHP 7.0 большинство внутренних системных сбоев 
(например, вызов несуществующей функции) вызывали фатальные ошибки (`Fatal Error`), 
которые намертво останавливали скрипт. С PHP 7.0 появился класс `Error`.
`Error`, как и `Exception`, наследует общий интерфейс `Throwable`. 
Теперь такие критические сбои, как `TypeError` (ошибка типов) или `ParseError` (ошибка синтаксиса), 
можно перехватывать внутри `try-catch`, не позволяя приложению упасть.

```php
try {
    $result = nonExistentFunction(); // Фатальная ошибка в старом PHP
} catch (Error $e) {
    echo "Поймали системную ошибку: " . $e->getMessage();
}
```

##### set_exception_handler и set_error_handler

Глобальные перехватчики, которые используются для создания единой точки логирования ошибок на всем сайте.

* `set_error_handler` — регистрирует функцию для перехвата старых классических предупреждений PHP (Notice, Warning). 
   Позволяет превратить обычные предупреждения в полноценные исключения (через класс `ErrorException`), 
   чтобы их тоже можно было ловить в try-catch.

* `set_exception_handler` — регистрирует функцию, которая сработает, если исключение (или Error) 
   не было поймано ни в одном блоке catch. 
   Это "последний рубеж" обороны приложения. 
   Обычно здесь красиво выводят заглушку "500 Ошибка сервера" для пользователя и пишут лог для разработчиков.

```php
// Глобальный перехватчик забытых исключений
set_exception_handler(function (Throwable $e) {
    Log::write($e->getMessage());
    echo "Упс, что-то пошло не так. Мы уже чиним.";
    exit;
});
```

## 🌐 Что такое PSR

`PSR` (PHP Standard Recommendations) — это стандарты рекомендаций по разработке на PHP, 
которые создает и развивает группа PHP-FIG (PHP Framework Interop Group). 
В эту группу входят создатели крупнейших фреймворков (Laravel, Symfony, Yii) и CMS.

Главная цель PSR — обеспечить интероперабельность, то есть сделать так, 
чтобы код разных разработчиков и разные библиотеки могли легко 
и без конфликтов работать вместе в одном проекте.

##### 📝 PSR-1: Basic Coding Standard

Базовый стандарт кодирования, определяющий минимальные технические 
требования к оформлению кода для обеспечения совместимости.

* Кодировка: файлы должны быть строго в `UTF-8` без BOM.
* Теги: разрешено использовать только длинные теги `<?php` и короткие `<?=.`
* Разделение обязанностей: файл должен либо объявлять структуры (классы, функции, константы), 
  либо вызывать побочные эффекты (выводить текст, менять настройки), 
  но не делать и то, и другое одновременно.
* Имена: имена классов пишутся в StudlyCaps (CamelCase с большой буквы), 
  константы — в UPPER_CASE, а методы — в camelCase (с маленькой буквы).

##### 🎨 PSR-12: Extended Coding Style Guide

Расширенное руководство по стилю кода. 
Оно заменяет старый стандарт PSR-2 и подробно описывает, 
где ставить пробелы, табы и фигурные скобки, 
чтобы код всей команды выглядел одинаково.

* Отступы: строго 4 пробела, использовать табы (Tab) запрещено.
* Скобки: открывающая фигурная скобка класса и метода ставится на следующей строке. 
  Для управляющих конструкций (if, for, foreach) — на той же строке.
* Ключевые слова: все ключевые слова PHP (типы данных, `true`, `false`, `null`) должны быть в нижнем регистре.
* Строки: длина строки не должна жестко превышать 120 символов.

```php
namespace App\Service;

class UserService // Скобка на новой строке (PSR-12)
{
    public function updateStatus(int $id, string $status): void 
    {
        if ($status === 'active') { // Скобка на той же строке
            // код
        }
    }
}
```

##### 🪵 PSR-3: Logger Interface

Стандарт определяет единый PHP-интерфейс для библиотек логирования 
(самая известная реализация — Monolog).

Он содержит 8 методов для разных уровней критичности логов 
(отладочные, информационные, критические ошибки): 

- `debug()`, 
- `info()`, 
- `notice()`, 
- `warning()`, 
- `error()`, 
- `critical()`, 
- `alert()`, 
- `emergency()`.

Благодаря этому стандарту вы можете в любой момент 
заменить одну библиотеку логирования на другую 
во всем проекте, не меняя код вызова логов.

```php
// Любой PSR-3 совместимый логгер будет работать так:
$logger->info("Пользователь обновил профиль", ['user_id' => 42]);
$logger->error("Не удалось подключиться к API платежной системы");
```

##### 📂 PSR-4: Autoloading Standard

Стандарт автоматической загрузки классов из файлов. 
Он связывает пространство имен (namespace) вашего кода со структурой папок на диске.

* Позволяет отказаться от ручного подключения файлов через require или include.
* Устанавливает правило: базовый `namespace` (например, `App\`) привязывается к корневой папке (например, `src/`). Дальнейшая вложенность пространств имен в точности повторяет подпапки.
* Класс `App\Controller\MainController` автозагрузчик будет искать строго по пути `src/Controller/MainController.php`.


##### 📦 PSR-11: Container Interface

Стандарт описывает интерфейс для DI-контейнеров (Dependency Injection), 
которые управляют созданием и зависимостями объектов в приложении.
Интерфейс содержит всего два основных метода:

* `get($id)` — достает нужный объект (сервис) из контейнера по его имени или интерфейсу.
* `has($id)` — проверяет, зарегистрирован ли такой объект в контейнере.

Это позволяет фреймворкам и внешним библиотекам одинаково общаться с контейнером, 
независимо от того, какой именно компонент используется под капотом (PHP-DI, Symfony Container и т.д.).

##### ⚡ PSR-16: Simple Cache

Стандарт "простого кэша" (простой интерфейс кэширования для базовых задач вроде "сохранить-получить-удалить").
Он предоставляет лаконичный интерфейс `CacheInterface` со знакомыми методами: 

- `get()`, 
- `set()`, 
- `delete()`,
- `clear()`,
- `has()`, 

а также их flat-версии для работы с массивами (`getMultiple()`, `setMultiple()`). 
Используя PSR-16, вы можете легко переключать хранилище кэша вашего приложения между файлами,
Redis или Memcached, используя один и тот же код.

```php
// Пример работы с любым PSR-16 кэшем
if (!$cache->has('users_list')) {
    $cache->set('users_list', $db->getUsers(), 3600); // Кэш на 1 час
}
$users = $cache->get('users_list');
```

## Генераторы (Generators)

##### ⚙️ Что такое генератор

Генератор — это особая функция в PHP, которая позволяет перебирать большие объемы данных 
(например, строки из тяжелого файла или миллионы записей из базы) 
без загрузки всего этого массива в оперативную память.
В отличие от обычной функции, которая возвращает всё значение целиком через return и завершает работу, 
генератор отдает элементы по одному по мере необходимости, 
временно "замораживая" свое состояние между шагами цикла foreach.


##### 🔑 yield ключевое слово

Ключевое слово yield — это сердце любого генератора. 
Оно работает похоже на return, но вместо окончательного завершения функции 
оно лишь временно приостанавливает её выполнение и отдает текущее значение наружу.
Когда вызывающий цикл foreach запрашивает следующий элемент, 
генератор "просыпается" и продолжает работу ровно с того места, 
где остановился (сохраняя значения всех локальных переменных).

```php
function getNumbers() {
    yield 1;
    yield 2;
    yield 3;
}

foreach (getNumbers() as $number) {
    echo $number; // Выведет: 123
}
```

##### 🔍 Generator методы

Когда вы вызываете функцию-генератор, PHP автоматически возвращает встроенный объект класса Generator. 
Этот класс реализует интерфейс `Iterator`, предоставляя методы для ручного управления потоком данных:

- `current()` — возвращает текущее значение, выданное через `yield`.
- `key()` — возвращает текущий ключ (индекс) элемента.
- `next()` — возобновляет работу генератора и переходит к следующему `yield`.
- `valid()` — проверяет, не завершился ли генератор (есть ли еще данные для перебора).
- `send($value)` — уникальный метод, который позволяет не только брать данные из генератора, 
   но и передавать значения внутрь него прямо в процессе работы, превращая генератор в двусторонний канал (сопрограмму/корутину).
- `getReturn()` — позволяет получить финальное значение, если в конце генератора был использован классический return (доступно с PHP 7.0).

##### ⛓️ yield from (PHP 7.0+)

Оператор `yield from` позволяет делегировать генерацию данных другому источнику. 
Это может быть другой генератор, обычный массив или любой объект, 
который можно перебрать в цикле (реализующий интерфейс Traversable).
Это избавляет от необходимости писать вложенные циклы foreach 
для перебора подмассивов или дочерних генераторов, делая код чище и быстрее.

```php
function countToFive() {
    yield from [1, 2]; // Делегирование массиву
    yield from getRestNumbers(); // Делегирование другому генератору
}

function getRestNumbers() {
    yield 3;
    yield 4;
    yield 5;
}
```

##### 📊 Генераторы vs массивы

Главная битва между генераторами и массивами идет за ресурсы сервера — память против процессора.

| Критерий | Массивы 📂 | Генераторы ⚙️ |
|---|---|---|
| Расход памяти | Пропорционален размеру массива. Миллион строк займет сотни мегабайт RAM. | Константный (минимальный). В памяти всегда находится только один текущий элемент. |
| Скорость старта | Медленный. Нужно дождаться, пока сгенерируются или загрузятся все элементы. | Мгновенный. Функция начинает отдавать первые данные сразу без задержки. |
| Повторный перебор | Можно перебирать сколько угодно раз в любых направлениях. | Одноразовый. После завершения работы генератор нельзя перезапустить, его нужно вызывать заново. |
| Функционал | Доступны встроенные функции count(), array_map(), доступ по индексу $arr[5]. | Нельзя посчитать через count(), нельзя получить элемент по индексу без полного перебора. |

- Когда выбирать массивы: когда данных немного (до пары тысяч элементов), 
  или вам нужно постоянно возвращаться к старым элементам, сортировать их или искать по индексам.

- Когда выбирать генераторы: когда вы читаете гигабайтные логи, 
  парсите огромные CSV-файлы или делаете выборку миллионов строк из БД.


## Рефлексия (Reflection API)

##### 🪞 Что такое Reflection

`Reflection` (рефлексия или отражение) — это мощный механизм PHP, 
который позволяет программе исследовать собственную структуру прямо во время выполнения (в рантайме).

С помощью рефлексии код может «заглянуть внутрь» самого себя: узнать, какие в системе есть классы, 
какие методы и свойства в них объявлены, какие у них типы данных и уровни доступа (`public`, `protected`, `private`).

Это фундамент для создания продвинутых инструментов: DI-контейнеров, 
тест-фреймворков (PHPUnit) и ORM для работы с базами данных.


##### 🏗️ ReflectionClass

Класс, предназначенный для полного анализа конкретного класса, интерфейса или трейта.
Он позволяет узнать всё о целевом объекте: 
имя его родительского класса, реализованные интерфейсы, а также получить списки его методов и свойств. 

Более того, `ReflectionClass` позволяет принудительно создать экземпляр класса, 
даже если его конструктор объявлен как приватный (часто используется в паттерне `Singleton` при тестировании).


```php
$reflector = new ReflectionClass(UserService::class);

echo $reflector->getName(); // Выведет имя класса
$hasMethod = $reflector->hasMethod('save'); // Проверит наличие метода
```

##### 🛡️ ReflectionProperty

Класс для анализа и управления свойствами (переменными) внутри класса.

Главная суперсила `ReflectionProperty` — возможность обходить модификаторы доступа. 
С его помощью можно программно сделать приватное свойство доступным (`setAccessible(true)`), 
прочитать его текущее значение или изменить его в обход стандартных методов класса.

```php
$property = new ReflectionProperty(User::class, 'password');
// Позволяет читать/писать даже private-свойства (в PHP 8.1+ работает автоматически)
$property->setAccessible(true); 
$property->setValue($userObject, 'new_secret_password');

```

##### ⚙️ ReflectionMethod

Класс для исследования и динамического вызова методов класса.

Он позволяет узнать уровень доступа метода, является ли он статическим или абстрактным, 
а также запустить его на выполнение с помощью метода `invoke()` или `invokeArgs()`. 
Как и в случае со свойствами, `ReflectionMethod` умеет вызывать `private` и `protected` методы, 
что незаменимо при написании юнит-тестов для скрытой логики.

```php
$method = new ReflectionMethod(Invoice::class, 'calculateTax');
// Вызов приватного метода calculateTax() на объекте $invoice
$result = $method->invoke($invoiceObject, $arg1, $arg2);
```

##### 📋 ReflectionParameter

Класс, который предоставляет подробную информацию о конкретном аргументе (параметре) функции или метода.
С его помощью DI-контейнеры понимают, какие зависимости нужно передать в конструктор класса. Он позволяет узнать:

- Имя параметра.
- Его ожидаемый тип (тайпхинт).
- Является ли параметр обязательным.
- Какое у него значение по умолчанию (если оно есть).

```php
$method = new ReflectionMethod(OrderController::class, '__construct');
foreach ($method->getParameters() as $param) {
    echo $param->getName(); // Имя аргумента (например, 'db')
    echo $param->getType(); // Его тип (например, 'DatabaseConnection')
}
```

##### 🏷️ Атрибуты (Attributes, PHP 8.0+)

Атрибуты (в других языках называемые аннотациями) — это официальный синтаксис PHP 
для добавления структурированных метаданных к классам, методам, свойствам и константам. 
Они заменили собой старые текстовые комментарии `/** @Route("/api") */`.

Сами по себе атрибуты — это просто декоративные метки в коде, которые ничего не делают. 
Но в сочетании с Reflection они превращаются в мощный инструмент настройки программы. 
Фреймворки считывают эти метки через рефлексию, 
чтобы автоматически настраивать маршруты (роутинг), правила валидации или связи в БД.

```php
// 1. Объявляем класс атрибута
#[Attribute]
class Route {
    public function __construct(public string $path) {}
}

// 2. Используем его на методе контроллера
class ProductController {
    #[Route('/products')]
    public function list() {}
}

// 3. Читаем атрибут через Рефлексию в ядре системы

$methodRef = new ReflectionMethod(ProductController::class, 'list');
$attributes = $methodRef->getAttributes(Route::class);

foreach ($attributes as $attribute) {
    $routeInstance = $attribute->newInstance(); // Создает объект Route
    echo $routeInstance->path; // Выведет: /products
}
```

## Новые возможности PHP 8.x

##### 🏷️ Named Arguments (Именованные аргументы, PHP 8.0)

Позволяют передавать значения в функцию, указывая имя параметра, а не только его порядковый номер.

- Гибкость: можно передавать аргументы в любом порядке.
- Пропуск дефолтных значений: если у функции 10 параметров со значениями по умолчанию, а вам нужно изменить только последний, больше не нужно переписывать все предыдущие.
- Читаемость: код становится самодокументируемым.

```php
// Раньше: приходилось писать null/шаблонные значения для пропуска

setcookie('name', 'value', 0, '', '', true, true);

// С PHP 8.0: передаем только то, что нужно, в любом порядке
setcookie(name: 'name', value: 'value', httponly: true, secure: true);
```

##### 🔀 Union Types (Объединения типов, PHP 8.0)

Позволяют указывать несколько возможных типов для одного аргумента, 
свойства или возвращаемого значения, разделяя их вертикальной чертой |.
До PHP 8.0 разработчикам приходилось полагаться на комментарии PHPDoc `/** @param string|int $id */`. 
Теперь типы проверяются строго на уровне ядра PHP во время выполнения. 
Также появился псевдотип false (полезно для функций, возвращающих результат или false при ошибке) 
и тип mixed, заменяющий собой любое значение.

```php
class Target {
    // Свойство может быть строкой или числом
    private int|string $id; 

    // Метод принимает массив или объект, а возвращает строку или null
    public function process(array|object $data): string|null {
        // ...
    }
}
```

##### 🎯 Match Expression (Выражение match, PHP 8.0)

Современная, безопасная и лаконичная альтернатива старому оператору `switch`.

- Возвращает значение: результат `match` можно сразу присвоить переменной.
- Строгое сравнение: использует проверку типов (`===`), в отличие от `switch`, который сравнивает нестрого (`==`).
- Нет проваливаний: не требует писать break после каждого условия.
- Обязательность: если совпадение не найдено и нет блока `default`, PHP выбросит исключение `UnhandledMatchError`.

```php
$status = 200;

$message = match ($status) {

    200, 201 => 'Успешно',
    404 => 'Не найдено',
    500 => 'Ошибка сервера',
    default => 'Неизвестный статус',

};
```

##### 🛡️ Nullsafe Operator ?-> (PHP 8.0)

Оператор для безопасной работы с цепочками объектов, 
которые могут возвращать null.
Вместо написания громоздких вложенных проверок `if ($user !== null) ...`, 
вы просто ставите знак вопроса перед стрелочкой. 
Если любой элемент в цепочке вернет null, выполнение всей строки мгновенно остановится, 
и итоговый результат станет `null`, не вызывая фатальной ошибки Call to a member function on null.

```php
// Раньше
$country = null;
if ($session !== null) {
    $user = $session->getUser();
    if ($user !== null) {
        $country = $user->getAddress()->getCountry();
    }
}

// С PHP 8.0
$country = $session?->getUser()?->getAddress()?->getCountry();
```

##### 🏗️ Constructor Property Promotion (Объявление свойств в конструкторе, PHP 8.0)

Синтаксический сахар, который позволяет объединить объявление свойств класса 
и их инициализацию через конструктор в одну короткую строку.
Вы просто добавляете модификатор доступа (`public`, `protected`, `private`) прямо к аргументам конструктора. 
PHP под капотом сам создаст эти свойства у класса и присвоит им переданные значения. 
Это сокращает количество шаблонного кода (boilerplate) примерно в 4 раза.

```php
// Раньше: писали имя свойства в 3 разных местах
class User {
    public string $name;
    public function __construct(string $name) {
        $this->name = $name;
    }
}

// С PHP 8.0: магия
class User {
    public function __construct(
        public string $name,
        private int $age = 18
    ) {}
}
```

##### 🔒 Readonly Properties (Свойства только для чтения, PHP 8.1)

Модификатор `readonly` гарантирует, что свойство класса может быть записано только один раз 
(обычно при инициализации в конструкторе).

Любая попытка изменить значение этого свойства позже, в процессе работы программы, 
приведет к выбросу исключения `Error`. 
Это идеальный инструмент для создания `Value Objects` (объектов-значений) 
и неизменяемых структур данных (`Data Transfer Objects` — `DTO`), защищающий код от случайных скрытых багов.

```php
class BlogPost {
    // В PHP 8.2 можно объявить readonly весь класс целиком
    public function __construct(
        public readonly string $title,
        public readonly DateTime $createdAt
    ) {}
}

$post = new BlogPost('Заголовок', new DateTime());
$post->title = 'Другой заголовок'; // Ошибка! Error: Cannot modify readonly property
```

##### 🧭 Enums (Перечисления, PHP 8.1)

Долгожданный инструмент для создания фиксированного набора именованных вариантов (перечислений), 
заменяющий использование обычных строковых или числовых констант.
Энумы в PHP — это не просто списки, это полноценные объекты, внутри которых можно объявлять методы, 
реализовывать интерфейсы и задавать типизированные значения (`Backed Enums`). 
Это полностью исключает ситуацию, когда в метод передается некорректная строка или число.

```php
// Объявление типизированного энума (Backed Enum)
enum OrderStatus: string {
    case Pending = 'pending';
    case Paid = 'paid';
    case Shipped = 'shipped';
    
    public function isFinal(): bool {
        return $this === self::Shipped;
    }
}

// Использование в качестве тайпхинта
function updateStatus(OrderStatus $status) { ... }

updateStatus(OrderStatus::Paid); // Безопасно

```

##### 🚀 Другие ключевые новшества PHP 8.x

* `JIT` (Just-In-Time) компиляция (8.0): 
  компилирует части кода в машинные инструкции прямо во время работы. 
  Дает огромный прирост скорости в математических вычислениях 
  и фоновых скриптах (но мало влияет на обычные веб-сайты, упирающиеся в БД).

* Тип `never` (8.1): тип возвращаемого значения для функций, 
  которые гарантированно никогда не возвращают управление 
  (например, вызывают `exit()` или всегда выбрасывают исключение).

* Оператор `intersection types` (8.1): тип `PureIntersection&Iterable`, 
  требующий, чтобы объект удовлетворял сразу нескольким условиям (интерфейсам) одновременно через знак &.

* Фиберы (Fibers, 8.1): низкоуровневый механизм для реализации легковесного асинхронного параллелизма 
  (сопрограмм) без блокировки основного потока.

* Readonly-классы (8.2): возможность пометить весь класс как `readonly`, 
  делая все его свойства неизменяемыми автоматически.

* Типы в DNF (Disjunctive Normal Form, 8.2): 
  позволяют комбинировать Union и Intersection типы (например: (`HasId&HasName`)|`null`).

* Типизированные константы классов (8.3): 
  теперь для констант внутри классов можно строго указывать тип данных (например, `const string VERSION = '1.0'`;).
