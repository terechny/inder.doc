
# Api development


## Что такое REST API?

REST (Representational State Transfer / Передача состояния представления) — это архитектурный стиль взаимодействия компонентов веб-приложения, основанный на протоколе HTTP. REST API использует стандартные механизмы сети вместо создания собственных протоколов, что делает его универсальным стандартом для связи бэкенда с фронтендом (SPA), мобильными приложениями и сторонними сервисами.


##### 1. Ключевые принципы REST

Чтобы API считалось по-настоящему RESTful, оно должно соответствовать базовым принципам:

* Stateless (Без состояния): Сервер не должен хранить контекст (состояние) клиента между запросами. Каждый HTTP-запрос от клиента должен содержать абсолютно всю информацию, необходимую для его обработки (например, токен авторизации в заголовке Authorization: Bearer).
* Единообразие интерфейса (Uniform Interface): Методы взаимодействия с сервером должны быть стандартизированы. Независимо от того, запрашивается ли пост, пользователь или заказ, способ формирования путей и использования HTTP-методов остается одинаковым.
* Ресурсы через URI: Всё в REST является ресурсом (сущностью). Доступ к ресурсам осуществляется через уникальные идентификаторы (URI). В именовании URI принято использовать существительные во множественном числе (например, /api/posts, а не /api/get-all-posts).
* Использование HTTP-методов: Действия над ресурсами определяются строго стандартными методами HTTP-протокола, а не текстом в названии ссылки.


##### 2. Сводная таблица HTTP-методов и эндпоинтов

В RESTful-архитектуре один и тот же URL (URI) в сочетании с разными HTTP-методами выполняет совершенно разные действия над ресурсом:

| HTTP-метод | URI (Эндпоинт) | Описание действия | Идемпотентность |
|---|---|---|---|
| GET | /api/posts | Получить список всех постов | Да |
| GET | /api/posts/1 | Получить один конкретный пост с ID = 1 | Да |
| POST | /api/posts | Создать новый пост (данные передаются в теле запроса) | Нет |
| PUT | /api/posts/1 | Обновить полностью пост №1 (заменяет весь объект целиком) | Да |
| PATCH | /api/posts/1 | Обновить частично пост №1 (изменяет только переданные поля) | Нет (обычно) |
| DELETE | /api/posts/1 | Удалить пост №1 | Да |

* Что такое Идемпотентность? (Частый вопрос на собеседовании).

Метод называется идемпотентным, если повторный вызов одного и того же запроса с одинаковыми данными приводит к одному и тому же результату и состоянию сервера.

* GET — идемпотентный (сколько раз ни запрашивай список, состояние сервера не изменится).
* POST — не идемпотентный (если отправить запрос 5 раз подряд, в базе данных создадутся 5 одинаковых постов с разными ID).


##### 3. Инструменты REST API в Laravel

Laravel предоставляет мощный и элегантный инструментарий для мгновенного развертывания RESTful API.

##### A. Автоматический роутинг через Route::apiResource()

Вместо ручного прописывания шести маршрутов для CRUD-операций в файле routes/api.php, Laravel позволяет зарегистрировать их одной строкой:

```php
use App\Http\Controllers\Api\PostController;
use Illuminate\Support\Facades\Route;

// Создает сразу 5 стандартных маршрутов (без методов create и edit, нужных только для HTML-форм)
Route::apiResource('posts', PostController::class);
```

##### B. API Resources (Трансформация данных)

Никогда не стоит отдавать клиенту сырую модель Eloquent из базы данных (`return Post::all();`), так как это раскрывает внутреннюю структуру БД и может случайно слить конфиденциальные поля (например, password или скрытые статусы).
API Resources — это слой трансформации (презентер), который превращает модели Eloquent в строго контролируемый JSON-ответ.
Генерация класса: `php artisan make:resource PostResource`

```php

namespace App\Http\Resources;

use Illuminate\Http\Request;
use Illuminate\Http\Resources\Json\JsonResource;

class PostResource extends JsonResource
{
    public function toArray(Request $request): array
    {
        // Четко определяем, какие поля и в каком формате увидит клиент API
        return [
            'id' => $this->id,
            'title' => $this->title,
            'content' => $this->content,
            'published_at' => $this->created_at->toIso8601String(), // форматируем дату
            'author' => new UserResource($this->whenLoaded('user')), // загружаем связь, только если она была запрошена
        ];
    }
}

```

Использование в контроллере:

```
public function show(Post $post)
{
    return new PostResource($post); // Вернет красивый JSON, обернутый в объект {"data": {...}}
}
```

##### 4. Стандартные HTTP-статусы ответов (Status Codes)

REST API общается с клиентом не только через JSON-тело, но и через правильные цифровые коды ответов в заголовках HTTP:

* 200 OK — запрос выполнен успешно (обычно возвращается для успешных GET, PUT, PATCH).
* 201 Created — ресурс успешно создан (обязательный ответ для метода POST при успешном создании сущности).
* 204 No Content — запрос выполнен успешно, но сервер ничего не возвращает в теле ответа (идеально для метода DELETE).
* 400 Bad Request — сервер не может обработать запрос из-за синтаксической ошибки на стороне клиента.
* 401 Unauthorized — клиент не аутентифицирован (отсутствует или невалиден Bearer-токен).
* 403 Forbidden — клиент аутентифицирован, но у него нет прав доступа к этому ресурсу (срабатывание Policy / Gate).
* 404 Not Found — запрашиваемый ресурс не существует на сервере.
* 422 Unprocessable Entity — ошибка валидации данных. Стандартный ответ Laravel, если присланные в FormRequest поля не прошли проверку (например, email уже занят или пароль слишком короткий).

##### 🔥 Вопросы на собеседовании по REST API

   1. В чем разница между методами PUT и PATCH?
   * Ответ: Метод PUT производит полную замену ресурса. Клиент должен прислать объект целиком со всеми обязательными полями. Поля, которые не были присланы, будут стерты или заменены на дефолтные значения. Метод PATCH выполняет частичную модификацию ресурса. Клиент может прислать всего одно поле (например, {'status': 'archived'}), и сервер обновит только его, оставив остальные данные нетронутыми.
   
   2. Почему REST называется Stateless и в чем его преимущество?
   * Ответ: Изоляция от состояния означает, что сервер не запоминает сессию конкретного пользователя на жестком диске или в оперативной памяти. Каждый запрос автономен. Это дает колоссальное преимущество в горизонтальном масштабировании (Scalability): если ваше приложение вырастет, вы сможете поставить 10 одинаковых серверов за балансировщик нагрузки, и любой сервер сможет обработать любой запрос от любого пользователя, так как токен авторизации и все параметры передаются внутри самого запроса.
   
   3. Как в Laravel переопределить HTTP-код ответа при ошибке валидации с 422 на другой?
   * Ответ: По умолчанию Laravel автоматически перехватывает исключение ValidationException и преобразует его в JSON со статусом 422. Переопределить это поведение можно в глобальном обработчике исключений (в Laravel 11+ это делается через метод ->renderable() в файле bootstrap/app.php, а в старых версиях фреймворка — в классе app/Exceptions/Handler.php). Однако менять статус 422 не рекомендуется, так как это общепринятый стандарт REST API для ошибок валидации.


## GraphQL

GraphQL — это язык запросов для API и среда для их выполнения, разработанная Facebook. В отличие от REST API, где структура ответа жестко определяется сервером на конкретном эндпоинте, в GraphQL клиент сам указывает, какие именно поля ему нужны.
Это решает две главные проблемы REST-архитектуры:

   1. Over-fetching (Избыток данных): Когда вам нужно только имя пользователя, но REST-эндпоинт /api/users/1 возвращает массив из 50 полей, включая адрес, телефон и историю заказов.
   2. Under-fetching (Недостаток данных): Когда на одной странице нужно отобразить пост, его автора и комментарии. В REST вам пришлось бы сделать 3 отдельных запроса: /api/posts/1, /api/users/1 и /api/posts/1/comments. В GraphQL это делается одним запросом.


##### 1. Основные концепции GraphQL

* Один эндпоинт: Все запросы (чтение, запись, схема) всегда отправляются на один и тот же URL (обычно POST /graphql).
* Query (Запросы): Используются исключительно для чтения данных (аналог методов GET в REST).
* Mutation (Мутации): Используются для изменения данных — создания, обновления, удаления (аналог POST, PUT, PATCH, DELETE в REST).
* Type (Типы): Строго типизированная схема данных, которая описывает, какие объекты существуют в системе и какие поля у них есть (например, тип Post, тип User).
* Resolver (Резолвер): Специальная функция на бэкенде, которая отвечает за сбор и загрузку данных для конкретного поля или типа. Именно здесь пишется код обращения к базе данных (через Eloquent) или к кэшу.


##### 2. Пример взаимодействия: Клиент и Сервер## Запрос от клиента (Query):

Клиент запрашивает пост с конкретным ID, но просит вернуть только заголовок поста и имя его автора.

```js

query GetPostWithAuthor {
    post(id: 1) {
        title
        author {
            name
        }
    }
}

```

##### Ответ от сервера (JSON):

Сервер вернет структуру, в точности повторяющую форму запроса. Никаких лишних полей.

```js

{
    "data": {
        "post": {
            "title": "Изучаем GraphQL в Laravel",
            "author": {
                "name": "Иван Иванов"
            }
        }
    }
}

```

##### 3. Подходы к созданию GraphQL API: Code-first vs Schema-first

При проектировании GraphQL API на бэкенде существует два принципиальных подхода:

##### A. Code-first (Сначала код)

Схема GraphQL генерируется автоматически на основе написанного вами PHP-кода (классов, аннотаций, атрибутов).

* Как это выглядит: Вы пишете стандартные PHP-классы для типов и резолверов, а библиотека парсит их и строит общую схему.
* Инструмент в Laravel: Пакет `rebing/graphql-laravel`.
* Плюсы: Полный контроль над логикой на уровне PHP, удобный рефакторинг в IDE, типы и код находятся в одном месте.

##### B. Schema-first (Сначала схема)

Сначала пишется файл схемы на специальном языке SDL (Schema Definition Language), а затем под эту схему пишутся резолверы в коде.

* Как это выглядит: Вы создаете файл schema.graphql, описываете там типы вручную, а библиотека маппит их на ваши методы.
* Инструмент в Laravel: Пакет Lighthouse (самый популярный инструмент для GraphQL в Laravel).
* Плюсы: Схема является чистой документацией. Фронтенд-разработчики могут согласовать схему с бэкендерами до написания кода и сразу начать делать UI, используя моки.

##### 4. Реализация GraphQL в Laravel

Самым мощным и «Laravel-подобным» инструментом является Lighthouse (Schema-first подход). Он позволяет связывать схему GraphQL с моделями Eloquent с помощью встроенных директив, минимизируя написание кода.

##### Пример схемы в Lighthouse (graphql/schema.graphql):

```php

type User {
    id: ID!
    name: String!
    email: String!
    posts: [Post!]! @hasMany # Нативная связь Laravel Eloquent
}
type Post {
    id: ID!
    title: String!
    content: String
    user: User! @belongsTo
}
type Query {
    # Директива @find автоматически сделает Post::find($id)
    post(id: ID! @eq): Post @find 
    
    # Директива @all вернет все записи Post::all()
    posts: [Post!]! @all
}
type Mutation {
    # Автоматическое создание записи в БД через Eloquent
    createPost(title: String!, content: String!, user_id: ID!): Post @create
}
```

Благодаря директивам @find, @all, @create, @belongsTo вам вообще не нужно писать PHP-код для базовых CRUD операций — Lighthouse сделает всё под капотом. Для сложной логики вы можете написать кастомный класс-резолвер с помощью директивы 

`@field(resolver: "App\\GraphQL\\Queries\\MyCustomQuery")`.

##### ⚠️ Главная проблема GraphQL: Проблема N+1 запросов

Поскольку клиент может динамически запрашивать вложенные связи (например, получить список постов, для каждого поста — автора, а для каждого автора — его топ-5 комментариев), GraphQL по умолчанию начинает вызывать резолверы для каждой строки изолированно. Это мгновенно приводит к проблеме N+1 запросов к базе данных [422].

* Как это решается в Laravel:
* В Lighthouse встроен механизм автоматического отложенного рендеринга и группировки запросов (Deferred Execution / Batch Loading). Когда используются директивы вроде @belongsTo или @hasMany, Lighthouse собирает все необходимые ID постов, делает один общий запрос к БД (WHERE IN) и распределяет данные, защищая сервер от перегрузки.

##### 🔥 Вопросы на собеседовании по GraphQL

   1. В каких случаях GraphQL лучше, чем REST API, а в каких — хуже?
   * Ответ: GraphQL лучше, когда у приложения сложный, динамичный интерфейс (например, дашборды), где на экранах постоянно требуются разные наборы данных. Он идеален для мобильных приложений, так как экономит мобильный трафик за счет исключения over-fetching'а. GraphQL хуже, когда логика приложения — это простой CRUD (REST пишется быстрее). Также в GraphQL гораздо сложнее настроить классическое HTTP-кэширование, потому что все запросы идут методом POST на один эндпоинт.
   2. Как работает кэширование в GraphQL, если все запросы идут через POST?
   * Ответ: Традиционное кэширование на уровне HTTP-шлюзов (Varnish, Nginx) по URL-адресу здесь не работает. Кэширование в GraphQL реализуется либо на стороне клиента (с помощью умных клиентов типа Apollo Client, которые кэшируют нормализованные объекты по их ID), либо на стороне сервера внутри конкретных резолверов (кэширование данных через Redis перед обращением к Eloquent).
   3. Как реализовать авторизацию (проверку прав доступа) в GraphQL в Laravel?
   * Ответ: Аутентификация (Bearer token) проходит стандартно через middleware на эндпоинте /graphql (например, Sanctum). Для авторизации действий (Policies) в пакете Lighthouse есть специальные директивы @can или @guard. Вы можете повесить их прямо в схеме: posts: [Post!]! @all @guard(with: ["sanctum"]) @can(ability: "viewAny").
   

## Swagger и OpenAPI?

OpenAPI — это публичный общепринятый стандарт (спецификация) для описания RESTful API. Он позволяет описать всю структуру вашего API (эндпоинты, входные параметры, структуры ответов, методы авторизации) в едином файле формата JSON или YAML.
Swagger — это набор инструментов, построенных вокруг спецификации OpenAPI. Самый известный из них — Swagger UI. Он берет JSON/YAML-файл спецификации и превращает его в интерактивную веб-страницу. На этой странице разработчики (фронтендеры, тестировщики или сторонние интеграторы) могут не только читать документацию, но и отправлять реальные запросы к API прямо из браузера (кнопка Try it out).


##### 1. Реализация Swagger в Laravel

В экосистеме Laravel стандартом для интеграции Swagger является пакет darkaonline/l5-swagger (обертка над библиотекой swagger-php).

##### Как устроен процесс разработки (Аннотации / Атрибуты)

Документация пишется прямо в PHP-коде (внутри контроллеров, моделей или реквестов) с использованием специальных PHPDoc-комментариев (аннотаций) или современных PHP-атрибутов (начиная с PHP 8+). Специальная artisan-команда парсит весь ваш код, собирает эти аннотации и генерирует итоговый файл api-docs.json.

##### 2. Примеры аннотаций в коде Laravel

Чтобы описать эндпоинт, структуру данных и метод авторизации, используются специальные блоки @OA\.

##### A. Глобальное описание API и авторизации

Обычно размещается в базовом контроллере (Controller.php):

```php

/**
 * @OA\Info(
 *      version="1.0.0",
 *      title="Laravel REST API Documentation",
 *      description="Документация серверной части для мобильного приложения"
 * )
 * 
 * @OA\SecurityScheme(
 *      securityScheme="bearerAuth",
 *      type="http",
 *      scheme="bearer",
 *      bearerFormat="JWT"
 * )
 */
class Controller extends BaseController { ... }

```

##### B. Описание структуры данных (Schema для моделей)

Чтобы не описывать поля ответа для каждого эндпоинта заново, структура выносится в общую схему (обычно вешается над моделью Eloquent или API Resource):

```php

/**
 * @OA\Schema(
 *      schema="Post",
 *      title="Модель Поста",
 *      required={"id", "title"}
 * )
 */
class Post extends Model
{
    /**
     * @OA\Property(property="id", type="integer", example=1)
     * @OA\Property(property="title", type="string", example="Изучаем Swagger в Laravel")
     * @OA\Property(property="content", type="string", nullable=true, example="Текст поста...")
     */
}

```

##### C. Описание эндпоинта в Контроллере

Классический пример документирования метода GET `/api/posts/{id}`:

```php

/**
 * @OA\Get(
 *      path="/api/posts/{id}",
 *      operationId="getPostById",
 *      tags={"Posts"},
 *      summary="Получить информацию о конкретном посте",
 *      security={{"bearerAuth":{}}},
 *      
 *      @OA\Parameter(
 *          name="id",
 *          in="path",
 *          required=true,
 *          description="ID поста",
 *          @OA\Schema(type="integer")
 *      ),
 *      
 *      @OA\Response(
 *          response=200,
 *          description="Успешный ответ",
 *          @OA\JsonContent(ref="#/components/schemas/Post") -- Ссылка на схему выше
 *      ),
 *      @OA\Response(
 *          response=401,
 *          description="Не авторизован"
 *      ),
 *      @OA\Response(
 *          response=404,
 *          description="Пост не найден"
 *      )
 * )
 */
public function show($id)
{
    return new PostResource(Post::findOrFail($id));
}

```

##### 3. Рабочий процесс и Artisan-команды

После того как аннотации написаны в коде, вам нужно сгенерировать саму документацию:

   1. Генерация JSON-файла:
   
   ```bash
   php artisan l5-swagger:generate
   ```
   
   2. Доступ в браузере:
   По умолчанию пакет разворачивает интерактивный UI по адресу /api/documentation. Вы заходите туда, видите красивый интерфейс, можете нажать кнопку Authorize, ввести свой Bearer-токен и тестировать API прямо из браузера.
   3. Автогенерация на локальной машине:
   В файле config/l5-swagger.php можно выставить параметр 'generate_always' => true. Тогда пакет будет автоматически пересобирать документацию при каждом обновлении страницы в браузере (полезно при активной разработке, но должно быть выключено на продакшене для экономии ресурсов).


##### ⚖️ Достоинства и недостатки подхода "Документация в коде"

Плюсы:

* Синхронность: Документация лежит прямо над методом контроллера. Когда разработчик меняет логику метода, он видит аннотацию и с большей вероятностью обновит и её (меньше шансов получить устаревшую документацию).
* Единый источник правды: Не нужно вести внешние файлы в Postman или Confluence. Весь проект задокументирован внутри Git-репозитория.

Минусы:

* Засорение кода (Boilerplate): Аннотации Swagger очень объемные. Часто PHPDoc-комментарий над методом занимает 50–100 строк кода, в то время как сам PHP-метод состоит всего из 3 строк. Код становится тяжело читать.
* Сложный синтаксис: Опечатка в кавычках или фигурных скобках внутри текстовой аннотации приведет к ошибке парсинга при генерации, которую бывает трудно дебажить.

##### 🔥 Вопросы на собеседовании по Swagger / OpenAPI

   1. Какая разница между OpenAPI и Swagger?
   * Ответ: OpenAPI — это название самой спецификации (стандарта), описывающей правила составления файлов документации. Swagger — это название бренда и набора инструментов (Swagger UI, Swagger Editor), которые работают с этой спецификацией. Спецификация OpenAPI когда-то называлась Swagger Specification, пока её не передали независимому консорциуму Linux Foundation.
   2. Где должна генерироваться документация Swagger — локально или на продакшене?
   * Ответ: Документация (JSON/YAML файл) должна генерироваться локально или на этапе CI/CD пайплайна перед деплоем и коммититься/заливаться на сервер как готовый статический файл. Запускать сканирование всего кода (l5-swagger:generate) на продакшн-сервере при запросе пользователя — это грубая ошибка, так как процесс парсинга аннотаций через рефлексию очень ресурсоемкий и сильно нагружает процессор.
   3. Как скрыть Swagger-документацию на продакшн-среде от посторонних глаз?
   * Ответ: В файле config/l5-swagger.php маршруты для Swagger можно защитить с помощью middleware (например, разрешить доступ только администраторам или проверять IP-адрес). Также в среде production можно полностью отключить роут документации, оставив его доступным только на local или staging серверах.
   

## Rate Limiting  

Rate Limiting (Ограничение частоты запросов) — это механизм, который ограничивает количество HTTP-запросов, выполняемых пользователем или IP-адресом за определенный промежуток времени.

##### 🛡 Зачем нужен Rate Limiting?

* Защита от DDoS и Brute Force: Предотвращает падение сервера от шквала автоматических запросов злоумышленников (перебор паролей, скрапинг данных, флуд).
* Экономия ресурсов: Отрезает неэффективно написанные скрипты со стороны клиентов (например, когда фронтенд отправляет тяжелый поисковый запрос на каждый символ при вводе текста).
* Монетизация API: Позволяет продавать разные тарифные планы с лимитами (например, 100 запросов в день на бесплатном тарифе и 10 000 — на премиуме).


##### 🚀 Реализация Rate Limiting в Laravel

Laravel предоставляет очень мощный и гибкий функционал для управления лимитами с помощью встроенного Rate Limiter API.

##### 1. Базовое ограничение через Middleware throttle

Самый простой способ защитить роуты — повесить стандартное middleware throttle. Параметры передаются через двоеточие в формате throttle:кол_во_запросов,минуты.

```php

// Разрешить строго 60 запросов в минуту для этого роута
Route::middleware('throttle:60,1')->get('/api/search', [SearchController::class, 'index']);
```
##### 2. Кастомные лимиты через RateLimiter::for()

Начиная с Laravel 8+, лимиты принято выносить в именованные конфигурации. В Laravel 11+ это делается в app/Providers/AppServiceProvider.php (в старых версиях — в RouteServiceProvider).
Вы можете динамически рассчитывать лимиты на основе авторизованного пользователя, его статуса в БД, IP-адреса или типа запроса.

##### А. Разделение лимитов: Гости vs Авторизованные

```php

use Illuminate\Cache\RateLimiting\Limit;
use Illuminate\Support\Facades\RateLimiter;
use Illuminate\Http\Request;

public function boot(): void
{
    RateLimiter::for('api', function (Request $request) {
        // Если пользователь авторизован, даем 100 запросов в минуту (идентификация по ID)
        // Если гость — всего 10 запросов в минуту (идентификация по IP)
        return $request->user()
            ? Limit::perMinute(100)->by($request->user()->id)
            : Limit::perMinute(10)->by($request->ip());
    });
}

```

Применение в роутере: `Route::middleware('throttle:api')->group(...)`;

##### Б. Разделение по ролям: Премиум-пользователи

```php

RateLimiter::for('downloads', function (Request $request) {
    $user = $request->user();

    if ($user && $user->is_premium) {
        return Limit::none(); // Для премиум-пользователей лимитов нет!
    }

    return Limit::perHour(5)->by($user?->id ?? $request->ip()); // Гостям — 5 скачиваний в час
});

```
##### В. Разделение на Чтение и Запись (Read vs Write)

Тяжелые изменяющие операции (`POST`, `PUT`, `DELETE`) должны быть защищены гораздо строже, чем обычное чтение (GET). Вы можете создать конфигурацию, которая реагирует на HTTP-метод:

```php

RateLimiter::for('strict-api', function (Request $request) {
    return $request->isMethod('GET')
        ? Limit::perMinute(60)->by($request->ip())
        : Limit::perMinute(5)->by($request->ip()); // На создание/удаление — всего 5 запросов в минуту
});

```

##### 🌐 Распределенные системы и Redis

По умолчанию Laravel хранит счетчики Rate Limiter в стандартном кэше (обычно это файловый кэш file).

* Проблема: Если ваше приложение масштабировано горизонтально и работает на нескольких серверах за балансировщиком нагрузки, файловый кэш на каждом сервере будет свой. Пользователь сможет сделать по 60 запросов на Сервер №1, Сервер №2 и Сервер №3, суммарно обойдя лимит.
* Решение: Переключить драйвер кэша на Redis или Memcached в файле .env (CACHE_STORE=redis). Redis является централизованным in-memory хранилищем, доступным всем серверам приложения. Счетчики запросов будут синхронизированы глобально, обеспечивая корректный distributed (распределенный) rate limiting.

##### 📋 Что получает клиент при превышении лимита?

Когда лимит исчерпан, Laravel автоматически прерывает запрос, возвращает HTTP-статус 429 Too Many Requests и отправляет стандартные HTTP-заголовки ответа:

* X-RateLimit-Limit: Общий лимит за период (например, 60).
* X-RateLimit-Remaining: Сколько запросов осталось в текущем окне (будет 0).
* Retry-After: Количество секунд, которое клиенту нужно подождать перед следующей попыткой.


##### 🔥 Вопросы на собеседовании по Rate Limiting

   1. Каков алгоритм работы встроенного Rate Limiter в Laravel?
   * Ответ: Laravel использует алгоритм Token Bucket (или Leaky Bucket / Фиксированное окно) в зависимости от настроек. В стандартном throttle применяется алгоритм фиксированного временного окна (Fixed Window Counter). В Redis пишется ключ с уникальным ID пользователя/IP, значение счетчика увеличивается на 1 при каждом запросе, а для ключа выставляется TTL (время жизни), равное интервалу ограничения (например, 60 секунд). По истечении TTL окно сбрасывается.
   2. Как правильно протестировать Rate Limiting в Feature-тестах Laravel?
   * Ответ: Не нужно писать цикл на 60 итераций в тесте — это сильно замедлит его выполнение. В Laravel для этого есть удобные хелперы фасада RateLimiter. Вы можете вручную сымитировать превышение лимита для конкретного ключа с помощью метода `RateLimiter::hit()` или `RateLimiter::clear()`, а затем отправить один HTTP-запрос и проверить, что бэкенд вернул $response->assertStatus(429);.
   3. Как защитить роут, если злоумышленник обходит лимит по IP с помощью прокси-серверов или VPN?
   * Ответ: Если атака идет со множества разных IP (DDoS/DDoS-брутфорс), ограничение по IP перестает быть эффективным. В таком случае нужно:
      1. Использовать Limit::by($request->input('email')) для роутов авторизации (привязка к аккаунту, а не к сети).
         2. Переносить защиту на уровень выше приложения — на уровень инфраструктуры (например, настраивать правила фильтрации трафика в Cloudflare, AWS WAF или на уровне Nginx-модуля ngx_http_limit_req_module).
      


## API Versioning

##### API Versioning (Версионирование API)

API Versioning — это практика управления версиями веб-сервиса, которая позволяет развивать функционал приложения, изменять бизнес-логику и структуру данных, не ломая работу существующих клиентов (мобильных приложений, фронтенда или сторонних интеграций), использующих старые версии [1, 2].

##### 🚨 Когда нужно выпускать новую версию (Breaking Changes)?

Новая версия API создается исключительно при наличии мажорных, ломающих изменений (Breaking Changes).

* Нужно версионировать:
* Удаление полей или эндпоинтов из старого контракта.
   * Изменение структуры JSON-ответа (например, поле author: "Иван" переписали в объект `author: {"id": 1, "name": "Иван"}`).
   * Изменение типов данных (строка стала числом).
   * Кардинальное изменение логики работы эндпоинта.
* НЕ нужно версионировать:
* Добавление новых (необязательных) полей в JSON-ответ. Старые клиенты их просто проигнорируют.
   * Добавление новых эндпоинтов.
   * Внутренний рефакторинг или оптимизация SQL-запросов (контракт API остается прежним).

##### 🛠 Основные методы реализации версионирования

Существует три основных подхода к передаче версии от клиента к серверу:

##### 1. URI Versioning (Версионирование через URL) — Самый популярный метод
Версия жестко зашивается прямо в адресную строку HTTP-запроса [2].

* Пример: GET `https://example.com` и GET `https://example.com` [2]
* Плюсы: Максимальная прозрачность. Ссылки легко читать, тестировать в браузере и кэшировать на уровне Nginx/Cloudflare.
* Минусы: Нарушается принцип REST (один ресурс должен иметь один уникальный URI, а тут один и тот же пост имеет два разных адреса). При изменении версии приходится переписывать весь роутинг.

##### 2. Header Versioning (Версионирование через заголовки)

URL остается чистым, а версия передается в кастомном HTTP-заголовке или внутри заголовка Accept (Media Type) [2].

* Пример кастомного заголовка: X-API-Version: 2
* Пример через Accept: Accept: application/vnd.company.v2+json [2]
* Плюсы: Самый «RESTful» подход. URL указывает чисто на ресурс (`/api/posts`), а версия управляет лишь представлением этого ресурса.
* Минусы: Сложнее тестировать вручную (нельзя просто открыть ссылку в браузере, нужен Postman). Труднее настроить кэширование на прокси-серверах (приходится учитывать заголовок в директиве Vary).

##### 3. Query Parameter Versioning (Версионирование через параметры запроса)

Версия передается как GET-параметр в конце URL [2].

* Пример: GET `https://example.com` [2]
* Плюсы: Просто реализовать. Позволяет легко выставлять дефолтную версию, если параметр не передан.
* Минусы: Параметры строки запроса могут некорректно кэшироваться некоторыми старыми прокси-серверами. URL выглядит перегруженным.

##### 🚀 Реализация версионирования в Laravel

Чаще всего в Laravel используют URI Versioning, так как он органично ложится на встроенный механизм групп маршрутов (Route::group).

##### Архитектурная структура папок проекта

Для поддержки чистоты кода контроллеры разделяют по пространствам имен (Namespaces):

```
app/Http/Controllers/Api/
├── V1/
│   ├── PostController.php
│   └── UserController.php
└── V2/
    └── PostController.php (только для измененной логики)
```

##### Настройка роутинга (routes/api.php)

В Laravel маршруты группируются с подставлением префиксов и пространств имен:

```php

use Illuminate\Support\Facades\Route;

// Маршруты Первой Версии API
Route::prefix('v1')
    ->namespace('App\Http\Controllers\Api\V1')
    ->group(function () {
        Route::apiResource('posts', PostController::class); // Вызовет V1\PostController
        Route::apiResource('users', UserController::class);
    });

// Маршруты Второй Версии API
Route::prefix('v2')
    ->namespace('App\Http\Controllers\Api\V2')
    ->group(function () {
        // Переопределенный контроллер для V2
        Route::apiResource('posts', PostController::class); // Вызовет V2\PostController
        
        // Если для пользователей изменений нет, можно проксировать на V1 
        // или продублировать маршрут, чтобы не ломать архитектуру V2
    });

```

##### Использование API Resources для разделения версий

Версионирование должно затрагивать не только контроллеры, но и слой трансформации данных (Resources). Для V1 и V2 создаются отдельные классы PostResource, чтобы гарантировать изоляцию форматов ответов.

##### 🔥 Вопросы на собеседовании по API Versioning

   1. Как избежать дублирования кода (DRY) в контроллерах, если в версии V2 изменился только один метод из пяти?
   * Ответ: Контроллер версии V2\PostController должен наследоваться от контроллера V1\PostController. В классе V2 мы переопределяем только тот конкретный метод (например, show), контракт которого изменился. Все остальные методы (index, store, delete) автоматически унаследуются из V1 и будут работать по старой, проверенной логике.
   2. Как реализовать версионирование через заголовки (Header Versioning) в Laravel, если стандартный роутинг завязан на URL?
   * Ответ: Для этого пишется кастомное Middleware (или плагин для роутера). Middleware перехватывает входящий запрос до этапа маршрутизации, считывает значение заголовка X-API-Version, и на основе этого динамически подменяет префикс или пространство имен для текущего запроса, перенаправляя его в нужную папку контроллеров.
   3. Что такое семантическое версионирование (SemVer) и применимо ли оно к REST API?
   * Ответ: SemVer — это формат МАЖОРНАЯ.МИНОРНАЯ.ПАТЧ (например, 2.1.4). Для REST API классический SemVer в URL не используется (писать /api/v2.1.4/ — антипаттерн). В URL API выносят только мажорную версию (v1, v2), так как только она сигнализирует о ломающих изменениях. Минорные обновления и патчи бэкенд выкатывает незаметно для клиента в рамках текущей мажорной версии.


## CORS (Cross-Origin Resource Sharing)


##### Что такое CORS?

CORS (Cross-Origin Resource Sharing / Совместное использование ресурсов разных источников) — это механизм безопасности, реализованный в браузерах. По умолчанию в вебе действует правило Same-Origin Policy (SOP): скрипт с сайта frontend.com не может прочитать данные из запроса к сайту backend.com, если домен, протокол или порт отличаются.
CORS — это способ для сервера официально сказать браузеру: «Я доверяю сайту frontend.com, разреши его скриптам делать ко мне запросы и читать ответы».

##### 🚨 Заголовки CORS и Preflight-запросы (OPTIONS)

Когда фронтенд (например, SPA на Vue/React) делает запрос к вашему Laravel API на другом домене, браузер делит эти запросы на две категории:

##### 1. Простые запросы (Simple Requests)

Это стандартные GET или POST запросы с базовыми заголовками. Браузер выполняет их сразу, отправляя заголовок Origin: `http://frontend.com`. Сервер в ответ должен прислать заголовок Access-Control-Allow-Origin: `http://frontend.com`. Если заголовка нет — браузер блокирует ответ.

##### 2. Предварительные запросы (Preflight Requests)

Если запрос использует методы PUT, PATCH, DELETE или содержит кастомные заголовки (например, Authorization: Bearer), браузер сначала отправляет скрытый проверочный запрос методом OPTIONS.

* Цель OPTIONS: Спросить сервер: «Можно ли сайту frontend.com отправить метод DELETE с заголовком Authorization?».
* Если сервер отвечает успешным статусом 200 OK и правильными CORS-заголовками, браузер совершает второй — реальный DELETE запрос. Если ответ сервера отрицательный, реальный запрос даже не отправится, а в консоли JS упадет CORS error.

##### 🚀 Настройка CORS в Laravel

В Laravel управление CORS реализовано «из коробки» на уровне Middleware. Вся конфигурация вынесена в файл `config/cors.php`.

##### Пример безопасной настройки для Production:

```php

return [
    /*

    | Конкретные пути, для которых включен CORS. Обычно это вся зона /api/*
    */
    'paths' => ['api/*', 'sanctum/csrf-cookie'],

    /*
    | Список разрешенных HTTP-методов (GET, POST и т.д.)
    */
    'allowed_methods' => ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],

    /*

    | Белый список доменов (Origins), которым разрешен доступ к API
    */
    'allowed_origins' => [
        'https://my-frontend-app.com',
        'https://my-frontend-app.com',
    ],

    'allowed_origins_patterns' => [],

    /*
    | Разрешенные заголовки от клиента (Content-Type, Authorization, X-Requested-With)
    */
    'allowed_headers' => ['*'],

    'exposed_headers' => [],

    /*

    | Кэширование OPTIONS-запроса в браузере (в секундах), чтобы не спамить сервер проверочными запросами
    */
    'max_age' => 86400, // 24 часа

    /*
    | Важно! Разрешает ли сервер передавать куки сессий и HttpOnly куки в кросс-доменных запросах
    */
    'supports_credentials' => true,
];

```

##### ⚠️ Важные правила безопасности (Для собеседования)## 1. Использование Wildcard (*)

Выставить `'allowed_origins' => ['*']` означает разрешить абсолютно любому сайту в интернете делать запросы к вашему API из браузера.

* Когда можно: Только для публичных API, где нет авторизации пользователей и приватных данных (например, API погоды, курсов валют).
* Когда категорически нельзя: Если в API есть личные кабинеты, персональные данные или финансовые операции. Злоумышленник сможет сделать сайт-обертку и воровать данные пользователей.

##### 2. Конфликт supports_credentials и Wildcard

Если вы выставите `'supports_credentials' => true` (для работы авторизации через сессионные куки) и одновременно укажете `'allowed_origins' => ['*']`, Laravel выдаст ошибку, а браузер заблокирует запросы.

* Правило спецификации веба: Нельзя разрешать передачу учетных данных (кук) для «всего мира». Источники (origins) в таком случае должны быть прописаны строго и явно.

##### 🔥 Вопросы на собеседовании по CORS

   1. CORS — это защита сервера или защита клиента?
   * Ответ: CORS — это защита клиента (пользователя), выполняемая силами его браузера. Сервер в любом случае выполнит код DELETE-запроса к базе данных, даже если CORS-настройки не совпадают. Но именно браузер не даст JavaScript-коду вредоносного сайта прочитать конфиденциальный ответ от сервера или заблокирует отображение данных, защищая пользователя. Защита самого сервера — это задачи авторизации, валидации и файрволов (WAF).
   2. Запрос падает с CORS error в браузере, но успешно выполняется через Postman или curl. Почему?
   * Ответ: Потому что Postman, curl и консольные утилиты — это не браузеры. Они не подчиняются правилам Same-Origin Policy и игнорируют CORS-заголовки. CORS-ограничения существуют исключительно внутри среды браузера для защиты обычных пользователей, переходящих по сайтам.
   3. Что делать, если в Laravel роуты защищены, в config/cors.php всё прописано верно, но OPTIONS-запрос всё равно возвращает CORS error со статусом 500?
   * Ответ: Скорее всего, ошибка происходит на этапе инициализации приложения до того, как отработает CORS Middleware, либо внутри самого контроллера (например, падает синтаксическая ошибка PHP или ошибка подключения к БД). Если Laravel падает с ошибкой 500 Internal Server Error, он прерывает стандартный жизненный цикл ответа и может не прикрепить CORS-заголовки к ответу. Браузер видит статус 500 и отсутствие заголовков, интерпретируя это как ошибку CORS. Нужно смотреть логи бэкенда (storage/logs/laravel.log).
