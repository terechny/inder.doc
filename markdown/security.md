
# Security

## XSS (Cross-Site Scripting)

XSS (Cross-Site Scripting / Межсайтовый скриптинг) — это тип уязвимости веб-приложений, при которой злоумышленнику удается внедрить вредоносный код (обычно на JavaScript) в страницу, просматриваемую другими пользователями.
В отличие от SQL-инъекций, где целью является база данных на сервере, XSS атакует браузер клиента.


##### Чем опасен XSS?
Поскольку вредоносный скрипт выполняется в браузере жертвы в контексте доверенного сайта, он имеет доступ ко всему, к чему имеет доступ обычный скрипт этого сайта:

* Кража куки и сессий (Cookie Hijacking): Скрипт может прочитать document.cookie, отправить сессионный токен злоумышленнику, и тот сможет зайти в аккаунт жертвы без ввода пароля.
* Фишинг на лету: Скрипт может динамически перерисовать форму авторизации на странице, чтобы пользователь сам ввел свой пароль или данные карты.
* Кража конфиденциальных данных: Чтение содержимого страницы, личных сообщений, токенов из localStorage.
* Перенаправление (Редирект): Отправка пользователя на вредоносные сторонние ресурсы.


##### Основные типы XSS

   1. Хранимая (Stored / Persistent XSS)
   * Суть: Вредоносный скрипт сохраняется на сервере приложения (в базе данных, логах, комментариях).
      * Пример: Злоумышленник пишет комментарий к статье: `<script>stealCookies();</script>`. Этот текст сохраняется в БД. Каждый пользователь, который откроет эту статью, загрузит комментарий из базы, и его браузер автоматически выполнит этот код.
      * Опасность: Самый опасный тип. Атака работает «на автомате» на неограниченный круг пользователей.
   2. Отраженная (Reflected / Non-Persistent XSS)
   * Суть: Скрипт не сохраняется на сервере, а передается в параметрах запроса (например, через URL) и сразу же «отражается» сервером обратно в браузер.
      * Пример: Ссылка на поиск по сайту выглядит так: `https://example.com<script>attack();</script>`. Если сервер выведет на странице текст: «Результаты поиска по запросу: attack();» без экранирования, код выполнится.
      * Опасность: Чтобы атака сработала, злоумышленнику нужно заставить жертву кликнуть по специально подготовленной ссылке (например, через фишинговое письмо).
   3. DOM-based XSS
   * Суть: Уязвимость полностью находится на стороне фронтенда. Сервер может вообще не знать об атаке. Вредоносный код выполняется потому, что скрипты самого сайта некорректно (небезопасно) обрабатывают данные из адресной строки или окружения браузера.
      * Пример: На странице есть JavaScript-код: document.getElementById('output').innerHTML = location.hash;. Если злоумышленник скинет ссылку `https://example.com<img src=x onerror=attack()>`, браузер сам вставит этот код в DOM, и он выполнится.
   

##### Защита от XSS на PHP и в Laravel## 1. Как это устроено в Laravel (Из коробки)

Laravel по умолчанию защищает разработчика от XSS в шаблонизаторе Blade.
Когда вы выводите переменные через двойные фигурные скобки, Laravel автоматически прогоняет их через PHP-функцию htmlspecialchars(), превращая опасные символы в безопасные HTML-сущности:

```php
// Если $name = "<script>alert(1)</script>"

{{ $name }} 
// Выведет в HTML: &lt;script&gt;alert(1)&lt;/script&gt; (Браузер просто покажет текст)

{!! $name !!} 
// ВНИМАНИЕ! Выведет сырой HTML. Скрипт выполнится. Использовать только для проверенного HTML!

```

##### 2. Защита при работе с чистым PHP

Если вы пишете код без фреймворка, всегда экранируйте вывод данных, полученных от пользователя ($_GET, $_POST, данные из БД):

`echo htmlspecialchars($userInput, ENT_QUOTES, 'UTF-8')`;

##### 3. Что делать, если пользователю НАДО разрешить ввод HTML?

Если вы делаете блог или форум, где пользователи используют текстовый редактор (WYSIWYG) и должны форматировать текст (`<b>`, `<i>`, `<img>`), обычный `htmlspecialchars` не подойдет, так как он сломает всю верстку.

* Решение: Использовать библиотеки-очистители (HTML Purifier / HTML Sanitizer). Они парсят HTML и вырезают из него все опасные теги (`<script>`, `<iframe>`) и атрибуты (onload, onerror). В Laravel для этого часто используют пакет mewebstudio/purifier.

##### 4. Защита заголовками (HTTP Headers)

* `HttpOnly` куки: При создании сессионных кук обязательно выставляйте флаг `HttpOnly`. Это полностью запрещает JavaScript доступ к этим кукам через document.cookie. Даже если злоумышленник найдет XSS уязвимость на сайте, он не сможет украсть сессию. В Laravel этот флаг включен по умолчанию в `config/session.php`.
* `Content Security Policy (CSP)`: Специальный заголовок ответа сервера, который говорит браузеру, с каких источников можно загружать и выполнять скрипты. Настройка CSP вида script-src 'self' полностью запретит выполнение любых инлайновых скриптов (включая те, что внедрили через XSS).

##### 🔥 Вопросы на собеседовании по XSS

   1. В чем разница между XSS и CSRF?
   * Ответ: При XSS злоумышленник внедряет свой код на ваш сайт и выполняет его в браузере жертвы, получая доступ к данным сайта. При CSRF злоумышленник заманивает пользователя на свой (сторонний) вредоносный сайт и оттуда отправляет скрытые запросы на ваш сайт, используя то, что пользователь на вашем сайте уже авторизован. (XSS — это воровство данных внутри сайта, CSRF — это обманные действия извне).
   2. Защищает ли валидация входных данных (`$request->validate()`) от XSS?
   * Ответ: Валидация типов (например, integer или email) помогает снизить риски, но полноценной защитой не является. Главное правило безопасности: фильтровать на входе (валидация), экранировать на выходе (контекстное кодирование). Текст с тегом `<script>` — это валидная строка (string), поэтому валидатор её пропустит. Защита должна происходить в момент вывода данных в HTML-код.
   3. Зачем нужен флаг HttpOnly для кук?
   * Ответ: Он защищает сессионные куки от кражи посредством XSS-атак, запрещая доступ к ним из JavaScript.
   


## CSRF?

CSRF (Cross-Site Request Forgery / Межсайтное заимствование запроса) — это вид атаки, при которой вредоносный сайт заставляет браузер пользователя выполнить нежелательное действие на другом (доверенном) сайте, где пользователь в данный момент авторизован.
В отличие от XSS, где вредоносный код внедряется внутрь уязвимого сайта, при CSRF-атаке вредоносный код находится на чужом сайте, но использует особенности работы браузеров с куками (Cookies).

##### Механизм атаки (Как это работает)

   1. Условие: Пользователь залогинен в свой онлайн-банк на сайте bank.com. Браузер хранит сессионную куку для этого домена.
   2. Уловка: Пользователь, не выходя из банка, переходит по ссылке на вредоносный сайт (например, evil.com), который прислали ему в спаме.
   3. Атака: На сайте `evil.com` скрыто загружается форма или JavaScript-скрипт, который автоматически отправляет POST-запрос на адрес `https://bank.com` с параметрами `to=attacker&amount=1000.
   4. Результат: Браузер видит, что запрос отправлен на bank.com, и автоматически прикрепляет к нему куку сессии пользователя. Сервер банка принимает запрос, видит валидную куку, думает, что это действие самого пользователя, и списывает деньги.

##### Защита от CSRF в Laravel

Laravel предоставляет мощную и автоматическую защиту от CSRF-атак «из коробки» с помощью CSRF-токенов.

##### 1. Механизм CSRF-токена

Для каждого активного пользователя Laravel автоматически генерирует уникальный, случайный и криптографически стойкий секретный ключ (токен) и сохраняет его в сессии.
Когда сервер получает изменяющий состояние запрос (POST, PUT, PATCH, DELETE), он сравнивает токен из запроса с токеном, сохраненным в сессии пользователя. Если они не совпадают или токен отсутствует, Laravel блокирует запрос и возвращает ошибку 419 Page Expired.

##### 2. Защита форм через Blade

Внутри любой HTML-формы, отправляющей данные методом POST, необходимо обязательно указывать директиву `@csrf`:

```html

<form action="/profile" method="POST">
    @csrf <!-- Генерирует скрытый input с токеном -->
    
    <!-- Развернется в: <input type="hidden" name="_token" value="xyz123..."> -->
    
    <input type="text" name="name">
    <button type="submit">Обновить</button>
</form>

```

##### 3. Защита для AJAX / API (X-CSRF-TOKEN)

Если вы делаете запросы через JavaScript (Axios, Fetch), передавать токен в теле каждого запроса неудобно. Laravel решает это через HTTP-заголовки:

   1. В главном шаблоне Blade токен регистрируется в метатеге:
      
   2. JavaScript-библиотека (например, Axios) автоматически считывает этот тег и прикрепляет его значение к каждому запросу в заголовок X-CSRF-TOKEN. Middleware Laravel проверяет этот заголовок так же, как и скрытое поле формы.

##### 4. Как это устроено под капотом (Middleware)

За проверку токенов отвечает глобальный middleware `Illuminate\Foundation\Http\Middleware\VerifyCsrfToken` (в Laravel 11+ конфигурация находится в файле bootstrap/app.php).
Если вам необходимо отключить CSRF-защиту для конкретных адресов (например, для вебхуков от сторонней платежной системы типа Stripe, которая не знает ваш токен), их нужно добавить в исключения:

```php

// Laravel 11+ в bootstrap/app.php
->withMiddleware(function (Middleware $middleware) {
    $middleware->validateCsrfTokens(except: [
        'webhook/stripe',
        'api/*' // Для чистых API роутов CSRF обычно отключают
    ]);
})

```

##### Дополнительный уровень защиты: Куки SameSite

Современные браузеры помогают бороться с CSRF на уровне конфигурации кук с помощью атрибута SameSite. В Laravel его поведение настраивается в config/session.php (параметр same_site).

* SameSite=Strict: Браузер вообще не отправит куку, если запрос идет со стороннего сайта (даже если пользователь просто кликнул по обычной ссылке на ваш сайт).
* SameSite=Lax (По умолчанию в Laravel): Куки отправляются при безопасных переходах по ссылкам (GET), но блокируются при отправке кросс-доменных POST-запросов с чужих сайтов. Это нивелирует классический CSRF.

##### 🔥 Вопросы на собеседовании по CSRF

   1. Нужен ли CSRF-токен для stateless REST API (например, на мобильных приложениях)?
   * Ответ: Нет, не нужен. CSRF-атака возможна только тогда, когда аутентификация пользователя основана на куках, которые браузер подставляет автоматически. Если ваше API использует Bearer-токены (JWT, Laravel Sanctum / Passport tokens), которые передаются в заголовке Authorization, то браузер сам ничего никуда не прикрепит. Злоумышленник с сайта evil.com не имеет доступа к локальному хранилищу (localStorage) вашего приложения, чтобы украсть этот токен (если только на сайте нет XSS).
   2. Защищает ли CSRF-токен от GET-запросов?
   * Ответ: По умолчанию VerifyCsrfToken в Laravel игнорирует методы GET, HEAD и OPTIONS. Именно поэтому критически важно соблюдать стандарты HTTP: GET-запросы должны быть безопасными (идемпотентными) и только читать данные, но никогда не изменять состояние системы (не удалять и не создавать записи). Если сделать удаление сущности через GET-роут, CSRF-защита Laravel не сработает.
   3. Что означает ошибка 419 в Laravel?
   * Ответ: Это специфичный для Laravel статус-код (Page Expired), который означает, что отправленная форма не содержала CSRF-токена, либо сессия пользователя истекла, и токен стал невалидным.


## Что такое SQL-инъекция?

SQL-инъекция (SQL Injection / SQLi) — это критическая уязвимость веб-приложений, при которой злоумышленник внедряет произвольный SQL-код в запрос к базе данных. Это происходит, когда входные данные от пользователя конкатенируются (склеиваются) напрямую с телом SQL-запроса без предварительной фильтрации или экранирования.
В отличие от XSS (атака на браузер) и CSRF (обман браузера), SQL-инъекция — это прямая атака на серверную базу данных.

##### Чем опасна SQL-инъекция?

Успешная SQL-инъекция дает злоумышленнику возможность:

* Обходить авторизацию: Входить в аккаунт любого пользователя (включая администратора) без знания пароля.
* Красть данные (Data Leaks): Читать конфиденциальную информацию из любых таблиц (пароли, персональные данные, номера карт).
* Уничтожать или модифицировать данные: Удалять таблицы (DROP TABLE), менять баланс счетов, изменять права пользователей.
* Захватить сервер (RCE): В некоторых СУБД (например, MS SQL или MySQL при определенных настройках) через SQL-запросы можно читать/писать файлы на сервере или выполнять системные команды.

##### Механизм атаки (Уязвимый код на чистом PHP)

Представим классический уязвимый код авторизации, где переменные из формы вставляются прямо в строку запроса:

```php

// УЯЗВИМЫЙ КОД! ТАК ДЕЛАТЬ НЕЛЬЗЯ!
$username = $_POST['username'];
$password = $_POST['password'];

$query = "SELECT * FROM users WHERE username = '$username' AND password = '$password'";
$result = $db->query($query);

```

##### Сценарий обхода авторизации:

Злоумышленник в поле username вводит следующую строку: admin' -- (или admin' OR '1'='1).
В итоге сформированный SQL-запрос превращается в:

`SELECT * FROM users WHERE username = 'admin' -- ' AND password = '...'`

В SQL символы -- (или #) означают комментарий. База данных выполнит только первую часть запроса (SELECT * FROM users WHERE username = 'admin'), а всю проверку пароля просто проигнорирует. Злоумышленник успешно зайдет под администратором.

##### Как защититься от SQL-инъекций? ## 1. Подготовленные запросы (Prepared Statements) — Главный способ защиты

Это абсолютный стандарт безопасности. Суть метода в том, что структура SQL-запроса и пользовательские данные компилируются базой данных раздельно.
Вместо подстановки переменных в строку используются плейсхолдеры (знаки ? или именованные параметры вроде :username). База данных сначала жестко фиксирует логику запроса, а затем подставляет данные как «безопасный текст», который физически не может изменить логику SQL-команды.
Пример на чистом PHP через PDO:

```php
// БЕЗОПАСНЫЙ КОД
$stmt = $pdo->prepare('SELECT * FROM users WHERE username = :username AND password = :password');

// Передаем данные отдельно от структуры запроса
$stmt->execute([
    'username' => $_POST['username'],
    'password' => $_POST['password']
]);
$user = $stmt->fetch();
```

##### 2. Как защита устроена в Laravel (Из коробки)

В Laravel за работу с базой данных отвечает Eloquent ORM и Query Builder. Под капотом они автоматически используют подготовленные запросы (PDO Prepared Statements) для всех базовых операций.

```php

// Полностью БЕЗОПАСНО в Laravel:
$user = User::where('username', $request->input('username'))->first();
```

Laravel сам создаст плейсхолдер под капотом, и инъекция здесь невозможна.

##### ⚠️ Где в Laravel можно случайно создать уязвимость? (Важно для собеседования)

Уязвимость возникает, когда разработчик использует методы с приставкой Raw (сырой SQL) и вручную конкатенирует строки вместо использования плейсхолдеров.

```php
// УЯЗВИМЫЙ КОД в Laravel!
User::whereRaw("username = '" . $request->input('username') . "'")->first();

// БЕЗОПАСНОЕ исправление с использованием плейсхолдеров:
User::whereRaw("username = ?", [$request->input('username')])->first();
```

Другие зоны риска: методы orderByRaw(), selectRaw(), havingRaw(). Если в них передаются неотфильтрованные данные напрямую (например, направление сортировки берется из URL без валидации), проект будет уязвим.

##### Основные разновидности SQLi (Для глубоких вопросов на собеседовании)

   1. In-band SQLi (Классическая): Самый простой тип. Злоумышленник использует один и тот же канал для атаки и сбора результатов. Например, данные из БД выводятся прямо на веб-страницу приложения (через оператор UNION).
   2. Inferential / Blind SQLi (Слепая инъекция): Данные из БД не выводятся на экран. Злоумышленник задает базе данных True/False вопросы и анализирует ответы.
   * Boolean-based: Страница меняет свой вид (или выдает ошибку), если SQL-условие истинно.
      * Time-based: Злоумышленник заставляет базу данных «уснуть» на 5 секунд (функция SLEEP(5)), если условие верно. Если сервер ответил с задержкой — значит, догадка верна. Так, по одной букве, можно выкачать всю БД.
   3. Out-of-band SQLi: Редкий тип. Используется, когда сервер не возвращает данные и не меняет поведение, но база данных имеет возможность делать внешние сетевые запросы (например, DNS или HTTP-запросы к серверу злоумышленника), передавая данные в поддоменах.

##### 🔥 Вопросы на собеседовании по SQL-инъекциям

   1. Защищает ли функция addslashes() или mysqli_real_escape_string() от SQL-инъекций на 100%?
   * Ответ: Нет, не защищает. Экранирование строк помогает в простых случаях, но имеет кучу уязвимостей, связанных с кодировками базы данных (например, Multi-byte encodings типа GBK), а также абсолютно бесполезно, если пользовательские данные вставляются в числовое поле без кавычек (WHERE id = $id). Единственная надежная защита — Prepared Statements.
   2. Как безопасно реализовать динамическую сортировку столбцов в Laravel?
   * Ответ: Нельзя передавать строку из $request->input('order_by') напрямую в orderByRaw(). Необходимо использовать белый список (Whitelisting) разрешенных полей. Если пришедшее значение есть в массиве разрешенных — сортируем, если нет — используем дефолтное значение (например, id).
   3. Какая разница между типами SQL-инъекций «Слепая» (Blind) и «Обычная»?
   * Ответ: При обычной инъекции результат запроса или текст ошибки СУБД виден сразу на странице приложения. При слепой инъекции веб-страница не выводит данные базы, и злоумышленнику приходится судить о результатах по косвенным признакам: изменению контента (Boolean) или времени ответа сервера (Time-based).
   

## Аутентификация


Аутентификация (Authentication) — это процесс проверки подлинности пользователя (обычно путем сопоставления введенного логина/пароля с данными в БД).
В экосистеме PHP и фреймворке Laravel авторизация и аутентификация разделены: за проверку «кто ты» отвечает Auth, а за проверку прав «что тебе разрешено делать» — Gates и Policies.


##### 1. Методы аутентификации## A. Session-based (Cookies) — для веб-приложений (Monolith)

* Как работает: Классический подход со схемой Stateful (сервер хранит состояние). Пользователь отправляет логин/пароль. Сервер проверяет их, создает файл сессии на сервере (или запись в Redis) и возвращает браузеру куку с идентификатором сессии (laravel_session). При каждом следующем запросе браузер автоматически прикрепляет эту куку.
* Плюсы: Высокая безопасность (куки защищены флагами HttpOnly и SameSite=Lax), сессию можно мгновенно отозвать (разлогинить пользователя со стороны админки).
* Минусы: Плохо масштабируется горизонтально (требуется централизованное хранилище сессий типа Redis для нескольких серверов), уязвимо к CSRF-атакам.

##### B. Token-based (Bearer) — для API / SPA / Мобильных приложений

* Как работает: Схема Stateless (сервер не хранит сессию). Пользователь отправляет данные формы, в ответ получает уникальную строку — токен (например, JWT или случайный хэш). Этот токен сохраняется на клиенте (localStorage или в памяти мобильного приложения) и вручную прикрепляется к каждому HTTP-запросу в заголовок Authorization: Bearer `<token>`.
* Плюсы: Полная независимость от бэкенда (удобно для микросервисов), иммунитет к CSRF.
* Минусы: Если токен украден (например, через XSS из localStorage), злоумышленник получит полный доступ. Отозвать токен (особенно JWT) сложнее, чем удалить сессию.
* Реализация в Laravel:
* Laravel Sanctum: Идеален для SPA (Vue/React) и простых мобильных приложений. Позволяет выдавать токены ($user->createToken('name')) или использовать безопасные Stateful-куки для SPA на одном домене.
   * Laravel Passport: Тяжеловесная реализация на базе полноценного OAuth2. Используется для сложных корпоративных систем.

##### C. OAuth 2.0 — для внешних провайдеров (Socialite)

* Как работает: Протокол делегирования авторизации. Позволяет пользователю войти на ваш сайт через сторонний сервис (Google, GitHub, Facebook), не передавая вашему сайту свой пароль. Ваш сайт перенаправляет пользователя на Google, тот подтверждает вход и возвращает вашему сайту временный Authorization Code, который бэкенд обменивает на Access Token для получения данных пользователя (email, аватар).
* Реализация в Laravel: Пакет Laravel Socialite. Интеграция в контроллере сводится к двум методам:

```php

// Перенаправление в Google
return Socialite::driver('google')->redirect();

// Обработка ответа (Callback)
$googleUser = Socialite::driver('google')->user();
$user = User::updateOrCreate(['email' => $googleUser->getEmail()], [...]);
Auth::login($user);

```

## 2. Безопасность и хранение паролей. Безопасное хранение: Hash::make()

Пароли категорически запрещено хранить в открытом виде или использовать простые алгоритмы вроде md5() или sha1(). Они легко взламываются по радужным таблицам (Rainbow Tables).
Laravel использует Фасад Hash [3], работающий по умолчанию на алгоритме Bcrypt или Argon2 (криптографические функции замедленного хэширования с автоматической солью).

```php

use Illuminate\Support\Facades\Hash;

// Хэширование при регистрации
$user->password = Hash::make($request->password);

// Проверка при авторизации (под капотом Auth::attempt())
if (Hash::check($request->password, $user->password)) {
    // Пароль верный
}
```

##### 3. Механизмы защиты и дополнительные уровни## A. Rate Limiting (Защита от Brute Force / Перебора)

Чтобы злоумышленники не могли бесконечно подбирать пароли роботами, количество попыток входа жестко ограничивают.
В Laravel защита от brute force настраивается через Rate Limiter (Middlewares). Из коробки для роутов авторизации применяется ограничение, завязанное на IP-адрес пользователя и его email.

```php

// Настройка в bootstrap/app.php или RateLimiterServiceProvider (Laravel 11+)
RateLimiter::for('login', function (Request $request) {
    return Limit::perMinute(5)->by($request->input('email').$request->ip());
});

// Если лимит превышен, Laravel выбросит ошибку 429 Too Many Requests

```

##### B. Email Verification (Подтверждение почты)

Защищает от регистрации ботов и фейковых аккаунтов. В Laravel реализуется добавлением интерфейса MustVerifyEmail к модели User.

```php

use Illuminate\Contracts\Auth\MustVerifyEmail;

class User extends Authenticatable implements MustVerifyEmail 
{
    use HasFactory, Notifiable;
}

```

После этого Laravel автоматически начнет отправлять письма со ссылкой подтверждения при регистрации, а роуты можно защитить встроенным middleware verified:

`Route::get('/dashboard', function () { ... })->middleware(['auth', 'verified'])`;

##### C. 2FA (Двухфакторная аутентификация)

Дополнительный слой безопасности, требующий помимо пароля ввести временный код (TOTP) из приложения (Google Authenticator, Authy).

* Как устроено: Сервер генерирует секретный ключ, показывает его пользователю в виде QR-кода. Приложение пользователя каждые 30 секунд генерирует 6-значный код на основе этого ключа и текущего времени. При входе бэкенд сверяет введенный код с математически ожидаемым.
* Реализация в Laravel: Готовые пакеты Laravel Fortify / Laravel Jetstream содержат встроенную поддержку 2FA «из коробки» (включая генерацию QR-кодов и резервных кодов восстановления). Также популярны пакеты вроде pragmarx/google2fa.

##### 🔥 Вопросы на собеседовании по Аутентификации

   1. Какая разница между JWT (JSON Web Token) и обычным токеном в Laravel Sanctum?
   * Ответ: JWT — это самодостаточный (stateless) токен. Он содержит внутри себя зашифрованные данные пользователя (payload) и подпись. Серверу не нужно лезть в БД, чтобы проверить его — достаточно валидировать подпись ключом. Токен Sanctum — это stateful токен. В базу данных пишется хэш токена (таблица personal_access_tokens), а клиенту отдается строка. При каждом запросе Laravel лезет в БД, чтобы проверить этот токен. Sanctum проще аннулировать, но он нагружает базу данных.
   2. Почему нельзя использовать md5($password) с солью вместо Hash::make()?
   * Ответ: Алгоритм md5 разработан как быстрая хэш-функция для проверки целостности файлов. Современный видеопроцессор (GPU) может перебирать миллиарды хэшей md5 в секунду, что делает соль бесполезной против брутфорса. Hash::make() использует Bcrypt / Argon2, которые специально созданы медленными адаптивными алгоритмами. Они задействуют много процессорного времени и памяти, из-за чего аппаратный перебор становится экономически невозможным.
   3. Что такое атака Session Fixation и как Laravel защищает от нее?
   * Ответ: Это атака, при которой злоумышленник подсовывает жертве свой ID сессии (например, через URL), дожидается, пока пользователь авторизуется, и заходит по этой же сессии. Laravel автоматически защищает от этого: при каждом успешном входе система вызывает метод Session::regenerate(), который полностью меняет ID сессии в куках, делая старый ID бесполезным.
   

## Авторизация?

Авторизация (Authorization) — это процесс проверки прав доступа пользователя, отвечающий на вопрос: «Что этому пользователю разрешено делать?».
В отличие от аутентификации («Кто ты такой?»), авторизация происходит после того, как система распознала пользователя. В Laravel механизмы авторизации работают с помощью двух основных инструментов: Gates и Policies. Под капотом оба инструмента используют один и тот же базовый компонент (`Illuminate\Auth\Access\Gate`), но служат для разных архитектурных задач.


##### 1. Gates (Шлюзы)

Gates — это простые проверки прав, основанные на замыканиях (Clousures). Они идеально подходят для действий, которые не привязаны к конкретной модели или сущности базы данных.

* Когда использовать: Для глобальных прав (например, «доступ к админ-панели», «просмотр логов системы», «перевод сайта в режим обслуживания»).
* Где объявлять: В Laravel 11+ гейты регистрируются в файле app/Providers/AppServiceProvider.php (или в AuthServiceProvider в старых версиях фреймворка) через одноименный Фасад Gate.

##### Пример объявления Gate:

```php

use Illuminate\Support\Facades\Gate;
use App\Models\User;

public function boot(): void
{
    // Первым аргументом Laravel всегда автоматически передает текущего аутентифицированного пользователя
    Gate::define('access-admin-panel', function (User $user) {
        return $user->role === 'admin' || $user->role === 'manager';
    });
}

```

------------------------------

##### 2. Policies (Политики)

Policies — это PHP-классы, которые группируют логику авторизации вокруг конкретной модели Eloquent.

* Когда использовать: Когда проверка прав жестко завязана на CRUD-действия над сущностью (например, «может ли пользователь редактировать эту конкретную статью?», «может ли он удалить этот комментарий?»).
* Создание: Генерируются через консоль: `php artisan make:policy PostPolicy --model=Post` . Laravel автоматически создаст класс с методами `viewAny`, `view`, `create`, `update`, `delete`, `restore`, `forceDelete`.

##### Пример реализации Policy (app/Policies/PostPolicy.php):

```php

namespace App\Policies;

use App\Models\Post;
use App\Models\User;

class PostPolicy
{
    /**
     * Определить, может ли пользователь обновить статью.
     */
    public function update(User $user, Post $post): bool
    {
        // Пользователь может редактировать только свои статьи или если он админ
        return $user->id === $post->user_id || $user->role === 'admin';
    }
}

```

##### 3. Как применять Gates и Policies в коде

##### A. В Контроллерах (Автоматический вызов через $this->authorize или хелпер)

Для политик Laravel автоматически сопоставляет переданную модель с нужным классом политики:

```php

public function update(Request $request, Post $post)
{
    // Вызовет метод update() внутри PostPolicy для текущего юзера
    $this->authorize('update', $post); 
    
    // Альтернативный вариант (в Laravel 10/11 через фасад Gate)
    Gate::authorize('update', $post);

    // Логика обновления...
}

```

##### B. В Middleware (На уровне роутинга)

Защита эндпоинта до того, как запрос попадет в контроллер:

```php

// Проверка через Gate
Route::get('/admin', [AdminController::class, 'index'])->middleware('can:access-admin-panel');

// Проверка через Policy (передаем имя метода и привязанный параметр маршрута {post})
Route::put('/posts/{post}', [PostController::class, 'update'])->middleware('can:update,post');

```

##### C. В Шаблонах Blade (Скрытие элементов интерфейса)

Скрытие кнопок управления от пользователей, у которых нет на это прав:

```html

@can('access-admin-panel')
    <a href="/admin">Войти в админку</a>
@endcan

@can('update', $post)
    <button>Редактировать статью</button>
@endcan

```

##### 🚀 Продвинутые фичи (Любимые вопросы на собеседовании)

##### 1. Глобальный обход проверок (Полномочия Суперадмина)

Чтобы не прописывать `|| $user->role === 'admin'` в каждом гейте и каждой политике, можно использовать метод `Gate::before()`. Он выполняется до любой другой проверки. Если замыкание возвращает true, доступ предоставляется автоматически.

```php

// В AppServiceProvider.php
Gate::before(function (User $user, string $ability) {
    if ($user->role === 'super-admin') {
        return true; // Обходит все проверки гейтов и политик
    }
    // Если вернуть null, то Laravel продолжит стандартную проверку внутри Policy/Gate
});

```

##### 2. Авторизация гостей (Неавторизованных пользователей)

По умолчанию, если пользователь не вошел в систему, все Gates и Policies автоматически возвращают false. Если вам нужно разрешить гостям какое-то действие (например, просмотр черновиков по секретной ссылке), нужно сделать тип аргумента User nullable (?User) или задать значение по умолчанию null:

```php

Gate::define('view-draft', function (?User $user, Post $post) {
    return $post->is_public_draft; // Сработает даже для неавторизованных гостей
});

```

##### 🔥 Вопросы на собеседовании по Авторизации

   1. В чем ключевое архитектурное отличие Gate от Policy?
   * Ответ: Gate — это процедурный подход (closure), используемый для глобальных действий, не привязанных к конкретным объектам БД (например, «открыть панель приборов»). Policy — это объектно-ориентированный подход, класс, логика которого сгруппирована вокруг конкретной модели Eloquent (CRUD-операции над моделью Order, User, Comment).
   2. Как Laravel понимает, какую именно Policy нужно вызвать для модели?
   * Ответ: По умолчанию в Laravel работает автоопределение (Auto-discovery). Фреймворк ищет класс политики в папке app/Policies, имя которого совпадает с именем модели + суффикс Policy (для модели Post ищется PostPolicy). Если структура проекта кастомная, соответствие моделей и политик можно прописать вручную через свойство $policies в AuthServiceProvider (в старых версиях) или зарегистрировать через Gate::policy().
   3. Что вернет метод авторизации, если пользователь не залогинен, а в политике тип юзера указан как User $user?
   * Ответ: Он сразу вернет false, даже не заходя внутрь метода политики. Laravel прерывает цепочку проверок для гостей, если метод жестко требует объект User. Чтобы пустить гостя, нужно объявить параметр как ?User $user.
   

## Шифрование и Криптография

В веб-разработке криптографические операции делятся на три принципиально разные категории: Шифрование (обратимое), Хеширование (необратимое) и Подписание данных (проверка целостности).

##### 1. Фундаментальные понятия

| Тип операции | Кол-во ключей | Направление | Для чего используется | Алгоритмы |
|---|---|---|---|---|
| Симметричное шифрование | Один секретный ключ | Двустороннее (Обратимое) | Конфиденциальные данные (токены, карты) | AES-256-CBC, AES-128-GCM |
| Асимметричное шифрование | Два ключа (Публичный + Приватный) | Двустороннее (Обратимое) | SSL/TLS, SSH, API-интеграции, JWT | RSA, ECC |
| Хеширование | Без ключа (но с солью) | Одностороннее (Необратимое) | Хранение паролей, контрольные суммы файлов | Bcrypt, Argon2, SHA-256 |


* Главное правило для собеседования: Шифрование используется для данных, которые системе нужно будет прочитать в исходном виде (например, выгрузить номер карты для отправки в банк). Хеширование используется для данных, исходный вид которых системе знать не нужно (например, пароли — система лишь сравнивает хэш введенного текста с хэшем в БД).

##### 2. Главный секрет: APP_KEY

В Laravel центральным элементом безопасности является переменная APP_KEY в файле .env.

* Что это такое: Это случайная строка из 32 байт, закодированная в Base64.
* Зачем нужен: Этот ключ используется как мастер-ключ для всего симметричного шифрования в приложении (сессии, куки, зашифрованные строки в БД).
* Генерация: Создается командой php artisan key:generate.
* ⚠️ Опасность: Если вы потеряете или измените APP_KEY на живом проекте, вы никогда не сможете расшифровать данные, которые были им зашифрованы ранее (например, токены пользователей или сессии превратятся в тыкву). Если ключ украдут, злоумышленники смогут подделывать сессионные куки и взломать приложение.

##### 3. Симметричное шифрование в Laravel

Laravel использует встроенные средства PHP (OpenSSL) для шифрования по алгоритмам AES-256-CBC или AES-128-GCM. За это отвечает фасад Crypt.
При шифровании Laravel автоматически генерирует IV (Initialization Vector) — случайную соль, благодаря которой один и тот же текст при каждом вызове шифрования будет выглядеть совершенно по-разному. Также к строке подмешивается MAC-подпись (Message Authentication Code), чтобы гарантировать, что зашифрованный текст не был изменен злоумышленником.

```php

use Illuminate\Support\Facades\Crypt;

// 1. Шифрование (строка превращается в длинный Base64-код, содержащий IV, MAC и payload)
$encrypted = Crypt::encryptString('4111_2222_3333_4444');

// 2. Расшифрование (обратный процесс)
$decrypted = Crypt::decryptString($encrypted); // '4111_2222_3333_4444'

// Если MAC-подпись нарушена (текст пытались подделать), выбросится DecreyptException

```

##### 4. Хеширование (Пароли)

Хеширование — это математическое преобразование строки любой длины в строку фиксированной длины (хэш) по одностороннему алгоритму. Восстановить исходную строку из хэша невозможно.
Laravel использует фасад Hash, поддерживающий современные стандарты безопасности: Bcrypt и Argon2 (Argon2id — победитель Password Hashing Competition, признан самым защищенным). Они намеренно сделаны «тяжелыми» для процессора, чтобы усложнить аппаратный перебор (Brute Force).

```php

use Illuminate\Support\Facades\Hash;

// Создание хэша (автоматически добавляет случайную уникальную соль для каждого пароля)
$hashedPassword = Hash::make('secret_password');

// Проверка (сравнивает хэш введенного пароля с хэшем из базы)
if (Hash::check('user_input_string', $hashedPassword)) {
    // Допуск разрешен
}

```

##### 5. Подписание URL (Signed URLs)

Signing (Подписание) — это механизм, который не прячет данные (URL остается читаемым), но гарантирует, что в параметры ссылки не были внесены изменения после её создания.
Laravel использует алгоритм HMAC-SHA256 на основе APP_KEY, чтобы прикрепить к URL хэш-подпись (параметр signature). Если злоумышленник попытается изменить хотя бы один символ в URL (например, поменять ID пользователя), подпись станет невалидной.

* Когда использовать: Ссылки для подтверждения Email, отписка от рассылок в один клик, временный доступ к скачиванию приватного файла.

##### Пример создания подписанного URL:

```php

use Illuminate\Support\Facades\URL;

// Создает ссылку: https://site.test...
$url = URL::signedRoute('unsubscribe', ['user' => 42]);

// Можно создать временную ссылку (сгорает через 30 минут)
$temporaryUrl = URL::temporarySignedRoute(
    'download', now()->addMinutes(30), ['file' => 'report.pdf']
);

## Пример проверки на бэкенде:
Вы можете проверить валидность ссылки вручную в контроллере или повесить встроенный middleware:

// Вариант 1: Вручную в контроллере
if (! $request->hasValidSignature()) {
    abort(401, 'Ссылка подделана или её срок действия истек!');
}

// Вариант 2: Защита на уровне роутинга (Middleware)
Route::get('/download/{file}', [FileController::class, 'download'])
    ->name('download')
    ->middleware('signed');

```

##### 🔥 Вопросы на собеседовании по Криптографии

   1. Почему нельзя использовать алгоритм md5 или sha256 для хэширования паролей?
   * Ответ: Эти алгоритмы создавались как быстрые функции для проверки целостности данных. На современном оборудовании (особенно на GPU) злоумышленник может перебирать сотни миллиардов хэшей md5 в секунду. Алгоритмы Bcrypt и Argon2 являются адаптивными и медленными — они требуют вычислительных мощностей и оперативной памяти, что делает brute force атаку экономически невыгодной.
   2. Зачем при шифровании Crypt::encrypt() используется IV (вектор инициализации)?
   * Ответ: Если шифровать одно и то же слово (например, статус "active") без IV, то на выходе всегда будет получаться одинаковая зашифрованная строка. Злоумышленник, анализируя базу данных, сможет выявить закономерности. IV — это случайная соль, которая подмешивается к каждому запросу. Благодаря IV, шифруя слово "active" 10 раз подряд, мы получим 10 абсолютно разных зашифрованных строк.
   3. Что произойдет с подписанным URL (URL::signedRoute()), если мы обновим APP_KEY?
   * Ответ: Все ранее сгенерированные подписанные ссылки мгновенно станут невалидными. Поскольку подпись (signature) рассчитывается на основе APP_KEY, при смене ключа сервер начнет рассчитывать хэш по-новому, и старые подписи проверку не пройдут.


## Протокол HTTPS и сетевая безопасность в PHP/Laravel

Безопасность веб-приложения не заканчивается на уровне кода (БД или сессий). Данные должны быть защищены в момент их транспортировки по сети между браузером пользователя и вашим сервером. За это отвечает связка протоколов HTTPS, заголовков безопасности и специализированных Middleware.

##### 1. Что такое HTTPS (SSL/TLS)?

HTTPS (HyperText Transfer Protocol Secure) — это расширение стандартного протокола HTTP, поддерживающее шифрование передаваемых данных.

* Как работает: Вместо отправки текста в открытом виде, HTTPS оборачивает обычный HTTP-трафик в криптографический протокол SSL/TLS.
* Двухэтапная схема:
1. При подключении сервер и клиент используют асимметричное шифрование (RSA/ECC), чтобы безопасно договориться о временном ключе сессии и проверить подлинность сервера (через SSL-сертификат).
   2. Весь последующий обмен данными внутри этой сессии происходит с помощью быстрого симметричного шифрования (AES) на базе этого временного ключа.

##### 2. От каких угроз защищает HTTPS?

Без HTTPS любой промежуточный узел в сети (провайдер интернета, администратор общественного Wi-Fi в кафе, магистральный маршрутизатор) может читать и модифицировать ваш трафик. HTTPS предотвращает следующие критические угрозы:

* Прослушивание трафика (Sniffing): Злоумышленник перехватывает сетевые пакеты. Без шифрования он увидит логины, пароли, номера кредитных карт и сессионные куки в виде обычного читаемого текста.
* Кража сессий (Session Hijacking): Если куки сессии передаются по HTTP, их легко перехватить. Получив чужую куку laravel_session, атакующий может подставить её в свой браузер и мгновенно войти в личный кабинет жертвы без ввода логина и пароля.
* Man-in-the-Middle (MitM / Атака «Человек посередине»): Злоумышленник тайно внедряется в канал связи между клиентом и сервером. Он может не только перехватывать данные, но и подменять их «на лету» (например, внедрить вредоносный XSS-скрипт в HTML-код сайта, который возвращается пользователю, или подменить реквизиты платежа). HTTPS блокирует MitM, так как браузер сразу заметит нарушение цифровой подписи сертификата.

##### 3. Инструменты развертывания и защиты## A. Let's Encrypt

Раньше SSL-сертификаты стоили дорого и требовали ручного продления. Let's Encrypt — это бесплатный, автоматизированный и открытый центр сертификации (Certificate Authority).

* Как используется: С помощью консольной утилиты (обычно Certbot) на сервере (Nginx/Apache) настраивается автоматический выпуск сертификата.
* Продление: Сертификаты выдаются на 90 дней, но Certbot вешается на cron и автоматически продлевает их каждые 60 дней, делая поддержку HTTPS полностью бесплатной и автономной.

##### B. HSTS Header (HTTP Strict Transport Security)

Даже если на сервере настроен редирект с `http://` на `https://`, пользователь при первом вводе сайта в строку браузера (или кликая по старой ссылке) сначала делает незащищенный HTTP-запрос. В этот короткий момент он уязвим для MitM-атаки (злоумышленник может перехватить запрос и заблокировать редирект).
HSTS — это специальный HTTP-заголовок ответа сервера, который жестко приказывает браузеру: «В течение следующего года общайся с этим сайтом ИСКЛЮЧИТЕЛЬНО по HTTPS. Даже если пользователь введет `http://`, принудительно переводи запрос на `https://` внутри самого браузера».

* Пример заголовка: `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`

##### C. Middleware TrustProxies в Laravel

В современных высоконагруженных архитектурах реальный веб-сервер с Laravel часто находится за балансировщиком нагрузки (Load Balancer, например, AWS ALB, Cloudflare или отдельный прокси-сервер Nginx).

* Проблема: Балансировщик принимает зашифрованный HTTPS-трафик от пользователя, расшифровывает его и пересылает внутрь вашей закрытой сети на сервер с Laravel по обычному HTTP. Из-за этого хелперы Laravel (например, `url()`, `route()`, `$request->secure()`) начинают думать, что сайт работает без шифрования, и генерируют небезопасные ссылки `http://`, ломая работу приложения и вызывая бесконечные циклы редиректов.
* Решение: Middleware TrustProxies. Оно заставляет Laravel доверять заголовкам X-Forwarded-Proto и X-Forwarded-For, которые балансировщик прикрепляет к запросу, чтобы передать бэкенду информацию об исходном протоколе пользователя.

Настройка в Laravel 11+ (в файле bootstrap/app.php):

```php

->withMiddleware(function (Middleware $middleware) {
    // Указываем IP-адреса ваших прокси/балансировщиков (или '*' для всех, если доверяете сети Cloudflare)
    $middleware->trustProxies(at: '*'); 
    
    // Теперь Laravel поймет, что пользователь зашел по HTTPS, 
    // даже если прокси переслал запрос на бэкенд по HTTP.
})

```

(В Laravel 10 и старше эта настройка живет в файле `app/Http/Middleware/TrustProxies.php` в свойстве $proxies).

##### 🔥 Вопросы на собеседовании по HTTPS и сетевой безопасности

   1. Помогает ли HTTPS защититься от SQL-инъекций или XSS?
   * Ответ: Нет, не помогает. HTTPS защищает данные только в процессе транспортировки от клиента к серверу от прослушивания и подмены третьими лицами. Но если сам пользователь умышленно отправляет вредоносный SQL-код в форму, HTTPS послушно зашифрует этот вредоносный код и доставит его в целости и сохранности прямо на ваш сервер. Защита от SQLi и XSS — это задача логики приложения (Prepared Statements, экранирование), а не сетевого протокола.
   2. Зачем нужен флаг secure для сессионных кук, если сайт уже работает на HTTPS?
   * Ответ: Если флаг secure у куки не выставлен, браузер может случайно отправить её в открытом виде, если пользователь каким-то образом перейдет по случайной ссылке `http://your-site.com`. Флаг secure гарантирует, что браузер прикрепит эту куку к запросу только тогда, когда соединение строго HTTPS. В Laravel этот параметр регулируется в `config/session.php` ('secure' => true).
   3. Что произойдет, если в TrustProxies указать at: '*' (доверять всем прокси), но приложение не находится за реальным балансировщиком?
   * Ответ: Это создаст уязвимость подмены IP и протокола. Любой злоумышленник сможет вручную сформировать HTTP-запрос к вашему серверу, подставить туда фейковые заголовки X-Forwarded-For (с чужим IP) и X-Forwarded-Proto: https, и ваше Laravel-приложение безоговорочно поверит этой информации. Доверять всем прокси (*) можно только тогда, когда ваш сервер гарантированно изолирован внутри приватной сети или находится строго за Cloudflare/AWS.
   

##  OWASP Top 10?

OWASP (Open Worldwide Application Security Project) — это международная некоммерческая организация, цель которой — повышение безопасности программного обеспечения.
OWASP Top 10 — это их флагманский, регулярно обновляемый документ (каждые 3–4 года), представляющий собой консенсус экспертов со всего мира о десяти самых критических уязвимостях веб-приложений. На собеседованиях знание этого списка (и понимание того, как защитить от них приложение на PHP/Laravel) показывает высокий уровень инженерии и зрелость разработчика.
Ниже приведена детальная расшифровка каждого пункта из актуального рейтинга OWASP Top 10, адаптированная под PHP и Laravel.


##### 1. Broken Access Control (Нарушение контроля доступа)

Эта уязвимость закрепилась на 1-м месте рейтинга. Она возникает, когда пользователи могут выполнять действия или получать доступ к данным за пределами своих фактических полномочий.

* Пример (IDOR): Пользователь заходит в личный кабинет и видит URL ://site.com. Он меняет ID в адресной строке на ://site.com и видит чужой заказ, потому что сервер проверил только факт авторизации, но не проверил, принадлежит ли заказ этому пользователю.
* Защита в Laravel: Использовать Policies. Каждое действие в контроллере, завязанное на ID, должно проходить через проверку: $this->authorize('view', $order);.

##### 2. Cryptographic Failures (Криптографические ошибки)
Ранее известная как Sensitive Data Exposure (Раскрытие конфиденциальных данных). Связана со слабым шифрованием или полным его отсутствием для важных данных.

* Пример: Хранение паролей в базе данных в открытом виде или использование устаревших алгоритмов типа md5/sha1, передача сессионных кук по незащищенному HTTP-протоколу.
* Защита в Laravel: Использовать Hash::make() (Bcrypt/Argon2) для паролей, Crypt::encryptString() для обратимых данных (номера карт), выставлять флаги secure и HttpOnly для кук в config/session.php, генерировать стойкий APP_KEY.

##### 3. Injection (Инъекции)
Сюда входят SQL-инъекции, XSS (Cross-Site Scripting), а также Command Injection (внедрение системных команд ОС). Уязвимость возникает, когда сырые (непроверенные) данные пользователя отправляются интерпретатору.

* Пример: Использование методов whereRaw() с прямой конкатенацией строк в Eloquent, или вывод текста пользователя через `{!! $userInput !!}` в Blade.
* Защита в Laravel: Использование подготовленных запросов (стандартный функционал Eloquent/Query Builder), автоматическое экранирование в Blade через двойные скобки `{{ $userInput }}`, валидация всех входящих данных.

##### 4. Insecure Design (Небезопасное проектирование)
Относительно новая категория, которая фокусируется на ошибках, допущенных на этапе архитектуры и дизайна, а не на этапе написания кода. Если архитектура изначально дырявая, идеальный код её не спасет.

* Пример: Проектирование механизма восстановления пароля без ограничения количества попыток (Rate Limiting) или без контрольных вопросов, позволяя злоумышленнику перебирать ссылки. Или отсутствие блокировки учетной записи после 10 неудачных попыток входа.
* Защита в Laravel: Применение Rate Limiting для критичных эндпоинтов, использование проверенных паттернов авторизации (Laravel Fortify, Sanctum), обязательное вовлечение практик моделирования угроз на этапе планирования фичи.

##### 5. Security Misconfiguration (Неправильная настройка безопасности)
Возникает из-за некорректной или дефолтной конфигурации серверов, фреймворков или инфраструктуры.

* Пример: Оставленная переменная APP_DEBUG=true на продакшене. При любой ошибке Laravel выведет на экран страницу Ignition со всеми паролями от БД, ключами API и переменными окружения из файла .env. Сюда же относится использование дефолтных паролей для Redis или базы данных.
* Защита в Laravel: Строго выставлять APP_DEBUG=false и APP_ENV=production на «живом» сервере. Отключать неиспользуемые роуты, закрывать порты Redis/БД от внешнего мира, вовремя обновлять веб-сервер (Nginx) и версию PHP.

##### 6. Vulnerable and Outdated Components (Уязвимые и устаревшие компоненты)
Использование сторонних библиотек, пакетов и плагинов, в которых содержатся известные уязвимости.

* Пример: Вы подключили старый популярный пакет через composer для обработки картинок или генерации PDF. В этом пакете нашли критическую уязвимость (RCE), позволяющую удаленно выполнить код на вашем сервере, но вы не обновляете зависимости.
* Защита в Laravel: Регулярно запускать команду composer audit для проверки установленных пакетов на наличие уязвимостей. Использовать инструменты автоматического обновления зависимостей в CI/CD (например, GitHub Dependabot).

##### 7. Identification and Authentication Failures (Сбои в идентификации и аутентификации)
Ранее называлась Broken Authentication. Связана со слабыми механизмами проверки подлинности пользователя.

* Пример: Отсутствие защиты от Brute force (перебора паролей), разрешение пользователям устанавливать слишком простые пароли (например, 123456), отсутствие механизма смены ID сессии при логине (Session Fixation).
* Защита в Laravel: Laravel защищает от Session Fixation автоматически через Session::regenerate(). Дополнительно нужно использовать RateLimiter для ограничения попыток входа, валидацию паролей по правилам сложности (Password::min(8)->mixedCase()->letters()->numbers()->uncompromised()), внедрять 2FA.

##### 8. Software and Data Integrity Failures (Нарушения целостности программного обеспечения и данных)
Связано с кодом и инфраструктурой, которые не защищены от несанкционированного изменения. Это касается обновлений плагинов, CI/CD пайплайнов и небезопасной десериализации данных.

* Пример: Использование нативной PHP-функции unserialize() для данных, которые пришли от пользователя через куки или API-запрос. Злоумышленник может сформировать специальный сериализованный объект (PHP Object Injection), что приведет к выполнению его кода на сервере.
* Защита в Laravel: Никогда не использовать unserialize() для пользовательских данных (использовать json_decode()). При передаче важных параметров в URL использовать механизмы подписания ссылок URL::signedRoute(). Внедрять проверку подписей пакетов в Composer (composer.lock).

##### 9. Security Logging and Monitoring Failures (Сбои в логировании и мониторинге безопасности)
Уязвимость заключается в том, что приложение либо вообще не логирует подозрительные действия, либо логирует их так, что администраторы не могут вовремя заметить и остановить хакерскую атаку.

* Пример: Злоумышленник в течение трех дней подбирает пароли к админке (брутфорсит), но в логи системы не пишется ничего. Система «слепа», и об атаке узнают только тогда, когда база данных будет слита в сеть.
* Защита в Laravel: Логировать критические события (неудачные попытки входа, смена паролей, изменение прав доступа) с помощью фасада Log. Настраивать уведомления о критических ошибках в Slack/Telegram, использовать внешние системы мониторинга логов (Sentry, Bugsnag, ELK stack).

##### 10. Server-Side Request Forgery (SSRF / Подделка запроса на стороне сервера)
Эта уязвимость замыкает топ-10. Она возникает, когда веб-приложение по запросу пользователя загружает данные с внешнего URL, но не валидирует адрес этого ресурса.

* Пример: В приложении есть фича «загрузить аватарку по ссылке». Пользователь отправляет POST /avatar с параметром `url=http://127.0.0.1:6379` (внутренний порт Redis на вашем сервере) или `url=http://169.254.169` (сервер метаданных AWS). Бэкенд делает этот запрос со своего IP-адреса, пробивая внутреннюю сеть, и отдает злоумышленнику секретные данные инфраструктуры.
* Защита в Laravel: Если приложение делает внешние запросы (через Http::get()), адрес URL должен строго валидироваться. Необходимо использовать белый список разрешенных доменов или запрещать запросы на внутренние (локальные) диапазоны IP-адресов (127.0.0.1, 192.168.*.*, 10.*.*.*).

##### 💡 Как уверенно ответить на собеседовании?
